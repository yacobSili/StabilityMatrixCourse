# 窗口系统概述与架构设计

## 引言

Android 窗口系统是连接应用 UI 与物理显示屏的核心桥梁，负责管理所有可见内容的创建、布局、合成与显示。作为 Android 图形栈的核心组成部分，窗口系统贯穿了从应用 View 到屏幕像素的整个技术栈，与 WindowManagerService、SurfaceFlinger、HWComposer 等核心组件紧密协作。

本文从架构师视角出发，建立对窗口系统的全局认知，帮助读者理解其分层设计、模块关系以及核心工作流程。

### 1.2 核心职责

窗口系统承担以下核心职责：

| 职责 | 描述 | 涉及层级 |
|-----|------|---------|
| 窗口管理 | 窗口的创建、更新、销毁、层级管理 | App/Framework |
| 布局计算 | 窗口大小、位置、Insets 计算 | Framework |
| Surface 管理 | 图形缓冲区的创建与生命周期管理 | Framework/Native |
| 图形合成 | 多个 Layer 合成为最终帧 | Native |
| 显示输出 | 将合成结果输出到物理屏幕 | Native/Hardware |
| 帧调度 | VSYNC 同步、帧节奏控制 | Native |
| 焦点管理 | 窗口焦点、输入焦点的维护 | Framework |

## 2. 四层架构设计

### 2.1 架构设计理念

Android 窗口系统采用**分层架构**设计，各层职责清晰：

```
┌─────────────────────────────────────────────────────────────────┐
│                     应用层 (Application)                         │
│  职责：UI 构建、View 绑定、绘制内容                               │
│  核心：Activity、Window、View、Canvas                            │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ WindowManager.addView()
                              │ IWindowSession.addToDisplay()
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                   Framework 层 (WMS)                             │
│  职责：窗口策略、布局计算、焦点管理、Surface 调度                  │
│  核心：WMS、WindowState、DisplayContent、SurfaceControl          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ SurfaceControl.Transaction
                              │ ISurfaceComposer
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    Native 层 (SurfaceFlinger)                    │
│  职责：Layer 管理、缓冲区管理、图形合成、VSYNC 调度               │
│  核心：SurfaceFlinger、Layer、BufferQueue、Scheduler             │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ HWC HAL
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      硬件层 (HWC/GPU/Display)                    │
│  职责：硬件合成、GPU 渲染、显示输出                               │
│  核心：HWComposer、GPU、Display Controller                       │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 各层职责详解

#### 应用层

**核心组件**：
- **Activity**：承载窗口的组件容器
- **PhoneWindow**：Window 的具体实现，管理 DecorView
- **DecorView**：窗口的根 View，包含标题栏和内容区
- **ViewRootImpl**：View 树与 WMS 的桥梁

**关键职责**：
1. 构建 View 层次结构
2. 处理用户输入事件
3. 触发 View 的测量、布局、绘制
4. 管理窗口属性（大小、标志等）

```java
// 应用层窗口创建流程
public class Activity {
    final void attach(Context context, ...) {
        // 1. 创建 PhoneWindow
        mWindow = new PhoneWindow(this, ...);
        mWindow.setWindowManager(...);
    }
    
    public void setContentView(int layoutResID) {
        // 2. 设置内容视图
        getWindow().setContentView(layoutResID);
    }
}
```

#### Framework 层 (WMS)

**核心组件**：
- **WindowManagerService**：窗口管理核心服务
- **WindowState**：单个窗口在 WMS 中的表示
- **WindowToken**：窗口令牌，关联一组窗口
- **DisplayContent**：显示屏内容管理

**关键职责**：
1. 窗口的添加、移除、更新
2. 窗口层级（Z-Order）管理
3. 窗口布局和大小计算
4. 焦点窗口管理
5. 窗口动画调度

```java
// WMS 添加窗口核心流程
public int addWindow(Session session, IWindow client, ...) {
    // 1. 权限检查
    // 2. 创建 WindowState
    WindowState win = new WindowState(this, session, client, ...);
    // 3. 添加到窗口列表
    win.mToken.addWindow(win);
    // 4. 创建 SurfaceControl
    win.createSurfaceControl(...)
}
```

#### Native 层 (SurfaceFlinger)

**核心组件**：
- **SurfaceFlinger**：图形合成服务
- **Layer**：可合成的图形层
- **BufferQueue**：生产者-消费者缓冲队列
- **Scheduler**：VSYNC 和帧调度

**关键职责**：
1. 管理所有 Layer 的层级和属性
2. 接收应用提交的图形缓冲区
3. 按照 VSYNC 节奏合成所有 Layer
4. 选择合成策略（GPU/HWC）
5. 提交合成结果到显示设备

```cpp
// SurfaceFlinger 主循环
void SurfaceFlinger::onMessageReceived(int32_t what) {
    switch (what) {
        case MessageQueue::INVALIDATE:
            // 1. 处理事务（窗口变化）
            handleMessageTransaction();
            // 2. 处理刷新
            handleMessageInvalidate();
            break;
        case MessageQueue::REFRESH:
            // 3. 执行合成
            handleMessageRefresh();
            break;
    }
}
```

#### 硬件层

**核心组件**：
- **HWComposer**：硬件合成器 HAL
- **GPU**：图形处理单元
- **Display Controller**：显示控制器

**关键职责**：
1. 硬件合成（叠加层直接合成）
2. GPU 合成（OpenGL ES 渲染）
3. 帧缓冲管理
4. 显示时序控制

## 5. 关键设计决策

### 5.1 为什么采用分层架构？

| 设计考量 | 解决方案 |
|---------|---------|
| 职责分离 | WMS 管窗口策略，SF 管图形合成 |
| 进程隔离 | 应用、WMS、SF 运行在不同进程 |
| 安全性 | 应用无法直接操作其他窗口的 Surface |
| 性能优化 | Native 层处理高频图形操作 |
| 硬件抽象 | HAL 层屏蔽硬件差异 |

### 5.2 为什么 WMS 和 SurfaceFlinger 分离？

1. **职责不同**：
   - WMS：窗口策略、布局、焦点（策略层）
   - SF：图形合成、显示输出（执行层）

2. **进程隔离**：
   - WMS 在 system_server 进程（Java）
   - SF 是独立 Native 进程（C++）

3. **性能考虑**：
   - SF 需要高频响应 VSYNC
   - 避免 Java GC 影响帧率

### 5.3 为什么使用 BufferQueue？

1. **异步通信**：生产者（App）和消费者（SF）解耦
2. **内存效率**：固定数量的缓冲区循环使用
3. **同步机制**：内置 Fence 机制保证同步
4. **零拷贝**：共享内存，避免数据拷贝
