## 实战案例：Android 设备全流程分析

### 案例设备

**设备信息**（示例）：
- 设备型号：Pixel 6（或你的实际设备）
- Android 版本：Android 13
- 架构：ARM64
- 分区方案：动态分区 + A/B 分区

### 第一步：分析源码配置

#### 1.1 设备配置位置

**查找设备配置**：
```bash
# 查找设备配置
find . -name "BoardConfig.mk" | grep pixel
# 假设找到：device/google/raviole/BoardConfig.mk
```

#### 1.2 分析 BoardConfig.mk

**关键配置**：
```makefile
# 架构
TARGET_ARCH := arm64
TARGET_ARCH_VARIANT := armv8-a

# 分区大小
BOARD_BOOTIMAGE_PARTITION_SIZE := 67108864
BOARD_SYSTEMIMAGE_PARTITION_SIZE := 3221225472
BOARD_VENDORIMAGE_PARTITION_SIZE := 1073741824

# 动态分区
BOARD_SUPER_PARTITION_SIZE := 6442450944
BOARD_SUPER_PARTITION_GROUPS := group_foo
BOARD_GROUP_FOO_SIZE := 6442450944
BOARD_GROUP_FOO_PARTITION_LIST := system vendor product system_ext
```

**分析**：
- 使用 ARM64 架构
- 使用动态分区
- 包含 system, vendor, product, system_ext

#### 1.3 分析 fstab

**查找 fstab**：
```bash
find device/google/raviole -name "fstab*"
```

**fstab 内容**（示例）：
```text
/dev/block/by-name/system     /system      ext4    ro,barrier=1  wait,first_stage_mount,avb=vbmeta_system,logical
/dev/block/by-name/vendor    /vendor      ext4    ro,barrier=1  wait,first_stage_mount,avb=vbmeta_vendor,logical
/dev/block/by-name/userdata  /data        f2fs    noatime       wait,check,formattable,resize,quota,fileencryption=aes-256-xts
```

**分析**：
- System 和 Vendor 使用 ext4，只读
- Data 使用 f2fs，可写，支持加密
- 使用动态分区（logical 标志）
- 使用 AVB 验证

### 第二步：理解分区布局

#### 2.1 查看分区表

**在设备上查看**：
```bash
# 查看所有分区
adb shell lsblk

# 查看分区列表
adb shell ls -la /dev/block/by-name/
```

**典型输出**（示例）：
```
boot_a
boot_b
system_a
system_b
vendor_a
vendor_b
...
```

**分析**：
- 使用 A/B 分区
- 每个分区有两个槽

#### 2.2 分析分区大小

**查看分区大小**：
```bash
adb shell df -h
```

**分析**：
- System 分区大小
- Vendor 分区大小
- Data 分区大小
- 其他分区大小

### 第三步：分析构建输出的镜像文件

#### 3.1 查看构建输出

**构建输出目录**：
```
out/target/product/raviole/
├── boot.img
├── system.img
├── vendor.img
├── vbmeta.img
└── ...
```

#### 3.2 分析 boot.img

**拆解 boot.img**：
```bash
unpack_bootimg --boot_img boot.img --out boot_unpacked/
```

**查看内容**：
```bash
ls -lh boot_unpacked/
# 应该看到：
# - kernel
# - ramdisk
# - dtb (如果包含)
```

**分析**：
- 内核大小
- Ramdisk 大小
- 设备树信息

#### 3.3 分析 system.img

**转换和挂载**：
```bash
# 转换 Sparse 镜像
simg2img system.img system_raw.img

# 挂载镜像
sudo mount -o loop system_raw.img /mnt

# 查看内容
ls -la /mnt/
```

**分析**：
- System 分区内容
- 文件系统类型
- 目录结构

### 第四步：模拟刷机过程

#### 4.1 准备镜像文件

**需要的镜像**：
- boot.img
- system.img
- vendor.img
- vbmeta.img
- 等等

#### 4.2 刷机命令

**使用 fastboot**：
```bash
# 进入 fastboot 模式
adb reboot bootloader

# 刷入镜像
fastboot flash boot boot.img
fastboot flash system system.img
fastboot flash vendor vendor.img
fastboot flash vbmeta vbmeta.img

# 重启
fastboot reboot
```

**分析**：
- 每个镜像刷入对应分区
- 刷机顺序很重要
- 验证启动信息

### 第五步：分析启动日志

#### 5.1 收集启动日志

**Kernel 日志**：
```bash
adb shell dmesg > kernel.log
```

**Init 日志**：
```bash
adb logcat -b all > init.log
```

#### 5.2 分析启动流程

**查找关键信息**：
```bash
# 查看 Bootloader 信息
grep -i "bootloader\|avb" kernel.log

# 查看分区挂载
grep -i "mount" kernel.log

# 查看 Init 启动
grep -i "init" init.log
```

**分析**：
- Bootloader 阶段
- Kernel 启动阶段
- Init 进程阶段
- 分区挂载过程

### 第六步：验证分区挂载

#### 6.1 查看挂载信息

**查看挂载表**：
```bash
adb shell mount
```

**分析**：
- 哪些分区已挂载
- 挂载点位置
- 挂载标志（ro/rw）

#### 6.2 验证挂载点

**检查关键挂载点**：
```bash
# 检查 System
adb shell ls -la /system

# 检查 Vendor
adb shell ls -la /vendor

# 检查 Data
adb shell ls -la /data
```

### 第七步：检查根目录结构

#### 7.1 查看根目录

**列出根目录**：
```bash
adb shell ls -la /
```

**分析**：
- 来自 Ramdisk 的目录
- 分区挂载点
- 虚拟文件系统
- 符号链接

#### 7.2 验证 System-as-root

**检查符号链接**：
```bash
adb shell ls -la / | grep "^l"
```

**应该看到**：
```
/system -> /
```

**分析**：
- System-as-root 实现
- 兼容性符号链接

### 第八步：完整流程总结

#### 8.1 流程梳理

**完整流程**：
1. **源码配置** → BoardConfig.mk, fstab
2. **构建镜像** → boot.img, system.img 等
3. **刷机** → fastboot 刷入分区
4. **启动** → Bootloader → Kernel → Init
5. **挂载** → 分区挂载到文件系统
6. **运行** → Zygote → System Server → Launcher

#### 8.2 关键点总结

**分区设计**：
- 使用动态分区
- 支持 A/B 分区
- 使用 AVB 验证

**启动流程**：
- System-as-root
- 两阶段挂载
- 验证启动链

**文件系统**：
- System/Vendor 使用 ext4（只读）
- Data 使用 f2fs（可写）
- 支持加密
