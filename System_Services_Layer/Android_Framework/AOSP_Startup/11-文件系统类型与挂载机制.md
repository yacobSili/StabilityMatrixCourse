# 文件系统类型与挂载机制

## 学习目标

- 理解 Android 支持的文件系统类型
- 掌握 ext4、f2fs、erofs 等文件系统的特点
- 理解只读挂载和可写挂载的区别
- 了解 Early mount、Late mount 等挂载机制
- 理解 dm-verity 与文件系统完整性

## 背景介绍

Android 设备使用多种文件系统来存储不同类型的数据。理解这些文件系统的特点和挂载机制，对于理解 Android 存储架构、优化性能、调试问题都至关重要。从 Android 13 开始，引入了 EROFS 只读文件系统，进一步优化了系统分区。

## 核心概念

### 支持的文件系统类型

#### ext4

**特点**：
- 传统 Linux 文件系统
- 成熟稳定
- 广泛使用

**使用场景**：
- System 分区（旧版本）
- Vendor 分区
- Data 分区（部分设备）

**优势**：
- 兼容性好
- 工具支持完善
- 性能稳定

**劣势**：
- 对 Flash 不够优化
- 写入性能一般

#### f2fs (Flash-Friendly File System)

**特点**：
- 专为 Flash 存储设计
- 优化写入性能
- 减少写入放大

**使用场景**：
- Data 分区（现代设备）
- Cache 分区
- Userdata 分区

**优势**：
- Flash 优化
- 写入性能好
- 减少磨损

**劣势**：
- 相对较新
- 某些场景性能不如 ext4

#### erofs (Enhanced Read-Only File System)

**特点**：
- 只读文件系统
- 高压缩比
- 快速挂载

**使用场景**：
- System 分区（Android 13+）
- Vendor 分区（Android 13+）
- 其他只读分区

**优势**：
- 压缩比高
- 挂载速度快
- 节省空间

**劣势**：
- 只读
- 相对较新

### 文件系统特性

#### 只读挂载

**特点**：
- 挂载为只读（`ro` 标志）
- 内容在构建时确定
- 运行时不能修改

**典型分区**：
- `system` - 系统框架
- `vendor` - 厂商实现
- `product` - 产品内容
- `system_ext` - 系统扩展
- `odm` - ODM 内容

**原因**：
1. **完整性保护**：防止系统文件被篡改
2. **性能优化**：只读文件系统可以优化
3. **安全**：配合 AVB 验证

**挂载示例**：
```text
/dev/block/by-name/system  /system  ext4  ro,barrier=1  wait,avb=vbmeta_system
```

#### 可写挂载

**特点**：
- 挂载为读写（`rw` 标志）
- 内容可以动态修改
- 需要文件系统检查

**典型分区**：
- `data` - 用户数据
- `cache` - 缓存

**文件系统特性**：
- 支持配额
- 支持加密
- 支持调整大小

**挂载示例**：
```text
/dev/block/by-name/userdata  /data  f2fs  noatime,nosuid,nodev  wait,check,formattable,resize,quota,fileencryption=aes-256-xts
```

### 挂载机制

#### Early Mount（早期挂载）

**时机**：Init 进程早期

**目的**：挂载必要的只读分区

**过程**：
1. 读取 fstab 文件
2. 查找 `first_stage_mount` 标志
3. 挂载分区
4. System-as-root 下挂载 system 为根

**关键标志**：
- `first_stage_mount` - 第一阶段挂载

**示例**：
```text
/dev/block/by-name/system  /system  ext4  ro  wait,first_stage_mount,avb=vbmeta_system
```

#### Late Mount（后期挂载）

**时机**：Init 进程后期

**目的**：挂载其他分区

**过程**：
1. 读取 fstab 文件
2. 挂载 data, cache 等分区
3. 设置挂载点权限

**关键标志**：
- `wait` - 等待设备就绪

**示例**：
```text
/dev/block/by-name/userdata  /data  f2fs  rw  wait,check,formattable
```

#### 延迟挂载

**特点**：
- 某些分区可能延迟挂载
- 按需挂载
- 减少启动时间

**使用场景**：
- 非关键分区
- 外部存储
- 可选分区

### dm-verity 与文件系统完整性

#### dm-verity 机制

**作用**：验证文件系统完整性

**原理**：
- 构建哈希树
- 运行时验证
- 防止文件系统被篡改

**工作流程**：
1. 构建时生成哈希树
2. 哈希树存储在分区
3. 运行时验证每个块
4. 验证失败阻止访问

#### 与 AVB 的关系

**AVB（Android Verified Boot）**：
- 验证分区签名
- 验证哈希树根
- 启动时验证

**dm-verity**：
- 运行时验证
- 验证文件系统块
- 持续保护

#### 配置

**fstab 标志**：
```text
avb=vbmeta_system
```

**验证链**：
```
vbmeta → system → dm-verity
```

## 实际应用

### 查看文件系统类型

```bash
# 查看挂载信息
mount | grep -E "system|vendor|data"

# 查看文件系统类型
df -T

# 查看块设备信息
lsblk -f
```

### 分析文件系统

```bash
# 查看 ext4 文件系统信息
tune2fs -l /dev/block/by-name/system

# 查看 f2fs 文件系统信息
fsck.f2fs -l /dev/block/by-name/userdata
```

### 调试挂载问题

```bash
# 查看挂载日志
dmesg | grep -i mount

# 查看 Init 日志
logcat -b all | grep -i mount

# 手动挂载测试
mount -t ext4 /dev/block/by-name/system /mnt
```

## 总结

### 核心要点

1. **Android 支持多种文件系统**：ext4, f2fs, erofs
2. **只读分区使用只读文件系统**：保护系统完整性
3. **可写分区使用优化文件系统**：f2fs 优化 Flash 性能
4. **挂载分两个阶段**：Early mount 和 Late mount
5. **dm-verity 提供运行时保护**：验证文件系统完整性

### 关键文件系统

- **ext4**：传统文件系统，兼容性好
- **f2fs**：Flash 优化，写入性能好
- **erofs**：只读文件系统，压缩比高
