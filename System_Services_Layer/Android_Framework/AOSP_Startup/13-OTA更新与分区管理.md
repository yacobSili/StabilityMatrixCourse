# OTA 更新与分区管理

## 学习目标

- 理解 OTA 更新流程
- 掌握 A/B 分区更新机制
- 理解 Seamless Updates 的工作原理
- 了解分区更新顺序
- 理解回滚机制
- 掌握动态分区更新

## 背景介绍

OTA（Over-The-Air）更新是 Android 设备接收系统更新的主要方式。理解 OTA 更新机制，对于理解系统更新流程、调试更新问题、进行系统定制都至关重要。从 Android 7.0 开始，A/B 分区机制实现了无缝更新。

## 核心概念

### OTA 更新流程

#### 更新阶段

**1. 检查更新**
- 系统检查更新服务器
- 下载更新包信息
- 验证更新包签名

**2. 下载更新包**
- 下载 OTA 包
- 验证包完整性
- 存储到缓存

**3. 准备更新**
- 验证更新包
- 检查系统状态
- 准备更新环境

**4. 应用更新**
- 写入新分区（A/B 设备）
- 或进入 Recovery 更新（传统设备）
- 更新分区内容

**5. 重启验证**
- 重启到新系统
- 验证更新结果
- 成功或回滚

#### 更新包格式

**ZIP 格式**：
- 包含分区镜像
- 包含更新脚本
- 包含元数据

**内容**：
- `META-INF/` - 元数据和脚本
- `system.new.dat` - System 分区数据
- `vendor.new.dat` - Vendor 分区数据
- 等等

### A/B 分区更新机制

#### 设计原理

**双槽设计**：
- 每个分区有两个槽：`_a` 和 `_b`
- 系统运行在一个槽
- OTA 更新写入另一个槽

#### 分区命名

**A/B 分区**：
```
boot_a / boot_b
system_a / system_b
vendor_a / vendor_b
...
```

**当前槽**：
- 系统运行在 Slot A 或 Slot B
- Bootloader 决定启动槽
- 可以通过属性查看

#### 工作流程

**1. 正常运行**
- 系统运行在 Slot A
- 用户正常使用设备

**2. OTA 更新**
- 后台下载更新包
- 验证更新包
- 写入 Slot B
- 更新不影响当前系统

**3. 重启切换**
- 用户重启设备
- Bootloader 切换到 Slot B
- 验证 Slot B 完整性

**4. 成功/失败**
- **成功**：继续使用 Slot B，Slot A 作为备份
- **失败**：自动回滚到 Slot A

#### 优势

**1. 无缝更新**
- 后台更新，用户无感知
- 不需要进入 Recovery
- 更新过程不中断使用

**2. 快速回滚**
- 更新失败立即回滚
- 不需要重新下载
- 提高可靠性

**3. 提高可靠性**
- 减少更新失败风险
- 保留原系统作为备份
- 支持验证和回滚

### Seamless Updates

#### 概念

**无缝更新**：
- 在后台更新系统
- 用户无感知
- 重启后使用新系统

#### 实现

**后台更新**：
- 下载更新包
- 验证更新包
- 写入非活动槽
- 不影响当前系统

**重启切换**：
- 用户重启设备
- Bootloader 切换到新槽
- 验证新系统
- 启动新系统

### 分区更新顺序

#### 更新顺序

**关键分区优先**：
1. **vbmeta** - 验证元数据
2. **boot** - 启动分区
3. **system** - 系统框架
4. **vendor** - 厂商实现
5. **其他分区** - 按依赖关系

#### 依赖关系

**验证链**：
- vbmeta 必须首先更新
- boot 依赖 vbmeta
- system 依赖 boot
- vendor 依赖 system

### 回滚机制

#### 自动回滚

**触发条件**：
- 新系统验证失败
- 新系统启动失败
- 新系统关键服务失败

**回滚过程**：
1. 检测到问题
2. 标记新槽为坏
3. 重启到原槽
4. 恢复原系统

#### 手动回滚

**用户操作**：
- 某些设备支持手动回滚
- 通过设置或 Recovery
- 恢复到之前版本

### 动态分区更新

#### 动态分区特点

**超级分区**：
- 多个逻辑分区共享空间
- 分区大小可以调整
- 支持 A/B 分区

#### 更新流程

**1. 准备更新**
- 计算新分区大小
- 检查空间是否足够
- 准备更新环境

**2. 应用更新**
- 调整分区大小
- 写入新分区内容
- 更新分区表

**3. 验证更新**
- 验证分区完整性
- 验证分区大小
- 验证分区内容

#### 优势

**1. 灵活分配**
- 根据实际需要分配空间
- 避免空间浪费
- 支持不同配置

**2. 支持 A/B**
- 无缝 OTA 更新
- 快速回滚
- 提高可靠性

### 增量更新

#### 概念

**增量更新**：
- 只更新变化的部分
- 减少下载大小
- 加快更新速度

#### 实现

**块级差异**：
- 比较新旧镜像
- 生成差异块
- 只传输差异块

**应用差异**：
- 读取差异包
- 应用到原分区
- 生成新分区

#### 优势

**1. 减少下载**
- 只下载变化部分
- 大幅减少下载大小
- 节省带宽

**2. 加快更新**
- 减少下载时间
- 减少写入时间
- 加快更新速度

## 实际应用

### 查看当前槽

```bash
# 查看当前槽
getprop ro.boot.slot_suffix

# 查看所有槽
ls /dev/block/by-name/ | grep -E "_a|_b"
```

### 分析更新问题

```bash
# 查看更新日志
logcat -b all | grep -i "ota\|update"

# 查看分区状态
getprop | grep -i "slot\|update"
```

### 手动触发更新

```bash
# 检查更新
adb shell am start -a android.settings.SYSTEM_UPDATE_SETTINGS

# 查看更新状态
getprop sys.ota.status
```

## 总结

### 核心要点

1. **OTA 更新分多个阶段**：检查、下载、准备、应用、验证
2. **A/B 分区实现无缝更新**：后台更新，重启切换
3. **分区更新有顺序**：关键分区优先，按依赖关系
4. **回滚机制提高可靠性**：自动回滚和手动回滚
5. **动态分区支持灵活更新**：调整分区大小，支持 A/B

### 关键机制

- **A/B 分区**：双槽设计，无缝更新
- **Seamless Updates**：后台更新，用户无感知
- **回滚机制**：自动回滚和手动回滚
- **动态分区**：灵活的空间管理
