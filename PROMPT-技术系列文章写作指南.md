# 技术系列文章写作 Prompting 指南

> 本文件是一套经过实战验证的 AI 提示词模板，用于指导 AI 生成**有深度、有广度、有源码依据**的技术系列文章。
> 适用于任何技术模块（内核子系统、Framework 服务、Native 库等）的系列化学习输出。

---

## 一、核心原则（写在最前面，每次对话都要强调）

```
你写的不是科普文，不是面试八股，而是给【资深架构师】看的技术深度文章。
必须满足以下硬性要求：

1. 言必有据，码必有源：每个技术论点必须标注对应的源码文件路径，关键流程必须贴核心源码片段。
2. 自顶向下：每个章节先讲"是什么、为什么需要它"，建立上下文后再深入源码细节。禁止上来就贴代码。
3. 关联实战：每个知识点必须关联到真实的工程问题（Crash/ANR/OOM/性能退化等），说清楚"这个机制会在什么场景下出问题"。
4. 实战案例收尾：每篇文章末尾必须有 1-2 个完整的线上问题排查案例（现象→分析→根因→修复）。
5. 内容量级：每篇文章不少于 300 行（约 8000-15000 字），确保广度和深度兼具。
```

---

## 二、系列文章的设计流程

### 第一步：定义角色与目标

```
我的角色是Android 稳定性高级工程师。

我要系统学习【填入模块名称，如：Binder / SurfaceFlinger / Linux 调度器】。

学习目标不是"了解原理"，而是：
- 能在遇到线上问题时，快速定位到该模块的对应子系统和源码文件
- 能理解问题的根因机制
- 能给出治理方案或监控手段
```

### 第二步：让 AI 先输出大纲（不要直接写文章）

```
请按照以下架构师思维链，为【模块名称】设计一套系列文章的大纲：

架构师看一个系统的逻辑链：
  它是什么？解决什么问题？（定位）
      ↓
  它在系统中处于什么位置？和谁协作？（边界与交互）
      ↓
  它内部是怎么运转的？（核心机制）
      ↓
  它会在什么地方出问题？（风险地图）
      ↓
  出了问题我怎么查？怎么防？（诊断与治理）

大纲要求：
1. 分为 3-5 个篇章（全局观 → 核心机制 → 实战/对比）
2. 每篇文章列出具体章节，每个章节说清楚：
   - 讲什么内容
   - 对应哪个源码文件/目录
   - 关联什么工程问题
3. 用表格形式呈现每篇文章的章节规划
```

### 第三步：逐篇撰写（给出写作约束）

```
请根据大纲撰写第 N 篇文章。

每篇文章必须遵循以下写作结构：

┌─ 背景与定义（是什么、为什么需要它、解决什么问题）
│     不要假设读者已经知道，先用 1-2 段建立上下文
│
├─ 架构与交互（在系统中的位置、上下游依赖、数据流向）
│     画架构图或层级图，说清楚边界
│
├─ 核心机制与源码（关键数据结构、核心流程、源码走读）
│     每段源码必须标注 AOSP/内核 源码路径
│     贴代码前先用自然语言解释这段代码要干什么
│     代码后紧跟"稳定性架构师视角"分析
│
├─ 风险地图（这个机制会在哪些场景下出问题）
│     列出具体的异常类型、日志特征、触发条件
│
└─ 实战案例（1-2 个线上真实问题的完整排查过程）
      格式：现象 → 分析思路 → 根因 → 修复方案
```

## 三、万能提示词模板

以下是一个可直接复制使用的完整提示词模板，只需替换 `【】` 中的内容：

```
# 角色设定
我是一名【Android 稳定性架构师】，正在系统学习【模块名称】。

# 写作标准
请为我撰写一篇关于【具体主题】的深度技术文章，要求如下：

## 硬性要求
1. 目标读者：资深架构师，不是初学者。不需要解释基础概念（如什么是线程、什么是指针），但需要解释该模块特有的术语和机制。
2. 每个章节必须先讲"这个东西是什么、为什么需要它、解决什么问题"，然后再深入源码。
3. 涉及源码时，必须标注源码文件路径（如 `art/runtime/gc/heap.cc`），并贴出关键代码片段（不要全贴，只贴核心逻辑）。
4. 每个技术点必须关联到实际的工程问题（如 Crash、ANR、OOM、JE、NE、Watchdog、性能退化、兼容性问题），说清楚"它会在什么场景下咬你一口"。
5. 文章末尾必须有 1-2 个完整的实战案例（现象 → 分析 → 根因 → 修复）。
6. 文章长度不少于 300 行。

## 章节结构
请按照以下章节结构组织内容：

1. 【章节 1 标题】
   - 要讲的内容：...
   - 核心源码路径：...
   - 关联的工程问题：...

2. 【章节 2 标题】
   - 要讲的内容：...
   - 核心源码路径：...
   - 关联的工程问题：...

...（按需添加）

## 禁止事项
- 禁止写成面试八股（罗列概念但不深入）
- 禁止上来就贴源码而不给上下文
- 禁止只讲原理不关联实际问题
- 禁止使用"据说"、"通常认为"等模糊表述，要么给源码依据，要么给数据依据
```

## 四、常见的质量检查清单

文章写完后，用以下清单自查：

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | **有没有回答"是什么"** | 文章开头 2 段内能让读者明白这个东西的定位和职责 |
| 2 | **有没有回答"为什么"** | 解释了这个设计选择的背景和动机，而不仅仅是描述现状 |
| 3 | **有没有架构图/层级图** | 至少有一张图说明该模块在系统中的位置和上下游关系 |
| 4 | **源码有没有标路径** | 每段源码都标注了对应的文件路径 |
| 5 | **源码前有没有上下文** | 每段源码前都有自然语言解释"这段代码要干什么" |
| 6 | **有没有关联实际问题** | 每个知识点都能回答"这和我排查线上问题有什么关系" |
| 7 | **有没有实战案例** | 文章末尾至少有 1 个完整的排查案例 |
| 8 | **深度够不够** | 不是停留在 API 使用层面，而是深入到数据结构和算法层面 |
| 9 | **广度够不够** | 覆盖了该主题的主要方面，没有重大遗漏 |

## 五、踩坑经验（从实战中总结）

### 反面教材 1：纯科普模式
```
❌ "ART 是 Android 的运行时环境，负责执行 Java 代码。它支持 JIT 和 AOT 两种编译模式..."
```
问题：没有源码、没有数据结构、没有关联问题，停留在百科词条级别。

### 反面教材 2：代码堆砌模式
```
❌ "## 1. 源码结构
   源码路径：art/runtime/runtime.cc

   ```cpp
   bool Runtime::Init(...) {
     heap_ = new gc::Heap(...);
     ...
   }
   ```"
```
问题：上来就贴源码，没有解释"为什么要看这段代码"、"它解决什么问题"、"它和我的日常工作有什么关系"。

### 正确姿势：自顶向下 + 有据可查
```
✅ "## 1. ART 是什么
   ART 是 Android 的核心运行时引擎，负责...（定义）

   ## 2. 为什么需要它
   在 ART 之前，Android 使用 Dalvik，存在以下痛点...（背景）

   ## 3. 它在系统中的位置
   ART 位于 Framework 之下、Native 库之上...（架构图）

   ## 4. 核心机制
   ART 的启动流程始于 Zygote 调用 JNI_CreateJavaVM...
   - 源码路径：art/runtime/java_vm_ext.cc
   - （贴代码前）这个函数做了三件事：创建 Runtime → 启动 → 绑定主线程
   - （贴代码后）稳定性视角：如果 Runtime::Start 失败，Zygote 崩溃，系统反复重启

   ## 5. 实战案例
   现象：线上大量 TimeoutException...
   分析：FinalizerWatchdogDaemon 检测到...
   根因：finalize() 中获取了竞争激烈的锁...
   修复：..."
```

## 六、适用场景示例

本指南已成功用于以下模块的系列文章生成：

- **ART 系列**（8 篇）：总览 → 字节码 → 编译执行 → 类加载 → 内存/GC → JNI → 对比JVM → 演进
- 可复用于：**Binder 系列**、**SurfaceFlinger 系列**、**Linux 调度器系列**、**Input 系列**、**Watchdog 系列** 等

使用时只需按照"第二步"先让 AI 出大纲，审核大纲后再按"第三步"逐篇撰写即可。
