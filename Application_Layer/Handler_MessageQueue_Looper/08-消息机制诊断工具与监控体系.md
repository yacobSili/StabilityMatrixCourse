# 08-消息机制诊断工具与监控体系

上一篇我们梳理了 Handler 的稳定性风险全景——内存泄漏、消息堆积、卡顿、线程泄漏。知道"会出什么问题"只是第一步，更关键的问题是：**出了问题怎么发现？发现了怎么定位根因？定位后怎么系统性治理？**

这就是本篇的核心主题：围绕 Android 消息机制，建立一套从**检测 → 定位 → 治理 → 预防**的完整工程体系。

```
稳定性治理四阶段：

检测（Detection）   ─ Looper.Printer / Observer / Choreographer.FrameCallback / Watchdog
    ↓
定位（Diagnosis）   ─ Systrace / Perfetto / ANR trace 解读 / 消息队列状态分析
    ↓
修复（Fix）         ─ 针对具体根因：IO 移出主线程 / 移除同步屏障 / 清理延迟消息
    ↓
预防（Prevention）  ─ 消息时间预算 / 静态 Handler / removeCallbacks / CI 卡顿检测
```

对于稳定性架构师来说，掌握这四个阶段的工具和方法论，才能将"救火"升级为"防火"。

## 1. Looper.setMessageLogging(Printer) — 最简单的监控入口

### 1.1 为什么从 Printer 说起

所有主流的卡顿监控框架——BlockCanary、ANRWatchDog、Matrix——都绕不开 `Looper.loop()` 中的一个设计：**每条消息分发前后，Looper 会调用一个 `Printer` 的 `println()` 方法**。这是 Android 框架暴露给开发者的最原始、最直接的消息级监控钩子。

理解这个机制的源码，才能理解所有基于它构建的监控方案的能力边界和性能代价。

### 1.2 源码解析

> Source: `frameworks/base/core/java/android/os/Looper.java`

```java
// Looper.java（简化，保留核心逻辑）
public static void loop() {
    final Looper me = myLooper();
    final MessageQueue queue = me.mQueue;

    for (;;) {
        Message msg = queue.next(); // 可能阻塞
        if (msg == null) {
            return; // Looper 已退出
        }

        // ★ 关键点 1：分发前打印
        final Printer logging = me.mLogging;
        if (logging != null) {
            logging.println(">>>>> Dispatching to " + msg.target + " "
                    + msg.callback + ": " + msg.what);
        }

        // ... Observer 回调（后文讨论）...

        msg.target.dispatchMessage(msg); // 实际分发

        // ★ 关键点 2：分发后打印
        if (logging != null) {
            logging.println("<<<<< Finished to " + msg.target + " "
                    + msg.callback);
        }

        msg.recycleUnchecked();
    }
}
```

两次 `println()` 输出的格式是固定的：

```
>>>>> Dispatching to Handler (com.example.MyActivity$1) {a1b2c3d} android.view.ViewRootImpl$ViewRootHandler: 6
<<<<< Finished to Handler (com.example.MyActivity$1) {a1b2c3d} android.view.ViewRootImpl$ViewRootHandler
```

- `>>>>>` 标记消息分发开始
- `<<<<<` 标记消息分发结束
- 两者之间的时间差就是这条消息的处理耗时

### 1.3 BlockCanary 的核心原理

BlockCanary 的思路极其简洁：

```java
public class BlockDetector implements Printer {
    private long mStartTime;
    private static final long BLOCK_THRESHOLD = 100; // ms

    @Override
    public void println(String x) {
        if (x.charAt(0) == '>') {
            // 消息开始分发，记录起始时间
            mStartTime = SystemClock.uptimeMillis();
            // 同时启动一个延迟任务，在 worker 线程中 dump 主线程堆栈
            startStackSampling();
        } else {
            // 消息分发结束，计算耗时
            long cost = SystemClock.uptimeMillis() - mStartTime;
            stopStackSampling();
            if (cost > BLOCK_THRESHOLD) {
                reportBlock(cost, getStackSamples());
            }
        }
    }
}

// 注册
Looper.getMainLooper().setMessageLogging(new BlockDetector());
```

关键技巧在于 `startStackSampling()`：在 worker 线程中周期性（通常 50ms - 200ms）调用 `Thread.getStackTrace()` 获取主线程堆栈。当消息超时后，采集到的堆栈序列可以帮助定位耗时操作的具体位置。

### 1.4 性能开销分析

Printer 方案的核心代价在于**字符串拼接**：

```java
logging.println(">>>>> Dispatching to " + msg.target + " "
        + msg.callback + ": " + msg.what);
```

这行代码在每条消息分发前执行，涉及：

| 操作 | 开销 |
| :--- | :--- |
| `msg.target.toString()` | Handler 的 toString 创建新 String |
| `msg.callback.toString()` | Runnable 的 toString（如果非 null） |
| StringBuilder 拼接 | 编译器生成 append 链 |
| 最终 String 对象 | 堆内存分配 |

**在 120Hz 设备上的估算：**

```
VSync 消息：    120 次/秒
Input 事件：    60-120 次/秒
系统消息：      ~50 次/秒
───────────────────────────
合计：          ~300 次/秒
× 2 次 println/消息 = 600 次/秒字符串拼接
× 约 3 个临时对象/次 ≈ 1800 个临时对象/秒
```

在低端设备上，这可能额外增加 ~1ms/消息 的开销，并触发更频繁的 Minor GC。因此 Printer 方案更适合**调试构建或线上采样**，不宜全量开启。

### 1.5 与稳定性的关联

| 维度 | 说明 |
| :--- | :--- |
| **优势** | 无需 Root、无需系统签名、API 1 就可用 |
| **局限** | 字符串开销在高频路径上可观；无法获取 Message 对象本身 |
| **适用场景** | Debug 构建全量开启；Release 按 1%-10% 采样 |
| **典型框架** | BlockCanary、Sliver、Hugo |

## 2. Looper.Observer (Android P+) — 零字符串开销的系统级监控

### 2.1 为什么需要 Observer

Printer 方案的最大痛点是字符串拼接开销。Google 在 Android P (API 28) 中引入了 `Looper.Observer` 接口——一个 `@hide` API，提供三个回调方法，直接传递 `Message` 对象引用，**完全不涉及字符串拼接**。

### 2.2 接口定义

> Source: `frameworks/base/core/java/android/os/Looper.java`

```java
// Looper.java（@hide API，Android P+）
public interface Observer {
    /**
     * 消息开始分发。
     * @return token，传递给后续回调用于关联
     */
    Object messageDispatchStarting();

    /**
     * 消息分发完成。
     * @param token     messageDispatchStarting 返回的 token
     * @param msg       被分发的 Message 对象
     * @param dispatchEnd 分发结束时间（uptimeMillis）
     */
    void messageDispatched(Object token, Message msg, long dispatchEnd);

    /**
     * 消息分发过程中抛出了异常。
     * @param token     messageDispatchStarting 返回的 token
     * @param msg       被分发的 Message 对象
     * @param exception 抛出的异常
     */
    void dispatchingThrewException(Object token, Message msg, Exception exception);
}
```

在 `Looper.loop()` 中的调用位置：

```java
// Looper.loop()（简化）
for (;;) {
    Message msg = queue.next();

    // Printer 回调（字符串拼接）
    if (logging != null) { logging.println(">>>>> ..."); }

    // ★ Observer 回调（无字符串拼接）
    final Observer observer = sObserver;
    Object token = null;
    if (observer != null) {
        token = observer.messageDispatchStarting();
    }

    try {
        msg.target.dispatchMessage(msg);
        if (observer != null) {
            observer.messageDispatched(token, msg, SystemClock.uptimeMillis());
        }
    } catch (Exception exception) {
        if (observer != null) {
            observer.dispatchingThrewException(token, msg, exception);
        }
        throw exception;
    }

    // Printer 回调
    if (logging != null) { logging.println("<<<<< ..."); }
}
```

### 2.3 Observer 的优势

| 维度 | Printer | Observer |
| :--- | :--- | :--- |
| 数据传递 | 拼接好的 String | Message 对象引用 |
| 字符串开销 | 每条消息 2 次拼接 | 无 |
| 可获取信息 | Handler className、msg.what | Message 全部字段（what、target、when、callback 等） |
| 异常捕获 | 无 | dispatchingThrewException 可捕获分发异常 |
| API 级别 | API 1+ | API 28+（@hide） |

### 2.4 通过反射访问 Observer

由于 Observer 是 `@hide` API，应用无法直接使用。线上监控框架通常通过反射设置：

```java
public static void setLooperObserver() {
    try {
        // 获取 Looper.sObserver 静态字段
        Field observerField = Looper.class.getDeclaredField("sObserver");
        observerField.setAccessible(true);

        // 创建动态代理
        Class<?> observerClass = Class.forName("android.os.Looper$Observer");
        Object proxy = Proxy.newProxyInstance(
            observerClass.getClassLoader(),
            new Class[]{observerClass},
            (proxyObj, method, args) -> {
                switch (method.getName()) {
                    case "messageDispatchStarting":
                        return SystemClock.uptimeMillis(); // token = 开始时间
                    case "messageDispatched":
                        long start = (long) args[0];
                        Message msg = (Message) args[1];
                        long end = (long) args[2];
                        long cost = end - start;
                        if (cost > 100) {
                            reportSlowMessage(msg, cost);
                        }
                        return null;
                    case "dispatchingThrewException":
                        reportDispatchException((Message) args[1],
                            (Exception) args[2]);
                        return null;
                    default:
                        return null;
                }
            });
        observerField.set(null, proxy);
    } catch (Exception e) {
        // 反射失败，回退到 Printer 方案
        fallbackToPrinter();
    }
}
```

**风险提示：**
- Android 9+ 的 Hidden API 限制可能导致反射失败（灰名单/黑名单）
- 每个 Android 版本需要验证字段名和类结构是否变更
- 建议使用 [FreeReflection](https://github.com/nicjansma/free-reflection) 或类似库绕过限制

### 2.5 系统内部的使用：LooperStats

Android 系统自身使用 Observer 来实现 `LooperStats`，用于统计系统服务线程的消息处理耗时：

```
# 通过 dumpsys 查看 LooperStats（需要 root 或调试设备）
adb shell dumpsys looper_stats
```

输出样例：

```
120ms total in Handler (android.app.ActivityThread$H)
  android.app.servertransaction.LaunchActivityItem: 45 calls, avg 2.67ms
  android.app.servertransaction.ResumeActivityItem: 38 calls, avg 1.21ms
  ...
```

### 2.6 与稳定性的关联

Observer 是目前**精度最高、开销最低**的消息级监控方案。对于稳定性架构师：

- **线上监控首选**：Android P+ 设备优先使用 Observer，P 以下回退到 Printer
- **异常兜底**：`dispatchingThrewException` 可以在消息分发异常时做上下文收集，辅助 Crash 分析
- **消息画像**：通过 `msg.what`、`msg.target` 等字段建立消息耗时分布模型，定位热点消息

## 3. Systrace / Perfetto 中的消息追踪

### 3.1 为什么需要 Trace 工具

Printer 和 Observer 告诉你"哪条消息慢了"，但无法回答更深层的问题：

- 这条消息内部做了什么？哪个方法最耗时？
- 消息与消息之间有没有"间隙"（gap）——即 Looper 空转或被阻塞？
- 消息处理过程中有没有发生 Binder 调用导致跨进程等待？

这些问题需要**系统级 Trace 工具**：Systrace（旧）和 Perfetto（新）。

### 3.2 抓取 Looper 消息 Trace

**Systrace 方式：**

```bash
# 抓取 view + input + am 分类的 trace，缓冲区 32MB
python systrace.py -b 32768 -t 5 view input am -o trace.html
```

**Perfetto 方式（推荐）：**

```bash
# 使用 perfetto 命令行工具
adb shell perfetto \
  -c - --txt \
  -o /data/misc/perfetto-traces/trace.perfetto-trace \
<<EOF
buffers: {
  size_kb: 32768
  fill_policy: RING_BUFFER
}
data_sources: {
  config {
    name: "linux.ftrace"
    ftrace_config {
      ftrace_events: "sched/sched_switch"
      ftrace_events: "power/cpu_frequency"
      atrace_categories: "view"
      atrace_categories: "input"
      atrace_categories: "am"
    }
  }
}
duration_ms: 5000
EOF
```

### 3.3 在 Perfetto 中识别消息

打开 Perfetto UI（`ui.perfetto.dev`）加载 trace 文件后，在主线程（通常是 PID 对应的第一个线程）的 trace 栏中可以看到：

```
主线程 Trace 示例（时间轴视图）：

│←── 16.6ms ──→│←── 16.6ms ──→│←── 32ms（掉帧！）──→│
┌──────────────┐┌──────────────┐┌────────────────────────────┐
│ doFrame      ││ doFrame      ││ handleMessage(LAUNCH_ACTIVITY)│
│ ├ measure    ││ ├ measure    ││ ├ inflate layout (12ms)      │
│ ├ layout     ││ ├ layout     ││ ├ Binder call (15ms) ★      │
│ └ draw       ││ └ draw       ││ └ initView (5ms)            │
└──────────────┘└──────────────┘└────────────────────────────┘
                                        ↑
                                  这条消息耗时 32ms → 掉了 1 帧
```

**关键识别技巧：**

| 模式 | 含义 | Trace 表现 |
| :--- | :--- | :--- |
| **长条** | 单条消息耗时过长 | 一个 slice 宽度远超 16.6ms |
| **间隙（Gap）** | 消息之间有空闲时间 | 两个 slice 之间出现空白 |
| **Binder 等待** | 消息内部做了 IPC 调用 | slice 内出现 `binder transaction` 子 slice |
| **锁等待** | 消息内部等待锁 | 主线程 state 变为 `sleeping`/`blocked` |
| **GC 暂停** | 垃圾回收暂停主线程 | 出现 `GC` 相关 slice |

### 3.4 Binder 调用关联分析

当主线程消息中出现 Binder 调用时，Perfetto 可以展示完整的跨进程调用链：

```
App 主线程：
│ handleMessage(BIND_SERVICE)                              │
│   ├ ServiceManager.getService("activity") ───────────┐   │
│   │                                                   │   │
│   │   ← Binder 等待 15ms →                           │   │
│   │                                                   │   │
│   ├ ←──────────────────────────────────────────────┘   │
│   └ onServiceConnected()                               │

system_server 线程：
                    │ AMS.getService()  │
                    │  ├ 权限检查 (2ms) │
                    │  └ 查询 (13ms)   │
```

这种可视化让你能够精确判断：消息耗时的根因是"自身代码慢"还是"在等系统服务响应"。

### 3.5 与稳定性的关联

Systrace/Perfetto 是**离线分析的主力工具**。线上监控（Printer/Observer）负责发现问题，Perfetto 负责在开发/测试阶段精确定位根因。两者是互补关系：

```
线上监控（Printer/Observer）         Perfetto/Systrace
──────────────────────           ──────────────────
发现"什么消息慢了"                   发现"为什么这条消息慢"
毫秒级精度                          微秒级精度
可全量采集                          离线分析，无法在线上使用
堆栈采样（近似）                     完整执行链路（精确）
```

## 4. ANR Trace 中的 Handler 信息解读

### 4.1 为什么 ANR Trace 解读是核心技能

对于稳定性架构师而言，**ANR trace 解读是价值最高的实战技能**。线上用户遇到的 ANR 无法用 Perfetto 复现，你唯一的信息源就是系统在 ANR 发生时 dump 的 `traces.txt`。而 ANR 的本质是消息机制问题——理解 Handler/Looper/MessageQueue 在 trace 中的表现，是准确定位根因的关键。

### 4.2 ANR Trace 的基本结构

```
----- pid 12345 at 2025-01-15 10:30:45 -----
Cmd line: com.example.myapp
...

"main" prio=5 tid=1 Blocked
  | group="main" sCount=1 dsCount=0 flags=1 obj=0x72c09e58 self=0xb400007a
  | sysTid=12345 nice=-10 cgrp=top-app sched=0/0 handle=0x7b9abcdef0
  | state=S schedstat=( 1234567890 987654321 5678 ) utm=100 stm=50 core=4 HZ=100
  | stack=0x7ffc000000-0x7ffc002000 stackSize=8192KB
  | held mutexes=
  at com.example.myapp.DataManager.loadFromDatabase(DataManager.java:125)
  - waiting to lock <0x0a1b2c3d> (a java.lang.Object) held by thread 15
  at com.example.myapp.MainActivity.onResume(MainActivity.java:67)
  at android.app.Instrumentation.callActivityOnResume(Instrumentation.java:1456)
  at android.app.Activity.performResume(Activity.java:8130)
  ...
  at android.os.Handler.dispatchMessage(Handler.java:106)
  at android.os.Looper.loop(Looper.java:223)
  at android.app.ActivityThread.main(ActivityThread.java:7656)
```

### 4.3 四种典型的主线程状态

**状态一：主线程在处理耗时消息**

```
"main" prio=5 tid=1 Native
  at android.os.BinderProxy.transactNative(Native Method)
  at android.os.BinderProxy.transact(BinderProxy.java:540)
  at android.content.ContentProviderProxy.query(ContentProviderNative.java:472)
  at com.example.myapp.SyncManager.syncData(SyncManager.java:89)
  at android.os.Handler.dispatchMessage(Handler.java:106)
  at android.os.Looper.loop(Looper.java:223)
```

**解读：** 主线程正在执行一条消息（`SyncManager.syncData`），并且卡在 Binder 调用上（等 ContentProvider 返回）。这是最直接的 ANR 模式——单条消息耗时超标。

**状态二：主线程空闲（nativePollOnce）**

```
"main" prio=5 tid=1 Native
  at android.os.MessageQueue.nativePollOnce(Native Method)
  at android.os.MessageQueue.next(MessageQueue.java:335)
  at android.os.Looper.loop(Looper.java:206)
  at android.app.ActivityThread.main(ActivityThread.java:7656)
```

**解读：** 主线程阻塞在 `nativePollOnce`——说明主线程**此刻是空闲的**。ANR 发生但主线程空闲，可能的原因：

| 可能原因 | 排查方向 |
| :--- | :--- |
| 同步屏障未移除 | 检查是否有 `postSyncBarrier` 未配对 `removeSyncBarrier` |
| 消息还在队列中未到执行时间 | 检查队列中延迟消息的 `when` 字段 |
| ANR 发生在"消息间隙"中 | 前一条消息耗时导致超时炸弹已触发，但当前已处理完 |
| Binder 线程池满 | 检查 Binder 线程是否全部阻塞 |

**状态三：主线程等待锁**

```
"main" prio=5 tid=1 Blocked
  at com.example.myapp.CacheManager.getCache(CacheManager.java:45)
  - waiting to lock <0x0a1b2c3d> (a java.lang.Object) held by thread 15
  at android.os.Handler.dispatchMessage(Handler.java:106)
  at android.os.Looper.loop(Looper.java:223)

"worker-thread-3" prio=5 tid=15 Native
  at android.os.BinderProxy.transactNative(Native Method)
  ...
  - locked <0x0a1b2c3d> (a java.lang.Object)
```

**解读：** 主线程在处理消息时试图获取一把锁，但锁被 thread 15 持有。thread 15 正在做 Binder 调用，持锁不放。**典型的"锁反转"导致的 ANR**。

**状态四：主线程在执行 SharedPreferences.apply()**

```
"main" prio=5 tid=1 Waiting
  at java.lang.Object.wait(Native Method)
  at java.lang.Object.wait(Object.java:442)
  at android.app.QueuedWork.waitToFinish(QueuedWork.java:176)
  at android.app.ActivityThread.handleStopActivity(ActivityThread.java:4562)
  at android.os.Handler.dispatchMessage(Handler.java:106)
  at android.os.Looper.loop(Looper.java:223)
```

**解读：** Activity Stop 时，`QueuedWork.waitToFinish()` 会**同步等待所有 SharedPreferences.apply() 完成**。如果 apply() 的写入操作尚未完成，主线程会在此阻塞。这是一个被广泛忽视但极其常见的 ANR 根因。

### 4.4 检查 MessageQueue 状态

在 ANR trace 中，有时可以看到 MessageQueue 的 dump 信息（需要系统支持）：

```
Message Queue (when=now):
  Message 0: { when=-2s500ms what=110 target=android.app.ActivityThread$H }
  Message 1: { when=-1s200ms what=159 target=android.app.ActivityThread$H }
  Message 2: { when=-500ms what=0 target=com.example.MyHandler callback=... }
  (Total messages: 47, polling=false, quitting=false)
  Sync barriers:
    token=12345 when=-3s (★ 未移除的同步屏障！)
```

**关键指标：**

- `when` 为负值：消息已到期但尚未被处理，说明队列拥堵
- `Total messages` 过多：消息堆积严重
- `Sync barriers` 存在未移除的屏障：所有同步消息被阻塞

### 4.5 与稳定性的关联

ANR trace 解读的核心方法论：

```
第一步：看主线程状态
    → 在处理消息？→ 看具体方法调用链，找耗时操作
    → nativePollOnce？→ 主线程空闲，查同步屏障 / Binder 线程池 / 消息时序

第二步：看锁信息
    → waiting to lock <xxx> held by thread N → 找 thread N 在做什么

第三步：看 MessageQueue
    → 待处理消息数量、是否有同步屏障、消息到期时间

第四步：看其他关键线程
    → Binder 线程是否全部阻塞？是否有 finalizer 线程死锁？
```

## 5. 卡顿监控方案对比

### 5.1 为什么需要多种方案

没有一种卡顿监控方案是完美的——每种方案在精度、开销、版本兼容性和适用场景上都有取舍。稳定性架构师需要理解各方案的原理，才能针对不同场景选型。

### 5.2 方案一：Looper Printer

前文已详细分析。核心原理是利用 `Looper.loop()` 中消息分发前后的 Printer 回调，计算每条消息耗时。

```java
Looper.getMainLooper().setMessageLogging(printer);
```

### 5.3 方案二：Choreographer.FrameCallback

Choreographer 是 Android UI 渲染的"节拍器"。每当 VSync 信号到来，Choreographer 会执行注册的 `FrameCallback`。利用连续两帧之间的时间差，可以检测掉帧：

```java
public class FrameMonitor implements Choreographer.FrameCallback {
    private long mLastFrameTimeNanos;
    private static final long FRAME_INTERVAL_NANOS = 16_666_667L; // 16.6ms @ 60fps

    @Override
    public void doFrame(long frameTimeNanos) {
        if (mLastFrameTimeNanos != 0) {
            long jitterNanos = frameTimeNanos - mLastFrameTimeNanos;
            int droppedFrames = (int) (jitterNanos / FRAME_INTERVAL_NANOS) - 1;
            if (droppedFrames > 0) {
                reportDroppedFrames(droppedFrames,
                    TimeUnit.NANOSECONDS.toMillis(jitterNanos));
            }
        }
        mLastFrameTimeNanos = frameTimeNanos;
        Choreographer.getInstance().postFrameCallback(this); // 注册下一帧
    }
}

// 启动
Choreographer.getInstance().postFrameCallback(new FrameMonitor());
```

**Matrix 的 FrameTracer** 就是基于此原理，配合 Choreographer 内部的各阶段（Input、Animation、Traversal、Commit）时间戳，精确定位是 measure/layout/draw 哪个阶段导致掉帧。

### 5.4 方案三：主线程 Watchdog

ANRWatchDog 的思路完全不依赖 Looper 内部回调。它在后台线程周期性地向主线程 post 一个 Runnable，检测该 Runnable 是否在超时时间内被执行：

```java
public class MainThreadWatchdog extends Thread {
    private volatile boolean mCompleted = true;
    private final Handler mMainHandler = new Handler(Looper.getMainLooper());
    private static final long TIMEOUT = 5000; // 5s

    private final Runnable mTicker = () -> mCompleted = true;

    @Override
    public void run() {
        while (!isInterrupted()) {
            mCompleted = false;
            mMainHandler.post(mTicker);
            try {
                Thread.sleep(TIMEOUT);
            } catch (InterruptedException e) {
                return;
            }
            if (!mCompleted) {
                // 主线程在 5s 内没有执行 mTicker → 主线程卡住了
                reportMainThreadStuck(Looper.getMainLooper().getThread()
                    .getStackTrace());
            }
        }
    }
}
```

### 5.5 方案对比表

| 维度 | Looper Printer | FrameCallback | 主线程 Watchdog |
| :--- | :--- | :--- | :--- |
| **检测粒度** | 单条消息 | 单帧（16.6ms） | 秒级（通常 5s） |
| **精度** | 高（消息级） | 中（帧级） | 低（秒级） |
| **性能开销** | 中（字符串拼接） | 低 | 极低 |
| **版本要求** | API 1+ | API 16+ | API 1+ |
| **能检测的问题** | 消息级卡顿 | 掉帧/渲染卡顿 | 长时间无响应（类 ANR） |
| **检测不到的问题** | 非消息导致的卡顿 | 非渲染路径的卡顿 | 短暂卡顿（< 5s） |
| **适用场景** | Debug 全量 + Release 采样 | 渲染性能监控 | ANR 预警 |
| **典型框架** | BlockCanary | Matrix FrameTracer | ANRWatchDog |

### 5.6 组合策略

实际工程中，三种方案通常组合使用：

```
┌────────────────────────────────────────────────────────────┐
│                   卡顿监控体系                                │
├────────────────────────────────────────────────────────────┤
│                                                            │
```

## 6. 治理最佳实践

### 6.1 从"能查"到"能治"到"能防"

监控只是手段，治理才是目的。以下是从无数线上案例中沉淀的 Handler 消息机制治理最佳实践。

### 6.2 消息时间预算

60fps 要求每帧 16.6ms，但主线程每帧不仅处理 UI 渲染，还要处理 Input 事件、系统回调等消息。因此**单条消息的时间预算应远小于 16.6ms**：

```
一帧的时间预算分配（60fps / 16.6ms）：

Input 处理：       ~2ms
Animation：        ~2ms
Measure/Layout：   ~4ms
Draw/Record：      ~4ms
余量（其他消息）：   ~4ms
───────────────────────
合计：              16ms

→ 单条非渲染消息的安全上限：< 5ms
→ 单条消息的告警阈值：> 16ms（一定掉帧）
→ 单条消息的严重阈值：> 100ms（用户可感知卡顿）
```

### 6.3 静态内部类 + WeakReference

Handler 内存泄漏是 Android 最经典的泄漏模式。规则很简单但必须严格执行：

```java
public class MyActivity extends AppCompatActivity {

    // ✅ 静态内部类，不持有外部类引用
    private static class SafeHandler extends Handler {
        private final WeakReference<MyActivity> mActivityRef;

        SafeHandler(MyActivity activity) {
            super(activity.getMainLooper());
            mActivityRef = new WeakReference<>(activity);
        }

        @Override
        public void handleMessage(@NonNull Message msg) {
            MyActivity activity = mActivityRef.get();
            if (activity == null || activity.isFinishing()) {
                return;
            }
            // 安全地使用 activity
        }
    }

    private final SafeHandler mHandler = new SafeHandler(this);

    @Override
    protected void onDestroy() {
        super.onDestroy();
        mHandler.removeCallbacksAndMessages(null); // ★ 清除所有未执行的消息
    }
}
```

### 6.4 removeCallbacksAndMessages(null) 的重要性

`removeCallbacksAndMessages(null)` 会**移除该 Handler 关联的所有消息和回调**。在 `onDestroy` 中调用它，可以：

1. **阻断引用链**：Message 从 MessageQueue 中移除 → Message.target（Handler）的引用被清除 → Handler → Activity 的引用链断开
2. **防止"幽灵回调"**：Activity 销毁后，延迟消息到期仍会执行回调 → 操作已销毁的 View → Crash
3. **减少消息堆积**：不需要的消息及时清理，减轻 MessageQueue 负担

### 6.5 IdleHandler：只做轻量工作，永远 return false

`IdleHandler` 在消息队列空闲时被调用，常被用来做延迟初始化。但两个关键约束经常被忽视：

```java
// ✅ 正确用法
Looper.myQueue().addIdleHandler(() -> {
    // 轻量工作：< 5ms
    preloadSmallConfig();
    return false; // ★ 一次性执行后移除
});

// ❌ 错误用法 1：耗时操作
Looper.myQueue().addIdleHandler(() -> {
    loadLargeDatabase(); // 200ms+ → 阻塞后续消息处理
    return false;
});

// ❌ 错误用法 2：return true 导致重复执行
Looper.myQueue().addIdleHandler(() -> {
    doSomething();
    return true; // 每次空闲都执行 → 主线程永远不会真正空闲 → 影响 GC 和内存回收
});
```

**IdleHandler 的执行时机在 `MessageQueue.next()` 中，在 `nativePollOnce` 阻塞之前**。如果 IdleHandler 耗时过长，会延迟 Looper 进入真正的空闲状态，间接影响后续消息的及时处理。

### 6.6 异步消息：应用层不应使用 FLAG_ASYNCHRONOUS

同步屏障 + 异步消息是系统保障 UI 渲染优先级的机制。`Message.setAsynchronous(true)` 会让消息穿透同步屏障。**应用层不应使用这个标志**，原因：

1. `postSyncBarrier()` 是 `@hide` API，应用无法插入屏障，设置异步标志毫无意义
2. 如果通过反射使用，可能干扰系统的渲染调度——导致非 UI 消息在渲染帧中被插队处理
3. 未来版本可能更改异步消息的行为

### 6.7 SharedPreferences.apply() 的 QueuedWork 陷阱

`SharedPreferences.apply()` 看似异步，实际上有一个隐藏的同步点：

```
apply() 调用链：

SharedPreferencesImpl.apply()
    → QueuedWork.addFinisher(writeToDiskRunnable)
        ↓
    后台线程异步写入磁盘
        ↓
Activity.onStop() / Service.onStop()
    → ActivityThread.handleStopActivity()
        → QueuedWork.waitToFinish()  // ★ 主线程同步等待所有 apply 完成！
```

如果 apply() 的后台写入还没完成，`waitToFinish()` 会在主线程**同步阻塞**，直到写入完毕。当 SharedPreferences 文件较大或磁盘 IO 繁忙时，这个等待可能长达数秒，直接导致 ANR。

**替代方案：**

| 方案 | 优势 | 适用场景 |
| :--- | :--- | :--- |
| **MMKV** | mmap 写入，无 QueuedWork 问题 | 通用 KV 存储 |
| **DataStore** | 协程驱动，完全异步 | Kotlin 项目 |
| **commit() + 后台线程** | 可控写入时机 | 需要确认写入结果的场景 |

### 6.8 HandlerThread 生命周期管理

HandlerThread 创建后会启动一个新线程并运行 Looper。如果不调用 `quit()` 或 `quitSafely()`：

- 线程永远不会退出 → 线程泄漏
- Looper 内的 epoll fd 不会释放 → FD 泄漏
- 累积到一定数量后触发 `Too many open files` 崩溃

```java
public class MyService extends Service {
    private HandlerThread mWorkerThread;
    private Handler mWorkerHandler;

    @Override
    public void onCreate() {
        super.onCreate();
        mWorkerThread = new HandlerThread("MyWorker");
        mWorkerThread.start();
        mWorkerHandler = new Handler(mWorkerThread.getLooper());
    }

    @Override
    public void onDestroy() {
        super.onDestroy();
        // ★ 必须在对应生命周期中 quit
        mWorkerThread.quitSafely();
        // quitSafely 会处理完已到期的消息后退出
        // quit 会丢弃所有消息立即退出
    }
}
```

### 6.9 治理检查清单

| 检查项 | 标准 | 检测方式 |
| :--- | :--- | :--- |
| 单条消息耗时 | < 16ms | Looper Observer/Printer |
| Handler 是否静态内部类 | 是 | Lint `HandlerLeak` 规则 |
| onDestroy 是否清理消息 | 是 | Code Review / 自定义 Lint |
| IdleHandler 是否 return false | 是（除非明确需要） | Code Review |
| HandlerThread 是否 quit | 是 | LeakCanary / 线程监控 |
| SharedPreferences 是否用 apply | 避免大文件 apply | QueuedWork 耗时监控 |
| 是否使用 FLAG_ASYNCHRONOUS | 否（应用层） | Code Review |
| postDelayed 是否有对应 remove | 是 | 自定义 Lint |

## 7. 稳定性实战案例

### 案例一：SharedPreferences.apply() 导致的批量 ANR

**背景：** 某电商 App 在大促期间 ANR 率从 0.1% 飙升到 1.2%，集中在 Activity 切换场景。

**现象：**
- ANR trace 中主线程堆栈一致指向 `QueuedWork.waitToFinish()`
- 只在 Android 8.0-11 设备上出现（Android 12 对 QueuedWork 做了优化）
- 大促期间比平时严重（因为业务配置写入更频繁）

**ANR Trace 关键片段：**

```
"main" prio=5 tid=1 Waiting
  at java.lang.Object.wait(Native Method)
  - waiting on <0x0abc1234> (a java.lang.Object)
  at java.lang.Object.wait(Object.java:442)
  at android.app.QueuedWork.waitToFinish(QueuedWork.java:176)
  at android.app.ActivityThread.handleStopActivity(ActivityThread.java:4562)
  at android.app.ActivityThread$H.handleMessage(ActivityThread.java:2012)
  at android.os.Handler.dispatchMessage(Handler.java:106)
  at android.os.Looper.loop(Looper.java:223)
  at android.app.ActivityThread.main(ActivityThread.java:7656)
```

**排查过程：**

1. **确认模式**：所有 ANR trace 都指向 `QueuedWork.waitToFinish`。主线程状态是 `Waiting`，不是在做计算，而是在等待锁。

2. **追溯 apply 调用**：全局搜索 `SharedPreferences.apply()` 调用点，发现大促期间新增了 3 个高频写入点：
   - 搜索历史记录（每次搜索写入一次）
   - A/B 实验配置缓存（每次打开页面写入）
   - 用户行为埋点临时缓存（每次滑动写入）

3. **计算写入量**：大促期间用户操作频繁，3 个写入点叠加后，SP 文件从 5KB 膨胀到 200KB+。每次 `apply()` 的后台写入从 1ms 增长到 50ms+。当用户快速切换 Activity 时，`handleStopActivity` → `waitToFinish` 需要等待积压的所有 apply 操作完成。

```
正常时期：
  SP 文件大小：5KB
  apply 写入耗时：~1ms
  waitToFinish 等待：~0ms（已完成）

大促期间：
  SP 文件大小：200KB+
  apply 写入耗时：~50ms/次
  快速切换积压 5 次 apply → waitToFinish 等待：~250ms+
  叠加磁盘 IO 竞争 → 等待可能 > 5s → ANR
```

**修复方案：**

1. **短期止血**：将高频写入的 3 个 SP 文件替换为 MMKV，消除 QueuedWork 问题

```java
// Before
SharedPreferences sp = getSharedPreferences("search_history", MODE_PRIVATE);
sp.edit().putString("last_query", query).apply();

// After
MMKV kv = MMKV.mmkvWithID("search_history");
kv.encode("last_query", query);
```

2. **长期方案**：
   - CI 流水线增加 Lint 规则，禁止新增 `SharedPreferences.apply()` 调用
   - 逐步将所有 SP 迁移到 MMKV 或 DataStore
   - 增加 `QueuedWork.waitToFinish` 耗时监控，在超过 100ms 时告警

**效果：** 替换后大促 ANR 率从 1.2% 降至 0.08%。

**稳定性启示：** `SharedPreferences.apply()` 的"异步"是一个危险的假象——它只是把写入放到后台线程，但在 Activity 生命周期回调中会被强制同步等待。**理解 QueuedWork 与 Handler 消息分发的交互关系，是避免这类 ANR 的关键。**

### 案例二：同步屏障泄漏导致的"幽灵卡死"

**背景：** 某社交 App 收到用户反馈"打开某个页面后，点击任何按钮都没反应，但画面不是完全卡死——动画还在播放"。

**现象：**
- 复现概率约 0.5%，在特定机型（MTK 芯片 + Android 10）上稍高
- 用户看到的效果是：页面渲染正常（动画流畅），但点击、滑动无反应
- 不会触发系统 ANR 弹窗
- 杀进程重启后恢复

**矛盾点：** 如果主线程卡死，动画不可能播放。但如果主线程没卡死，为什么不响应点击？

**排查过程：**

1. **抓取 Perfetto trace**：在复现设备上抓取 trace。发现主线程一直在处理 `doFrame`（Choreographer 渲染帧），但**完全没有 Input 事件被处理**。

```
Perfetto 时间轴：
│ doFrame │ doFrame │ doFrame │ doFrame │ doFrame │ ...（无限循环）
              ↑ 没有任何 Input dispatchTouchEvent 出现
```

2. **怀疑同步屏障**：`doFrame` 中的消息是异步消息（setAsynchronous(true)），它们能穿透同步屏障。Input 事件是同步消息，会被同步屏障阻塞。如果存在一个**未移除的同步屏障**，就会出现"动画正常但不响应输入"的症状。

3. **通过反射 dump MessageQueue**：

```java
// 紧急调试代码
MessageQueue queue = Looper.getMainLooper().getQueue();
Field messagesField = MessageQueue.class.getDeclaredField("mMessages");
messagesField.setAccessible(true);
Message msg = (Message) messagesField.get(queue);

while (msg != null) {
    if (msg.target == null) {
        // target == null → 同步屏障！
        Log.e("BARRIER", "Found sync barrier! when=" + msg.when
            + " arg1(token)=" + msg.arg1);
    }
    Field nextField = Message.class.getDeclaredField("next");
    nextField.setAccessible(true);
    msg = (Message) nextField.get(msg);
}
```

**确认结果：** 队列中存在一条 `target == null` 的消息（同步屏障），token 值与某个自定义 View 的 `scheduleTraversals` 有关。

4. **定位根因**：一个自定义的 `SurfaceView` 子类在某个异常路径中调用了 `ViewRootImpl.scheduleTraversals()`（通过 `requestLayout()`），插入了同步屏障。但后续的 `performTraversals` 由于异常被中断，**`unscheduleTraversals()` 未被调用**，屏障永远留在队列中。

```
正常流程：
  scheduleTraversals() → postSyncBarrier()
      → VSync 信号到来
          → doTraversal() → unscheduleTraversals() → removeSyncBarrier()
              → 屏障移除，同步消息恢复处理

异常流程（Bug 路径）：
  scheduleTraversals() → postSyncBarrier()
      → VSync 信号到来
          → doTraversal() 内部抛异常 → unscheduleTraversals() 未执行
              → 屏障永远不会被移除
              → 所有同步消息（包括 Input 事件）被永久阻塞
              → 但异步消息（doFrame 渲染）正常执行
              → 用户看到"画面流畅但不响应点击"
```

**修复方案：**

1. **修复异常路径**：在自定义 View 的关键路径添加 try-finally，确保 `unscheduleTraversals()` 一定被调用

2. **增加同步屏障泄漏检测**：

```java
public class SyncBarrierDetector {
    private static final long BARRIER_TIMEOUT = 3000; // 3s

    public static void startDetection() {
        Handler handler = new Handler(Looper.getMainLooper());
        handler.postDelayed(new Runnable() {
            @Override
            public void run() {
                // 如果这条同步消息能被执行，说明没有同步屏障
                // 如果超时未执行，说明可能存在同步屏障泄漏
                handler.postDelayed(this, BARRIER_TIMEOUT);
            }
        }, BARRIER_TIMEOUT);

        // 在后台线程检测：向主线程 post 一个同步消息，如果 N 秒内未执行 → 报警
        new Thread(() -> {
            while (true) {
                AtomicBoolean executed = new AtomicBoolean(false);
                handler.post(() -> executed.set(true));
                try { Thread.sleep(BARRIER_TIMEOUT); } catch (InterruptedException e) { return; }
                if (!executed.get()) {
                    reportSyncBarrierLeak();
                }
            }
        }).start();
    }
}
```

3. **建立兜底机制**：当检测到同步屏障泄漏时，尝试通过反射移除屏障恢复主线程

**效果：** 修复后该类问题完全消失。同步屏障检测机制上线后，还发现了 2 个其他场景的屏障泄漏并提前修复。

**稳定性启示：** 同步屏障泄漏是最难排查的消息机制问题之一——它**不会触发 ANR（因为异步消息正常处理）、不会触发 Crash（没有异常抛出）**，但会导致应用"假死"。只有深入理解同步屏障与异步消息的机制，才能解释"画面正常但不响应"这种反直觉现象，并建立有效的检测手段。

## 总结

本篇围绕 Handler 消息机制，建立了从检测到治理的完整工程体系：

| 工具/方案 | 核心能力 | 适用阶段 |
| :--- | :--- | :--- |
| **Looper.Printer** | 消息级耗时监控，最简单 | 开发/测试 + 线上采样 |
| **Looper.Observer** | 零字符串开销的消息监控 | 线上全量（Android P+） |
| **Systrace/Perfetto** | 微秒级全链路分析 | 开发/测试阶段定位根因 |
| **ANR Trace 解读** | 线上 ANR 根因定位 | 线上问题分析 |
| **FrameCallback** | 掉帧检测 | 线上全量 |
| **主线程 Watchdog** | ANR 级别卡顿预警 | 线上全量 |
| **治理最佳实践** | 从检测到修复到预防 | 全生命周期 |

核心方法论：**监控是手段，治理是目的，预防是终极目标。** 好的稳定性体系不是等问题出现后去"救火"，而是通过消息时间预算、编码规范、CI 自动检测，将问题消灭在上线之前。
