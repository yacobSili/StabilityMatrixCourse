# 05-内存与GC：OOM 的七种死法

这是稳定性架构师的**主战场**。当你的线上 Crash 报表中 OOM 稳居 Top 3，当你发现不同类型的 OOM 日志长得完全不一样，当你被"堆内存明明还有 100MB 为什么还是 OOM"的诡异现象折磨——这篇文章就是为你准备的。

本篇将从 ART 的 Heap 空间划分出发，自底向上地拆解内存分配器、GC 算法、GC 触发时机、Finalizer 机制，以及**五种不同类型 OOM 的触发条件和排查手段**。目标是让你在看到任何一条 OOM 日志时，能在 3 秒内判断出它属于哪种类型，并知道从哪里入手排查。

---

## 1. Heap 空间划分

### 1.1 为什么要划分多个 Space

一个简单的问题：为什么 ART 不像 C 的 `malloc` 一样只维护一块大内存，需要时就分配、不需要时就释放？

答案是 **GC 效率**。不同类型的对象有完全不同的生命周期特征和移动需求：

*   **核心类对象**（如 `java.lang.String` 的 Class 对象）在 Zygote 阶段就已创建，所有 App 进程共享，永远不会被回收——给它们分配到一个只读的共享区域，GC 永远不需要扫描这块区域。
*   **App 创建的普通对象**（如 `new ArrayList()`）生命周期短暂，频繁创建和销毁——需要一个高效的分配和回收区域。
*   **大对象**（如 `byte[4MB]`）分配和拷贝成本极高——把它们隔离到独立区域，避免在 GC 的压缩（Compacting）阶段被移动。
*   **Class 元数据**（`mirror::Class` 对象）不能被 GC 移动（因为所有实例对象都持有指向 Class 的指针，移动 Class 需要更新所有指针，代价巨大）——需要一个不可移动区域。

基于这些需求，ART 将 Java Heap 划分为 **5 个核心 Space**。

### 1.2 五大 Space 的职责与边界

*   **源码路径：** `art/runtime/gc/heap.h`, `art/runtime/gc/space/`

```cpp
// art/runtime/gc/heap.h (简化)
class Heap {
  // Image Space: boot.art 映射的只读区域，包含预初始化的核心类对象
  space::ImageSpace* boot_image_space_;

  // Zygote Space: Zygote 进程 fork 前分配的对象，COW 共享
  space::ZygoteSpace* zygote_space_;

  // Allocation Space (也叫 Main Space/Region Space): App 对象的主分配区域
  space::RegionSpace* region_space_;        // CC GC 使用
  // 或 space::DlMallocSpace* main_space_;  // CMS GC 使用

  // Large Object Space: 大于 primitive_array_threshold_ (默认 3 页 = 12KB) 的数组
  space::LargeObjectSpace* large_object_space_;

  // Non-Moving Space: 不可被 GC 移动的对象 (mirror::Class, DexCache 等)
  space::MallocSpace* non_moving_space_;

  // 关键参数
  size_t max_allowed_footprint_;     // 当前允许的最大堆大小 (动态增长)
  size_t growth_limit_;              // 堆增长上限 (dalvik.vm.heapgrowthlimit)
  size_t max_heap_size_;             // 堆绝对上限 (dalvik.vm.heapsize)
};
```

各 Space 的详细对比：

| Space | 存放内容 | 生命周期 | 是否可被 GC 移动 | 典型大小 |
| :--- | :--- | :--- | :--- | :--- |
| **Image Space** | `boot.art` 中预初始化的核心类和字符串对象 | 与系统同生共死 | 否（只读 mmap） | 60-120MB |
| **Zygote Space** | Zygote fork 前分配的对象（预加载类的实例） | 与进程同生共死 | 否（COW 保护） | 10-30MB |
| **Allocation Space** | App 运行时 `new` 出来的所有普通对象 | 由 GC 管理 | 是（CC GC 会移动） | 受 `growth_limit_` 限制 |
| **Large Object Space** | 大于 12KB 的原始类型数组（`byte[]`, `int[]` 等） | 由 GC 管理 | 否（移动成本太高） | 无独立上限，共享堆配额 |
| **Non-Moving Space** | `mirror::Class`、`DexCache`、JNI Global Reference 指向的对象 | 由 GC 管理，但不可移动 | 否 | 通常 64-128MB |

### 1.4 稳定性：各 Space 容量配置与 OOM 的关系

*   **`growth_limit_` 与 `max_heap_size_` 的区别：** `growth_limit_` 是 App 默认的堆上限（对应 `dalvik.vm.heapgrowthlimit`，通常 256MB）。当 App 在 `AndroidManifest.xml` 中声明了 `android:largeHeap="true"` 后，ART 将 `max_heap_size_`（对应 `dalvik.vm.heapsize`，通常 512MB）作为堆上限。**但 largeHeap 不是免费午餐**——更大的堆意味着 GC 扫描范围更大、STW 时间更长、内存压力更大时被 LMK（Low Memory Killer）杀掉的概率更高。
*   **Non-Moving Space 独立 OOM：** Non-Moving Space 有自己的容量上限。即使 Allocation Space 还有大量空余，如果 Non-Moving Space 满了（加载了太多类），也会触发 OOM。此时日志中的关键特征是 `space: Non-Moving Space`。
*   **Large Object Space 共享堆配额：** Large Object Space 与 Allocation Space 共享 `growth_limit_` 配额。一个 App 如果频繁分配大 Bitmap（每张占 4-16MB），Large Object Space 会迅速挤压 Allocation Space 的可用空间，导致普通对象分配失败。

## 2. 内存分配器

### 2.1 为什么需要专用分配器

操作系统提供的 `malloc`（在 Android 上是 Bionic 的 `jemalloc` 或 `scudo`）是通用内存分配器，不了解 Java 对象的特性。Java 对象分配有以下特殊需求：

1.  **极高频率：** 一个典型 App 每秒可能分配数万个对象。通用 `malloc` 在多线程高并发下需要频繁加锁。
2.  **大小相对集中：** Java 对象的大小通常集中在 16B-256B 区间（小对象），分配器可以针对这些大小做预分配。
3.  **GC 需要遍历所有存活对象：** 分配器需要能快速枚举所有已分配的对象和空闲区域。

ART 为此实现了两个专用分配器：**RosAlloc**（默认，用于 CMS GC）和 **Region-based allocator**（用于 CC GC）。

### 2.2 RosAlloc：Run-of-Slots + TLAB

RosAlloc（Runs-of-Slots Allocator）是 ART 为小对象分配设计的高性能分配器，核心思想是**按大小分桶 + 线程本地缓存**。

*   **源码路径：** `art/runtime/gc/allocator/rosalloc.cc`

```cpp
// art/runtime/gc/allocator/rosalloc.cc (概念性简化)
class RosAlloc {
  // Run: 一块连续的内存区域，被切分为相同大小的 Slot
  struct Run {
    uint8_t magic_num_;
    uint8_t size_bracket_idx_;   // 大小类别 (如 16B, 32B, 48B...)
    uint8_t is_thread_local_;    // 是否属于 TLAB
    uint32_t alloc_bit_map_[];   // 位图，标记哪些 Slot 已被分配
    // Slot 数组紧随其后
  };

  // 按大小类别分成 42 个桶 (8B, 16B, 24B, 32B, ..., 2048B)
  // 每个桶维护一个 Run 的链表
  Run* current_runs_[kNumOfSizeBrackets];

  // TLAB (Thread-Local Allocation Buffer)
  // 每个线程持有自己的 Run，分配时无需加锁
  // Thread::rosalloc_runs_[] 存储每个线程的 TLAB
};
```

**TLAB（Thread-Local Allocation Buffer）的无锁分配原理：**

```
线程 A 分配一个 32 字节对象的过程：

1. 查找线程 A 的 TLAB 中大小类别为 32B 的 Run
2. 在 Run 的 alloc_bit_map_ 中找到第一个未分配的 Slot (bit 为 0)
3. 将 bit 置为 1，返回对应 Slot 的地址
4. 整个过程无锁！因为 TLAB 是线程私有的

当 TLAB 的 Run 用完时：
5. 从全局的 current_runs_[bracket_idx] 中获取一个新的 Run (需要加锁)
6. 将新 Run 设为当前线程的 TLAB
7. 这就是"慢速路径"——有锁竞争的开销
```

### 2.3 Region-based Allocator（CC GC 的选择）

Android 8.0 引入 CC GC 后，Allocation Space 改用 **Region Space**——将整个堆划分为固定大小的 Region（通常 256KB），每个 Region 可以处于"已分配"、"空闲"、"TLAB"等状态。

*   **源码路径：** `art/runtime/gc/space/region_space.h`

```cpp
// art/runtime/gc/space/region_space.h (简化)
class RegionSpace : public ContinuousMemMapAllocSpace {
  static constexpr size_t kRegionSize = 256 * KB;

  struct Region {
    enum class RegionState : uint8_t {
      kRegionStateFree,          // 空闲
      kRegionStateAllocated,     // 已分配（包含存活对象）
      kRegionStateLarge,         // 大对象（跨多个 Region）
      kRegionStateLargeTail,     // 大对象的后续 Region
    };

    RegionState state_;
    uint8_t* top_;               // 当前分配指针 (bump pointer)
    uint8_t* end_;               // Region 尾部
    Thread* tlab_owner_;         // 如果是 TLAB，属于哪个线程
  };

  Region* regions_;              // Region 数组
  size_t num_regions_;           // Region 总数
};
```

Region Space 中 TLAB 的分配更简单——直接 **bump pointer**（指针碰撞）：将分配指针向前移动对象大小，返回旧指针地址。这比 RosAlloc 的位图查找更快，是 CC GC 能降低分配延迟的原因之一。

### 2.4 稳定性：TLAB 快慢路径与碎片化

**快速路径（Fast Path）vs 慢速路径（Slow Path）的性能差异：**

| 路径 | 操作 | 耗时 | 是否加锁 |
| :--- | :--- | :--- | :--- |
| **快速路径** | TLAB 内分配（位图/指针碰撞） | < 100ns | 否 |
| **慢速路径 1** | TLAB 用完，从全局 Run/Region 池获取新 TLAB | ~ 1μs | 是（短锁） |
| **慢速路径 2** | 全局池也不足，触发 GC 后重试 | ~ 1-10ms | 是（GC 锁） |
| **慢速路径 3** | GC 后仍不足，扩展堆到 growth_limit_ | ~ 10-100μs | 是 |
| **最终失败** | 堆已达 growth_limit_ 且 GC 无法回收更多 | - | 抛出 OOM |

正常情况下，99% 以上的分配走快速路径。但**内存碎片化**可能导致一种诡异的"虚假 OOM"——堆的总空闲空间足够，但没有任何一块连续空闲区域能容纳请求的对象大小。

**碎片化导致的虚假 OOM 典型日志：**

```
java.lang.OutOfMemoryError: Failed to allocate a 1048592-byte allocation
with 15728640 free bytes and 14MB until OOM
```

注意：这里有 **15MB 空闲空间** 且距离 OOM 还有 14MB，但一个 1MB 的分配却失败了。这就是碎片化——空闲空间被打散成大量小块，无法拼凑出一块连续的 1MB。这种情况在使用 CMS GC（标记-清除，不压缩）的旧版本 Android 上更常见。CC GC 通过并发复制和压缩（Compacting）大幅缓解了碎片化问题。

## 3. OOM 的多重面孔

### 3.1 为什么 OOM 不只是"堆内存不够"

当开发者听到 OOM，第一反应往往是"Java 堆满了"。但在 Android 上，`java.lang.OutOfMemoryError` 被 ART 用作了**多种资源耗尽的统一异常**——至少有 **5 种完全不同的根因** 都会抛出同一个 `OutOfMemoryError`，但排查方向和治理方案截然不同。

*   **源码路径：** `art/runtime/gc/heap.cc`, `art/runtime/thread.cc`, `art/libartbase/base/mem_map.cc`

### 3.2 五种 OOM 的详细对比

#### 类型一：Java Heap OOM

**触发条件：** 分配对象时，Java 堆已达到 `growth_limit_`（或 largeHeap 的 `max_heap_size_`），且 GC 后仍无法回收足够空间。

**源码触发点：**

```cpp
// art/runtime/gc/heap.cc (简化)
mirror::Object* Heap::AllocObjectWithAllocator(Thread* self,
                                                ObjPtr<mirror::Class> klass,
                                                size_t byte_count,
                                                AllocatorType allocator,
                                                /* ... */) {
  mirror::Object* obj = TryToAllocate(self, allocator, byte_count, ...);
  if (UNLIKELY(obj == nullptr)) {
    // 快速分配失败 → 触发 GC 后重试
    obj = AllocateInternalWithGc(self, allocator, byte_count, ...);
    if (obj == nullptr) {
      // GC 后仍然失败 → 抛出 OOM
      ThrowOutOfMemoryError(self, byte_count, allocator_type);
      return nullptr;
    }
  }
  return obj;
}
```

**日志特征：**

```
java.lang.OutOfMemoryError: Failed to allocate a 4194320-byte allocation
with 524288 free bytes and 256MB until OOM,
target footprint 268435456, growth limit 268435456
```

关键字段解读：
*   `4194320-byte allocation`：试图分配的对象大小（约 4MB）
*   `524288 free bytes`：当前空闲字节（512KB，不足以分配 4MB）
*   `growth limit 268435456`：堆上限 256MB

**排查手段：** 用 Android Studio Profiler 或 LeakCanary 分析堆转储，找到内存泄漏的对象引用链。重点检查 `Activity`、`Fragment`、`Bitmap`、`WebView` 等大对象的生命周期管理。

---

#### 类型二：FD（File Descriptor）OOM

**触发条件：** 进程打开的文件描述符数量达到内核限制（通常 1024 或 32768）。当 ART 需要创建新线程（`pthread_create` 内部需要打开 `/proc/self/task/` 下的文件）或 `mmap` 新文件时，因 FD 耗尽而失败。

**日志特征：**

```
java.lang.OutOfMemoryError: Could not allocate JNI Env:
Failed to create thread: pthread_create (1040KB stack) failed: Try again
```

或者：

```
java.lang.OutOfMemoryError: Could not allocate JNI Env
```

**源码触发点：**

```cpp
// art/runtime/thread.cc (简化)
void Thread::CreateNativeThread(JNIEnv* env, jobject java_peer,
                                 size_t stack_size, bool is_daemon) {
  // 尝试创建 pthread
  int pthread_create_result = pthread_create(&new_pthread, &attr,
                                              Thread::CreateCallback, child_thread);
  if (pthread_create_result != 0) {
    // 创建失败 → 抛出 OOM (虽然本质不是堆内存问题)
    env->ThrowNew(WellKnownClasses::java_lang_OutOfMemoryError,
                  "Could not allocate JNI Env");
  }
}
```

**排查手段：**
1.  `ls -la /proc/<pid>/fd | wc -l` —— 查看当前 FD 数量
2.  `ls -la /proc/<pid>/fd` —— 查看每个 FD 指向的文件/socket
3.  常见 FD 泄漏源：未关闭的 `InputStream`/`OutputStream`、`Cursor`、`ParcelFileDescriptor`、`HandlerThread`（每个线程消耗 1-2 个 FD）

---

#### 类型三：Thread OOM

**触发条件：** 创建新线程时，无法为线程的 Native 栈分配虚拟内存（每个线程的栈通常需要 1MB 的连续虚拟地址空间）。在 32 位进程中，虚拟地址空间只有约 3GB，大量线程会迅速耗尽。

**日志特征：**

```
java.lang.OutOfMemoryError: pthread_create (1040KB stack) failed: Out of memory
```

**排查手段：**
1.  `ls /proc/<pid>/task | wc -l` —— 查看当前线程数
2.  `cat /proc/<pid>/status | grep Threads` —— 另一种查看方式
3.  正常 App 的线程数通常在 50-150 之间。如果超过 500，大概率存在线程泄漏——通常是 `new Thread()` 没有使用线程池、`HandlerThread` 未 quit、或 `AsyncTask` 被滥用。

---

#### 类型四：VSS（Virtual Memory）OOM

**触发条件：** 进程的虚拟内存空间（VSS, Virtual Set Size）耗尽。在 32 位进程中，用户空间虚拟地址只有约 3GB。每个 `mmap` 映射（加载 so 库、Dex 文件、OAT 文件）、每个线程栈、每个匿名内存分配都消耗虚拟地址空间。

**源码触发点：**

```cpp
// art/libartbase/base/mem_map.cc (简化)
MemMap MemMap::MapAnonymous(const char* name,
                             uint8_t* addr,
                             size_t byte_count,
                             int prot,
                             /* ... */) {
  void* actual = mmap(addr, byte_count, prot,
                      MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
  if (actual == MAP_FAILED) {
    // mmap 失败 → 虚拟内存不足
    *error_msg = StringPrintf("Failed anonymous mmap(%p, %zd, ...): %s",
                              addr, byte_count, strerror(errno));
    return Invalid();
  }
  return MemMap(name, ...);
}
```

**日志特征：**

```
java.lang.OutOfMemoryError: Failed to allocate a 16777232-byte allocation with ...
...
Cause: Failed anonymous mmap(0x0, 16777232, 0x3, ...): Out of memory
```

**排查手段：**
1.  `cat /proc/<pid>/maps | wc -l` —— 查看 VMA 数量
2.  `cat /proc/<pid>/status | grep VmPeak` —— 查看虚拟内存峰值
3.  重点排查大量 so 库加载（插件化框架）、大量 Dex 文件映射（热修复 Patch 累积）、WebView 内核的内存占用。
4.  **最有效的治理手段**：将 App 的 `targetSdkVersion` 升至 29+，使用 64 位 ABI。64 位进程的虚拟地址空间高达 256TB，VSS OOM 基本不再出现。

---

#### 类型五：VMA（Virtual Memory Area）OOM

**触发条件：** Linux 内核限制每个进程的 VMA 条目数量（`/proc/sys/vm/max_map_count`，Android 上通常为 65536）。每个 `mmap` 区域（包括文件映射、匿名映射）占用一个 VMA 条目。当 VMA 数量达到上限时，新的 `mmap` 调用失败。

**日志特征：**

```
java.lang.OutOfMemoryError:
Failed to allocate a 24-byte allocation with 67108864 free bytes and 64MB until OOM;
...
Cause: mmap R-- anon failed: Cannot allocate memory (errno: 12)
```

注意关键矛盾：**有 64MB 空闲内存**，但一个 **24 字节**的分配却失败了。这就是 VMA OOM 的典型特征——不是内存不够，是 VMA 条目数耗尽。

**排查手段：**
1.  `cat /proc/<pid>/maps | wc -l` —— 查看当前 VMA 数量，如果接近 65536 即为 VMA OOM
2.  `cat /proc/<pid>/maps | awk '{print $NF}' | sort | uniq -c | sort -rn | head -20` —— 找出 VMA 占用最多的文件
3.  常见 VMA 泄漏源：大量小文件的 `mmap`、`Parcel` 的 `ashmem` 映射未释放、GPU 驱动的内存分配

### 3.3 五种 OOM 的快速鉴别表

| OOM 类型 | 日志关键字 | 本质原因 | 排查工具 | 治理方向 |
| :--- | :--- | :--- | :--- | :--- |
| **Java Heap** | `Failed to allocate ... growth limit` | 堆内存耗尽 | MAT / LeakCanary | 修复内存泄漏、减少大对象 |
| **FD** | `Could not allocate JNI Env` | FD 数量超限 | `ls /proc/<pid>/fd` | 关闭未释放的流和连接 |
| **Thread** | `pthread_create ... failed` | 线程数过多或虚拟内存不足 | `ls /proc/<pid>/task` | 使用线程池、控制线程数 |
| **VSS** | `Failed anonymous mmap` | 虚拟地址空间耗尽（32 位） | `VmPeak in /proc/<pid>/status` | 迁移到 64 位 |
| **VMA** | 大量空闲内存 + 小对象分配失败 | VMA 条目数耗尽 | `wc -l /proc/<pid>/maps` | 减少 mmap 区域数量 |

## 4. GC 算法演进

### 4.1 为什么 GC 算法一直在变

GC（Garbage Collection）的核心矛盾是：**回收效率 vs 暂停时间**。更彻底的回收（如全堆标记-压缩）能减少碎片、提高空间利用率，但需要更长时间暂停所有 Java 线程（Stop-The-World, STW）。而任何超过 16ms 的 STW 都会导致 UI 掉帧。

Android 的 GC 算法经历了三代演进：

| 版本 | GC 算法 | STW 时间 | 核心思想 |
| :--- | :--- | :--- | :--- |
| Android 5.0-7.0 | **CMS**（Concurrent Mark-Sweep） | 5-50ms | 并发标记、STW 清除 |
| Android 8.0-9.0 | **CC**（Concurrent Copying） | < 1ms | 并发复制、读屏障 |
| Android 10.0+ | **Generational CC** | < 0.5ms | 分代 + 并发复制 |

### 4.2 CMS：并发标记-清除

CMS 是 ART 最早使用的并发 GC 策略（Android 5.0-7.0 默认），分为四个阶段：

```
阶段 1: Initial Mark (STW)
   - 暂停所有线程
   - 只标记 GC Roots 直接引用的对象 (很快, <1ms)
   
阶段 2: Concurrent Mark
   - 恢复所有线程
   - GC 线程与 App 线程并发运行，遍历标记所有可达对象
   
阶段 3: Remark (STW)
   - 再次暂停所有线程
   - 修正并发标记期间被 App 修改的引用关系 (通常 2-10ms)
   
阶段 4: Concurrent Sweep
   - 恢复所有线程
   - GC 线程并发清除未被标记的死亡对象
```

**CMS 的缺陷：**
1.  **内存碎片化：** 清除（Sweep）只是标记空闲区域，不移动存活对象，堆越用越碎。
2.  **STW 不可控：** Remark 阶段的耗时取决于并发标记期间的对象修改量。如果 App 在标记期间大量创建/修改对象（如 RecyclerView 快速滑动），Remark 可能飙到 50ms+。

### 4.3 CC GC：并发复制

CC GC（Concurrent Copying）是 Android 8.0 引入的革命性 GC 算法。它的核心思想：**不是"标记死对象并清除"，而是"复制活对象到新区域"——旧区域直接整体回收**。

*   **源码路径：** `art/runtime/gc/collector/concurrent_copying.cc`

```cpp
// art/runtime/gc/collector/concurrent_copying.cc (概念性简化)
void ConcurrentCopying::RunPhases() {
  // 阶段 1: InitializePhase (极短 STW, < 0.1ms)
  //   翻转 Region 空间的 from-space/to-space 语义
  //   安装读屏障 (Read Barrier)
  FlipThreadRoots();

  // 阶段 2: CopyingPhase (并发，无 STW)
  //   GC 线程扫描所有 GC Roots
  //   遇到 from-space 中的对象时，将其复制到 to-space
  //   更新所有引用指向新地址
  ProcessMarkStack();

  // 阶段 3: ReclaimPhase (极短 STW, < 0.1ms)
  //   确认所有 from-space 对象已被复制
  //   回收 from-space 的所有 Region
  ReclaimRegions();
}
```

**读屏障（Read Barrier）—— CC GC 的核心机制：**

在并发复制过程中，App 线程和 GC 线程同时运行。App 可能在 GC 复制对象的同时访问该对象。**读屏障**确保 App 线程总是访问到对象的**最新副本**。

*   **源码路径：** `art/runtime/read_barrier.h`

```cpp
// art/runtime/read_barrier.h (概念性简化)
template <typename MirrorType>
inline MirrorType* ReadBarrier::Barrier(MirrorType* obj) {
  if (UNLIKELY(obj != nullptr && obj->GetReadBarrierState() == gray_state_)) {
    // 对象处于"灰色"状态 —— 可能正在被 GC 移动
    // 检查 forwarding pointer：如果对象已被复制到 to-space，
    // 返回 to-space 中的新地址
    return Mark(obj);
  }
  return obj;
}
```

编译器在**每一次对象引用的读取操作**之前插入读屏障代码。例如 `a.b` 这个字段访问，在机器码层面变成：

```
// 伪代码
obj_a = ReadBarrier(a);  // 确保 a 指向最新副本
result = obj_a.b;        // 然后再读取字段
```

### 4.4 Generational CC：分代并发复制

Android 10.0 引入的 Generational CC 基于**分代假说（Generational Hypothesis）**——大多数对象"朝生夕灭"，只有少数对象长期存活。

将堆分为 **Young Generation（年轻代）** 和 **Old Generation（老年代）**：
*   新分配的对象进入年轻代
*   经过多次 GC 存活的对象被提升（Promote）到老年代
*   **Minor GC** 只扫描年轻代（对象少，极快）
*   **Major GC** 扫描全堆（较慢，但触发频率低）

效果：在正常场景下，90% 的 GC 都是 Minor GC，耗时不到 0.5ms，用户完全无感知。

### 4.5 稳定性：CC GC 的读屏障如何影响 Hook 框架

CC GC 的读屏障对 **Java Method Hook 框架（SandHook、Epic、Pine）** 产生了深远影响：

**问题一：Hook 代码中的对象引用可能失效**

Hook 框架将方法的 `entry_point_` 替换为自定义代码。如果自定义代码中持有对 Java 对象的 Native 指针（而非通过 JNI Handle），CC GC 在移动对象后，这些裸指针指向旧地址，下次访问触发 `SIGSEGV`。

**问题二：读屏障开销对 Hook 性能的影响**

每次通过 Hook 代码访问 Java 对象时，读屏障都会被触发。如果 Hook 的方法是高频调用的（如 `View.draw()`），读屏障的额外开销可能导致可感知的性能退化。

**问题三：ArtMethod 的 `entry_point_` 在 GC 期间可能被更新**

CC GC 在移动 `ArtMethod` 关联的对象（如 `DexCache`）时，可能需要更新 `ArtMethod` 中的某些指针。如果 Hook 框架此时正在修改同一个 `ArtMethod`，形成竞态条件，可能导致 `ArtMethod` 的状态不一致。

**治理建议：**
*   使用经过充分验证的 Hook 框架版本，确保其已适配 CC GC 的读屏障
*   Hook 代码中的 Java 对象引用必须通过 JNI Handle（`jobject`）而非裸指针
*   避免 Hook 高频调用的系统方法（如 `View.draw`、`Canvas.drawBitmap`）

## 5. GC 触发时机与日志

### 5.1 为什么需要理解 GC 触发时机

GC 的触发时机直接决定了 App 的**流畅度感知**。同样是一次 2ms 的 GC，如果发生在用户快速滑动列表时（16ms 渲染窗口内），可能导致掉帧；如果发生在 App 空闲时，用户完全无感。理解 GC 的触发时机，是诊断"内存抖动"和"卡顿与 GC 关联性"的基础。

*   **源码路径：** `art/runtime/gc/heap.cc`

### 5.2 三种 GC 触发原因

```cpp
// art/runtime/gc/heap.cc (简化)
enum GcCause {
  kGcCauseForAlloc,      // 分配失败触发 (最紧急)
  kGcCauseBackground,    // 后台并发触发 (阈值触发)
  kGcCauseExplicit,      // 显式调用 System.gc()
  kGcCauseForNativeAlloc,// Native 内存压力触发
  kGcCauseCollectorTransition, // GC 收集器切换
};
```

#### GC_FOR_ALLOC（分配失败触发）

**触发条件：** 当 `TryToAllocate` 在快速路径和慢速路径上都无法找到足够空间时，在抛出 OOM 之前，先触发一次阻塞式 GC，然后重试分配。

```cpp
// art/runtime/gc/heap.cc (概念性简化)
mirror::Object* Heap::AllocateInternalWithGc(Thread* self, ...) {
  // 触发一次同步 GC (会等待 GC 完成)
  CollectGarbageInternal(collector_type_, kGcCauseForAlloc, false);

  // GC 完成后重试分配
  mirror::Object* obj = TryToAllocate(self, allocator, byte_count, ...);
  if (obj != nullptr) {
    return obj;
  }

  // 仍然失败 → 尝试更激进的 GC (清除软引用)
  CollectGarbageInternal(collector_type_, kGcCauseForAlloc, true);
  obj = TryToAllocate(self, allocator, byte_count, ...);

  return obj;  // 如果仍为 nullptr，调用者抛 OOM
}
```

**稳定性含义：** `GC_FOR_ALLOC` 是**同步阻塞的**——触发它的线程会等待 GC 完成。如果这是主线程，GC 的耗时直接加到帧渲染时间上。**频繁出现 `GC_FOR_ALLOC` 是"内存抖动"的铁证**——意味着 App 在高频分配对象、又几乎不释放，堆持续在上限附近挣扎。

#### GC_CONCURRENT（阈值触发）

**触发条件：** 当堆已用空间超过 `concurrent_start_bytes_`（通常为 `max_allowed_footprint_` 的一个百分比，如 75%）时，ART 的 `HeapTaskDaemon` 触发一次后台并发 GC。

```cpp
// art/runtime/gc/heap.cc (概念性简化)
void Heap::RequestConcurrentGCAndSaveObject(Thread* self, ...) {
  if (GetBytesAllocated() >= concurrent_start_bytes_) {
    // 堆使用率达到阈值 → 触发后台并发 GC
    task_processor_->AddTask(self, new ConcurrentGCTask(target_time));
  }
}
```

**稳定性含义：** 这是最"正常"的 GC 触发方式——在后台并发执行，对 App 性能影响最小。健康的 App 应该以 `GC_CONCURRENT` 为主要 GC 触发方式。

#### GC_EXPLICIT（手动触发）

**触发条件：** 代码中显式调用 `System.gc()` 或 `Runtime.getRuntime().gc()`。

**稳定性含义：** ART 默认情况下会忽略 `System.gc()` 请求（通过 `-XX:+DisableExplicitGC`）。但某些 OEM ROM 或系统服务可能开启了 explicit GC。如果你在日志中看到频繁的 `GC_EXPLICIT`，检查是否有代码在循环中调用 `System.gc()`——这是一种**反模式**，不仅无法解决内存问题，反而会增加 GC 开销。

### 5.3 GC 日志格式解读

ART 的 GC 日志是排查内存问题的最重要线索之一。以下是一条典型的 CC GC 日志：

```
I/art: Explicit concurrent copying GC freed 28423(1568KB) AllocSpace objects,
12(512KB) LOS objects, 49% free, 5MB/10MB, paused 0.087ms total 12.542ms
```

各字段含义：

| 字段 | 含义 | 示例值 |
| :--- | :--- | :--- |
| `Explicit` | GC 触发原因 | Explicit / Alloc / Background |
| `concurrent copying` | GC 算法 | concurrent copying / mark sweep |
| `freed 28423(1568KB)` | 回收了多少对象和空间 | 28423 个对象，共 1568KB |
| `12(512KB) LOS objects` | Large Object Space 回收 | 12 个大对象，共 512KB |
| `49% free` | GC 后的堆空闲百分比 | 49% |
| `5MB/10MB` | 已用/总堆大小 | 已用 5MB，总堆 10MB |
| `paused 0.087ms` | STW 暂停时间 | 0.087ms |
| `total 12.542ms` | GC 总耗时 | 12.542ms |

### 5.4 稳定性：GC 日志中的危险信号

| 日志特征 | 含义 | 严重程度 | 处理方案 |
| :--- | :--- | :--- | :--- |
| 连续多条 `Alloc` 触发的 GC | 内存抖动 | **高** | 减少对象分配频率，使用对象池 |
| `paused` > 5ms | GC 暂停时间过长 | **中** | 检查是否仍在用 CMS GC，升级到 CC GC |
| `0% free` 后仍有 `Alloc` GC | 堆已满，即将 OOM | **极高** | 紧急排查内存泄漏 |
| GC 频率 > 10 次/秒 | 对象分配速率远超回收速率 | **高** | 检查热循环中的对象创建（如 `String +`） |
| `freed 0(0B)` | GC 什么也没回收 | **极高** | 所有对象都是存活的，严重泄漏 |

## 6. Finalizer 机制

### 6.1 为什么 finalize() 是稳定性毒药

Java 的 `Object.finalize()` 方法允许对象在被 GC 回收前执行清理逻辑。听起来很美好，但实际上它是 Android 稳定性的**头号毒药**之一。原因有三：

1.  **不确定性：** `finalize()` 的执行时机完全不可预测——可能在 GC 后立即执行，也可能延迟数分钟。
2.  **单线程瓶颈：** 所有对象的 `finalize()` 方法由同一个 `FinalizerDaemon` 线程串行执行。一个慢的 `finalize()` 会阻塞后面所有对象的回收。
3.  **超时即杀：** `FinalizerWatchdogDaemon` 监控 `FinalizerDaemon` 的执行，如果单个 `finalize()` 超过 **10 秒** 未完成，直接 abort 进程。

### 6.2 FinalizerDaemon 的工作原理

*   **源码路径：** `libcore/libart/src/main/java/java/lang/Daemons.java`

```java
// libcore/libart/src/main/java/java/lang/Daemons.java (简化)
private static class FinalizerDaemon extends Daemon {
    // 待执行 finalize() 的对象队列
    private final ReferenceQueue<Object> queue = FinalizerReference.queue;

    @Override public void runInternal() {
        while (isRunning()) {
            // 从队列中取出下一个需要 finalize 的对象 (阻塞等待)
            FinalizerReference<?> finalizingReference = (FinalizerReference<?>) queue.poll();
            if (finalizingReference != null) {
                // 标记开始时间 (WatchdogDaemon 据此判断超时)
                finalizingObject = finalizingReference.get();

                // 执行 finalize()
                doFinalize(finalizingReference);

                // 清除标记
                finalizingObject = null;
            }
        }
    }

    private void doFinalize(FinalizerReference<?> reference) {
        Object object = reference.get();
        reference.clear();
        try {
            object.finalize();  // 调用对象的 finalize() 方法
        } catch (Throwable ex) {
            // finalize() 中的异常被静默吞掉！
        }
    }
}
```

### 6.3 FinalizerWatchdogDaemon 的 10 秒超时

*   **源码路径：** `libcore/libart/src/main/java/java/lang/Daemons.java`

```java
// libcore/libart/src/main/java/java/lang/Daemons.java (简化)
private static class FinalizerWatchdogDaemon extends Daemon {
    // 超时时间：10 秒
    private static final long FINALIZER_TIMEOUT_MILLIS = 10 * 1000;

    @Override public void runInternal() {
        while (isRunning()) {
            // 等待 FinalizerDaemon 开始执行某个 finalize()
            waitForFinalizationStarted();

            // 等待超时
            boolean finalized = waitForFinalizationCompleted(FINALIZER_TIMEOUT_MILLIS);

            if (!finalized) {
                // 超时！FinalizerDaemon 卡死了
                // 获取 FinalizerDaemon 的当前堆栈
                Object finalizing = FinalizerDaemon.INSTANCE.finalizingObject;
                Thread finalizerThread = FinalizerDaemon.INSTANCE.getThread();
                StackTraceElement[] stack = finalizerThread.getStackTrace();

                // 构造 TimeoutException 并 abort 进程
                TimeoutException ex = new TimeoutException(
                    finalizing.getClass().getName() +
                    ".finalize() timed out after " +
                    FINALIZER_TIMEOUT_MILLIS / 1000 + " seconds");
                ex.setStackTrace(stack);

                // 直接终止进程
                Thread.UncaughtExceptionHandler h =
                    Thread.getDefaultUncaughtExceptionHandler();
                h.uncaughtException(Thread.currentThread(), ex);
            }
        }
    }
}
```

### 6.4 Finalizer 超时的完整触发链

```
1. GC 标记阶段发现对象 A 不可达且重写了 finalize()
      ↓
2. GC 不直接回收 A，而是将 A 加入 FinalizerReference 队列
   （A 在这次 GC 中"幸存"了，占用的内存不会被释放）
      ↓
3. FinalizerDaemon 线程从队列中取出 A
      ↓
4. FinalizerDaemon 调用 A.finalize()
   ──── FinalizerWatchdogDaemon 开始计时 ────
      ↓
5a. A.finalize() 在 10 秒内完成
    → 计时器重置，处理下一个对象
    → A 在下次 GC 时被真正回收

5b. A.finalize() 超过 10 秒未完成（如等锁、I/O 阻塞）
    → FinalizerWatchdogDaemon 判定超时
    → 抛出 TimeoutException，abort 进程
```

### 6.5 稳定性：TimeoutException 的根因与治理

**典型的 TimeoutException 日志：**

```
FATAL EXCEPTION: FinalizerWatchdogDaemon
java.util.concurrent.TimeoutException:
android.content.res.Resources$Theme.finalize() timed out after 10 seconds
    at android.content.res.Resources$Theme.finalize(Resources.java:1233)
    at java.lang.Daemons$FinalizerDaemon.doFinalize(Daemons.java:250)
```

**常见根因分析：**

| finalize() 所在类 | 卡死原因 | 发生频率 |
| :--- | :--- | :--- |
| `Resources$Theme` | 调用 Native 释放方法时等 Binder 锁 | 高 |
| `Bitmap` | `nativeRecycle()` 等待 GPU 释放纹理 | 中 |
| `FileInputStream` / `FileOutputStream` | `close()` 触发磁盘 flush，I/O 延迟 | 中 |
| `Cursor` / `SQLiteDatabase` | `close()` 需要获取数据库写锁 | 中 |
| `WebView` | Native 资源释放涉及复杂的跨线程同步 | 高 |

**为什么 finalize() 容易超时：**

1.  **系统整体负载高：** `FinalizerDaemon` 是低优先级线程（`THREAD_PRIORITY_FOREGROUND` - 2），当 CPU 密集时可能长时间得不到调度。
2.  **锁竞争：** `finalize()` 中调用的 Native 方法可能需要获取全局锁（如 Binder 锁、GPU 锁），在高负载下等锁时间超过 10 秒。
3.  **队列积压：** 如果大量对象同时被 GC 标记为可回收（如 Activity 销毁释放大量 View），队列积压导致前面的 `finalize()` 正常完成、但后面的在等待过程中已经超时。

**治理方案：**

1.  **最根本的治理：不要使用 finalize()。** 使用 `try-with-resources` 或 `Closeable` 接口显式释放资源。
2.  **对于无法修改的系统类（如 `Resources$Theme`）：**
    *   通过 `PhantomReference` + `ReferenceQueue` 替代 `finalize()`——在 ReferenceQueue 的消费线程中执行清理逻辑，可以自主控制超时和线程优先级。
    *   在 `Activity.onDestroy()` 中主动调用 `theme.close()`（如果有 close 方法），避免依赖 `finalize()`。
3.  **监控 FinalizerDaemon 队列深度：** 通过反射获取 `FinalizerReference.queue` 的大小，当超过阈值（如 1000）时触发告警，提前发现泄漏。
4.  **对于低端设备：** 部分 OEM 将 Finalizer 超时时间从 10 秒调整到 20 秒或 60 秒，以降低 TimeoutException 的误杀率，但这只是治标不治本。

## 稳定性实战案例

### 案例 1：碎片化导致"虚假 OOM"—— 堆里还有大量空间却分配失败

**现象：** 某电商 App 在 Android 7.0 的低端机上 OOM 率突然升高。OOM 日志显示：

```
java.lang.OutOfMemoryError: Failed to allocate a 2097168-byte allocation
with 41943040 free bytes and 38MB until OOM,
target footprint 268435456, growth limit 268435456
```

App 试图分配一个 2MB 的 `byte[]`，**堆中有 40MB 空闲空间**，远远大于 2MB，但分配还是失败了。

**分析过程：**

1.  首先排除了 Non-Moving Space / VMA / FD 等其他类型的 OOM——日志明确指向 Allocation Space。
2.  Android 7.0 使用 **CMS GC**（标记-清除，不压缩）。`byte[2MB]` 需要一块 **2MB 连续** 的空闲区域。
3.  通过 `adb shell dumpsys meminfo <pid>` 分析堆内存分布，发现 Large Object Space 碎片严重——此前频繁分配和释放 Bitmap 对应的 `byte[]`（1-4MB 不等），在 Large Object Space 中留下了大量空洞。
4.  40MB 的空闲空间被碎成了上百块 0.1-1.5MB 的小区域，没有一块能容纳 2MB。

**根因：** CMS GC 的标记-清除不做内存压缩，Long-running 的 App 中频繁分配/释放不等大小的大对象导致 Large Object Space 严重碎片化。

**修复方案：**

1.  **短期：** 对 Bitmap 使用 `inBitmap` 选项复用已有的 `byte[]` 缓冲区，减少大对象的分配/释放频率。
2.  **中期：** 将 targetSdkVersion 升至 26+，利用 Android 8.0 的 CC GC（并发复制 + 压缩），从根本上消除碎片化。
3.  **监控：** 添加内存碎片化指标——在分配失败的 `catch (OutOfMemoryError)` 中，记录当前堆的空闲总量和最大连续空闲块大小的比值。如果空闲总量充裕但最大连续空闲块远小于请求大小，判定为碎片化 OOM，上报到 APM 系统做专项治理。

---

### 案例 2：FinalizerDaemon 阻塞导致 Bitmap 内存无法释放 → Java Heap OOM

**现象：** 某短视频 App 在快速滑动信息流时，约 3-5 分钟后出现 Java Heap OOM。MAT 分析堆转储发现大量 `android.graphics.Bitmap` 对象（总计 180MB），但每个 Bitmap 的 `mRecycled` 字段均为 `false`（未被回收）。

**分析过程：**

1.  第一感觉是 Bitmap 泄漏——但检查代码发现 ImageView 的引用链路正常，Activity/Fragment 的生命周期管理也正确。
2.  深入分析发现，这些 Bitmap 已经**不被任何 GC Root 引用**了（即 GC 已标记为"不可达"），但仍然占用内存。
3.  关键发现：`Bitmap` 类重写了 `finalize()` 方法（调用 `nativeRecycle()` 释放 Native 内存）。虽然 GC 已标记 Bitmap 为不可达，但 Bitmap 需要经过 **FinalizerDaemon** 执行 `finalize()` 后，才能在下一次 GC 中被真正回收。
4.  检查 `FinalizerDaemon` 线程状态，发现它卡在一个 `SQLiteDatabase.finalize()` 上——某个被泄漏的 `SQLiteDatabase` 实例在 `finalize()` 中尝试关闭数据库，但需要获取数据库写锁。该写锁被一个长事务持有（一个后台线程正在执行大批量数据写入，耗时 15-20 秒）。
5.  `FinalizerDaemon` 被阻塞在 `SQLiteDatabase.finalize()` 上等锁，导致后面排队的几百个 `Bitmap.finalize()` 全部等待。这些 Bitmap 虽然"不可达"但无法被真正回收，持续占用 180MB 堆内存，最终触发 OOM。
6.  之所以没有触发 `TimeoutException`，是因为 `SQLiteDatabase.finalize()` 中的 `close()` 方法在等锁 9.8 秒后恰好获得了锁，刚好没超过 10 秒超时阈值——但这 9.8 秒的阻塞已经导致了大量 Bitmap 积压。

**根因：** `FinalizerDaemon` 是单线程串行执行所有 `finalize()` 的。一个慢的 `finalize()`（如 `SQLiteDatabase.close()` 等锁）会**阻塞整个 Finalizer 队列**，导致所有依赖 `finalize()` 释放资源的对象（如 Bitmap）都无法被及时回收，形成**级联内存积压**。

**修复方案：**

1.  **修复 SQLiteDatabase 泄漏：** 确保所有 `SQLiteDatabase` 实例通过 `try-with-resources` 显式关闭，不依赖 `finalize()`。
2.  **优化数据库长事务：** 将大批量写入拆分为多个小事务（每次写入 100-500 条），降低写锁持有时间。
3.  **Bitmap 内存管理：** 在 `ImageView.onDetachedFromWindow()` 中主动调用 `bitmap.recycle()` 释放 Native 内存，而非依赖 `finalize()`。
4.  **添加 FinalizerDaemon 监控：**
    *   定期检查 `FinalizerDaemon` 线程的堆栈，如果发现其长时间停留在同一个 `finalize()` 方法上，上报告警。
    *   监控 `FinalizerReference.queue` 的深度（通过反射），当积压超过 500 个对象时触发 `System.gc()` + 告警。

**教训：** `finalize()` 是一个**全局共享的单线程瓶颈**。任何一个"重"的 `finalize()` 都会产生级联效应，阻塞其他无关对象的回收。这就是为什么我们说"`finalize()` 是稳定性毒药"——它的危害不仅限于自身，而是会扩散到整个进程。

## 7. 总结

本篇从 Heap 空间划分出发，自底向上地拆解了 ART 内存与 GC 子系统的核心机制：

**核心要点：**

1.  **五大 Space 各司其职：** Image Space（只读共享）、Zygote Space（COW）、Allocation Space（App 对象）、Large Object Space（大对象）、Non-Moving Space（Class 元数据）。理解它们的边界是分析 OOM 的前提。
2.  **TLAB 是内存分配性能的关键：** 快速路径（无锁 TLAB 分配）与慢速路径（加锁全局分配/GC）的性能差异可达 100 倍。碎片化可能导致"虚假 OOM"。
3.  **OOM 不只是"堆满了"：** Java Heap OOM、FD OOM、Thread OOM、VSS OOM、VMA OOM 五种类型的根因、日志特征和排查手段截然不同。3 秒内判断 OOM 类型是稳定性架构师的基本功。
4.  **GC 从 CMS → CC → Generational CC 的演进**是 STW 从 50ms 降到 0.5ms 的关键。CC GC 的读屏障是理解现代 ART 内存行为的核心概念。
5.  **GC 触发原因的鉴别：** `GC_FOR_ALLOC` 频繁出现 = 内存抖动铁证；`GC_CONCURRENT` 是健康的后台回收；`GC_EXPLICIT` 通常是反模式。
6.  **finalize() 是单线程串行的全局瓶颈：** 一个慢的 `finalize()` 会阻塞整个 Finalizer 队列，导致无关对象也无法回收。`FinalizerWatchdogDaemon` 的 10 秒超时机制是 `TimeoutException` 的来源。**永远不要使用 finalize()。**
