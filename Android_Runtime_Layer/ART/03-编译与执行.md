# 03-编译与执行：从字节码到机器码的三条路

## 1. 三种执行模式的全景

ART 并不是只有一种方式来执行你的代码。它同时维护着**三种执行模式**——解释器（Interpreter）、JIT（Just-In-Time）、AOT（Ahead-Of-Time），并在运行时根据方法的"热度"和编译状态动态切换。这三条路径的存在，是 Android 在启动速度、运行性能和存储空间之间反复博弈的产物。

### 1.1 三种模式的工作场景

| 模式 | 何时工作 | 性能特征 | 典型场景 |
| :--- | :--- | :--- | :--- |
| **解释器** | 方法首次执行、无 OAT 文件、debug 模式 | 最慢（逐条翻译指令） | 新安装 App 的首次冷启动 |
| **JIT** | 方法被反复调用达到热度阈值后 | 中等（编译后接近 AOT，但编译本身消耗 CPU） | App 运行中的热点代码 |
| **AOT** | 设备空闲时后台 `dex2oat` 编译，或安装时根据 Profile 编译 | 最快（直接执行机器码，零运行时编译开销） | 后台编译完成后的再次启动 |

### 1.2 模式切换的时序

一个方法在其生命周期中，可能经历如下状态变迁：

```
首次安装 → 解释执行 → 热度达标 → JIT 编译 → 机器码执行
                                    ↓
                        Profile 记录热点方法
                                    ↓
                    设备空闲充电 → dex2oat AOT 编译
                                    ↓
                    再次启动 → 直接加载 OAT 机器码执行
```

### 1.3 为什么同一个 Bug 在不同条件下表现不同

这是稳定性架构师必须理解的关键：**同一段 Java 代码，在解释模式、JIT 模式和 AOT 模式下的执行路径完全不同。**

*   **解释模式**下，代码逐条经过 `ExecuteSwitchImpl` 的 switch-case 分发。每条指令前都有安全点检查（Safepoint Poll），GC 可以随时暂停线程。
*   **JIT 模式**下，热点方法被编译为机器码，安全点检查被嵌入到编译后的代码中（如循环回边处）。某些解释器中的隐式检查可能被优化掉。
*   **AOT 模式**下，代码在安装时就已编译完毕，编译器可能做了更激进的优化（内联、常量折叠、死代码消除），执行路径与前两种截然不同。

这意味着：
1.  一个只在 AOT 模式下触发的编译器 Bug，在新安装（纯解释模式）时不会复现。
2.  一个 JIT 编译后才出现的竞态条件（Race Condition），在解释模式下由于执行速度较慢，时序窗口不同，可能永远不触发。
3.  同一个 App 在不同用户的手机上，由于 Profile 积累程度不同、dex2oat 是否完成不同，实际执行模式可能完全不同——这就是为什么"有人能复现、有人不能复现"。

## 2. 解释器深入

### 2.1 为什么还需要解释器

既然 JIT 和 AOT 都能生成更快的机器码，为什么 ART 还要保留解释器？因为解释器是**兜底方案**：

1.  **冷启动的第一次执行：** App 刚安装后还没有任何编译产物，必须用解释器执行。
2.  **Debug 模式：** 断点调试时需要逐条执行指令，只有解释器能做到。
3.  **Deoptimization（反优化）回退：** 当 JIT/AOT 编译的代码因为假设被打破（如类层次变化导致内联失效）时，需要回退到解释器重新执行。
4.  **安全点保证：** 解释器在每条指令前都检查安全点，是 GC 能暂停线程的最可靠保证。

### 2.2 两种解释器实现

ART 提供了两种解释器实现，各有侧重：

*   **源码路径：** `art/runtime/interpreter/interpreter.cc`

**Switch Interpreter（C++ 实现）：**

上一篇已经详细分析过的 `ExecuteSwitchImpl`，本质是一个巨大的 `while(true) { switch(opcode) { ... } }` 循环。优点是可读性好、易于维护和调试；缺点是 switch-case 分发在现代 CPU 上分支预测命中率极低，性能较差。

**Mterp（汇编优化解释器）：**

为了解决 Switch Interpreter 的性能问题，ART 针对每个目标架构（ARM、ARM64、x86、x86_64）手写了一套汇编解释器——Mterp（名字源自 Dalvik 时代的 "Machine Interpreter"）。

Mterp 的核心优化思想是**计算跳转（Computed Goto）**：不使用 switch-case，而是维护一张**跳转表（Handler Table）**，每个操作码对应一个固定大小的代码块（Handler）。取到操作码后，直接计算出 Handler 的地址并跳转，避免了 switch-case 的多层分支判断。

```
; 伪代码 (ARM64 风格)
; 取下一条指令的操作码
ldrb  w0, [x_pc]            ; 读取 PC 处的操作码 (1 字节)
; 计算跳转：handler_base + opcode * handler_size
add   x1, x_handler_table, x0, lsl #7   ; 每个 handler 128 字节
br    x1                     ; 直接跳转到对应的 handler
```

这种方式对 CPU 的间接分支预测器（Indirect Branch Predictor）更友好，性能比 Switch Interpreter 提升约 **30%-50%**。

### 2.3 ShadowFrame 栈帧结构

无论使用哪种解释器，每个方法调用都对应一个 `ShadowFrame`（影子栈帧），它是解释器的"工作台"。

*   **源码路径：** `art/runtime/stack.h`

```cpp
// art/runtime/stack.h (简化)
class ShadowFrame {
  ArtMethod* method_;          // 当前执行的方法
  uint32_t dex_pc_;            // 当前程序计数器（指向 Dex 字节码偏移）
  ShadowFrame* link_;          // 链向上一个栈帧（形成调用链）
  uint32_t number_of_vregs_;   // 虚拟寄存器数量
  uint32_t vregs_[0];          // 虚拟寄存器数组（变长，紧跟结构体后）
  // ... references 位图（标记哪些寄存器是对象引用）
};
```

每次方法调用（`invoke-*` 指令），解释器都会在**当前线程的栈上**分配一个新的 `ShadowFrame`，通过 `link_` 指针串成链表。方法返回时，弹出当前帧，恢复上一帧的执行位置。

### 2.4 稳定性：StackOverflowError 的两种来源

`StackOverflowError` 在 ART 中有**两种截然不同的触发机制**，排查时必须区分：

**来源一：Java 栈溢出（ShadowFrame 链表过深）**

ART 在每次方法调用时会检查 `ShadowFrame` 的分配是否超过了线程的 Java 栈大小限制（默认约 1MB，可通过 `Thread` 构造函数设置）。如果递归过深，`ShadowFrame` 的累计大小超出限制，ART 主动抛出 `StackOverflowError`。

```cpp
// art/runtime/interpreter/interpreter.cc (概念性)
// 进入方法前检查栈空间
if (UNLIKELY(reinterpret_cast<uint8_t*>(shadow_frame) <=
             self->GetStackEnd())) {
  self->SetException(StackOverflowError);
  return;
}
```

**来源二：Native 栈溢出（C++ 调用栈触碰 Guard Page）**

每个 Java 线程底层对应一个 `pthread`，拥有一个固定大小的 Native 栈（通常 1MB）。当深度递归导致 C++ 的 `ExecuteSwitchImpl` 本身递归调用过深，Native 栈指针触碰到栈底的 Guard Page（一段标记为不可读写的内存页），CPU 触发 `SIGSEGV`，ART 的 `FaultHandler` 捕获后判定为栈溢出，转化为 Java 层的 `StackOverflowError`。

**区分方法：** 如果 Crash 堆栈中能看到完整的 Java 调用链且最后一帧是 `StackOverflowError`，多数是来源一。如果堆栈中出现 `signal 11 (SIGSEGV), code 2 (SEGV_ACCERR)` 且地址接近栈底，是来源二。来源二更危险——如果 `FaultHandler` 自身的处理逻辑也消耗栈空间（栈上的空间已经极度紧张），可能导致二次 `SIGSEGV`，变成无法恢复的 Native Crash。

## 3. JIT 编译

### 3.1 为什么需要 JIT

解释器太慢——每条指令都要经过"取指→解码→分发→执行"四步，性能只有原生机器码的 **1/10 到 1/50**。JIT 的职责是识别出被频繁执行的"热点方法"，将其实时编译为机器码，替换掉解释执行的入口点，让后续调用直接走机器码。

JIT 的核心挑战在于**编译的收益必须大于编译本身的开销**。如果一个方法只被调用一次，花费几毫秒去编译它反而得不偿失。因此 JIT 需要一个**热度检测机制**来决定"什么时候值得编译"。

### 3.2 热度计数器与阈值触发

*   **源码路径：** `art/runtime/jit/jit.cc`

ART 的 JIT 使用**方法调用计数器**和**回边计数器（Back-edge Counter，循环体的回跳次数）**来衡量方法热度。

```cpp
// art/runtime/jit/jit.cc (简化)
void Jit::AddSamples(Thread* self, ArtMethod* method,
                     uint16_t samples, bool with_backedges) {
  // 递增该方法的热度计数
  uint16_t new_warm_count = method->IncrementCounter(samples);

  // 检查是否达到编译阈值
  if (new_warm_count >= warm_method_threshold_) {
    // 达到"温热"阈值 → 开始收集 Profile 信息
    MaybeEnqueueCompilation(method, self);
  }
  if (with_backedges && new_warm_count >= hot_method_threshold_) {
    // 达到"热点"阈值 → 提交编译任务
    MaybeEnqueueCompilation(method, self);
  }
}
```

默认阈值配置（可通过系统属性调整）：
*   `warm_method_threshold_`：约 1000 次调用 → 开始采集 Profile
*   `hot_method_threshold_`：约 10000 次调用 → 触发 JIT 编译
*   `osr_method_threshold_`：约 20000 次回边 → 触发 OSR（栈上替换）

### 3.3 编译任务：从 JitCompileTask 到机器码

当方法达到热度阈值后，ART 将编译任务封装为 `JitCompileTask`，投入 JIT 线程池。

*   **源码路径：** `art/runtime/jit/jit.cc`

```cpp
// art/runtime/jit/jit.cc (概念性简化)
class JitCompileTask final : public SelfDeletingTask {
  void Run(Thread* self) override {
    // 1. 通过 Optimizing Compiler 将 Dex 字节码编译为机器码
    bool success = compiler_driver_->CompileMethod(method_);

    if (success) {
      // 2. 将编译后的机器码存入 JitCodeCache
      const void* code = jit_code_cache_->CommitCode(method_, ...);

      // 3. 替换方法的入口点（关键一步）
      method_->SetEntryPointFromQuickCompiledCode(code);
    }
  }
};
```

第 3 步 `SetEntryPointFromQuickCompiledCode` 是整个 JIT 的**核心切换点**：它将 `ArtMethod` 结构体中的 `entry_point_from_quick_compiled_code_` 字段从解释器桥（`art_quick_to_interpreter_bridge`）替换为新编译的机器码地址。此后任何对该方法的调用都直接跳转到机器码执行，不再经过解释器。

### 3.4 JitCodeCache 的结构与 GC

编译后的机器码存储在一块专用的内存区域——`JitCodeCache`。

*   **源码路径：** `art/runtime/jit/jit_code_cache.cc`

```cpp
// art/runtime/jit/jit_code_cache.cc (概念性)
class JitCodeCache {
  // 代码区：存放编译后的机器码（mmap 分配，标记为 RX 可读可执行）
  MemMap code_map_;
  // 数据区：存放编译过程中生成的元数据（根表、栈映射等）
  MemMap data_map_;

  // 代码区的容量上限（通常 32MB-64MB）
  size_t max_capacity_;
  // 当前已使用的容量
  size_t current_capacity_;
};
```

`JitCodeCache` 有容量上限。当接近满时，ART 会进行 **Code Cache GC**——回收那些不再被使用的编译产物（例如，已经被 AOT 编译覆盖的方法，或者由于 Deoptimization 被废弃的代码）。

### 3.5 稳定性：JIT 引发的两类问题

**问题一：JIT 线程抢占前台 CPU → Jank**

JIT 编译是 CPU 密集型操作（涉及 SSA 构建、寄存器分配、指令选择等优化 Pass）。JIT 线程池通常有 1-2 个线程，在大核上运行时可能与 UI 线程竞争 CPU 时间片，导致帧率下降。

典型表现：App 启动后的前 5-10 秒（JIT 密集编译期间）出现可感知的 Jank，之后随着热点方法编译完毕，流畅度恢复。

治理方案：
*   利用 Baseline Profile 将核心路径的方法预编译为 AOT，减少运行时 JIT 的工作量。
*   通过 `setprop dalvik.vm.usejitprofiles true` 确保 JIT 仅编译 Profile 中记录的方法，避免无差别编译。

**问题二：Code Cache 满 → 性能退化**

当 `JitCodeCache` 达到 `max_capacity_` 后，新的热点方法无法被编译，只能回退到解释模式。如果此时 Code Cache GC 无法回收足够的空间（所有编译产物都仍在使用），性能会持续退化。

典型表现：App 长时间运行后（如后台存活数小时），突然变卡。Logcat 中出现 `JIT code cache full, ...` 警告。

治理方案：
*   通过 `-Xjitcodecachesize` 系统属性增大 Code Cache 容量。
*   减少方法数（R8 优化、减少不必要的包装类/Lambda），降低 Code Cache 压力。

## 4. AOT 编译与 dex2oat

### 4.1 为什么需要 AOT

JIT 解决了"运行中"的性能问题，但它有一个天然缺陷：**编译产物只存在于内存中，进程退出即丢失**。每次冷启动都要重新经历"解释 → JIT 编译 → 机器码"的升温过程。

AOT 的思路是**把编译提前到安装时或空闲时**，将编译产物持久化为文件，下次启动直接加载。这就是 `dex2oat` 工具的职责。

### 4.2 Vdex / Odex / Art 文件三件套

`dex2oat` 编译后会生成三个文件，它们各司其职：

| 文件 | 全称 | 内容 | 典型路径 |
| :--- | :--- | :--- | :--- |
| `.vdex` | Verified Dex | 经过校验的 Dex 文件（去除了重复校验的开销） + 快速查找结构 | `/data/app/.../oat/arm64/base.vdex` |
| `.odex` | Optimized Dex（本质是 OAT 文件） | 编译后的机器码 + OAT Header + ELF 格式容器 | `/data/app/.../oat/arm64/base.odex` |
| `.art` | App Image | 预初始化的堆对象快照（Class 对象、String 对象等），直接 mmap 到内存 | `/data/app/.../oat/arm64/base.art` |

三者的协作关系：

```
                   ┌─────────────┐
                   │   .vdex     │  校验后的 Dex 原始字节码
                   │  (Dex 原文)  │  供解释器/反优化回退使用
                   └──────┬──────┘
                          │ 引用
    ┌─────────────────────┼─────────────────────┐
    │                     │                     │
    ▼                     ▼                     ▼
┌─────────┐      ┌──────────────┐      ┌─────────────┐
│  .odex  │      │    .art      │      │  boot.art   │
│ (OAT)   │      │ (App Image)  │      │ (Boot Image)│
│ 机器码   │ ◄──── 类对象引用   │      │ 核心类快照   │
│ + ELF   │      │ 预初始化堆   │      │             │
└─────────┘      └──────────────┘      └─────────────┘
```

### 4.3 OAT 文件结构

OAT 文件本质上是一个**定制的 ELF 文件**，包含了 ART 特有的 Header 和编译后的机器码。

*   **源码路径：** `art/runtime/oat_file.h`

```cpp
// art/runtime/oat_file.h (简化)
class OatHeader {
  uint8_t magic_[4];                    // "oat\n"
  uint8_t version_[4];                  // OAT 版本号 (如 "199\0")
  uint32_t adler32_checksum_;           // 校验和

  InstructionSet instruction_set_;      // 目标指令集 (ARM, ARM64, x86...)
  InstructionSetFeatures instruction_set_features_;  // CPU 特性 (如 NEON)

  uint32_t dex_file_count_;             // 包含的 Dex 文件数
  uint32_t oat_dex_files_offset_;       // OatDexFile 数组的偏移

  uint32_t key_value_store_size_;       // 键值存储区大小 (编译参数等)
  uint8_t key_value_store_[0];          // 编译选项（如 --compiler-filter）
};
```

`instruction_set_` 和 `instruction_set_features_` 决定了 OAT 文件的硬件兼容性。如果一个在 ARM64 设备上编译的 OAT 文件被放到 ARM32 设备上，ART 会检测到不匹配，拒绝加载，回退到解释执行。

### 4.4 dex2oat 编译流程

*   **源码路径：** `art/dex2oat/dex2oat.cc`

`dex2oat` 是一个独立的可执行文件，由 `PackageManagerService`（安装时）或 `BackgroundDexOptService`（空闲时）调用。

```cpp
// art/dex2oat/dex2oat.cc (编译主流程，概念性简化)
static dex2oat::ReturnCode Dex2oat(int argc, char** argv) {
  // 1. 解析编译参数
  //    --compiler-filter=speed-profile  (按 Profile 编译)
  //    --dex-file=/data/app/.../base.apk
  //    --oat-file=/data/app/.../oat/arm64/base.odex
  Dex2Oat dex2oat(&argc, &argv);

  // 2. 打开并校验 Dex 文件
  dex2oat.OpenDexFiles();
  dex2oat.VerifyDexFiles();

  // 3. 创建 CompilerDriver
  //    内部初始化 Optimizing Compiler（寄存器分配、内联、死代码消除等）
  dex2oat.CreateCompilerDriver();

  // 4. 编译所有方法（按 compiler-filter 决定编译范围）
  //    speed-profile: 只编译 Profile 中记录的热点方法
  //    speed:         编译所有方法
  //    verify:        只做字节码校验，不编译
  dex2oat.Compile();

  // 5. 写出 OAT 文件 (.odex) 和 Vdex 文件 (.vdex)
  dex2oat.WriteOutputFiles();

  // 6. 可选：生成 App Image (.art)
  dex2oat.WriteAppImage();

  return dex2oat::ReturnCode::kNoFailure;
}
```

### 4.5 Compiler Filter：编译的"油门"

`--compiler-filter` 参数控制 `dex2oat` 的编译力度，直接影响编译耗时和产物质量：

| Filter | 行为 | 耗时 | 运行时性能 | 典型场景 |
| :--- | :--- | :--- | :--- | :--- |
| `verify` | 只做字节码校验，不编译 | 极快（秒级） | 最差（纯解释+JIT） | 安装时的默认行为（Android 7.0+） |
| `quicken` | 校验 + 简单优化（指令替换） | 快 | 较差 | 已弃用 |
| `speed-profile` | 只编译 Profile 中的热点方法 | 中（十几秒） | 好（热点方法是 AOT，其余 JIT） | 后台空闲编译 |
| `speed` | 编译所有方法 | 慢（分钟级） | 最好 | 系统应用、核心库 |
| `everything` | 编译所有方法 + 生成最详细的调试信息 | 最慢 | 最好 | 开发调试 |

### 4.6 稳定性：AOT 引发的两类问题

**问题一：后台 dex2oat I/O 风暴 → 前台 ANR**

`BackgroundDexOptService` 在设备充电且空闲时触发 `dex2oat`。但"空闲"的判断有时并不准确——用户可能突然解锁使用设备。此时 `dex2oat` 正在大量读写磁盘（读取 Dex、写入 OAT），与前台 App 的 I/O 竞争，导致前台 App 读取数据库或 SharedPreferences 时 I/O 延迟飙升，主线程阻塞超过 5 秒触发 ANR。

典型日志特征：ANR 的 `traces.txt` 中主线程卡在 `read()` 或 `write()` 系统调用上，同时 `top` 或 `dumpsys cpuinfo` 显示 `dex2oat` 进程的 CPU 和 I/O 占用极高。

治理方案：
*   OEM 厂商通过 cgroup 限制 `dex2oat` 的 I/O 带宽和 CPU 优先级。
*   Android 10+ 引入了 `setIoSchedulingClass(IOPRIO_CLASS_IDLE)` 将 `dex2oat` 的 I/O 优先级降至最低。

**问题二：OAT 文件损坏 → SIGBUS**

OAT 文件通过 `mmap` 映射到进程的虚拟地址空间后被直接执行。如果文件在映射后被损坏（如磁盘坏块、系统异常断电导致写入不完整、ROM 空间不足导致截断），CPU 在读取损坏的机器码指令时会触发 `SIGBUS`（Bus Error）信号，进程直接崩溃。

典型日志特征：`signal 7 (SIGBUS), code 2 (BUS_ADRERR)`, fault address 位于 OAT 文件的 mmap 范围内。

治理方案：
*   ART 在加载 OAT 文件时会校验 `OatHeader` 中的 `adler32_checksum_`。如果校验失败，拒绝加载，回退到解释执行。
*   设备端监控 `/data` 分区的剩余空间，低于阈值时暂停 `dex2oat`。

## 5. PGO 与 Baseline Profile

### 5.1 Profile-Guided Optimization 的必要性

ART 的混合编译模式面临一个核心矛盾：**安装时不知道哪些方法是热点**。如果全量编译（`speed` filter），安装时间和存储空间不可接受；如果不编译（`verify` filter），首次启动性能很差。

PGO 的思路是：**先观察用户的实际使用行为，记录热点方法，再据此精准编译**。

### 5.2 Profile 文件的收集

*   **源码路径：** `art/runtime/jit/profile_saver.cc`

`ProfileSaver` 是 ART 中一个专用的后台线程，定期将 JIT 采集到的热点方法信息持久化到磁盘。

```cpp
// art/runtime/jit/profile_saver.cc (概念性简化)
void ProfileSaver::Run() {
  while (!ShuttingDown(self)) {
    // 每隔 N 秒（默认约 20 秒）执行一次
    WaitForSavingRequest(save_period_ms_);

    // 从 JIT 的热度计数器中收集热点方法
    // 包括：方法签名、调用次数、是否包含循环
    uint32_t num_new_methods = FetchAndCacheResolvedClassesAndMethods();

    if (num_new_methods > 0) {
      // 写入 Profile 文件
      // 路径：/data/misc/profiles/cur/0/<package_name>/primary.prof
      ProcessProfilingInfo(/*force_save=*/false, &number_of_new_methods);
    }
  }
}
```

Profile 文件的格式包含：
*   **Hot Methods：** 热点方法的 Dex 方法索引 + 所属 Dex 文件的校验和
*   **Startup Methods：** 标记为启动路径的方法
*   **Post-Startup Methods：** 启动后使用的方法
*   **Classes：** 需要预加载的类列表

### 5.3 Cloud Profile 下发

Android 9.0 引入了 Cloud Profile 机制：Google Play 在分发 APK 时，会聚合海量用户的 Profile 数据，将最通用的热点方法 Profile 随 APK 一并下载。这样新用户在安装时就能获得接近最优的 AOT 编译效果，不再需要等待本地 Profile 积累。

### 5.4 Baseline Profile 的打包与安装时编译

Android 12+ 引入了 **Baseline Profile**：开发者在 CI/CD 流程中通过 Macrobenchmark 运行核心用户路径（启动、滑动列表、页面跳转等），采集 Profile 数据，将其打包进 APK/AAB 的 `assets/dexopt/baseline.prof` 中。

安装时，`PackageManagerService` 检测到 Baseline Profile 的存在，会指示 `dex2oat` 使用 `speed-profile` filter 按照该 Profile 编译。这等同于**在安装时就完成了原本需要等待数天才能积累的 PGO 优化**。

### 5.5 稳定性：冷启动耗时波动

**问题：同一个 App，在不同用户设备上的冷启动时间差异巨大（300ms 到 3000ms）。**

根因分析：
1.  **Profile 积累不足：** 新安装用户如果没有 Cloud Profile 或 Baseline Profile，首次启动完全走解释+JIT，耗时 3x-5x。
2.  **dex2oat 尚未完成：** 后台编译需要设备充电+空闲，如果用户从不充电到 100%（或频繁使用），`dex2oat` 可能数天都不执行。
3.  **Profile 不完整：** 如果用户的使用路径与 Profile 记录的热点方法不一致（如跳过了首页直接进入深层页面），部分关键方法仍走解释器。

治理方案：
*   **必须打包 Baseline Profile**。这是目前治理冷启动波动最有效的手段。
*   在 App 启动链路的关键方法上添加 `@androidx.annotation.Keep` + 手动标记为热点。
*   通过 `adb shell cmd package compile -m speed-profile -f <package>` 强制触发编译以验证效果。

## 6. 蹦床机制（Trampolines）

### 6.1 什么是蹦床

在 ART 中，每个 Java 方法（`ArtMethod`）都有一个 `entry_point_from_quick_compiled_code_` 字段，指向该方法的实际执行入口。但一个方法可能处于不同的状态（未解析、已解释、已 JIT 编译、已 AOT 编译），不可能为每种状态都生成一份完整的入口逻辑。

**蹦床（Trampoline）** 就是一组短小的汇编代码片段，充当不同执行路径之间的"转发器"。方法的入口点先跳到蹦床，蹦床再根据当前状态决定实际去向。

### 6.2 三个核心蹦床

*   **源码路径：** `art/runtime/arch/arm64/quick_entrypoints_arm64.S`

**`art_quick_to_interpreter_bridge`：** 当方法尚未编译（或被反优化）时，入口点指向此蹦床。它负责：
1.  从 `ArtMethod` 获取 `CodeItem`
2.  创建 `ShadowFrame`
3.  跳转到解释器（`ExecuteSwitchImpl` 或 Mterp）

**`art_quick_generic_jni_trampoline`：** 当 Java 代码调用一个 Native 方法时，入口点指向此蹦床。它负责：
1.  将 Java 的调用约定（虚拟寄存器）转换为 C/C++ 的调用约定（物理寄存器 + 栈参数）
2.  查找 Native 函数的地址（`dlsym` 或 `RegisterNatives` 注册的地址）
3.  切换线程状态为 `kNative`
4.  调用 Native 函数
5.  返回后切换回 `kRunnable`，检查异常

**`art_quick_resolution_trampoline`：** 当方法的目标尚未解析（首次调用）时，入口点指向此蹦床。它负责：
1.  调用 `ClassLinker` 解析方法签名，找到真正的 `ArtMethod`
2.  根据方法状态（已编译/未编译），更新入口点
3.  跳转到正确的执行入口

```asm
// art/runtime/arch/arm64/quick_entrypoints_arm64.S (概念性)
ENTRY art_quick_resolution_trampoline
    // 保存调用者上下文
    SETUP_SAVE_REFS_AND_ARGS_FRAME

    // 调用 C++ 的 artQuickResolutionTrampoline()
    // 该函数完成方法解析、入口点更新
    bl    artQuickResolutionTrampoline

    // 恢复上下文
    RESTORE_SAVE_REFS_AND_ARGS_FRAME

    // 跳转到解析后的真正入口
    br    x0
END art_quick_resolution_trampoline
```

### 6.3 稳定性：Hook 框架修改 EntryPoint 引发的 Crash

蹦床机制是各种 Java Method Hook 框架（如 Epic、SandHook、Pine）的**核心利用点**。这些框架通过修改 `ArtMethod` 的 `entry_point_from_quick_compiled_code_`，将方法的入口替换为自定义的 Hook 代码。

但这种做法风险极高：

1.  **GC 期间的一致性问题：** CC GC（Concurrent Copying）在移动对象时需要更新所有指向该对象的引用（包括编译后代码中的嵌入式指针）。如果 Hook 框架替换了入口点，GC 的更新逻辑可能遗漏 Hook 代码中的对象引用，导致访问已移动的旧地址 → `SIGSEGV`。
2.  **编译器假设被打破：** AOT 编译时，编译器可能已经将被调用方法的代码内联到调用方中。如果此时 Hook 了被调用方，内联的代码不会受影响，导致 Hook 不生效或行为不一致。
3.  **版本兼容性：** `ArtMethod` 结构体的内存布局在不同 Android 版本间可能变化（字段偏移不同）。Hook 框架如果 hardcode 了某个版本的偏移量，在其他版本上会写错位置，导致不可预测的 Crash。

## 7. OSR（On-Stack Replacement）

### 7.1 为什么需要 OSR

常规的 JIT 编译只能在**方法级别**生效：方法被编译后，下一次调用才走机器码。但如果一个方法内部有一个非常长的循环（如数据处理循环执行百万次迭代），等循环结束再换成机器码已经太晚了。

OSR 的核心思想是：**在循环体执行过程中，当检测到回边计数超过阈值时，在不退出当前方法的情况下，将当前解释器栈帧"原地"替换为编译后的机器码栈帧，继续从循环的当前迭代位置开始以机器码执行。**

### 7.2 OSR 的触发与实现

*   **源码路径：** `art/runtime/jit/jit.cc`

```cpp
// art/runtime/jit/jit.cc (概念性简化)
// 在解释器的回边（循环跳回）处调用
bool Jit::MaybeDoOnStackReplacement(Thread* self,
                                    ArtMethod* method,
                                    uint32_t dex_pc,
                                    ...) {
  // 1. 检查该方法在当前 dex_pc 处是否有已编译的 OSR 版本
  const void* osr_code = jit_code_cache_->LookupOsrMethodHeader(method);
  if (osr_code == nullptr) {
    // 没有 OSR 编译版本，提交编译任务
    EnqueueOsrCompilation(method, dex_pc);
    return false;
  }

  // 2. 将解释器栈帧（ShadowFrame）中的虚拟寄存器
  //    映射到编译后代码的物理寄存器/栈槽位
  PrepareForOsr(self, shadow_frame, osr_code);

  // 3. 跳转到编译后代码的对应位置（当前循环迭代处）
  //    从此刻起，当前方法的剩余执行完全由机器码接管
  JumpToCompiledCode(osr_code, dex_pc);

  return true;
}
```

### 7.3 稳定性：OSR 过程中的罕见 Native Crash

OSR 是 ART 中**最复杂的状态切换之一**，因为它需要在运行时将解释器的虚拟寄存器状态精确地映射到编译后代码的物理寄存器布局。这个映射过程中存在若干微妙的风险：

**风险一：OSR 过程中触发 GC**

OSR 的 `PrepareForOsr` 阶段需要分配内存（用于存储映射信息）。如果此时触发了 GC，GC 需要扫描当前线程的栈帧——但此刻栈帧正处于"半解释器半编译码"的中间状态，GC 的栈遍历器（Stack Walker）可能无法正确识别所有的对象引用，导致活跃对象被错误回收，后续访问该对象时触发 `SIGSEGV`。

**风险二：OSR 过程中抛出异常**

如果在 `PrepareForOsr` 过程中发生异常（如 `OutOfMemoryError`），异常展开（Stack Unwinding）需要回退到正常的栈帧状态。但 OSR 的中间状态下，解释器栈帧已经部分被修改，回退可能不完整，导致后续的异常处理逻辑读到脏数据。

**风险三：编译器 OSR 入口点的正确性**

OSR 不是从方法的起始地址开始执行，而是从循环体的某个中间位置开始。编译器需要为这个中间入口点生成正确的初始化代码（将虚拟寄存器映射到物理寄存器）。如果编译器在某些边界情况下（如循环变量是 long/double 类型，跨两个虚拟寄存器）生成了错误的映射代码，执行时会读到错误的值。

这些问题极其罕见（百万级 DAU 的 App 每天可能只有个位数的上报），但一旦发生就是无法捕获的 Native Crash，堆栈通常指向编译后的代码区域，难以直接关联到 Java 源码。

## 【稳定性实战案例】

### 案例 1：后台 dex2oat 风暴导致批量 ANR

**现象：** 某社交 App 在 Android 11 低端机（4GB RAM、eMMC 存储）上 ANR 率突然飙升，集中在每天凌晨 2-4 点。ANR 类型是 `Input dispatching timed out`，主线程卡在 `SharedPreferences.edit().commit()` 上。

**分析过程：**
1.  提取 ANR 的 `traces.txt`，主线程调用栈：
    ```
    at android.app.SharedPreferencesImpl.commitToMemory (SharedPreferencesImpl.java:475)
    at android.app.SharedPreferencesImpl$EditorImpl.commit (SharedPreferencesImpl.java:420)
    - waiting to lock <0x... held by thread 15>
    ```
    主线程在等一个锁，但持锁线程（thread 15）的堆栈显示它卡在 `fsync` 系统调用上。
2.  检查 ANR 时段的 `dumpsys cpuinfo` 日志，发现 `dex2oat` 进程持续占用 **60%-80% 的 I/O 带宽**。eMMC 的 4K 随机写 IOPS 本就有限（约 5000），`dex2oat` 在写 OAT 文件时产生大量 4K 写操作，导致前台 App 的 `fsync` 延迟从正常的 5ms 飙升到 **800ms-2000ms**。
3.  ANR 集中在凌晨 2-4 点的原因：用户通常在这个时段充电，设备判定为"空闲"，触发 `BackgroundDexOptService` 批量编译所有 App。

**根因：** `dex2oat` 的 I/O 优先级在 Android 11 的该 OEM ROM 中未正确设置为 `IOPRIO_CLASS_IDLE`，导致后台编译与前台 I/O 平等竞争带宽。

**修复方案：**
1.  OEM 侧：修复 I/O 优先级设置，确保 `dex2oat` 使用 `IOPRIO_CLASS_IDLE`。
2.  App 侧：将 `SharedPreferences.commit()` 替换为 `apply()`（异步写入），避免主线程阻塞在 `fsync` 上。对于必须同步写入的场景，使用 MMKV 替代 SharedPreferences（MMKV 使用 `mmap` 而非 `write`+`fsync`，受 I/O 风暴影响更小）。

### 案例 2：JIT Code Cache 耗尽导致游戏卡顿

**现象：** 某大型手游在连续运行 2 小时后，帧率从稳定 60fps 下降到 30-40fps。重启 App 后恢复。用户反馈"越玩越卡"。

**分析过程：**
1.  通过 `adb shell dumpsys meminfo <pid>` 观察内存分布，发现 `JIT Code Cache` 的使用量持续增长，2 小时后达到上限（64MB）。
2.  Logcat 中出现周期性的 `JIT code cache collection` 日志，但每次 GC 只能回收约 2-3MB（因为大部分编译产物仍在使用）。
3.  游戏使用了大量 Lua/JS → Java 反射调用（热更新框架），每个反射调用路径在 JIT 看来都是不同的方法签名，导致大量"长尾"方法被编译但调用频率极低。
4.  Code Cache 满后，新进入的关卡中的热点方法无法被 JIT 编译，退化到解释执行，帧率骤降。

**根因：** 热更新框架通过反射动态调用大量方法，导致 JIT 编译了大量实际并非热点的方法，耗尽了 Code Cache 容量。

**修复方案：**
1.  通过 `dalvik.vm.extra-opts=-Xjitcodecachesize:128m` 将 Code Cache 上限从 64MB 提升到 128MB（需要与内存预算权衡）。
2.  优化热更新框架，将高频反射调用替换为预生成的 Wrapper 类（编译期生成，避免运行时反射），减少"虚假热点"的 JIT 编译数量。
3.  为核心游戏逻辑（渲染循环、物理引擎调用）打包 Baseline Profile，确保这些方法在安装时就完成 AOT 编译，不占用运行时 Code Cache。

## 8. 总结

本篇从"三种执行模式的全景"出发，自顶向下地剖析了 ART 的编译与执行架构：

**核心要点：**
1.  **ART 的解释器、JIT、AOT 三位一体**是理解应用启动速度和运行性能差异的基础。同一个 Bug 在不同执行模式下可能表现不同，这是"偶现 Crash"难以复现的根因之一。
2.  **JIT 的热度计数器 → 编译任务 → 入口替换**是方法从慢到快的完整升级路径。Code Cache 是 JIT 的有限资源，耗尽后性能退化。
3.  **dex2oat 是 AOT 的执行者**，Vdex/Odex/Art 三件套各司其职。后台 dex2oat 的 I/O 风暴是低端设备 ANR 的常见根因。
4.  **Baseline Profile 是治理冷启动波动最有效的手段**，本质是将 PGO 从"运行时积累"提前到"开发期采集"。
5.  **蹦床机制是 ART 方法分发的枢纽**，也是 Hook 框架的核心利用点和风险源头。
6.  **OSR 是 ART 最精密的状态切换**，其过程中的 GC/异常风险是罕见 Native Crash 的来源。
