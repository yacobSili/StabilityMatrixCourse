# 05-栈回溯与符号化：从地址到代码行

上一篇我们拆解了 debuggerd 与 Tombstone 的生成流程——crash_dump 进程通过 ptrace 附着到崩溃进程，读取寄存器和内存，然后调用 libunwindstack 进行堆栈回溯。但"堆栈回溯"这四个字背后，隐藏着一整套精密的工程：**如何从崩溃瞬间的 PC/SP/FP 寄存器值，一帧一帧地还原出完整的函数调用链？还原出来的每一帧只是一个十六进制地址（如 `pc 0x7f8a3b2c`），又如何将它翻译成人类可读的 `foo.cpp:42 in MyClass::doSomething()`？**

这就是栈回溯（Stack Unwinding）与符号化（Symbolication）要解决的问题。它们是 Tombstone 可读性的命脉——回溯失败意味着 backtrace 残缺不全，符号化失败意味着 backtrace 全是 `<unknown>`。对于稳定性架构师来说，理解这两个环节的原理和失败模式，直接决定了你能否从一份"残缺"的 Tombstone 中提取出有效信息。

---

## 1. 栈回溯是什么

### 1.1 为什么需要栈回溯

当 Native Crash 发生时，CPU 停在了触发异常的那条指令上。此时我们能直接从寄存器中获取的信息只有：

- **PC（Program Counter）**：当前正在执行的指令地址——即"在哪里崩溃的"
- **SP（Stack Pointer）**：当前栈顶地址
- **FP/X29（Frame Pointer）**：当前栈帧的基地址（如果编译时保留了 Frame Pointer）
- **LR/X30（Link Register）**：调用当前函数的返回地址——即"谁调用了当前函数"

但仅凭 PC 和 LR 只能知道"崩溃点"和"直接调用者"两帧信息。要还原出完整的调用链（`main() → A() → B() → C() → crash`），就需要**栈回溯（Stack Unwinding）**——从当前帧的寄存器状态出发，逐帧恢复上一帧的 PC/SP/FP，直到回溯到栈底（通常是线程的入口函数 `__start_thread` 或 `main()`）。

Tombstone 中的 backtrace 就是栈回溯的产物：

```
backtrace:
    #00 pc 0x0004a2bc  /vendor/lib64/libcamera_hal.so (CameraDevice::processRequest+188)
    #01 pc 0x0004b890  /vendor/lib64/libcamera_hal.so (RequestThread::threadLoop+240)
    #02 pc 0x00012f64  /system/lib64/libutils.so (android::Thread::_threadLoop+280)
    #03 pc 0x000b1234  /apex/com.android.runtime/lib64/bionic/libc.so (__pthread_start+64)
    #04 pc 0x00052a90  /apex/com.android.runtime/lib64/bionic/libc.so (__start_thread+40)
```

每一行的 `pc` 值就是该帧的返回地址（Return Address），括号中的函数名和偏移量是符号化的结果。

### 1.2 两种主流回溯算法

栈回溯有两种主流方法，它们的可靠性和性能差异巨大：

| 维度 | FP-based 回溯 | DWARF CFI 回溯 |
| :--- | :--- | :--- |
| **依赖** | 编译时保留 Frame Pointer（`-fno-omit-frame-pointer`） | ELF 文件中的 `.eh_frame` / `.debug_frame` 段 |
| **速度** | 极快——只需要沿 FP 链表遍历 | 较慢——需要解析 DWARF CFI 字节码指令 |
| **可靠性** | 低——依赖编译选项和调用约定 | 高——能处理无 FP、叶函数、异步中断等场景 |
| **适用场景** | 性能采样（perf/simpleperf 的快速回溯模式） | 崩溃分析（debuggerd/libunwindstack 的默认方式） |

Android 的 debuggerd 和 libunwindstack **默认使用 DWARF CFI 回溯**，但会在 DWARF 信息缺失时回退到 FP-based 回溯作为 fallback。

**稳定性架构师视角：** 理解两种回溯方式的差异至关重要。当你在 Tombstone 中看到 backtrace 在某个 so 库处突然截断、或者出现明显不合理的帧（如跳到了完全无关的地址），第一反应应该是：这个 so 库的回溯信息是否完整？它是否被编译时 strip 掉了 `.eh_frame` 段？它是否使用了 `-fomit-frame-pointer` 且缺少 DWARF 信息？

## 2. FP-based 回溯

### 2.1 Frame Pointer 是什么

在 ARM64（AArch64）的标准调用约定（AAPCS64）中，每个函数的 prologue（入口代码）会做两件事：

1. 将调用者的 FP（X29）和 LR（X30）保存到当前栈帧
2. 将 SP 的值赋给 FP，使 FP 指向当前栈帧的"帧底"

```
高地址
┌─────────────────────┐
│ caller's caller FP   │ ← caller 的 FP 指向这里
│ caller's caller LR   │
├─────────────────────┤
│ caller's local vars  │
├─────────────────────┤
│ caller's FP (saved)  │ ← 当前帧的 FP（X29）指向这里
│ caller's LR (saved)  │   保存了调用者的返回地址
├─────────────────────┤
│ current local vars   │
│ ...                  │
├─────────────────────┤
│                      │ ← SP 指向当前栈顶
低地址
```

这形成了一条**FP 链表**：当前帧的 FP 指向保存在栈上的"上一帧 FP"，上一帧 FP 又指向再上一帧的 FP……一直到栈底。

### 2.2 FP-based 回溯的实现

FP-based 回溯的算法极其简单——沿着 FP 链表遍历即可：

```
function fp_unwind(fp, lr):
    frame[0] = { pc: crash_pc, sp: crash_sp }
    frame[1] = { pc: lr }       // LR 就是第一帧的返回地址
    
    i = 2
    while fp != 0 and fp is valid:
        saved_fp = *(fp)        // FP 指向的位置存储了上一帧的 FP
        saved_lr = *(fp + 8)    // FP+8 处存储了上一帧的 LR（即返回地址）
        frame[i] = { pc: saved_lr }
        fp = saved_fp           // 跳到上一帧
        i++
```

ARM64 的编译器生成的典型函数 prologue：

```asm
// 一个典型的 ARM64 函数入口
stp x29, x30, [sp, #-16]!    // 将 FP(x29) 和 LR(x30) 压栈，SP 减 16
mov x29, sp                   // FP = SP，建立当前帧的 Frame Pointer
```

对应的 epilogue：

```asm
ldp x29, x30, [sp], #16      // 从栈上恢复 FP 和 LR，SP 加 16
ret                           // 跳转到 LR（返回调用者）
```

### 2.3 为什么 `-fomit-frame-pointer` 会破坏回溯

编译器优化选项 `-fomit-frame-pointer` 会省略 prologue 中保存 FP 的操作，将 X29 当作普通的通用寄存器使用。这样做的好处是多了一个可用寄存器，能提升几个百分点的性能。代价是 FP 链表断裂——回溯器沿着 FP 往上追溯时，遇到不保存 FP 的函数就"断链"了。

```
正常（保留 FP）：
  FP → saved_FP → saved_FP → saved_FP → 0 (栈底)
  ✅ 完整回溯

使用 -fomit-frame-pointer 的函数：
  FP → saved_FP → ??? (X29 被用作普通寄存器，值是随机的)
  ❌ 链表断裂，后续帧全部丢失或错误
```

**稳定性架构师视角：** Android NDK 从 r23 开始，默认在 Release 构建中启用 `-fomit-frame-pointer`（因为 DWARF CFI 信息已足够用于回溯）。但如果你使用自定义编译工具链或者旧版 NDK，需要确认 `.eh_frame` 段是否被保留。如果一个 so 库同时缺少 FP 和 `.eh_frame`，那它在 Tombstone 中就是一个"黑洞"——回溯到这个库就会截断。

**典型场景：** 第三方 SDK 提供的预编译 so 库（如某些加固/加密 SDK、某些游戏引擎），经常被 strip 得很彻底（连 `.eh_frame` 都删了），又使用了 `-fomit-frame-pointer`。当崩溃发生在这些库内部时，Tombstone 的 backtrace 可能只有一帧甚至零帧。

## 3. DWARF CFI 回溯

### 3.1 为什么需要 DWARF

FP-based 回溯虽然快，但有三个根本性缺陷：

1. **依赖编译选项**：一旦 `-fomit-frame-pointer` 生效，FP 链表就断了
2. **无法处理叶函数**：叶函数（不调用其他函数的函数）可能不保存 LR 到栈上，FP 回溯无法获取其返回地址
3. **无法处理异步中断**：信号可能在函数的任意位置中断执行——可能在 prologue 之前（FP 还没建立）、在 epilogue 之后（FP 已恢复），此时 FP 指向的不是当前帧

**DWARF（Debugging With Attributed Record Formats）** 标准定义了一种与编译器无关的方式来描述"在程序的任意位置，如何恢复上一帧的寄存器状态"。这些信息被编码成 **CFI（Call Frame Information）** 指令，存储在 ELF 文件的特殊段中。

### 3.2 `.eh_frame` 与 `.debug_frame`

DWARF CFI 信息存储在两个可能的 ELF 段中：

| 段名 | 用途 | 是否在 Release 中保留 | 使用者 |
| :--- | :--- | :--- | :--- |
| `.eh_frame` | 运行时异常处理（C++ exception）和回溯 | **是**——即使 `strip` 也默认保留 | libunwindstack、C++ runtime |
| `.debug_frame` | 纯调试用途的 CFI 信息 | 否——`strip` 时被移除 | 调试器（gdb/lldb） |

**关键认知：** `.eh_frame` 是 `strip` 后仍然保留的，因为它被标记为 `SHF_ALLOC`（需要加载到内存）。Android 系统库和 NDK 构建的 so 库默认都包含 `.eh_frame`。这就是为什么即使 Release 版本的 so 库也能进行 DWARF 回溯。

### 3.3 CFI 指令：描述寄存器保存规则

CFI 的核心思想：**对于程序中的每一个指令地址，描述如何计算 CFA（Canonical Frame Address，标准帧地址）以及每个寄存器的保存位置。**

一条 CFI 规则大致表达的含义是：

```
在 PC = 0x1234 处：
  CFA = SP + 32                    （当前帧的标准基地址 = 栈指针 + 32）
  X29(FP) 保存在 CFA - 16 处      （FP 被保存到了栈上的某个位置）
  X30(LR) 保存在 CFA - 8 处       （LR/返回地址被保存到了栈上的某个位置）
```

有了这些规则，回溯器就能在程序的任意位置恢复上一帧的状态：

1. 根据当前 PC 查找对应的 CFI 规则
2. 用规则计算 CFA（通常是 `SP + offset` 或 `FP + offset`）
3. 从 CFA 加偏移处读取保存的寄存器值（尤其是 LR 和 FP）
4. LR 就是上一帧的 PC，重复以上步骤

用 `readelf --debug-dump=frames` 可以查看一个 so 库的 CFI 信息：

```
$ readelf --debug-dump=frames libfoo.so

00000014 0000001c 00000000 FDE cie=00000000 pc=00001234..00001280
  DW_CFA_advance_loc: 4 to 00001238
  DW_CFA_def_cfa_offset: 32             // CFA = SP + 32
  DW_CFA_offset: r30 (x30) at cfa-8     // LR 保存在 CFA - 8
  DW_CFA_offset: r29 (x29) at cfa-16    // FP 保存在 CFA - 16
  DW_CFA_advance_loc: 4 to 0000123c
  DW_CFA_def_cfa_register: r29 (x29)    // CFA 改为用 FP 作为基准
```

每条 `DW_CFA_*` 指令对应一个 CFI 操作，它们按照 PC 递增的顺序排列，描述了函数从入口到出口过程中寄存器保存状态的变化。

### 3.4 DWARF 回溯的工作流程

```
┌────────────────────────────────────────────────────────────────┐
│  DWARF CFI 回溯流程                                              │
│                                                                 │
│  输入: 崩溃时的 PC, SP, FP, LR 寄存器值                           │
│                                                                 │
│  ① 查找 PC 所在的 ELF 文件                                       │
│     通过 /proc/<pid>/maps 确定 PC 属于哪个 so 库                  │
│     │                                                           │
│     ▼                                                           │
│  ② 在该 ELF 的 .eh_frame 中查找 PC 对应的 FDE                    │
│     FDE（Frame Description Entry）包含一个 PC 区间的 CFI 规则      │
│     │                                                           │
│     ▼                                                           │
│  ③ 执行 CFI 指令，计算 CFA 和寄存器恢复规则                        │
│     逐条执行 DW_CFA_* 指令直到当前 PC                              │
│     │                                                           │
│     ▼                                                           │
│  ④ 恢复上一帧的寄存器状态                                          │
│     从栈内存中读取保存的 LR（上一帧 PC）和 FP                      │
│     新 SP = CFA                                                  │
│     │                                                           │
│     ▼                                                           │
│  ⑤ 重复 ①-④ 直到回溯到栈底                                       │
│     终止条件: PC = 0 或 SP 不再递增或达到最大帧数                   │
└────────────────────────────────────────────────────────────────┘
```

**稳定性架构师视角：** DWARF CFI 回溯比 FP 回溯更可靠，但也更慢（需要解析二进制格式的 CFI 指令）。在 crash_dump 的场景中，性能不是关键问题（只回溯一次），可靠性才是。但在性能采样工具（如 simpleperf）中，每秒可能需要回溯上千次，此时通常会优先使用 FP 回溯，DWARF 作为 fallback。

DWARF 回溯失败的常见原因：
- **`.eh_frame` 被 strip 删除**：某些激进的 strip 配置会删除 `.eh_frame`（使用 `--remove-section=.eh_frame`）
- **CFI 信息不准确**：手写汇编代码或内联汇编通常不带 CFI 注解，编译器无法为其生成正确的 CFI
- **栈被破坏**：即使 CFI 规则正确，如果栈上保存的寄存器值已被覆盖（如 buffer overflow 破坏了保存的 LR），读取到的返回地址就是错误的

## 4. libunwindstack

### 4.1 为什么需要 libunwindstack

栈回溯不是一个"调一个函数就行"的简单操作。它需要：

1. 读取目标进程的寄存器（可能是另一个进程，通过 ptrace）
2. 读取目标进程的内存（栈内容、ELF 段内容）
3. 解析 `/proc/<pid>/maps` 获取内存映射信息
4. 解析多个 ELF 文件的 `.eh_frame`、`.debug_frame`、`.gnu_debugdata` 段
5. 执行 DWARF CFI 指令恢复寄存器
6. 处理各种边界情况（信号帧、JIT 编译代码、PLT/GOT 桩）

Android 将这些复杂逻辑统一封装在 **libunwindstack** 中。它是 ART 运行时和 debuggerd 共用的核心回溯库，也是 simpleperf 等性能工具的回溯引擎。

**源码路径：** `system/unwinding/libunwindstack/`

### 4.2 三大核心类

libunwindstack 的架构围绕三个核心类展开：

#### Regs：寄存器状态

`Regs` 类封装了 CPU 寄存器的读取和管理。对于 ARM64，它包含 X0-X30、SP、PC 共 33 个寄存器。

**源码路径：** `system/unwinding/libunwindstack/include/unwindstack/Regs.h`

```cpp
// system/unwinding/libunwindstack/include/unwindstack/RegsArm64.h
class RegsArm64 : public RegsImpl<uint64_t> {
 public:
  RegsArm64();

  // 从 ucontext 结构体初始化（崩溃时内核保存的寄存器快照）
  void SetFromUcontext(unwindstack::Arm64Ucontext* ucontext);

  // 获取 PC 和 SP
  uint64_t pc() override { return regs_[ARM64_REG_PC]; }
  uint64_t sp() override { return regs_[ARM64_REG_SP]; }

  // 关键方法：通过回溯恢复上一帧的寄存器状态
  bool StepIfSignalHandler(uint64_t elf_offset, Elf* elf,
                           Memory* process_memory) override;
};
```

#### Maps：内存映射

`Maps` 类解析 `/proc/<pid>/maps`，将每个内存映射区域封装为 `MapInfo` 对象。回溯器通过 Maps 实现"PC 地址 → 所属 so 库"的查找。

**源码路径：** `system/unwinding/libunwindstack/include/unwindstack/Maps.h`

```cpp
// system/unwinding/libunwindstack/include/unwindstack/Maps.h
class Maps {
 public:
  // 解析 /proc/<pid>/maps 文件
  bool Parse();

  // 通过 PC 地址查找所属的 MapInfo
  std::shared_ptr<MapInfo> Find(uint64_t pc);

  // 所有 MapInfo 的列表
  std::vector<std::shared_ptr<MapInfo>> maps_;
};

// MapInfo 代表一个内存映射区域（对应 VMA）
struct MapInfo {
  uint64_t start;         // 区域起始地址
  uint64_t end;           // 区域结束地址
  uint64_t offset;        // 文件偏移
  uint16_t flags;         // 权限标志（PROT_READ/WRITE/EXEC）
  std::string name;       // 映射名称（如 /system/lib64/libc.so）
  std::shared_ptr<Elf> elf; // 关联的 ELF 解析器（懒加载）
};
```

**稳定性架构师视角：** Maps 的准确性直接影响回溯质量。如果一个 so 库是通过 `dlopen` 动态加载的，且加载后 `/proc/<pid>/maps` 更新不及时（极少见但理论存在），Maps 可能找不到 PC 对应的映射——此时该帧会被标记为 `<unknown>`。更常见的问题是 JIT 编译的代码不在任何 so 库中，libunwindstack 需要通过 ART 提供的 JIT debug info 来回溯这些帧。

#### Elf：ELF 解析与 CFI 执行

`Elf` 类是 libunwindstack 最复杂的部分。它负责解析 ELF 文件格式，提取 `.eh_frame`、`.debug_frame`、`.gnu_debugdata` 段，并执行 DWARF CFI 指令来恢复寄存器。

**源码路径：** `system/unwinding/libunwindstack/Elf.cpp`

```cpp
// system/unwinding/libunwindstack/Elf.cpp
bool Elf::Step(uint64_t rel_pc, Regs* regs, Memory* process_memory,
               bool* finished, bool* is_signal_frame) {
  // rel_pc = PC - MapInfo.start + MapInfo.offset（相对于 ELF 文件的偏移）

  // 优先尝试 .eh_frame 段中的 CFI 信息
  if (valid_ && interface_->Step(rel_pc, regs, process_memory, finished,
                                  is_signal_frame)) {
    return true;
  }

  // .eh_frame 失败，尝试 .gnu_debugdata 中的 CFI 信息
  if (gnu_debugdata_interface_ &&
      gnu_debugdata_interface_->Step(rel_pc, regs, process_memory,
                                     finished, is_signal_frame)) {
    return true;
  }

  return false;
}
```

### 4.3 Unwinder：回溯的主循环

`Unwinder` 类是对外的主入口，它将 Regs、Maps、Elf 三者组合，执行逐帧回溯的主循环。

**源码路径：** `system/unwinding/libunwindstack/Unwinder.cpp`

```cpp
// system/unwinding/libunwindstack/Unwinder.cpp
void Unwinder::Unwind(const std::vector<std::string>* initial_map_names_to_skip,
                      const std::vector<std::string>* map_suffixes_to_ignore) {
  frames_.clear();
  bool return_address_attempt = false;

  for (size_t frame_num = 0; frame_num < max_frames_; frame_num++) {
    uint64_t cur_pc = regs_->pc();
    uint64_t cur_sp = regs_->sp();

    // ① 通过 Maps 查找 PC 所属的 MapInfo
    std::shared_ptr<MapInfo> map_info = maps_->Find(cur_pc);

    // ② 构造 FrameData（一帧的信息）
    FrameData* frame = &frames_.emplace_back();
    frame->num = frame_num;
    frame->pc = cur_pc;
    frame->sp = cur_sp;
    frame->rel_pc = cur_pc;

    if (map_info != nullptr) {
      // 计算相对 PC（相对于 so 库基地址的偏移）
      frame->rel_pc = map_info->GetRelPc(cur_pc);
      frame->map_name = map_info->name();

      // ③ 获取/初始化 Elf 对象
      Elf* elf = map_info->GetElf(process_memory_, arch_);

      // ④ 尝试 DWARF CFI 回溯：恢复上一帧的寄存器
      bool finished = false;
      bool stepped = false;
      if (elf->valid()) {
        stepped = elf->Step(frame->rel_pc, regs_.get(),
                            process_memory_.get(), &finished,
                            &frame->is_signal_frame);
      }

      if (finished) {
        break;  // 到达栈底
      }

      if (!stepped) {
        // ⑤ DWARF 失败，尝试 FP-based 回溯作为 fallback
        if (!FpUnwind(cur_pc, cur_sp)) {
          break;  // 两种方式都失败，回溯终止
        }
      }
    } else {
      // PC 不在任何映射区域中——回溯终止
      break;
    }

    // ⑥ 安全检查：SP 必须单调递增（栈从高地址向低地址增长，回溯时 SP 应该递增）
    if (regs_->sp() <= cur_sp) {
      break;  // SP 没有增长，说明回溯出了问题
    }
  }
}
```

**稳定性架构师视角：** 这段代码揭示了回溯失败的几个关键原因：

1. **`map_info == nullptr`**：PC 不在任何已知的内存映射中——通常是 JIT 代码或者栈被破坏导致 PC 值错误。
2. **`elf->Step()` 返回 false**：DWARF CFI 回溯失败——可能是 `.eh_frame` 被 strip、CFI 信息不完整、或者栈上保存的寄存器值已被破坏。
3. **`regs_->sp() <= cur_sp`**：SP 没有递增，说明回溯进入了死循环（通常是 CFI 信息错误导致计算出了错误的 SP）。
4. **`frame_num >= max_frames_`**：达到最大帧数限制（默认 256 帧）——极深的递归调用可能超过此限制。

### 4.4 信号帧的特殊处理

当崩溃发生时，内核会在用户栈上构造一个 **sigframe**（参见 02 篇的 `handle_signal`）。这个 sigframe 不是正常的函数调用产生的栈帧——它是内核"插入"的。libunwindstack 需要识别这种特殊帧并正确处理。

**源码路径：** `system/unwinding/libunwindstack/RegsArm64.cpp`

```cpp
// system/unwinding/libunwindstack/RegsArm64.cpp
bool RegsArm64::StepIfSignalHandler(uint64_t elf_offset, Elf* elf,
                                     Memory* process_memory) {
  // 检查当前 PC 是否指向 VDSO 中的 sigreturn trampoline
  // 即 __kernel_rt_sigreturn
  uint64_t data;
  if (!process_memory->ReadFully(pc(), &data, sizeof(data))) {
    return false;
  }

  // ARM64 的 sigreturn 指令序列：
  // mov x8, #__NR_rt_sigreturn (0xd2801168)
  // svc #0                     (0xd4000001)
  if (data == 0xd4000001d2801168ULL) {
    // 这是一个信号帧！
    // sigframe 中的 ucontext 包含了信号到达时的完整寄存器快照
    // 从 sigframe 中恢复所有寄存器
    uint64_t sp = regs_[ARM64_REG_SP];
    // ucontext 在 sigframe 中的偏移
    uint64_t uc_offset = sp + offsetof(struct rt_sigframe, uc);
    // 从 ucontext.uc_mcontext 中读取所有寄存器
    ReadFromMcontext(process_memory, uc_offset);
    return true;
  }

  return false;
}
```

**稳定性架构师视角：** 信号帧的正确识别极为重要。如果 libunwindstack 未能识别出信号帧，它会尝试用普通的 DWARF CFI 规则去回溯 sigreturn trampoline——这必然失败，导致 backtrace 在信号处理函数处截断。Tombstone 中如果你看到 backtrace 在 `[vdso]` 或 `__restore_rt` 处结束，很可能就是信号帧识别失败。

## 5. 符号化工具链

### 5.1 为什么需要符号化

栈回溯的产物是一组 `(PC地址, so库路径)` 对。但一个十六进制地址（如 `pc 0x0004a2bc`）对人类毫无意义。**符号化（Symbolication）** 的目标是将地址翻译成源码级的信息：函数名、源文件名、行号。

```
符号化前：
  #00 pc 0x0004a2bc  /vendor/lib64/libcamera_hal.so

符号化后：
  #00 pc 0x0004a2bc  /vendor/lib64/libcamera_hal.so (CameraDevice::processRequest+188)
                                                       camera_device.cpp:342
```

### 5.2 ELF 中的符号信息层次

ELF 文件中有多个段承载不同粒度的符号信息：

| ELF 段 | 内容 | strip 后是否保留 | 提供的信息 |
| :--- | :--- | :--- | :--- |
| `.symtab` | 完整符号表（所有函数、全局变量） | **否** | 函数名 |
| `.dynsym` | 动态符号表（导出的和导入的符号） | **是**——动态链接必需 | 部分函数名（仅导出的） |
| `.debug_info` | DWARF 调试信息（类型、变量、编译单元） | 否 | 函数名、参数类型 |
| `.debug_line` | DWARF 行号信息（PC → 源文件:行号 映射） | 否 | 源文件名和行号 |
| `.gnu_debugdata` | 压缩的 MiniDebugInfo（LZMA 压缩的精简 `.symtab`） | **是**——Android 特有 | 函数名（包括 static 函数） |

**关键认知：** `strip` 操作会移除 `.symtab`、`.debug_info`、`.debug_line` 等纯调试段，但会保留 `.dynsym`（动态链接必需）和 `.gnu_debugdata`（Android 特有，见第 6 节）。这就是为什么 Release 版 so 库仍然能部分符号化——`.dynsym` 提供导出函数名，`.gnu_debugdata` 提供更多内部函数名。

### 5.3 addr2line

`addr2line` 是最常用的符号化工具。它读取 ELF 文件的 `.debug_line` 段，将一个相对地址翻译为源文件名和行号。

```bash
# 基本用法：-C 开启 C++ 名称反修饰，-f 显示函数名，-e 指定 ELF 文件
$ aarch64-linux-android-addr2line -Cfe libcamera_hal.so 0x0004a2bc
CameraDevice::processRequest(CaptureRequest*)
camera_device.cpp:342

# 如果 so 文件已 strip，需要使用未 strip 的版本
# 通常在编译输出中的 obj/ 目录下（NDK）或 symbols/ 目录下（系统构建）
$ aarch64-linux-android-addr2line -Cfe out/target/product/xxx/symbols/vendor/lib64/libcamera_hal.so 0x0004a2bc
```

**原理：** addr2line 解析 `.debug_line` 段中的行号程序（Line Number Program）——这是一个状态机，它的输入是一系列操作码（opcodes），输出是 `(PC地址, 文件名, 行号)` 的映射表。addr2line 在这个映射表中二分查找目标地址，得到对应的源文件和行号。

### 5.4 ndk-stack

`ndk-stack` 是 Android NDK 提供的便捷工具，它能直接解析 logcat 或 Tombstone 中的 backtrace，批量符号化所有帧。

```bash
# 从 logcat 输入
$ adb logcat | ndk-stack -sym /path/to/symbols/

# 从 Tombstone 文件输入
$ ndk-stack -sym /path/to/symbols/ -dump tombstone_00
```

ndk-stack 内部对每一帧调用 addr2line，并自动匹配 so 库名和符号目录中的对应文件。

### 5.5 llvm-symbolizer

`llvm-symbolizer` 是 LLVM 工具链提供的符号化工具，与 addr2line 功能类似但支持更多特性（如内联函数展开）。

```bash
# 将地址符号化，显示内联函数链
$ llvm-symbolizer --obj=libfoo.so --inlining=true 0x12345
std::__1::vector<int>::operator[](unsigned long)
include/c++/v1/vector:1432
 (inlined by) MyClass::processData()
my_class.cpp:87
```

**稳定性架构师视角：** `--inlining=true` 选项在排查优化编译（`-O2`/`-O3`）的崩溃时极其有用。优化编译器会大量内联小函数，Tombstone 中一个地址可能实际对应了三四层嵌套的内联调用。不展开内联信息，你看到的函数名可能是"外层包装函数"而不是真正出问题的"内层函数"。

### 5.6 符号化失败的常见原因

| 现象 | 原因 | 解决方案 |
| :--- | :--- | :--- |
| 函数名显示为 `<unknown>` | so 库被彻底 strip（`.dynsym` 中无该函数、无 `.gnu_debugdata`） | 使用未 strip 的 so 文件重新符号化 |
| 有函数名但无行号 | `.debug_line` 已被 strip | 使用带 debug info 的 so 文件 |
| 行号偏差（指向错误的源文件行） | 编译器优化导致指令重排 | 检查编译优化等级；使用 `-O0` 编译的 so 验证 |
| `addr2line` 输出 `??:0` | 地址不在任何编译单元范围内 | 确认 Tombstone 中的 `rel_pc` 计算是否正确；确认 so 版本匹配 |
| C++ 函数名显示为修饰后的形式（如 `_ZN7MyClass7doStuffEv`） | 未开启 demangle | 使用 `-C` 参数开启 C++ 名称反修饰 |

## 6. MiniDebugInfo

### 6.1 为什么 Release 版 so 需要保留部分符号

在 Android 的 Release 构建中，so 库会经过 `strip` 移除所有调试信息以减小体积。但这带来了一个严峻问题：**Tombstone 中的 backtrace 全是地址，无法直接判断崩溃位置**。

理想的做法是保留完整的 `.symtab`，但 `.symtab` 通常占 so 文件体积的 20%-40%，在存储寸土寸金的移动设备上不可接受。

Android 的解决方案是 **MiniDebugInfo**——将精简版的符号表用 LZMA 算法压缩后，存储在 ELF 的 `.gnu_debugdata` 段中。压缩后的体积通常只有原始 `.symtab` 的 5%-10%，但能覆盖大部分函数名（包括 static 函数和内部函数），极大改善了 Release 版本的 backtrace 可读性。

### 6.2 `.gnu_debugdata` 的结构

`.gnu_debugdata` 段的内容是一个 **LZMA 压缩的完整 ELF 文件**。这个嵌入的 ELF 文件只包含精简的 `.symtab` 和 `.eh_frame` 段（以及必要的 `.shstrtab`）。

```
原始 ELF (libfoo.so):
┌────────────────────────┐
│ .text (代码)            │
│ .rodata (只读数据)      │
│ .dynsym (动态符号)      │
│ .eh_frame (CFI 信息)    │
│ .gnu_debugdata ─────────┼──┐ LZMA 压缩的嵌入 ELF
│ ...                     │  │
└────────────────────────┘  │
                            ▼
                   嵌入的 Mini ELF:
                   ┌──────────────────┐
                   │ .symtab (精简版)  │  只包含函数符号（去掉了对象/文件符号）
                   │ .strtab (字符串表)│  函数名字符串
                   │ .eh_frame (可选)  │  补充的 CFI 信息
                   └──────────────────┘
```

### 6.3 libunwindstack 如何使用 MiniDebugInfo

libunwindstack 在解析 ELF 时，会检查是否存在 `.gnu_debugdata` 段。如果存在，将其解压为内存中的 ELF 文件，并用它来补充符号查找和 CFI 回溯。

**源码路径：** `system/unwinding/libunwindstack/Elf.cpp`

```cpp
// system/unwinding/libunwindstack/Elf.cpp
void Elf::InitGnuDebugdata(Memory* memory) {
  // 查找 .gnu_debugdata 段
  uint64_t offset, size;
  if (!interface_->GetGnuDebugdataInfo(&offset, &size)) {
    return;  // 没有 .gnu_debugdata 段
  }

  // 读取 .gnu_debugdata 段的原始数据
  std::vector<uint8_t> compressed(size);
  memory->ReadFully(offset, compressed.data(), size);

  // LZMA 解压
  std::vector<uint8_t> decompressed;
  if (!DecompressLzma(compressed, &decompressed)) {
    return;  // 解压失败
  }

  // 将解压后的数据作为一个新的 ELF 文件解析
  auto gnu_debugdata_memory = std::make_shared<MemoryBuffer>(
      decompressed.data(), decompressed.size());
  gnu_debugdata_interface_.reset(CreateInterfaceFromMemory(gnu_debugdata_memory));

  if (gnu_debugdata_interface_) {
    gnu_debugdata_interface_->Init();
  }
}
```

在符号查找时，libunwindstack 会先在主 ELF 的 `.dynsym` 中查找，找不到再去 `.gnu_debugdata` 的 `.symtab` 中查找：

```cpp
// system/unwinding/libunwindstack/Elf.cpp
bool Elf::GetFunctionName(uint64_t addr, SharedString* name,
                          uint64_t* func_offset) {
  // 先从主 ELF 的 .dynsym 查找
  if (interface_->GetFunctionName(addr, name, func_offset)) {
    return true;
  }

  // 再从 .gnu_debugdata 的 .symtab 查找
  if (gnu_debugdata_interface_ &&
      gnu_debugdata_interface_->GetFunctionName(addr, name, func_offset)) {
    return true;
  }

  return false;
}
```

### 6.4 MiniDebugInfo 的生成流程

Android 的构建系统在 `strip` so 库时自动生成 MiniDebugInfo：

```bash
# Android 构建系统中的 strip + MiniDebugInfo 生成流程（简化版）

# 1. 从完整 so 中提取精简的 .symtab（只保留函数符号）
llvm-objcopy --only-keep-debug libfoo.so libfoo.debug
llvm-objcopy -S --keep-section=.symtab --keep-section=.strtab \
    libfoo.debug libfoo.mini

# 2. 用 LZMA 压缩
xz -c libfoo.mini > libfoo.mini.xz

# 3. 将压缩数据作为 .gnu_debugdata 段添加到 stripped so 中
llvm-objcopy --strip-all \
    --add-section .gnu_debugdata=libfoo.mini.xz \
    libfoo.so libfoo.stripped.so
```

**稳定性架构师视角：** MiniDebugInfo 是 Android 在"体积"和"可调试性"之间的精妙平衡。有了它，即使是 Release 版本的系统库，Tombstone 中也能显示大部分函数名。但 MiniDebugInfo 有几个局限：

1. **不包含行号信息**：只能还原到函数名 + 偏移量，无法还原到源文件:行号。要获取行号，仍需使用未 strip 的 so 配合 addr2line。
2. **不包含内联信息**：优化编译的内联函数不在 MiniDebugInfo 的符号表中。
3. **LZMA 解压有 CPU 开销**：在 crash_dump 中解压 `.gnu_debugdata` 需要几毫秒到几十毫秒，在大量 so 库的进程中这些开销会累加。
4. **第三方 so 可能没有**：只有 Android 构建系统自动添加 MiniDebugInfo。第三方 SDK 提供的预编译 so 如果没有经过相同处理，就缺少 `.gnu_debugdata`。

当你在 Tombstone 中看到某些帧有函数名（来自 `.dynsym` 或 `.gnu_debugdata`）而某些帧只有地址，通常就是后者的 so 库缺少 MiniDebugInfo。

## 7. 回溯与符号化的常见失败模式

在实际的稳定性工作中，你会频繁遇到"backtrace 不完整"或"符号化失败"的情况。以下是一份系统化的失败模式清单：

| 失败现象 | 根因 | 排查方向 |
| :--- | :--- | :--- |
| backtrace 只有 1-2 帧就截断 | 崩溃发生在缺少 `.eh_frame` 且使用 `-fomit-frame-pointer` 的 so 中 | 检查该 so 是否包含 `.eh_frame`（`readelf -S libfoo.so`） |
| backtrace 中某一帧后出现 `<unknown>` 然后截断 | PC 跳转到了一个不在任何映射区域的地址（如跳转表损坏、函数指针野指针） | 检查前一帧的代码是否有间接调用（虚函数、函数指针） |
| backtrace 出现明显不合理的帧（如跳到了 libc 的 `malloc` 然后又到了 `libGLES`） | 栈被 buffer overflow 破坏，保存的 LR 值被覆盖为垃圾值 | 检查 backtrace 中"断裂点"附近的函数是否有栈上 buffer |
| 所有帧都只有地址没有函数名 | so 库被彻底 strip 且缺少 `.gnu_debugdata` | 获取未 strip 的 so 文件后离线符号化 |
| 函数名正确但行号偏移 | so 文件版本不匹配（build ID 不同） | 对比 Tombstone 中的 build-id 和本地 so 文件的 build-id |
| backtrace 在 `[vdso]` 处截断 | 信号帧识别失败 | 检查 libunwindstack 版本；可能是非标准的 signal trampoline |
| JIT 编译的 Java 帧显示为 `<anonymous:jit-cache>` | libunwindstack 未获取到 ART 的 JIT debug descriptor | 确认 crash_dump 是否成功读取了 ART 的 JIT 元数据 |

## 8. 稳定性实战案例

### 案例一：第三方 SDK 的 so 库缺少回溯信息导致 Tombstone "黑洞"

**现象：**

某应用集成了一个第三方图像处理 SDK（`libimgproc.so`），线上频繁出现 SIGSEGV 崩溃，但 Tombstone 的 backtrace 极度简短，几乎无法提供有用信息：

```
signal 11 (SIGSEGV), code 1 (SEGV_MAPERR), fault addr 0x0000000000000018

backtrace:
    #00 pc 0x000a3f40  /data/app/com.example.app/lib/arm64/libimgproc.so
    #01 pc 0x00052a90  /apex/com.android.runtime/lib64/bionic/libc.so (__start_thread+40)
```

只有 2 帧：崩溃帧（在 `libimgproc.so` 中，但没有函数名）和线程入口帧。中间的调用链完全丢失。

**分析思路：**

1. **backtrace 只有 2 帧且中间缺失**：说明从 `#00` 帧回溯到上一帧时就失败了，libunwindstack 无法恢复上一帧的寄存器状态。
2. **`#00` 帧没有函数名**：说明 `libimgproc.so` 既缺少 `.gnu_debugdata`（否则能显示函数名），`.dynsym` 中也没有该函数（说明它是内部函数，非导出函数）。
3. **检查 `libimgproc.so` 的 ELF 段**：

   ```bash
   $ readelf -S libimgproc.so | grep -E "eh_frame|debug|gnu_debug"
   # 输出为空——没有 .eh_frame、没有 .debug_frame、没有 .gnu_debugdata
   ```

4. **检查编译选项**：通过反汇编查看函数 prologue，发现没有 `stp x29, x30, [sp, ...]` 指令——确认使用了 `-fomit-frame-pointer`。

5. **结论**：该第三方 so 库被极度 strip（连 `.eh_frame` 都被删除），又使用了 `-fomit-frame-pointer`。两种回溯方式都无法工作。

**根因：**

第三方 SDK 的构建脚本使用了过于激进的 strip 命令：`llvm-strip --strip-all --remove-section=.eh_frame`。正常的 `strip` 命令不会删除 `.eh_frame`（因为它标记了 `SHF_ALLOC`），但 `--remove-section` 强制删除了。

**修复方案：**

1. **短期修复**：联系 SDK 提供方，要求其保留 `.eh_frame` 段。提供修改后的 strip 命令建议：

   ```bash
   # 正确的 strip 方式：保留 .eh_frame 和添加 .gnu_debugdata
   llvm-strip --strip-debug libimgproc.so
   # 或更精确地：
   llvm-objcopy --strip-debug --keep-section=.eh_frame libimgproc.so
   ```

2. **长期治理**：
   - 在 CI 中增加自动化检查脚本，对所有集成的第三方 so 库验证：
     - 是否包含 `.eh_frame` 段
     - 是否包含 `.gnu_debugdata` 段
     - 使用 `readelf -S` 自动扫描并在缺失时报警
   - 建立第三方 SDK 接入规范，明确要求提供带 `.eh_frame` 的 so 库和独立的符号文件。

3. **应急措施**（在 SDK 方修复之前）：要求 SDK 方提供未 strip 的 so 文件作为符号文件，用于离线符号化。结合 `fault addr = 0x18`（低地址，典型空指针解引用 + 偏移）和 `rel_pc = 0xa3f40`，使用 addr2line 离线还原：

   ```bash
   $ aarch64-linux-android-addr2line -Cfe libimgproc_unstripped.so 0xa3f40
   ImageProcessor::applyFilter(ImageBuffer*)
   image_processor.cpp:287
   ```


### 案例二：Build-ID 不匹配导致符号化行号偏移，误导排查方向

**现象：**

线上某版本出现 SIGSEGV 崩溃，Tombstone 中 backtrace 信息完整：

```
signal 11 (SIGSEGV), code 1 (SEGV_MAPERR), fault addr 0x0000000000000000
backtrace:
    #00 pc 0x0001b840  /data/app/com.example.app/lib/arm64/libnetwork.so (HttpClient::sendRequest+224)
    #01 pc 0x0001c100  /data/app/com.example.app/lib/arm64/libnetwork.so (HttpClient::execute+180)
    #02 pc 0x0001a320  /data/app/com.example.app/lib/arm64/libnetwork.so (NetworkManager::dispatch+96)
```

工程师使用 addr2line 将 `0x0001b840` 符号化到 `http_client.cpp:156`，这一行是一个完全无害的日志打印语句：

```cpp
// http_client.cpp:156
LOG_DEBUG("Sending request to %s", url.c_str());  // 这行怎么会崩溃？
```

花了两天排查这行"不可能崩溃"的代码，未果。

**分析思路：**

1. **"不可能崩溃"的代码行 = 符号化结果不可信**：当 addr2line 指向一行明显不可能崩溃的代码时，首先怀疑 so 文件版本不匹配。
2. **检查 Build-ID**：Tombstone（Android 12+）中会记录 so 库的 build-id：

   ```
   #00 pc 0x0001b840  /data/app/.../libnetwork.so (BuildId: 7a3f012c8b...)
   ```

   对比本地的 so 文件：

   ```bash
   $ readelf -n libnetwork.so | grep "Build ID"
   Build ID: a1b2c3d4e5...
   ```

3. **Build-ID 不匹配！** Tombstone 中的 build-id 是 `7a3f012c8b...`，而工程师使用的符号文件 build-id 是 `a1b2c3d4e5...`。两者属于不同的构建版本。
4. **根因**：该版本发布后进行了一次 hotfix（修改了几个源文件并重新编译），但符号文件服务器上只更新了 APK，未更新对应的符号文件。工程师使用的是 hotfix 前的符号文件。
5. **使用正确的符号文件重新符号化**：

   ```bash
   $ aarch64-linux-android-addr2line -Cfe libnetwork_correct.so 0x0001b840
   HttpClient::sendRequest(Request const&)
   http_client.cpp:162
   ```

   正确的行号是 162，这一行是 `request->connection->write(...)`，当 `connection` 为 NULL 时触发空指针解引用——完全合理。

**根因：**

符号文件版本与线上运行的 so 库版本不一致。由于 hotfix 修改了源文件，编译器生成的指令布局发生了变化，导致同一个相对 PC 地址在不同版本中对应不同的源码行。

**修复方案：**

1. **短期修复**：从正确的构建产物中获取匹配的符号文件，重新符号化确认崩溃根因——`connection` 在特定错误路径下未正确初始化，添加 NULL 检查即可。

2. **长期治理**：
   - **符号文件管理体系**：建立自动化的符号文件归档流水线——每次构建（包括 hotfix）自动将 so 文件的 build-id 和对应的未 strip 符号文件上传到符号服务器。
   - **符号化工具自动匹配 Build-ID**：在内部的 Tombstone 分析平台中，根据 Tombstone 中的 build-id 自动从符号服务器拉取对应版本的符号文件，避免人工匹配出错。
   - **CI 强制校验**：发布流程中增加卡点——检查符号服务器上是否已存在当前 build-id 对应的符号文件，缺失则阻止发布。

## 9. 总结

本篇从"原理 → 实现 → 工具 → 失败模式"四个维度，系统拆解了栈回溯与符号化的完整技术栈。

核心要点：

1. **栈回溯有两种方式**——FP-based（快但不可靠）和 DWARF CFI（慢但可靠）。Android 的 libunwindstack 默认使用 DWARF CFI，FP 作为 fallback。
2. **`-fomit-frame-pointer` 破坏 FP 链表**，但只要 `.eh_frame` 段保留，DWARF 回溯仍然有效。真正的"回溯黑洞"是既删除了 `.eh_frame` 又省略了 FP 的 so 库。
3. **DWARF CFI 通过 `.eh_frame` 段中的指令描述每个 PC 位置的寄存器保存规则**——这些规则让回溯器能在程序的任意位置恢复上一帧的状态，包括信号处理函数的异步中断点。
4. **libunwindstack 是 Android 回溯的核心引擎**，Regs/Maps/Elf 三大类协作完成"寄存器读取 → PC 到 so 映射 → DWARF CFI 执行 → 逐帧回溯"的完整流程。
5. **符号化依赖 ELF 中的多层信息**——`.dynsym`（strip 后保留）→ `.gnu_debugdata`（Android 特有的压缩符号）→ `.symtab` + `.debug_line`（完整调试信息）。Release 版本只能还原函数名，要获取行号需要未 strip 的符号文件。
6. **MiniDebugInfo（`.gnu_debugdata`）是 Android 在体积和可调试性之间的精妙平衡**——LZMA 压缩的精简符号表，让 Release 版 so 库的 backtrace 也能显示函数名。
7. **Build-ID 是符号化正确性的校验码**——so 文件版本不匹配是符号化行号偏移的头号原因，必须建立 build-id 到符号文件的自动映射机制。
