# 07-检测工具体系：ASan / HWASan / MTE / GWP-ASan

上一篇我们拆解了 Tombstone 的逐字段解读方法，能够从崩溃现场"倒推"问题方向。但有一个残酷的现实：**你在 Tombstone 中看到的崩溃点，往往不是真正的根因点。** 一个 Use-After-Free 可能在内存释放后的第 1000 次分配时才被覆盖，又在被覆盖后的第 500 次访问时才恰好触发 SIGSEGV——此时 backtrace 中的函数和真正释放内存的函数可能完全不相关。

这就是内存错误的"冰山模型"：水面上那 10% 是你看到的崩溃，水面下那 90% 是沉默的内存腐蚀（silent corruption）。要想从根本上解决 Native 层的内存安全问题，你需要**在错误发生的瞬间就捕获它**——而不是等到它在不相关的地方爆炸。

本篇将系统讲解 Android 生态中四大内存错误检测工具——ASan、HWASan、MTE、GWP-ASan——的原理、能力、开销和适用场景，并给出开发/测试/线上三阶段的选型矩阵。

---

## 1. 为什么需要检测工具

### 1.1 内存错误的"冰山模型"

内存错误（memory corruption）是 Native Crash 中最隐蔽、最致命的一类。它们之所以难以排查，根本原因在于**错误的发生时刻和崩溃的表现时刻之间存在巨大的时间差和空间差**。

考虑一个经典的 Use-After-Free 场景：

```
时刻 T1: Thread-A 释放了对象 X 的内存（free(X)）
          ↓ 内存进入 allocator 的 free list
时刻 T2: Thread-B 分配了新对象 Y，恰好复用了 X 的内存
          ↓ Y 的数据覆盖了 X 的旧内容
时刻 T3: Thread-C 仍然持有 X 的指针，通过 X->field 访问
          ↓ 实际读到的是 Y 的数据（错误数据，但不崩溃）
时刻 T4: Thread-C 用错误数据计算出一个非法地址，解引用
          ↓ SIGSEGV！但 backtrace 指向 Thread-C 的代码
          ↓ 真正的 bug 在 Thread-A（T1 时刻的提前释放）
```

**冰山模型的核心观点：**

```
                      ┌─────────────┐
                      │  可见的崩溃   │  ← 10%：SIGSEGV/SIGABRT，有 Tombstone
                      │  (水面以上)   │     崩溃点 ≠ 根因点
              ~~~~~~~~│~~~~~~~~~~~~~│~~~~~~~~~~~~~~~~~~~~~~~~
                      │  沉默的腐蚀   │  ← 60%：数据被悄悄改写，程序行为异常
                      │  (水面以下)   │     但未触发崩溃，表现为"偶发逻辑 bug"
                      │             │
                      │  完全无症状   │  ← 30%：越界读了相邻对象的 padding 区域
                      │             │     恰好数值合理，程序"正常"运行
                      └─────────────┘
```

这意味着：你在 APM 后台看到的 NE 数量，只是实际内存错误数量的冰山一角。那些没崩溃的内存错误，可能以"偶发数据异常"、"间歇性逻辑错误"、"仅特定机型复现的 UI 异常"等形式长期潜伏。

### 1.2 为什么仅靠 Tombstone 不够

Tombstone 是"死后验尸报告"，它只能告诉你**崩溃瞬间**的状态：

| Tombstone 能告诉你的 | Tombstone 不能告诉你的 |
| :--- | :--- |
| 崩溃时访问了什么地址（fault addr） | 这块内存是什么时候被释放的 |
| 崩溃时的调用栈（backtrace） | 是谁释放的（freed-by backtrace） |
| 崩溃时各线程的状态 | 内存被释放后被谁复用了 |
| 崩溃时的内存映射（maps） | 越界写入了多少字节、覆盖了什么 |

检测工具的价值在于：**它们在错误发生的瞬间就触发报告**，而不是等到错误最终导致崩溃。ASan 能在 `free()` 后第一次访问就报 UAF；MTE 能在越界写入的那一条指令就触发异常。这将排查效率从"大海捞针"提升到"精确制导"。

### 1.3 检测工具的分类视角

从实现层面，内存错误检测工具分为三大类：

| 类型 | 代表 | 原理 | 特点 |
| :--- | :--- | :--- | :--- |
| **编译时插桩** | ASan, HWASan | 编译器在每次内存访问前/后插入检查代码 | 检测能力强，但性能开销大 |
| **硬件辅助检测** | MTE | CPU 硬件在每次内存访问时自动检查标签 | 零 CPU 开销，但需要新硬件 |
| **运行时采样** | GWP-ASan | 概率性地将 malloc/free 替换为带 Guard Page 的版本 | 开销极低，可用于线上，但检测率低 |

接下来逐一深入。

## 2. ASan（AddressSanitizer）

### 2.1 ASan 是什么

ASan（AddressSanitizer）是 Google 开发的编译时内存错误检测工具，最初集成在 LLVM/Clang 中（2012 年），是目前最成熟、检测能力最全面的内存安全工具。它通过**编译时插桩 + 运行时库**的方式，在每一次内存访问前检查目标地址是否合法。

**核心源码路径：** `compiler-rt/lib/asan/`

**一句话原理：** ASan 维护了一块与应用内存 1:8 映射的 **Shadow Memory**，每 8 字节应用内存对应 1 字节 Shadow 值。编译器在每次内存访问前插入代码，读取对应的 Shadow 值来判断该访问是否合法。

### 2.2 Shadow Memory 机制

Shadow Memory 是 ASan 的核心数据结构。理解它，就理解了 ASan 的一切。

**映射关系：**

```
应用内存地址:     [Addr]
Shadow 地址:     [Addr >> 3] + ShadowOffset

即：每 8 字节应用内存 → 1 字节 Shadow

Shadow 值的含义：
  0           → 8 字节全部可访问（正常内存）
  1-7         → 前 N 字节可访问（部分可访问，常见于结构体尾部对齐）
  负值(0x80+) → 不可访问（各种错误类型用不同的 magic number 标记）
```

**不同类型的 Shadow 标记值：**

```c
// compiler-rt/lib/asan/asan_internal.h

static const u8 kAsanHeapLeftRedzoneMagic    = 0xfa;  // 堆 left redzone
static const u8 kAsanHeapFreeMagic           = 0xfd;  // 已释放的堆内存
static const u8 kAsanStackLeftRedzoneMagic   = 0xf1;  // 栈 left redzone
static const u8 kAsanStackMidRedzoneMagic    = 0xf2;  // 栈变量之间的 redzone
static const u8 kAsanStackRightRedzoneMagic  = 0xf3;  // 栈 right redzone
static const u8 kAsanStackAfterReturnMagic   = 0xf5;  // 函数返回后的栈内存
static const u8 kAsanGlobalRedzoneMagic      = 0xf9;  // 全局变量 redzone
```

**插桩代码示例：** 编译器将一次简单的内存访问：

```c
*addr = value;  // 原始的 8 字节写入
```

转换为：

```c
// 编译器插入的检查代码（伪码）
shadow_addr = (addr >> 3) + kShadowOffset;
shadow_value = *shadow_addr;
if (shadow_value != 0) {
    // 如果 shadow 不为 0，需要进一步检查
    if ((addr & 7) + access_size > shadow_value) {
        // 非法访问！调用 ASan 报错函数
        __asan_report_store8(addr);
    }
}
*addr = value;  // 通过检查后执行原始访问
```

**源码路径：** `compiler-rt/lib/asan/asan_mapping.h`

```c
// compiler-rt/lib/asan/asan_mapping.h

// Shadow Memory 的地址计算
#define MEM_TO_SHADOW(mem) (((mem) >> ASAN_SHADOW_SCALE) + (ASAN_SHADOW_OFFSET))

// ARM64 上的 Shadow 配置
// ASAN_SHADOW_SCALE = 3 (即 1:8 映射)
// ASAN_SHADOW_OFFSET 由运行时确定，通常为 0x0000100000000000
```

**稳定性架构师视角：** Shadow Memory 的 1:8 映射意味着 ASan 需要额外约 1/8 的虚拟地址空间来存放 Shadow。在 32 位进程中，这会严重压缩可用地址空间（4GB 中约 512MB 被 Shadow 占用），可能导致部分大内存应用直接 OOM。这也是为什么 ASan 在 ARM64 上表现远好于 ARM32 的原因之一——64 位地址空间充裕，Shadow 开销不是瓶颈。

### 2.3 检测能力矩阵

ASan 能检测的内存错误类型几乎覆盖了所有常见场景：

| 错误类型 | ASan 检测原理 | 典型场景 |
| :--- | :--- | :--- |
| **Heap Buffer Overflow** | 在堆分配的内存块两侧插入 Redzone（标记为 0xfa），越界访问命中 Redzone | `memcpy(buf, src, too_large_size)` |
| **Stack Buffer Overflow** | 在栈变量之间插入 Redzone（标记为 0xf1-0xf3） | `char buf[10]; buf[20] = 'x';` |
| **Global Buffer Overflow** | 在全局变量之间插入 Redzone（标记为 0xf9） | 全局数组越界 |
| **Use-After-Free** | `free()` 后将内存的 Shadow 标记为 0xfd，延迟回收到 quarantine | `free(p); *p = 1;` |
| **Use-After-Return** | 开启 `detect_stack_use_after_return` 后，函数返回时将栈帧标记为 0xf5 | 返回局部变量的指针 |
| **Double-Free** | 在 ASan 的 allocator 中维护分配状态，二次释放时检查状态 | `free(p); free(p);` |
| **Memory Leak** | 进程退出时扫描所有未释放的内存块（需开启 `detect_leaks`） | 未 free 的 malloc |

**Quarantine 机制：** ASan 检测 Use-After-Free 的关键是 **Quarantine（隔离区）**。当 `free()` 被调用时，ASan 不会立即将内存归还给分配器，而是将其放入一个 FIFO 队列（quarantine）。在 quarantine 中的内存，其 Shadow 被标记为 0xfd（freed），任何访问都会被检测到。只有当 quarantine 满时，最早的内存才会被真正归还。

```c
// compiler-rt/lib/asan/asan_allocator.cpp

void Deallocate(void *ptr) {
    // 1. 将内存的 Shadow 标记为 freed
    PoisonShadow(ptr, size, kAsanHeapFreeMagic);

    // 2. 放入 quarantine 而非直接归还
    quarantine.Put(ptr, size);

    // 3. 如果 quarantine 超过阈值，回收最老的块
    while (quarantine.GetSize() > max_quarantine_size) {
        void *old = quarantine.Pop();
        // 清除 Shadow 标记，归还给底层分配器
        UnpoisonShadow(old, old_size);
        RealFree(old);
    }
}
```

### 2.4 性能开销

ASan 的强大检测能力是以性能为代价的：

| 开销维度 | 量级 | 原因 |
| :--- | :--- | :--- |
| **CPU** | ~2x 减速 | 每次内存访问都要额外执行 Shadow 检查（3-5 条指令） |
| **内存** | ~3x 膨胀 | Shadow Memory（1/8）+ Redzone（每个分配块两侧各 16-256 字节）+ Quarantine |
| **代码体积** | ~2x 增长 | 插桩代码增加了二进制大小 |

这样的开销使得 ASan **只适合开发和测试阶段**，不可用于线上生产环境。

### 2.5 在 Android 上使用 ASan

#### 方式一：用 `wrap.sh` 为单个应用启用

Android NDK 提供了 `wrap.sh` 机制，可以在不修改系统的情况下为单个应用启用 ASan：

```bash
# 1. 编译 NDK 代码时加上 ASan 标志
# Application.mk 或 CMakeLists.txt
APP_STL := c++_shared
APP_CFLAGS := -fsanitize=address -fno-omit-frame-pointer
APP_LDFLAGS := -fsanitize=address

# 2. 创建 wrap.sh（放入 APK 的 lib/<abi>/ 目录）
#!/system/bin/sh
HERE="$(cd "$(dirname "$0")" && pwd)"
export ASAN_OPTIONS=detect_leaks=0:halt_on_error=0:print_legend=0
export LD_PRELOAD=$HERE/libclang_rt.asan-aarch64-android.so
exec "$@"

# 3. 在 AndroidManifest.xml 中启用 wrap.sh
# android:extractNativeLibs="true"
# 以及 debug 模式：android:debuggable="true"
```

#### 方式二：全系统 ASan（用于系统级调试）

```bash
# 在 AOSP 编译时为整个系统启用 ASan
# 修改 BoardConfig.mk
SANITIZE_TARGET := address

# 或者针对特定模块
LOCAL_SANITIZE := address
```

**稳定性架构师视角：** ASan 的 `wrap.sh` 方式在 Android 上有一个重要限制——它只能检测应用自身 NDK 代码中的内存错误，无法检测系统库（如 `libbinder.so`）中的问题，因为系统库没有被 ASan 编译。如果你需要检测系统库的内存错误，必须使用全系统 ASan 构建或转向 HWASan。

## 3. HWASan（Hardware-assisted AddressSanitizer）

### 3.1 HWASan 是什么

HWASan（Hardware-assisted AddressSanitizer）是 ASan 的进化版，它利用 ARM64 处理器的 **TBI（Top Byte Ignore）** 特性，将内存标签编码在指针的高位字节中，从而大幅降低内存开销。

**核心源码路径：** `compiler-rt/lib/hwasan/`

**一句话原理：** HWASan 在每次 `malloc()` 时，为分配的内存块随机生成一个 4-bit 标签（tag），同时将该标签写入返回指针的高 8 位。每次内存访问时，硬件自动忽略指针高 8 位（TBI），而 HWASan 插桩代码检查指针中的标签是否与内存中的标签匹配。

### 3.2 TBI（Top Byte Ignore）原理

ARM64 架构中，虚拟地址虽然是 64 位，但实际上只有低 48 位（或 52 位）用于地址翻译。高 8 位（bit 56-63）在默认配置下会被 MMU 忽略——这就是 TBI（Top Byte Ignore）特性。

```
64-bit 指针布局（启用 TBI）：

  63    56 55                                               0
  ┌──────┐┌───────────────────────────────────────────────────┐
  │ TAG  ││              实际虚拟地址（48 位有效）                │
  │(8bit)││                                                   │
  └──────┘└───────────────────────────────────────────────────┘
     ↑                          ↑
     │                          └── MMU 用这部分做地址翻译
     └── MMU 忽略这 8 位，但软件可以自由使用
```

HWASan 利用了这个"免费"的 8 位空间来存储内存标签：

```
指针: [TAG=0xA3] [0x0000007f12340000]
内存: 地址 0x7f12340000 处的 Shadow 中也存了 TAG=0xA3

访问时：
  1. CPU 用低 48 位做地址翻译（TBI，忽略高 8 位）→ 正常访问
  2. HWASan 插桩代码比较 指针中的 TAG(0xA3) vs 内存 Shadow 中的 TAG
     - 匹配 → 合法访问
     - 不匹配 → 报错！
```

### 3.3 Shadow Memory 对比：ASan vs HWASan

| 维度 | ASan | HWASan |
| :--- | :--- | :--- |
| Shadow 映射比 | 1:8（8 字节应用内存 → 1 字节 Shadow） | 1:16（16 字节应用内存 → 1 字节 Shadow） |
| Shadow 存储内容 | 可访问状态（0=OK, 负值=poison） | 4-bit 标签值 |
| 内存开销 | ~3x | ~1.15x（Shadow 更小，无需大 Redzone） |
| CPU 开销 | ~2x | ~1.5-2x |
| Redzone 大小 | 每个分配块两侧 16-256 字节 | 不需要 Redzone（通过标签不匹配检测） |

**源码路径：** `compiler-rt/lib/hwasan/hwasan_allocator.cpp`

```c
// compiler-rt/lib/hwasan/hwasan_allocator.cpp

void *HwasanAllocate(uptr size, uptr alignment) {
    // 1. 调用底层分配器分配内存
    void *ptr = allocator.Allocate(size, alignment);

    // 2. 生成随机标签
    tag_t tag = GenerateRandomTag();

    // 3. 将标签写入内存的 Shadow
    TagMemoryAligned((uptr)ptr, size, tag);

    // 4. 将标签编码到指针的高 8 位
    ptr = AddTagToPointer(ptr, tag);

    return ptr;  // 返回带标签的指针
}

void HwasanDeallocate(void *tagged_ptr) {
    // 1. 从指针中提取标签和实际地址
    tag_t old_tag = GetTagFromPointer(tagged_ptr);
    void *ptr = UntagPointer(tagged_ptr);

    // 2. 生成新的随机标签，标记已释放的内存
    tag_t new_tag = GenerateRandomTag();
    TagMemoryAligned((uptr)ptr, size, new_tag);

    // 3. 释放内存
    allocator.Deallocate(ptr);
}
```

**关键洞察：** HWASan 在 `free()` 时会给内存打上新的随机标签。之后如果有人通过旧指针（带旧标签）访问，标签不匹配就会被检测到。这比 ASan 的 quarantine 方案更优雅——ASan 需要延迟回收内存才能检测 UAF，而 HWASan 即使内存被重新分配，只要新标签不同（概率 255/256），就能检测到。

### 3.4 HWASan vs ASan 能力对比

| 能力 | ASan | HWASan | 说明 |
| :--- | :--- | :--- | :--- |
| Heap Buffer Overflow | ✅ | ✅ | ASan 靠 Redzone，HWASan 靠标签边界 |
| Stack Buffer Overflow | ✅ | ✅ | HWASan 给栈变量也打标签 |
| Use-After-Free | ✅ | ✅ | HWASan 不依赖 quarantine，检测窗口更长 |
| Use-After-Return | ✅（需额外配置） | ✅ | HWASan 默认支持 |
| Double-Free | ✅ | ✅ | |
| Wild pointer (arbitrary) | ❌ | ✅（概率性） | 任意地址的标签有 1/16 概率碰巧匹配 |
| 内存泄漏检测 | ✅ | ❌ | HWASan 不提供 leak 检测 |
| 32-bit 支持 | ✅ | ❌ | TBI 是 ARM64 特性，32 位不可用 |

**稳定性架构师视角：** HWASan 的最大优势在于**内存开销低**（~15% vs ASan 的 ~200%）。这使得 HWASan 可以用于更接近真实环境的测试：跑完整的 monkey test、stress test，而不会因为 OOM 导致测试环境本身崩溃。Google 的 Android 持续集成测试（CTS/VTS）已经在使用 HWASan 构建来进行日常测试。

### 3.5 在 Android 上使用 HWASan

```bash
# AOSP 全系统 HWASan 构建
# 修改 BoardConfig.mk
SANITIZE_TARGET := hwaddress

# 针对单个模块
LOCAL_SANITIZE := hwaddress

# HWASan 需要 ARM64 设备和较新的内核（支持 TBI）
# Android 10+ 开始正式支持 HWASan 构建
```

HWASan 不支持通过 `wrap.sh` 为单个应用启用——它需要整个 system image 都使用 HWASan 编译，因为标签需要在整个进程的所有 so 之间保持一致。

## 4. MTE（Memory Tagging Extension）

### 4.1 MTE 是什么

MTE（Memory Tagging Extension）是 ARM v8.5-A 架构引入的**硬件级内存标签**机制。如果说 HWASan 是"软件打标签 + 软件检查"，那 MTE 就是"软件打标签 + 硬件检查"——CPU 在每次内存访问时自动比较指针标签和内存标签，**零额外 CPU 指令开销**。

**核心源码路径：** `bionic/libc/`（Android 的 MTE 集成在 Bionic C 库的内存分配器中）

**一句话原理：** MTE 在硬件层面为每 16 字节内存分配一个 4-bit 标签，存储在专用的标签内存（Tag Memory）中。malloc 时给内存和指针打上相同标签，free 时更换标签。CPU 在每次内存访问时自动比对标签，不匹配则触发同步异常或异步报告。

### 4.2 硬件标签机制

MTE 引入了两类新的 ARM 指令和一块独立的标签存储空间：

```
MTE 的三层机制：

┌─────────────────────────────────────────────────────────┐
│ 1. Tag Memory（硬件管理，对软件透明）                       │
│    每 16 字节物理内存 → 4-bit 标签                         │
│    即额外 ~3% 的内存开销用于存储标签                         │
│                                                          │
│ 2. 新指令                                                 │
│    IRG：生成随机标签并写入指针高位                            │
│    STG：将指针中的标签写入 Tag Memory                        │
│    LDG：从 Tag Memory 读取标签                              │
│    ADDG/SUBG：带标签的指针算术                               │
│                                                          │
│ 3. 自动检查（硬件执行，无指令开销）                           │
│    每次 Load/Store 时，CPU 自动比较：                        │
│    指针高 4 位的标签 vs Tag Memory 中对应地址的标签            │
│    不匹配 → 触发 Tag Check Fault                            │
└─────────────────────────────────────────────────────────┘
```

**MTE 的标签位宽为 4-bit（值范围 0x0-0xF）**，编码在指针的 bit 59-56 中（与 HWASan 使用的 bit 63-56 不同，MTE 只用 4 位）。这意味着标签碰撞概率为 1/16（6.25%），虽然不如 HWASan 的 1/256，但考虑到零 CPU 开销，这是极佳的权衡。

### 4.3 同步模式 vs 异步模式

MTE 提供两种运行模式，适用于不同场景：

| 模式 | 行为 | 开销 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **Sync（同步模式）** | 标签不匹配时立即触发同步异常（类似 SIGSEGV），精确定位到出错指令 | CPU ~3-5% | 开发/测试：需要精确的错误位置 |
| **Async（异步模式）** | 标签不匹配时仅设置标志位，不立即异常，定期由内核检查并报告 | CPU ~1-2% | 线上/性能测试：需要最低开销 |

**同步模式的价值：** 崩溃时的 PC 精确指向出错的 Load/Store 指令，backtrace 完全可靠。这是调试内存错误的理想模式。

**异步模式的价值：** 几乎零性能影响，但报告可能延迟，且 PC 不一定精确指向出错指令。适合在线上长期开启，以概率性地发现内存错误。

**Android 的 MTE 配置：** `bionic/libc/` 中的 Scudo 分配器集成了 MTE 支持：

```c
// bionic/libc/bionic/scudo_wrapper.cpp（简化）

void *scudo_malloc(size_t size) {
    void *ptr = scudo::allocate(size);

#if defined(__aarch64__) && defined(USE_MTE)
    if (mte_enabled) {
        // 1. 生成随机标签（IRG 指令）
        ptr = __arm_mte_create_random_tag(ptr, 0);
        // 2. 将标签写入 Tag Memory（STG 指令），覆盖分配范围
        __arm_mte_set_tag(ptr, size);
    }
#endif

    return ptr;  // 返回带标签的指针
}

void scudo_free(void *ptr) {
#if defined(__aarch64__) && defined(USE_MTE)
    if (mte_enabled) {
        // 生成新的随机标签，覆盖旧标签
        // 之后通过旧指针访问 → 标签不匹配 → 检测到 UAF
        void *retagged = __arm_mte_create_random_tag(ptr, 0);
        __arm_mte_set_tag(retagged, size);
    }
#endif

    scudo::deallocate(ptr);
}
```

### 4.4 Android 的 MTE 支持状态

| Android 版本 | MTE 支持状态 |
| :--- | :--- |
| Android 12 (S) | 首次引入 MTE 支持，开发者选项中可启用 |
| Android 13 (T) | per-app MTE 配置（在 `AndroidManifest.xml` 中声明） |
| Android 14 (U) | 系统级 MTE 配置改进，支持 async 模式全局启用 |
| Android 15+ | 推进 MTE 作为默认启用选项（在支持的硬件上） |

**在 App 中启用 MTE（Android 13+）：**

```xml
<!-- AndroidManifest.xml -->
<application
    android:memtagMode="sync">  <!-- 可选值: off / async / sync -->
    ...
</application>
```

**在系统级启用 MTE：**

```bash

# 启动参数方式（bootconfig）

```

**稳定性架构师视角：** MTE 是内存安全的"终极方案"——硬件级零 CPU 开销、实时检测、无需重新编译应用代码（Bionic 层已集成）。但它有两个现实限制：(1) 需要 ARMv8.5+ 的 SoC，目前只有部分旗舰处理器（如 Google Tensor、部分 Dimensity 系列）支持；(2) 4-bit 标签的碰撞率为 1/16，即约 6% 的内存错误可能逃逸检测。即便如此，对于长时间运行的进程（如 system_server），累积的检测概率会远高于单次。

## 5. GWP-ASan

### 5.1 GWP-ASan 是什么

GWP-ASan（Google's "sampling" allocator for detecting memory safety bugs）是**唯一可以在线上生产环境使用**的内存错误检测工具。它的设计哲学是：不求检测每一次内存错误，但求以极低的开销在海量用户中采样到足够多的错误。

**核心源码路径：** `bionic/libc/bionic/gwp_asan_wrappers.cpp`

**一句话原理：** GWP-ASan 以一定的采样率（如每 1000 次 malloc 中有 1 次）将 `malloc/free` 替换为带有 Guard Page 保护的特殊版本。被采样的内存分配前后各有一个不可访问的 Guard Page，任何越界访问都会立即触发 SIGSEGV。

### 5.2 Guard Page 机制

GWP-ASan 的核心检测机制基于**虚拟内存页保护（mprotect）**：

```
普通 malloc 的内存布局：
┌──────────────────────────────────────┐
│ ... │ chunk_A │ chunk_B │ chunk_C │ ...│   ← 分配连续排列
└──────────────────────────────────────┘
    chunk_A 越界写入 chunk_B → 无法检测（直到 B 被使用时出错）

GWP-ASan 采样分配的内存布局：
┌───────────┬──────────┬───────────┬──────────┬───────────┐
│ Guard Page│ chunk_A  │ Guard Page│ chunk_B  │ Guard Page│
│ (不可访问) │ (正常)   │ (不可访问) │ (正常)   │ (不可访问) │
│ PROT_NONE │          │ PROT_NONE │          │ PROT_NONE │
└───────────┴──────────┴───────────┴──────────┴───────────┘
    chunk_A 越界 1 字节 → 触及 Guard Page → 立即 SIGSEGV！
```

**Use-After-Free 的检测：** 当采样分配的内存被 `free()` 时，GWP-ASan 不会将内存归还给系统，而是使用 `mprotect(ptr, size, PROT_NONE)` 将整个分配区域标记为不可访问。之后任何通过旧指针的访问都会触发 SIGSEGV。

### 5.3 核心源码解析

```c
// bionic/libc/bionic/gwp_asan_wrappers.cpp

// GWP-ASan 的初始化
void SetupGwpAsan(libc_globals* globals) {
    // 读取配置（采样率、最大分配数等）
    android_mallopt_gwp_asan_options_t options;
    options.enabled = true;
    options.sample_rate = GetGwpAsanSampleRate();  // 默认 1/1000
    options.max_allocs = kMaxSimultaneousAllocations;  // 默认 32

    gwp_asan::options::initOptions(
        options.sample_rate,
        options.max_allocs,
        /*installSignalHandlers=*/true);

    // 替换 malloc/free 为带采样的版本
    globals->malloc_dispatch = &gwp_asan_dispatch;
}
```

GWP-ASan 的分配池是预先通过 `mmap` 分配的一块大内存区域，分为多个 slot：

```c
// compiler-rt/lib/gwp_asan/guarded_pool_allocator.cpp

void *GuardedPoolAllocator::allocate(size_t Size) {
    // 1. 采样检查：每 N 次 malloc 只有 1 次进入 GWP-ASan
    if (--thread_local_counter > 0)
        return nullptr;  // 不采样，走正常 malloc
    thread_local_counter = sample_rate;  // 重置计数器

    // 2. 从预分配的 slot 池中找一个空闲 slot
    size_t SlotIdx = reserveSlot();
    if (SlotIdx == kInvalidSlotIdx)
        return nullptr;  // 池满，走正常 malloc

    // 3. 将 slot 对应的内存页设为可读写
    uintptr_t SlotAddr = slotToAddr(SlotIdx);
    mprotect((void*)getPageAddr(SlotAddr), getPageSize(), PROT_READ | PROT_WRITE);

    // 4. 右对齐分配（使越界写入更容易触及右侧 Guard Page）
    uintptr_t Ptr = alignUp(SlotAddr + getPageSize() - Size, Alignment);

    // 5. 记录分配时的 backtrace（用于报告）
    AllocationMetadata[SlotIdx].AllocationTrace = getBacktrace();
    AllocationMetadata[SlotIdx].RequestedSize = Size;

    return (void*)Ptr;
}

void GuardedPoolAllocator::deallocate(void *Ptr) {
    size_t SlotIdx = addrToSlot((uintptr_t)Ptr);

    // 1. 记录释放时的 backtrace
    AllocationMetadata[SlotIdx].DeallocationTrace = getBacktrace();

    // 2. 将内存页设为不可访问（检测 UAF）
    mprotect((void*)getPageAddr(Ptr), getPageSize(), PROT_NONE);

    // 3. 不立即释放 slot，保留元数据用于后续报告
    // slot 会在池压力较大时被回收
}
```

### 5.4 采样率与配置

GWP-ASan 的采样率决定了检测概率和性能开销之间的平衡：

| 参数 | 默认值 | 说明 |
| :--- | :--- | :--- |
| `sample_rate` | 1/1000 ~ 1/5000 | 每 N 次 malloc 有 1 次被 GWP-ASan 拦截 |
| `max_allocs` | 32 | GWP-ASan 池中最多同时存在的受保护分配数 |
| 内存开销 | ~数百 KB | `max_allocs × 2 × page_size`（每个 slot 两个 Guard Page） |
| CPU 开销 | < 1% | 绝大多数 malloc 只多了一次计数器递减和比较 |

**Android 应用中的配置（Android 11+）：**

```xml
<!-- AndroidManifest.xml -->
<application
    android:gwpAsanMode="always">  <!-- always / never -->
    ...
</application>
```

系统进程（如 `system_server`、`zygote`）默认开启 GWP-ASan。应用默认在 Android 14+ 对可调试应用启用。

### 5.5 GWP-ASan 报告格式

当 GWP-ASan 检测到错误时，它会生成一份比普通 Tombstone 更丰富的报告：

```
*** GWP-ASan detected use-after-free ***

Crash occurred at address 0x7a3f010040 in allocation at 0x7a3f010000

Allocation of size 64 at:
    #0  malloc
    #1  CameraDevice::allocateBuffer(int)
    #2  CameraDevice::configureStreams(...)

Deallocation at:
    #0  free
    #1  CameraDevice::releaseBuffer(int)
    #2  CameraDevice::closeStream(...)

Crash backtrace:
    #0  CameraDevice::processRequest(...)
    #1  RequestThread::threadLoop()
```

**GWP-ASan 报告的独特价值：** 它同时提供了**分配时堆栈**、**释放时堆栈**、**崩溃时堆栈**三份 backtrace。这是普通 Tombstone 无法提供的——有了释放时堆栈，你可以直接定位到"是谁提前释放了内存"，将排查效率提升一个数量级。

**稳定性架构师视角：** GWP-ASan 的检测率看似很低（只有被采样到的分配才能被保护），但考虑到线上的用户量，效果非常可观。假设采样率为 1/1000，线上日活用户 100 万，每个用户每天触发 10 万次 malloc，那么每天有 `100万 × 10万 / 1000 = 1000亿` 次受保护的分配。只要 bug 有一定的触发概率，GWP-ASan 几乎一定能在线上采样到。这就是"以量换率"的策略。

## 7. 实战案例

### 案例一：ASan 定位 Vendor 库的 Heap Buffer Overflow

**现象：**

某音视频应用在特定机型上播放 4K HDR 视频时，约有 0.5% 的概率触发 NE。Tombstone 如下：

```
signal 11 (SIGSEGV), code 2 (SEGV_ACCERR), fault addr 0x7b8c001000
    #00 pc 0x00028a4c  /vendor/lib64/libvideodec_hw.so (VDecoder::OutputFrame+316)
    #01 pc 0x0001f280  /vendor/lib64/libvideodec_hw.so (VDecoder::DecoderThread+480)
    #02 pc 0x00012f64  /system/lib64/libutils.so (android::Thread::_threadLoop+280)
```

fault addr `0x7b8c001000` 恰好在页边界上，`si_code = SEGV_ACCERR`（权限不足），初步怀疑是越界写入触及了只读页或 Guard Page。但 Tombstone 无法告诉我们"越界了多少"以及"写入的源头是哪个 buffer"。

**排查过程：**

1. **构建 ASan 版本**：获取 Vendor 库源码，加入 ASan 编译标志：
   ```bash
   LOCAL_SANITIZE := address
   LOCAL_CFLAGS += -fno-omit-frame-pointer
   ```

2. **复现得到 ASan 报告**：
   ```
   ==19432==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x62900001fc10
   WRITE of size 256 at 0x62900001fc10 thread T7 (DecoderThread)
       #0 memcpy
       #1 VDecoder::CopyOutputBuffer(OutputBuffer*, unsigned char*, int)
       #2 VDecoder::OutputFrame()

   0x62900001fc10 is located 16 bytes to the right of 65520-byte region
   [0x6290000000c0, 0x62900001faf0)
   allocated here:
       #0 malloc
       #1 VDecoder::AllocateOutputBuffers(int, int)
   ```

3. **分析**：ASan 精确指出——`CopyOutputBuffer` 中有一个 `memcpy` 越界写入了 16 字节（超过了 65520 字节的分配范围）。回查代码发现，4K HDR 帧的 stride 计算公式在 10-bit 色深下有一个对齐计算错误：`stride = width * 1.5`（10-bit YUV）应该向上对齐到 256 字节，但代码直接用了 `width * 2`（8-bit 路径）。

**根因：**

Vendor 解码器的 `AllocateOutputBuffers` 按 8-bit 色深计算 buffer 大小，但 `CopyOutputBuffer` 按实际的 10-bit HDR 色深拷贝数据，导致写入量超过分配量约 16 字节。在普通环境下这 16 字节恰好落在 malloc 的对齐 padding 中不崩溃；在 ASan 下 Redzone 精确捕获。

**修复方案：**

1. **短期修复**：修正 stride 计算公式，按实际色深对齐分配 buffer。
2. **长期治理**：在 Vendor 解码器的 CI 中加入 ASan 构建，每日跑 10-bit/HDR/Dolby Vision 等特殊格式的自动化测试用例。

```
修复前后对比：
┌────────────────────────┬──────────┬──────────┐
│ 指标                    │ 修复前    │ 修复后    │
├────────────────────────┼──────────┼──────────┤
│ 4K HDR 播放 NE 率       │ 0.5%     │ 0%       │
│ ASan 构建回归测试通过率   │ 87%      │ 100%     │
│ 相关用户投诉 / 日         │ ~120     │ 0        │
└────────────────────────┴──────────┴──────────┘
```

### 案例二：GWP-ASan 线上采样定位低频 Use-After-Free

**现象：**

某社交应用的线上 NE 聚类中，有一类长期存在的低频崩溃（日均 ~200 次，占总 NE 的 3%），信号为 `SIGSEGV (SEGV_MAPERR)`，fault addr 为随机大地址，backtrace 顶部是图片解码库 `libimgcodec.so`：

```
signal 11 (SIGSEGV), code 1 (SEGV_MAPERR), fault addr 0x55a3f01234c8
    #00 pc 0x0003b2a0  libimgcodec.so (ImageDecoder::decodeRow+128)
    #01 pc 0x0003a8f4  libimgcodec.so (ImageDecoder::decode+560)
    #02 pc 0x00022140  libimgloader.so (ImageLoader::loadFromCache+240)
```

随机大地址 + 图片解码 = 高度怀疑 Use-After-Free。但无法在测试环境复现——它只在特定的网络/缓存/并发条件下触发。

**排查过程：**

1. **确认 GWP-ASan 已启用**：该应用在 Android 14 设备上默认启用了 GWP-ASan（`android:gwpAsanMode="always"`）。

2. **等待线上采样**：上线一周后，GWP-ASan 捕获到 3 例带完整分配/释放堆栈的报告：

   ```
   *** GWP-ASan detected use-after-free on address 0x7b9c200040 ***

   Allocation of size 2048 at:
       #0  malloc
       #1  ImageCache::createEntry(std::string const&)
       #2  ImageCache::getOrCreate(std::string const&)

   Deallocation at:
       #0  free
       #1  ImageCache::evictLRU()
       #2  ImageCache::trimToSize(int)
       #3  ImageCache::onMemoryPressure(int)

   Crash backtrace:
       #0  ImageDecoder::decodeRow(unsigned char*, int)
       #1  ImageDecoder::decode(ImageCache::Entry*)
       #2  ImageLoader::loadFromCache(std::string const&)
   ```

3. **分析**：三份 backtrace 完整还原了事件链——
   - **分配**：`ImageCache::createEntry` 创建了一个缓存条目（2048 字节）
   - **释放**：系统内存压力回调 `onMemoryPressure` → `trimToSize` → `evictLRU` 淘汰了该条目
   - **崩溃**：`ImageLoader::loadFromCache` 仍持有指向已淘汰条目的指针，传给 `ImageDecoder::decode` 继续使用

**根因：**

`ImageCache` 的 LRU 淘汰在 `onMemoryPressure` 回调中执行（由系统 trim memory 触发），而 `ImageLoader::loadFromCache` 在另一个线程中通过裸指针持有缓存条目。两个操作之间没有同步机制——当内存压力回调淘汰缓存时，正在使用该条目的解码线程不知情，继续通过已释放的指针访问。

这是一个经典的**跨线程对象生命周期管理缺陷**。普通 Tombstone 只能看到 `decodeRow` 的崩溃栈，无法得知内存何时被谁释放。GWP-ASan 的三栈报告直接揭示了根因。

**修复方案：**

1. **短期修复**：将缓存条目的裸指针替换为 `std::shared_ptr`，`ImageLoader` 持有 `shared_ptr` 副本，保证在使用期间条目不会被释放。
2. **长期治理**：
   - 对所有缓存模块的条目访问引入引用计数或 `shared_ptr` 管理
   - 在 `onMemoryPressure` 回调中，仅标记条目为"待淘汰"，由使用方在释放引用时真正删除
   - 增加 GWP-ASan 采样率到 1/500（通过 `android_mallopt(M_GWP_ASAN_SAMPLE_RATE, ...)`），提高同类问题的采样概率

```
修复前后对比：
┌──────────────────────────┬──────────┬──────────┐
│ 指标                      │ 修复前    │ 修复后    │
├──────────────────────────┼──────────┼──────────┤
│ 图片解码 NE 日均量         │ ~200     │ ~5       │
│ GWP-ASan UAF 采样报告 / 周 │ ~3       │ 0        │
│ 图片加载成功率             │ 99.2%    │ 99.97%   │
│ OOM 关联的缓存异常         │ ~50/天   │ ~2/天    │
└──────────────────────────┴──────────┴──────────┘
```

## 8. 总结

本篇从"冰山模型"出发，系统讲解了 Android 生态中四大内存错误检测工具的原理、能力和适用场景。

核心要点：

1. **内存错误的崩溃点 ≠ 根因点**——这是所有检测工具存在的根本原因。Tombstone 只能告诉你"在哪里爆炸"，检测工具能告诉你"炸弹是谁埋的"。

2. **ASan 是开发阶段的第一选择**——Shadow Memory 1:8 映射 + Redzone + Quarantine，检测能力最全面（UAF/OOB/DF/Leak），但 2x CPU + 3x 内存的开销使其无法用于线上。

3. **HWASan 利用 ARM64 TBI 大幅降低内存开销**——标签编码在指针高位，内存开销从 3x 降至 ~15%，适合大规模自动化测试。但仍需重新编译且不适合线上。

4. **MTE 是硬件级的终极方案**——ARMv8.5+ 的 CPU 在每次 Load/Store 时自动检查标签，零 CPU 指令开销。sync 模式精确定位，async 模式可用于线上。限制是需要新硬件且标签碰撞率为 1/16。

5. **GWP-ASan 是唯一的线上检测方案**——概率性采样 + Guard Page，开销 < 1%。虽然单用户检测率低，但海量用户的聚合效应使其成为线上内存安全的核心守护者。它的三栈报告（分配栈 + 释放栈 + 崩溃栈）是排查 UAF 的"杀手锏"。

6. **三阶段策略**：开发用 ASan → 测试用 HWASan/MTE-sync → 线上用 GWP-ASan/MTE-async。不同阶段用不同工具，形成层层递进的内存安全防线。
