# 03-内存管理与保护：SIGSEGV 背后的真相

上一篇我们拆解了 Linux 信号从产生到传递到处理的完整生命周期。你已经知道，当 MMU 检测到非法内存访问时，内核会向进程发送 SIGSEGV。但"非法内存访问"这五个字背后，隐藏着一整套精密的内存管理与保护体系——**虚拟内存、页表、VMA、mmap、Guard Page、堆分配器、MTE**——它们共同构成了 SIGSEGV/SIGBUS/SIGABRT 的"发射场"。

本篇将从硬件（MMU/TLB）到内核（Page Fault/VMA/mmap）再到用户态（栈/堆/分配器），逐层拆解内存管理的核心机制，并在每个环节指出"它会在什么条件下向你的进程发射一颗致命信号"。

---

## 1. 虚拟内存与 MMU

### 1.1 为什么需要虚拟内存

每个 Android 进程都"以为"自己独享了一整块连续的 64 位地址空间（ARM64 上通常是 48 位有效，即 256TB）。但物理内存（RAM）可能只有 4GB~16GB。操作系统靠**虚拟内存（Virtual Memory）**实现了这个幻觉：每个进程有独立的虚拟地址空间，彼此隔离，由硬件和内核协作完成虚拟地址到物理地址的翻译。

虚拟内存解决了三个核心问题：
1. **隔离性**：进程 A 无法访问进程 B 的物理内存，即使两者的虚拟地址数值相同。
2. **超额分配**：所有进程的虚拟地址空间之和可以远大于物理内存，内核通过按需分配物理页（Demand Paging）和换页（Swap）来管理。
3. **权限控制**：同一块虚拟内存区域可以设置为只读（代码段 `.text`）、可读写（数据段 `.data`）、不可执行（栈/堆），硬件在每次访问时自动检查权限。

### 1.2 Page Table、MMU、TLB

虚拟地址到物理地址的翻译由三个组件协作完成：

```
┌──────────────────────────────────────────────────────────────────┐
│ CPU 发出虚拟地址 VA                                                │
│    │                                                              │
│    ▼                                                              │
│ TLB（Translation Lookaside Buffer）── 缓存最近的翻译结果           │
│    │                                                              │
│    ├─ TLB Hit → 直接拿到物理地址 PA，访问内存（< 1 cycle 延迟）     │
│    │                                                              │
│    └─ TLB Miss → 查 Page Table                                    │
│         │                                                         │
│         ▼                                                         │
│    MMU（Memory Management Unit）── 硬件 Page Table Walker          │
│         │                                                         │
│         ├─ 找到有效的 PTE（Page Table Entry）→ 翻译成功             │
│         │   · 检查权限位（R/W/X/User/Kernel）                      │
│         │   · 权限通过 → 返回 PA，填入 TLB                         │
│         │   · 权限不通过 → 触发 Permission Fault                   │
│         │                                                         │
│         └─ PTE 无效（unmapped / not present）                      │
│             → 触发 Translation Fault（Page Fault）                 │
│             → CPU 陷入内核态，进入异常处理程序                       │
└──────────────────────────────────────────────────────────────────┘
```

ARM64 使用 4 级页表（PGD → PUD → PMD → PTE），每级 9 位索引，加上 12 位页内偏移，覆盖 48 位虚拟地址空间。

**源码路径：** `arch/arm64/mm/`

```c
// arch/arm64/include/asm/pgtable-types.h
typedef struct { pteval_t pte; } pte_t;   // 页表项（Page Table Entry）
typedef struct { pgdval_t pgd; } pgd_t;   // 顶级页目录项

// arch/arm64/include/asm/pgtable-hwdef.h
#define PAGE_SHIFT      12               // 4KB 页大小
#define PTRS_PER_PTE    512              // 每级页表 512 个条目（2^9）
#define ARM64_HW_PGTABLE_LEVELS 4        // 4 级页表

// PTE 的权限位定义
#define PTE_VALID       (_AT(pteval_t, 1) << 0)   // 页表项有效
#define PTE_USER        (_AT(pteval_t, 1) << 6)   // 用户态可访问
#define PTE_RDONLY      (_AT(pteval_t, 1) << 7)   // 只读
#define PTE_PXN         (_AT(pteval_t, 1) << 53)  // 特权态不可执行
#define PTE_UXN         (_AT(pteval_t, 1) << 54)  // 用户态不可执行
```

**稳定性架构师视角：** Tombstone 中的 `fault addr` 就是 MMU 翻译失败时 CPU 报出的虚拟地址。两种 Fault 类型直接对应 Tombstone 中 `si_code` 的两个值：
- **Translation Fault**（PTE 无效/不存在）→ `SEGV_MAPERR`：地址根本没有映射。
- **Permission Fault**（PTE 存在但权限不足）→ `SEGV_ACCERR`：映射存在但操作违规（如写只读页）。

TLB 在稳定性排查中较少直接暴露，但有一个例外：**TLB 一致性问题**。在多核 SoC 上，如果内核修改了页表但未正确执行 TLB invalidate（`tlbi` 指令），其他 CPU 核心可能仍使用旧的 TLB 条目，导致间歇性的、难以复现的 SIGSEGV。这类问题通常出现在 Vendor 内核补丁中。

## 2. Page Fault 的三种结局

### 2.1 为什么 Page Fault 不等于崩溃

很多开发者一听到 "Page Fault" 就联想到 SIGSEGV，但实际上大多数 Page Fault 是**正常的、必需的**——它们是 Linux "按需分配"内存策略的核心机制。只有少数 Page Fault 才会导致信号。

当 MMU 触发 Page Fault 后，内核的 `do_page_fault()` 会判断这次 fault 属于以下三种情况中的哪一种：

```
                         Page Fault 发生
                              │
                              ▼
                    ┌──────────────────┐
                    │ find_vma(addr)    │  查找 fault addr 所在的 VMA
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
      地址在合法 VMA 内   地址在 VMA 内    地址不在任何
      且 PTE 标记为       但权限不匹配     VMA 内
      "not present"
              │              │              │
              ▼              ▼              ▼
      ┌──────────────┐ ┌──────────┐ ┌──────────────┐
      │ Minor Fault   │ │ SIGSEGV  │ │ SIGSEGV      │
      │               │ │ SEGV_    │ │ SEGV_MAPERR  │
      │ · Demand Page │ │ ACCERR   │ │              │
      │ · COW         │ │          │ │ 地址完全非法  │
      │ · Swap In     │ │ 权限违规  │ │ (空/野指针)   │
      │               │ │          │ │              │
      │ 分配物理页     │ │ 发送信号  │ │ 发送信号     │
      │ 更新 PTE      │ │ 终止进程  │ │ 终止进程     │
      │ 返回用户态继续 │ │          │ │              │
      └──────────────┘ └──────────┘ └──────────────┘
```

### 2.2 结局一：Demand Paging（正常缺页）

当进程通过 `mmap()` 或 `malloc()` 分配了一块虚拟内存时，内核并不会立即分配物理页。它只在页表中做一个"占位符"标记。当进程首次访问这块内存时，触发 Page Fault，内核这才真正分配一个物理页、建立映射、更新 PTE，然后让进程继续执行——进程对此毫无感知。

### 2.3 结局二：COW（Copy-On-Write）

`fork()` 之后，父子进程共享相同的物理页，页表项标记为只读。当其中一方试图写入时，MMU 触发 Permission Fault，内核检测到这是一个 COW 页，于是复制一份物理页给写入方，更新其 PTE 为可写，然后重新执行写入操作。

### 2.4 结局三：非法访问 → SIGSEGV

如果 Page Fault 的地址不在任何合法的 VMA 中，或者虽然在 VMA 中但操作权限不匹配（如写入只读 VMA），内核判定这是一次不可恢复的非法访问，向进程发送 SIGSEGV。

**源码路径：** `mm/memory.c`

```c
// mm/memory.c
vm_fault_t handle_mm_fault(struct vm_area_struct *vma,
                           unsigned long address,
                           unsigned int flags,
                           struct pt_regs *regs)
{
    /* 权限检查：写操作是否被 VMA 允许 */
    if (flags & FAULT_FLAG_WRITE) {
        if (!(vma->vm_flags & VM_WRITE))
            return VM_FAULT_BADACCESS;  // → SEGV_ACCERR
    }

    /* 执行权限检查 */
    if (flags & FAULT_FLAG_INSTRUCTION) {
        if (!(vma->vm_flags & VM_EXEC))
            return VM_FAULT_BADACCESS;  // → SEGV_ACCERR（NX 保护）
    }

    /* 进入具体的 fault 处理 */
    return __handle_mm_fault(vma, address, flags);
}

// mm/memory.c
static vm_fault_t __handle_mm_fault(struct vm_area_struct *vma,
                                     unsigned long address,
                                     unsigned int flags)
{
    pgd_t *pgd;
    p4d_t *p4d;
    pud_t *pud;
    pmd_t *pmd;

    /* 逐级遍历页表 */
    pgd = pgd_offset(vma->vm_mm, address);
    p4d = p4d_alloc(vma->vm_mm, pgd, address);
    pud = pud_alloc(vma->vm_mm, p4d, address);
    pmd = pmd_alloc(vma->vm_mm, pud, address);

    /* 最终到达 PTE 级别的处理 */
    return handle_pte_fault(vma, address, pmd, flags);
}
```

`handle_pte_fault` 内部会区分三种情况：PTE 不存在（分配新页 / 从文件读入 / 从 swap 读入）、PTE 存在但是 COW 页（复制页）、PTE 不存在且不在任何 VMA 中（非法访问）。

**稳定性架构师视角：** 理解 Page Fault 的三种结局，能帮你在 Tombstone 分析中做出关键判断：
- `SEGV_MAPERR`（code 1）：地址不在任何 VMA 中。查看 Tombstone 的 `memory maps` 段，确认 `fault addr` 是否落在某个映射区间内——如果不在任何区间内，就是真正的"无映射"访问（空指针/野指针/UAF 后 VMA 被回收）。
- `SEGV_ACCERR`（code 2）：地址在某个 VMA 中但权限不足。查看该 VMA 的权限标记（`r--p` / `rw-p` / `r-xp`），比对操作类型（读/写/执行），就能判断是哪种权限违规。

## 3. mmap 与内存映射

### 3.1 为什么要理解 mmap

Android 系统中几乎所有的内存区域都是通过 `mmap()` 建立的——so 库的加载、匿名内存的分配（`malloc` 底层）、文件的映射读取、Binder 的共享内存、GPU buffer 的映射……Tombstone 中的 `memory maps` 段就是进程所有 `mmap` 映射的快照。理解 `mmap`，你才能看懂 `memory maps`。

### 3.2 文件映射 vs 匿名映射

`mmap()` 有两种基本模式：

| 模式 | 描述 | 典型用途 | Tombstone maps 中的特征 |
| :--- | :--- | :--- | :--- |
| **文件映射** | 将磁盘文件的内容映射到虚拟地址空间 | so 库加载、dex/vdex/oat 加载、资源文件读取 | 显示文件路径，如 `/system/lib64/libc.so` |
| **匿名映射** | 不关联任何文件，纯内存 | `malloc()` 大块分配、线程栈、JIT 代码缓存 | 显示 `[anon:...]` 或无名 |

**源码路径：** `mm/mmap.c`

```c
// mm/mmap.c
unsigned long do_mmap(struct file *file, unsigned long addr,
                      unsigned long len, unsigned long prot,
                      unsigned long flags, unsigned long pgoff,
                      unsigned long *populate, struct list_head *uf)
{
    struct mm_struct *mm = current->mm;
    struct vm_area_struct *vma;

    /* 权限标志转换：PROT_xxx → VM_xxx */
    vm_flags_t vm_flags = calc_vm_prot_bits(prot, 0) |
                          calc_vm_flag_bits(flags);

    /* PROT_READ  → VM_READ   (可读)
     * PROT_WRITE → VM_WRITE  (可写)
     * PROT_EXEC  → VM_EXEC   (可执行)
     * PROT_NONE  → 无权限（Guard Page 使用）*/

    /* 寻找空闲的虚拟地址区间 */
    addr = get_unmapped_area(file, addr, len, pgoff, flags);

    /* 创建新的 VMA（Virtual Memory Area）结构体 */
    vma = vm_area_alloc(mm);
    vma->vm_start = addr;
    vma->vm_end = addr + len;
    vma->vm_flags = vm_flags;
    vma->vm_file = file;
    vma->vm_pgoff = pgoff;

    /* 将 VMA 插入进程的 VMA 红黑树 */
    vma_link(mm, vma, prev, rb_link, rb_parent);

    return addr;
}
```

### 3.3 VMA：虚拟内存区域

VMA（`vm_area_struct`）是内核管理进程虚拟地址空间的基本单元。一个进程的地址空间由一系列不重叠的 VMA 组成，每个 VMA 描述一段连续的虚拟地址区间及其属性（权限、关联文件等）。

```c
// include/linux/mm_types.h
struct vm_area_struct {
    unsigned long vm_start;       // 区间起始地址
    unsigned long vm_end;         // 区间结束地址（不包含）
    struct vm_area_struct *vm_next; // 链表中的下一个 VMA
    pgprot_t vm_page_prot;        // 页级权限
    unsigned long vm_flags;       // VMA 级标志（VM_READ/VM_WRITE/VM_EXEC/...）
    struct file *vm_file;         // 关联的文件（匿名映射为 NULL）
    unsigned long vm_pgoff;       // 文件映射的起始偏移（页为单位）
    // ...
};
```

Tombstone 中 `memory maps` 的每一行就对应一个 VMA：

```
memory map:
    7a1c000000-7a1c001000 r--p 00000000 /system/lib64/libc.so
    7a1c001000-7a1c0a0000 r-xp 00001000 /system/lib64/libc.so
    7a1c0a0000-7a1c0a5000 r--p 000a0000 /system/lib64/libc.so
    7a1c0a5000-7a1c0a8000 rw-p 000a4000 /system/lib64/libc.so
```

四列分别是：地址范围、权限（r=读/w=写/x=执行/p=私有/s=共享）、文件偏移、文件路径。

### 3.4 mmap 文件截断 → SIGBUS

这是 Android 上一个经典的稳定性问题。当一个文件通过 `mmap()` 映射到内存后，如果文件在映射期间被截断（文件大小变小），进程访问超出文件当前大小的映射区域时，不会触发 SIGSEGV——因为 VMA 仍然存在，权限也没问题——而是触发 **SIGBUS**（`BUS_ADRERR`）。

```c
// mm/filemap.c
static vm_fault_t filemap_fault(struct vm_fault *vmf)
{
    struct inode *inode = file_inode(vmf->vma->vm_file);
    loff_t size = i_size_read(inode);
    pgoff_t max_pgoff = (size - 1) >> PAGE_SHIFT;

    /* 访问的页偏移超出文件当前大小 → SIGBUS */
    if (unlikely(vmf->pgoff > max_pgoff)) {
        return VM_FAULT_SIGBUS;
    }

    /* 正常路径：从 page cache 读取页面 */
    // ...
}
```

**稳定性架构师视角：** 这个场景在 Android 上的典型触发路径是：
1. **应用热更新期间 so 文件被替换**：旧的 so 文件已经被 `dlopen` 加载（mmap 映射），更新流程删除/替换了文件，后续访问触发 SIGBUS。
2. **SQLite WAL 文件被意外截断**：数据库的 WAL 文件被 mmap 映射，如果另一个进程或线程截断了 WAL 文件，读操作触发 SIGBUS。
3. **Asset 文件在 APK 更新后失效**：通过 `AAsset_getBuffer()` 获取的指针实际是 mmap 映射的 APK 内容，APK 更新后旧映射失效。

Tombstone 特征：`signal 7 (SIGBUS), code 2 (BUS_ADRERR)`，`fault addr` 位于某个文件映射的 VMA 内。

## 4. 栈的结构与保护

### 4.1 线程栈的布局

每个 Android Native 线程在创建时，Bionic 会为其分配一块连续的虚拟内存作为栈空间。这块内存的布局如下：

```
高地址 ┌───────────────┐
       │ TLS (Thread    │  线程局部存储
       │ Local Storage) │
       ├───────────────┤
       │               │
       │  Stack        │  栈空间（默认 1MB，可配置）
       │  (grows ↓)    │  SP 从高地址向低地址增长
       │               │
       ├───────────────┤
       │ Guard Page    │  保护页（通常 4KB-64KB，PROT_NONE）
       │ (不可访问)     │  栈溢出时 SP 触碰到此处 → SIGSEGV
低地址 └───────────────┘
```

**源码路径：** `bionic/libc/bionic/pthread_create.cpp`

```cpp
// bionic/libc/bionic/pthread_create.cpp
static int __allocate_thread(pthread_attr_t* attr,
                             bionic_tcb** tcbp,
                             void** child_stk) {
    // 栈大小：默认 1MB（PTHREAD_STACK_SIZE_DEFAULT = 1024 * 1024）
    size_t stack_size = attr->stack_size;

    // Guard 大小：默认 PAGE_SIZE（4096），可通过 pthread_attr_setguardsize 配置
    size_t guard_size = attr->guard_size;

    // 总分配大小 = guard + stack + TLS
    size_t mmap_size = guard_size + stack_size + sizeof(bionic_tcb);

    // 使用 mmap 分配整块内存
    void* space = mmap(NULL, mmap_size, PROT_READ | PROT_WRITE,
                       MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

    // 将底部的 guard_size 字节设置为不可访问
    if (guard_size) {
        mprotect(space, guard_size, PROT_NONE);
    }

    // 栈从 mmap 区域的顶部开始向下增长
    *child_stk = (char*)space + mmap_size;
    return 0;
}
```

### 4.2 Guard Page 的保护机制

Guard Page 是栈溢出保护的最后一道防线。当函数调用嵌套过深（递归）或者局部变量过大时，SP 持续向低地址增长，最终"撞上"Guard Page。由于 Guard Page 的权限是 `PROT_NONE`（不可读、不可写、不可执行），MMU 触发 Permission Fault，内核发送 `SIGSEGV (SEGV_ACCERR)`。

**稳定性架构师视角：** Guard Page 虽然能检测栈溢出，但它有一个致命弱点——**Guard Page 跳跃（Stack Clash）**。如果一个函数声明了一个非常大的局部变量（如 `char buf[2*1024*1024]`），编译器生成的代码可能会直接将 SP 减去 2MB，**跳过** 4KB 的 Guard Page，直接落在 Guard Page 下方的其他合法映射上——此时没有任何 fault 发生，但栈数据会覆盖其他内存区域，导致后续不可预测的崩溃。

Linux 内核通过 **Stack Gap**（在 VMA 分配时预留 1MB 的间隙）和编译器的 **-fstack-clash-protection** 选项（逐页探测 SP）来缓解此问题。Android 14+ 的 NDK 默认启用了 `-fstack-clash-protection`。

Tombstone 中栈溢出的特征：
- `signal 11 (SIGSEGV), code 2 (SEGV_ACCERR)`
- `Cause: stack-guard-page-hit`（Android 10+ 的 Tombstone 会标注）
- backtrace 中有大量重复帧（递归调用的标志）
- `fault addr` 刚好位于栈 VMA 的底部边界

## 5. 堆的结构与分配器

### 5.1 为什么堆问题最难排查

栈的生命周期由编译器管理（函数进入/退出自动分配/释放），很少出错。而堆的生命周期完全由程序员控制——`malloc/free`、`new/delete`——这就是万恶之源。堆相关的内存安全问题（Double-Free、Use-After-Free、Heap Buffer Overflow）占 Android Native Crash 的 **30%+**，且它们有一个共同的可怕特征：**崩溃点往往不是根因点**。

一个 Double-Free 可能在第一次 `free()` 时没有任何异常，在第二次 `free()` 时也没有异常，但在之后的某次 `malloc()` 或 `free()` 时——可能隔了几秒、几千行代码之后——堆分配器检测到元数据损坏，调用 `abort()` 终止进程。Tombstone 中的 backtrace 指向的是检测到损坏的那个 `malloc/free`，而不是真正造成损坏的那个。

### 5.2 Scudo：Android 的默认堆分配器

从 Android 11 开始，Bionic 的默认堆分配器从 jemalloc 切换为 **Scudo**。Scudo 是 LLVM 项目的一部分，专为安全性设计，内置了多种堆损坏检测机制。

**源码路径：** `bionic/libc/bionic/malloc_common.cpp`

```cpp
// bionic/libc/bionic/malloc_common.cpp
extern "C" void* malloc(size_t bytes) {
    auto dispatch_table = GetDispatchTable();
    if (__predict_false(dispatch_table != nullptr)) {
        // 如果有 dispatch table（如开启了 malloc debug/hooks），走拦截路径
        return dispatch_table->malloc(bytes);
    }
    // 默认路径：直接调用 Scudo
    void* result = scudo_malloc(bytes);
    if (__predict_false(result == nullptr)) {
        errno = ENOMEM;
    }
    return result;
}

extern "C" void free(void* mem) {
    auto dispatch_table = GetDispatchTable();
    if (__predict_false(dispatch_table != nullptr)) {
        dispatch_table->free(mem);
        return;
    }
    scudo_free(mem);
}
```

### 5.3 Scudo 的安全检测机制

Scudo 在每个堆块（chunk）的头部维护了一个 **chunk header**，包含校验和（checksum）、大小信息、分配状态等元数据。每次 `free()` 和 `malloc()` 都会验证 chunk header 的完整性。

```
┌──────────────────────────────────────┐
│ Scudo Chunk 内存布局                   │
├──────────┬───────────────────────────┤
│ Chunk    │ ClassId: 分配器 size class  │
│ Header   │ State: 已分配/已释放/隔离中  │
│ (16 bytes│ Origin: malloc/new/memalign│
│  通常)   │ Checksum: CRC32 校验和     │
├──────────┼───────────────────────────┤
│ User     │                           │
│ Data     │ malloc() 返回的指针指向这里 │
│          │                           │
├──────────┼───────────────────────────┤
│ (可选)    │ 填充/对齐字节              │
└──────────┴───────────────────────────┘
```

当 Scudo 检测到以下异常时，会调用 `abort()` 终止进程（触发 SIGABRT）：

| 异常类型 | 检测方式 | Tombstone abort message 特征 |
| :--- | :--- | :--- |
| **Double-Free** | chunk header 的 State 已经是"已释放" | `"invalid chunk state"` |
| **Heap Buffer Overflow** | 后续 `free()` 时发现 chunk header 的 Checksum 不匹配 | `"corrupted chunk header"` |
| **Use-After-Free (写入)** | 隔离区（Quarantine）中的 chunk 内容被修改 | `"chunk header corruption"` |
| **Invalid Free** | 传入的指针不是 `malloc` 返回的地址 | `"invalid aligned pointer"` |

**源码路径：** Scudo 的核心检测逻辑

```cpp
// compiler-rt/lib/scudo/standalone/combined.h（Scudo standalone 实现）
void deallocate(void *Ptr, ...) {
    // 1. 获取 chunk header
    Chunk::UnpackedHeader Header;
    Chunk::loadHeader(Cookie, Ptr, &Header);

    // 2. 验证 checksum
    if (UNLIKELY(!Chunk::isValid(Cookie, Ptr, &Header))) {
        // Checksum 不匹配 → 堆元数据已被破坏
        reportInvalidChunkHeader(AllocatorAction::Deallocating, Ptr);
        // → 内部调用 abort()
    }

    // 3. 检查状态：是否已经被 free 过
    if (UNLIKELY(Header.State != Chunk::State::Allocated)) {
        // Double-Free 或 Use-After-Free
        reportMisalignedPointer(AllocatorAction::Deallocating, Ptr);
    }

    // 4. 标记为已释放
    Header.State = Chunk::State::Quarantined;
    Chunk::storeHeader(Cookie, Ptr, &Header);

    // 5. 放入隔离区（Quarantine）而非立即归还系统
    //    隔离区延迟复用，增大 UAF 的检测窗口
    quarantineOrDeallocateChunk(Options, Ptr, &Header, Size);
}
```

**稳定性架构师视角：** Scudo 的 Quarantine（隔离区）是一个精妙的设计——`free()` 后的内存不会立即被复用，而是放入隔离区保持一段时间。在这段时间内，如果有代码通过 dangling pointer 写入了这块内存，Scudo 在 chunk 最终出隔离区时能检测到内容变化。隔离区越大，UAF 的检测窗口越大，但内存开销也越大。Android 在 Scudo 的配置中使用了较保守的隔离区大小，以平衡安全性和内存占用。

### 5.4 堆分配器演进：jemalloc → Scudo

| 维度 | jemalloc (Android ≤10) | Scudo (Android 11+) |
| :--- | :--- | :--- |
| 设计目标 | 高性能（减少锁竞争、减少碎片） | 安全优先（检测堆损坏、缓解利用） |
| 元数据校验 | 无内置校验和 | Chunk header 带 CRC32 校验 |
| Double-Free 检测 | 不检测（可能导致堆静默损坏） | 检测 → SIGABRT |
| Quarantine | 无 | 有（延迟复用，增大 UAF 检测窗口） |
| 性能 | 更快（无额外校验开销） | 略慢（校验和计算 + 隔离区管理） |

**稳定性关联：** 从 jemalloc 切换到 Scudo 后，很多 App 发现 SIGABRT 增多——这不是因为引入了新 bug，而是 Scudo 把之前被 jemalloc "静默吞掉"的堆损坏暴露了出来。这些 bug 一直存在，只是之前没有被检测到。

## 6. MTE（Memory Tagging Extension）

### 6.1 为什么现有工具还不够

ASan 和 HWASan 虽然能高效检测内存错误，但它们有不可忽视的运行时开销（ASan: 2x CPU + 3x 内存; HWASan: ~15% CPU + ~15% 内存），无法在线上 Release 版本中启用。Scudo 的检测能力有限——它只能在 `free()` 和 `malloc()` 的入口处检查元数据，对于"已分配内存的越界访问"（Heap Buffer Overflow 在两次 malloc/free 之间的中间过程）无能为力。

ARM v8.5+ 引入的 **MTE（Memory Tagging Extension）** 是一种**硬件级**的内存安全方案，它用极低的开销（~3% CPU，~3% 内存）实现了对 UAF 和 Buffer Overflow 的实时检测，是未来 Android 内存安全的终极答案。

### 6.2 MTE 的工作原理

MTE 的核心思想：给每块内存和每个指针都打上一个 **4-bit 标签（tag）**，每次内存访问时硬件自动比对标签是否匹配。

```
┌─────────────────────────────────────────────────────────────┐
│  MTE 工作原理                                                │
│                                                              │
│  指针:                                                       │
│  ┌───────┬──────────────────────────────────────────────────┐│
│  │ Tag=5 │ 0x 0000 7a1c 0042 30                             ││
│  │(4-bit)│ (虚拟地址，Tag 存储在指针的高 4 位)                 ││
│  └───────┴──────────────────────────────────────────────────┘│
│                                                              │
│  内存:                                                       │
│  ┌───────┬───────┬───────┬───────┬───────┬──────┐           │
│  │Tag=5  │Tag=5  │Tag=5  │Tag=5  │Tag=3  │Tag=3 │           │
│  │16 byte│16 byte│16 byte│16 byte│16 byte│......│           │
│  └───────┴───────┴───────┴───────┴───────┴──────┘           │
│  ▲ 已分配给当前 malloc   ▲ 下一个 chunk（不同 tag）             │
│                                                              │
│  访问检查：                                                   │
│  · 指针 Tag(5) == 内存 Tag(5) → ✅ 访问通过                   │
│  · 指针 Tag(5) != 内存 Tag(3) → ❌ Tag Mismatch Fault         │
│    → 触发 SIGSEGV（si_code = SEGV_MTESERR 或 SEGV_MTEASERR） │
└─────────────────────────────────────────────────────────────┘
```

MTE 能检测两类核心问题：
- **Use-After-Free**：`free()` 后，分配器将内存区域的 tag 改为新的随机值。旧指针中的 tag 不再匹配 → 访问时立即 fault。
- **Buffer Overflow**：相邻的 chunk 拥有不同的 tag。越界访问时指针 tag 与目标内存 tag 不匹配 → 立即 fault。

### 6.3 Android 的 MTE 支持

Android 12 开始支持 MTE，Android 14+ 在 Pixel 设备上对系统进程默认启用。

**源码路径：** `bionic/libc/bionic/scudo_wrapper.cpp`

```cpp
// bionic/libc/bionic/scudo_wrapper.cpp
// Scudo 与 MTE 的集成：malloc 时为内存区域设置随机 tag

void* scudo_malloc(size_t size) {
    void* ptr = ScudoAllocator.allocate(size, ...);

#if defined(__aarch64__) && defined(ANDROID_EXPERIMENTAL_MTE)
    if (__predict_true(ptr != nullptr) && mte_supported()) {
        // 为分配的内存区域设置随机 tag
        // IRG: Insert Random Tag (ARM MTE 指令)
        ptr = __arm_mte_create_random_tag(ptr, 0);
        // STG: Store Allocation Tag (将 tag 写入内存的 tag storage)
        __arm_mte_set_tag(ptr, size);
    }
#endif

    return ptr;
}

void scudo_free(void* ptr) {
#if defined(__aarch64__) && defined(ANDROID_EXPERIMENTAL_MTE)
    if (mte_supported()) {
        // free 时将内存的 tag 改为新的随机值
        // 旧指针中的 tag 不再匹配 → UAF 访问会触发 fault
        void* tagged = __arm_mte_create_random_tag(ptr, 0);
        __arm_mte_set_tag(tagged, chunk_size);
    }
#endif

    ScudoAllocator.deallocate(ptr, ...);
}
```

### 6.4 MTE 的两种模式

| 模式 | 检测时机 | 性能开销 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **同步模式（Sync MTE）** | 每次内存访问时立即检查 tag，不匹配时立即触发 fault | ~3-5% CPU | 开发/测试阶段，需要精确的崩溃现场 |
| **异步模式（Async MTE）** | tag 检查结果累积，在下一次内核态切换时统一报告 | ~1-2% CPU | 线上 Release 版本，降低性能影响 |

**稳定性架构师视角：** 异步模式的代价是崩溃现场不精确——你知道"发生了 tag mismatch"，但 fault 报告的 PC 可能不是实际违规访问的那条指令（因为检查结果是延迟汇报的）。在排查 MTE 触发的崩溃时，需要结合代码逻辑和分配/释放的时间线来推断真实的违规点。

Tombstone 中 MTE 触发的信号特征：
- `signal 11 (SIGSEGV), code 9 (SEGV_MTESERR)`：同步模式 MTE 错误
- `signal 11 (SIGSEGV), code 8 (SEGV_MTEASERR)`：异步模式 MTE 错误
- `Cause: [MTE]: Use After Free` 或 `Cause: [MTE]: Buffer Overflow`（Android 14+ Tombstone 会标注）

## 7. 稳定性实战案例

### 案例一：热更新导致 SIGBUS——mmap 映射的 so 文件被替换

**现象：**

某应用在推送热更新补丁后，线上出现大量 SIGBUS 崩溃。崩溃集中在更新后的 10 分钟内，之后逐渐消失。Tombstone 信息如下：

```
signal 7 (SIGBUS), code 2 (BUS_ADRERR), fault addr 0x7b3a045000
backtrace:
    #00 pc 0x00045000  /data/app/com.example.app/lib/arm64/libplugin.so (PluginManager::execute+128)
    #01 pc 0x0003f200  /data/app/com.example.app/lib/arm64/libplugin.so (PluginManager::dispatch+96)
    #02 pc 0x0001a844  /data/app/com.example.app/lib/arm64/libapp.so (AppMain::onEvent+340)
```

**分析思路：**

1. **SIGBUS + BUS_ADRERR 的含义**：进程试图访问一个合法映射区域内的地址，但底层的物理页无法提供——通常是 mmap 映射的文件在运行时被截断或删除。
2. **`fault addr` 位于 `libplugin.so` 的映射区间内**：查看 Tombstone 的 `memory maps` 段，确认 `0x7b3a045000` 确实落在 `libplugin.so` 的 `r-xp`（代码段）映射内。
3. **"更新后 10 分钟内集中崩溃"**：高度暗示是热更新过程中文件被替换。旧的 `libplugin.so` 已经通过 `dlopen()` 加载到内存（mmap 映射），热更新流程将新版本的 so 文件覆盖到同一路径，导致旧文件的 inode 被删除/截断。
4. **为什么 10 分钟后消失**：旧进程逐渐被杀（用户关闭/系统回收），新启动的进程加载的是新版本 so 文件，映射正常。

**根因：**

热更新流程直接使用 `rename()` 将新的 `libplugin.so` 覆盖了旧文件。在 Linux 上，`rename()` 会原子地替换目标文件的 inode。已经被 mmap 映射的旧 inode 仍在内存中保持引用计数，但如果内核的 page cache 需要回收旧 inode 的页面（内存压力大时），后续访问这些页面会触发 `filemap_fault()` → 发现文件已被截断/删除 → 返回 `VM_FAULT_SIGBUS`。

**修复方案：**

1. **短期修复**：热更新后立即通知应用重启（或杀死旧进程），避免在旧映射上继续执行。
2. **长期治理**：
   - 不直接覆盖正在使用的 so 文件。将新版本 so 写入新的路径（如带版本号的子目录），通过 `dlopen()` 加载新路径，旧版本在下次冷启动时清理。
   - 使用 `O_TMPFILE` + `linkat()` 的原子替换模式：先写临时文件，写完后 `linkat()` 到目标路径。这样旧 inode 在所有引用释放前不会被真正删除。

```
修复前后对比：
┌─────────────────┬───────────┬──────────┐
│ 指标             │ 修复前     │ 修复后    │
├─────────────────┼───────────┼──────────┤
│ 热更新后 SIGBUS  │ 3,500/次  │ 0        │
│ 热更新成功率     │ 92%       │ 99.7%    │
│ 用户感知闪退     │ 280/次    │ 0        │
└─────────────────┴───────────┴──────────┘
```

### 案例二：Scudo 检测到 Double-Free 导致 SIGABRT，真实根因是多线程竞态

**现象：**

某 App 在 Android 11 设备上 SIGABRT 激增，但在 Android 10 设备上无此问题。Tombstone 中的 abort message 为：

```
signal 6 (SIGABRT), code -1 (SI_QUEUE)
Abort message: 'Scudo ERROR: invalid chunk state when deallocating address 0x7b4c023a80'
backtrace:
    #00 pc 0x00053a24  /apex/com.android.runtime/lib64/bionic/libc.so (abort+168)
    #01 pc 0x00054b10  /apex/com.android.runtime/lib64/bionic/libc.so (scudo::die+20)
    #02 pc 0x0005f234  /apex/com.android.runtime/lib64/bionic/libc.so (scudo::Allocator::deallocate+328)
    #03 pc 0x00053100  /apex/com.android.runtime/lib64/bionic/libc.so (free+32)
    #04 pc 0x0001b840  /data/app/com.example.app/lib/arm64/libnetwork.so (HttpClient::onResponse+224)
    #05 pc 0x0001c100  /data/app/com.example.app/lib/arm64/libnetwork.so (HttpClient::processQueue+180)
```

**分析思路：**

1. **`invalid chunk state when deallocating`**：Scudo 在 `free()` 时检测到 chunk header 的 State 字段不是 `Allocated`（已分配状态）。最常见的原因是 **Double-Free**——同一块内存被 `free()` 了两次。
2. **为什么只在 Android 11+ 出现**：Android 10 使用 jemalloc，不检测 Double-Free，内存被静默地双重释放但不会立即崩溃（只是堆结构被悄悄破坏，可能在后续的任意时刻以不相关的方式崩溃）。Android 11 切换到 Scudo 后，Double-Free 被即时检测。
3. **定位 Double-Free 的源头**：通过 `addr2line` 还原 `HttpClient::onResponse` 处的代码：

   ```cpp
   void HttpClient::onResponse(Response* resp) {
       processBody(resp->body);
       free(resp->body);       // 第一次释放
       resp->callback(resp);   // callback 内部可能再次释放 resp->body
   }
   ```

4. **进一步分析**：`resp->callback` 中的用户回调在某些路径上也会调用 `free(resp->body)`，造成 Double-Free。
5. **使用 MTE（Sync 模式）或 HWASan 验证**：在测试设备上启用 MTE 同步模式编译，复现后 MTE 报告精确定位了两次 `free()` 的调用点。

**根因：**

`HttpClient::onResponse` 释放了 `resp->body` 后未将指针置 NULL，后续的 callback 回调路径中存在条件分支会再次释放同一指针。在 jemalloc 时代这个 bug 表现为"偶发的堆损坏→随机崩溃"，很难归因；切换到 Scudo 后被精确检测。

**修复方案：**

1. **短期修复**：释放后立即置 NULL，并在 `free()` 前增加 NULL 检查：

   ```cpp
   void HttpClient::onResponse(Response* resp) {
       processBody(resp->body);
       free(resp->body);
       resp->body = nullptr;   // 置 NULL，防止 Double-Free
       resp->callback(resp);
   }
   ```

2. **长期治理**：
   - 将裸指针 + 手动 `free()` 替换为 `std::unique_ptr`，利用 RAII 机制自动管理生命周期。
   - 在 CI 中增加 HWASan 构建变体，每日运行自动化测试以捕获内存安全问题。
   - 对线上 Release 版本启用 GWP-ASan（概率性采样检测），持续监控堆安全问题。

```
修复前后对比：
┌──────────────────────┬────────────┬───────────┐
│ 指标                  │ 修复前      │ 修复后     │
├──────────────────────┼────────────┼───────────┤
│ Scudo SIGABRT 日均量  │ 4,200      │ 35        │
│ Android 11+ 崩溃率    │ 0.15%      │ 0.02%     │
│ 网络模块相关 NE       │ 2,800/天   │ 18/天     │
└──────────────────────┴────────────┴───────────┘
```

## 8. 总结

本篇从硬件（MMU/TLB）到内核（Page Fault/VMA/mmap）再到用户态（栈/堆/Scudo/MTE），逐层拆解了 SIGSEGV/SIGBUS/SIGABRT 背后的内存管理与保护机制。

核心要点：

1. **虚拟内存翻译由 MMU 硬件执行**，翻译失败的两种类型（Translation Fault → SEGV_MAPERR / Permission Fault → SEGV_ACCERR）直接决定 Tombstone 排查方向。
2. **Page Fault 有三种结局**——Demand Paging、COW、非法访问。只有最后一种才会产生信号，前两种是操作系统的正常行为。
3. **mmap 是 Android 内存布局的基石**。mmap 映射的文件被截断/删除会触发 SIGBUS，这在热更新和文件 I/O 场景中尤为常见。
4. **栈由 Guard Page 保护**，栈溢出触发 SEGV_ACCERR。但 Guard Page 可能被"跳过"（Stack Clash），编译器的 `-fstack-clash-protection` 是关键防线。
5. **Scudo 替代 jemalloc 后大幅提升了堆安全检测能力**——Double-Free、Heap BOF、Invalid Free 都能被即时检测并触发 SIGABRT。很多"新增的"SIGABRT 实际上是旧 bug 被暴露。
6. **MTE 是未来的终极方案**——硬件级 tag 检查，~3% 开销即可实时检测 UAF 和 Buffer Overflow，Android 14+ 已在系统进程中默认启用。

在下一篇中，我们将深入 Android 的崩溃捕获架构——debuggerd 和 Tombstone 的生成流程，理解 Tombstone 中每一行信息从何而来、为何有时信息不完整。
