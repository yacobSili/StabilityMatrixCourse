### Patch #2: ARM64 内存映射范围检查优化 (arch/arm64/mm/mmu.c)

**基本信息**：
- **Commit ID**：6f7006bd789999662b9d2057e78fb0f5d6871eea（内核升级 patch 的一部分）
- **提交时间**：2025-03-12 10:17:26 +0800
- **作者**：yulong.chen <yulong.chen@transsion.com>
- **模块**：ARM64 内存管理（MMU）
- **文件变更**：`arch/arm64/mm/mmu.c`
- **变更位置**：第 439 行（`create_mapping_noalloc`）、第 466 行（`update_mapping_prot`）

**变更内容**：

**修改前**：
```c
if ((virt >= PAGE_END) && (virt < VMALLOC_START)) {
    // 处理逻辑
}
```

**修改后**：
```c
if (virt < PAGE_OFFSET) {
    // 处理逻辑
}
```

**变更说明**：
- 简化了虚拟地址范围的检查条件
- 从检查 `PAGE_END` 到 `VMALLOC_START` 的范围，改为只检查是否小于 `PAGE_OFFSET`
- 这个变更出现在两个关键函数中：
  1. `create_mapping_noalloc`：创建内存映射（不分配页面）
  2. `update_mapping_prot`：更新映射保护属性

**代码分析**：

#### ARM64 内存布局（关键常量）

在 ARM64 架构中，虚拟地址空间布局如下：

```
┌─────────────────────────────────────┐
│  VMALLOC_START (~0xffff800000000000)│ ← 内核虚拟内存区域开始
│                                     │
│  VMALLOC 区域                       │
│                                     │
├─────────────────────────────────────┤
│  PAGE_END                           │ ← 页面映射区域结束
│                                     │
│  直接映射区域（Direct Mapping）     │
│  （线性映射，virt = phys + offset） │
│                                     │
├─────────────────────────────────────┤
│  PAGE_OFFSET (~0xffff800000000000)  │ ← 内核虚拟地址空间开始
│                                     │
│  用户空间                           │
│                                     │
└─────────────────────────────────────┘
```

**关键常量关系**：
- `PAGE_OFFSET`：内核虚拟地址空间的起始地址
- `PAGE_END`：直接映射区域的结束地址
- `VMALLOC_START`：vmalloc 区域的起始地址

**变更影响分析**：

1. **原条件**：`((virt >= PAGE_END) && (virt < VMALLOC_START))`
   - 检查的是 `PAGE_END` 到 `VMALLOC_START` 之间的范围
   - 这个范围通常很小或不存在（取决于配置）

2. **新条件**：`(virt < PAGE_OFFSET)`
   - 检查的是所有小于 `PAGE_OFFSET` 的地址
   - 这包括用户空间地址和部分内核地址

**潜在问题**：

⚠️ **关键问题**：如果 `PAGE_END` 和 `PAGE_OFFSET` 的关系不是预期的，这个变更可能导致：

1. **映射范围扩大**：
   - 如果 `PAGE_END < PAGE_OFFSET`，新条件会包含更多地址
   - 可能导致不应该被处理的地址被处理

2. **映射范围缩小**：
   - 如果原条件检查的范围在新条件下被排除，可能导致某些地址无法正确映射

**影响分析**：

#### IO 路径影响：**高风险** ⚠️

1. **Page Cache 映射影响**：
   - `create_mapping_noalloc` 和 `update_mapping_prot` 用于创建和更新内存映射
   - **如果映射范围判断错误，可能导致 Page Cache 页面无法正确映射**
   - Page Cache 是文件 IO 的核心机制，映射失败会导致：
     - 文件读取时无法使用 Page Cache，必须从磁盘读取
     - 文件写入时无法缓存到 Page Cache，必须直接写磁盘
     - **结果：IO 操作增加，iowait 上升**

2. **内存映射失败的影响**：
   - 如果某些内存区域无法正确映射，可能导致：
     - 内存分配失败或回退到更慢的路径
     - 触发更多的内存回收（kswapd）
     - 增加 swap IO 操作
     - **结果：间接导致 IO 压力增加**

#### 内存影响：**高风险** ⚠️

1. **内存映射范围改变**：
   - 如果映射范围判断错误，可能导致：
     - 某些内存区域无法正确映射
     - 内存分配可能失败或使用更慢的路径
     - 可能触发更多的内存回收

2. **Page Cache 影响**：
   - Page Cache 依赖于正确的内存映射
   - 映射问题可能导致 Page Cache 效率降低
   - 可能导致更多的内存回收和 swap IO

#### 调度影响：**中风险**

1. **内存分配延迟**：
   - 如果映射失败导致内存分配延迟，可能影响进程调度
   - 可能导致进程在等待内存时阻塞

#### 同步影响：**低风险**

- 此变更主要影响映射范围检查，不太可能引入新的同步问题

#### 性能影响：**高风险** ⚠️

1. **IO 性能下降**：
   - 如果 Page Cache 映射失败，会导致：
     - **缓存命中率下降**：无法使用 Page Cache，必须从磁盘读取
     - **IO 延迟增加**：每次文件操作都需要磁盘 IO
     - **IO 吞吐量下降**：无法利用 Page Cache 的批量操作
   - **结果：直接导致 iowait 增加和 ANR 增加**

2. **内存性能下降**：
   - 映射问题可能导致内存分配效率降低
   - 可能触发更多的内存回收，间接影响 IO

**风险评估**：

- **风险等级**：**高风险** ⚠️⚠️
- **风险原因**：
  1. **直接影响 Page Cache**：内存映射是 Page Cache 的基础，映射问题会直接影响 IO 性能
  2. **条件变更可能引入 bug**：从复杂条件改为简单条件，如果逻辑不完整，可能导致映射范围错误
  3. **时间相关性**：此变更与 IO 性能劣化时间高度相关
  4. **影响范围广**：内存映射影响所有使用 Page Cache 的文件操作

  - Page Cache 映射失败或效率降低
  - 文件 IO 必须直接访问磁盘，无法使用缓存
  - 导致 iowait 增加和 ANR 增加

### 6.2 根本原因

**主要原因**：

#### 1. arch/arm64/mm/mmu.c 的 VA-range 检查变更 ⚠️⚠️⚠️

**提交信息**：
- **Commit**: `621619f626cb`
- **标题**: `arm64: mm: fix VA-range sanity check`
- **作者**: Mark Rutland <mark.rutland@arm.com>
- **日期**: 2023-06-15

**变更内容**：
- 从 `((virt >= PAGE_END) && (virt < VMALLOC_START))` 改为 `(virt < PAGE_OFFSET)`
- 提交说明称这是"修复历史遗留问题"，但实际上改变了检查逻辑

**问题分析**：
- 原条件检查的是 `PAGE_END` 到 `VMALLOC_START` 之间的范围
- 新条件检查所有小于 `PAGE_OFFSET` 的地址
- **如果 `PAGE_END` 和 `PAGE_OFFSET` 的关系不是预期的，这个变更可能导致映射范围错误**
- 可能影响 Page Cache 的映射，导致文件 IO 无法使用缓存

**影响**：
- 直接影响 Page Cache 映射
- 文件 IO 必须直接访问磁盘
- 导致 iowait 增加和 ANR 增加

#### 2. mm/vmscan.c 的内存压缩限制变更 ⚠️⚠️

**提交信息**：
- **Commit**: `8d7132a67eeb`
- **标题**: `mm, vmscan: prevent infinite loop for costly GFP_NOIO | __GFP_RETRY_MAYFAIL allocations`
- **作者**: Vlastimil Babka <vbabka@suse.cz>
- **日期**: 2024-02-21

**变更内容**：
- 引入 `gfp_compaction_allowed()` 检查
- 限制内存压缩（compaction）的使用
- 目的是防止无限循环，但可能引入新问题

**问题分析**：
- 限制压缩可能导致内存碎片化
- 内存碎片化可能导致分配失败
- 分配失败可能触发更多 swap IO
- **间接导致 IO 压力增加**

**影响**：
- 内存分配效率降低
- 更多 swap IO
- 间接导致 IO 压力增加和 iowait 上升

**次要原因**：

#### 3. mm/page-writeback.c 的阈值限制

- 添加 UINT_MAX 检查，限制脏页阈值
- 可能导致写回更频繁

#### 4. block/blk-mq.c 的内存屏障

- 添加内存屏障，可能增加 IO 延迟

#### 机制 3：写回策略改变（次要影响）⚠️

**✅ 已确认在 2024-05_r2 → 2024-11_r3 中引入的关键变更**：

**内存管理相关（高优先级）**：

1. **arch/arm64/mm/mmu.c** ⚠️⚠️⚠️

   - **影响**: VA-range 检查逻辑改变，可能影响 Page Cache 映射

   - **风险**: 中，可能增加 IO 频率

4. **mm/filemap.c** ⚠️
   - **变更内容**: 添加 Android vendor hooks（trace 点）
   - **影响**: 仅添加 trace，不太可能影响性能
   - **风险**: 低

**IO 层相关（中优先级）**：

   - **影响**: 可能增加 IO 延迟，但主要是修复潜在的竞争条件
   - **风险**: 中，可能略微增加 IO 延迟

   - **风险**: 低

```bash
#!/bin/bash
# check_kernel_fixes.sh - 检查内核版本修复

REPO_URL="https://android.googlesource.com/kernel/common"
OLD_VERSION="android13-5.15-2024-11_r3"
NEW_VERSION="android13-5.15-2024-11_r4"

KEY_FILES=(
    "arch/arm64/mm/mmu.c"
    "mm/vmscan.c"
    "mm/page-writeback.c"
    "mm/filemap.c"
    "block/blk-mq.c"
    "fs/f2fs/data.c"
)

echo "=========================================="
echo "旧版本: $OLD_VERSION"
echo "新版本: $NEW_VERSION"
echo ""

# 检查仓库是否存在
if [ ! -d "common" ]; then
    echo "克隆仓库..."
    git clone $REPO_URL common
fi

cd common

# 获取标签
echo "获取标签..."
git fetch --tags

# 检查标签是否存在
if ! git rev-parse "$OLD_VERSION" >/dev/null 2>&1; then
    echo "错误: 标签 $OLD_VERSION 不存在"
    exit 1
fi

if ! git rev-parse "$NEW_VERSION" >/dev/null 2>&1; then
    echo "警告: 标签 $NEW_VERSION 不存在，尝试查找最新版本..."
    LATEST=$(git tag | grep "2024-11" | sort -V | tail -1)
    echo "找到最新版本: $LATEST"
    NEW_VERSION=$LATEST
fi

echo ""
echo "=========================================="
echo "对比关键文件变更"
echo "=========================================="

for file in "${KEY_FILES[@]}"; do
    echo ""
    echo "--- $file ---"
    changes=$(git diff $OLD_VERSION $NEW_VERSION -- "$file" | wc -l)
    if [ "$changes" -gt 0 ]; then
        echo "✅ 有变更 ($changes 行)"
        echo "变更摘要:"
        git diff $OLD_VERSION $NEW_VERSION --stat -- "$file"
        echo ""
        echo "关键变更:"
        git diff $OLD_VERSION $NEW_VERSION -- "$file" | grep -E "^\+|^\-" | head -20
    else
        echo "❌ 无变更"
    fi
done

echo ""
echo "=========================================="
echo "提交日志（相关修复）"
echo "=========================================="
git log $OLD_VERSION..$NEW_VERSION --oneline -- \
    "${KEY_FILES[@]}" | head -20
```

- **0-5%**：正常范围
- **5-10%**：需要关注
- **10-15%**：可能有问题
- **15-20%**：明显 IO 阻塞
- **>20%**：严重 IO 阻塞（本问题场景）

**IO PSI 阈值**：
- **some > 10%**：需要关注
- **some > 20%**：需要处理
- **full > 5%**：严重问题
