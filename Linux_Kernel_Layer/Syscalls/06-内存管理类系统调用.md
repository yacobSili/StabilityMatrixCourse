# 内存管理类系统调用

## 学习目标

- 理解内存管理类系统调用的作用
- 掌握 mmap/munmap 系统调用的实现
- 理解 brk/sbrk 系统调用的机制
- 了解 mprotect、madvise 等系统调用
- 掌握内存映射与内存管理子系统的交互
- 理解与你的性能问题的关联

## 背景介绍

内存管理类系统调用是应用程序管理内存的主要接口。理解这些系统调用的实现，有助于分析内存使用、优化内存分配，以及理解你遇到的性能问题。

## 核心概念

### 内存管理类系统调用分类

#### 1. 内存映射

- **mmap()** - 创建内存映射
- **munmap()** - 取消内存映射
- **mremap()** - 重映射内存区域

#### 2. 堆内存管理

- **brk()** - 调整堆边界
- **sbrk()** - brk 的封装（已废弃）

#### 3. 内存保护

- **mprotect()** - 修改内存保护属性
- **madvise()** - 提供内存使用建议
- **mincore()** - 检查内存页是否驻留

#### 4. 内存同步

- **msync()** - 同步内存映射到文件
- **sync()** - 同步所有文件系统

### mmap 系统调用详解

#### 函数签名

```c
SYSCALL_DEFINE6(mmap, unsigned long, addr, unsigned long, len,
                unsigned long, prot, unsigned long, flags,
                unsigned long, fd, unsigned long, off)
```

**参数**：
- `addr`：建议的映射地址（通常为 0，由内核选择）
- `len`：映射长度
- `prot`：保护标志（PROT_READ, PROT_WRITE, PROT_EXEC）
- `flags`：映射标志（MAP_SHARED, MAP_PRIVATE, MAP_ANONYMOUS 等）
- `fd`：文件描述符（匿名映射时为 -1）
- `off`：文件偏移

**返回值**：
- 成功：映射的虚拟地址
- 失败：MAP_FAILED (-1)

#### 实现流程

**位置**：`../common/arch/arm64/kernel/sys.c` 和 `../common/mm/mmap.c`

**关键步骤**：

1. **参数验证**
   ```c
   if (offset_in_page(off) != 0)
       return -EINVAL;
   ```

2. **调用通用实现**
   ```c
   return ksys_mmap_pgoff(addr, len, prot, flags, fd, off >> PAGE_SHIFT);
   ```

3. **ksys_mmap_pgoff 函数**
   - 验证参数
   - 创建 VMA（虚拟内存区域）
   - 建立页表映射
   - 返回虚拟地址

#### 内存映射类型

**1. 文件映射（MAP_SHARED / MAP_PRIVATE）**
- 映射文件到内存
- MAP_SHARED：修改会写回文件
- MAP_PRIVATE：修改不会写回文件（写时复制）

**2. 匿名映射（MAP_ANONYMOUS）**
- 不映射文件
- 用于分配内存
- 常用于 malloc 实现

#### 与内存管理子系统的交互

**VMA 管理**：
- 每个映射创建一个 VMA
- VMA 记录映射的地址范围、权限、文件等信息
- VMA 链表管理进程的所有映射

**页表管理**：
- 建立虚拟地址到物理地址的映射
- 延迟分配物理页面（按需分配）
- 使用页表项（PTE）记录映射信息

### munmap 系统调用详解

#### 函数签名

```c
SYSCALL_DEFINE2(munmap, unsigned long, addr, size_t, len)
```

**功能**：
- 取消内存映射
- 释放虚拟地址空间
- 释放物理页面（如果没有其他映射）

#### 实现流程

**位置**：`../common/mm/mmap.c`

**关键步骤**：

1. **查找 VMA**
   ```c
   struct vm_area_struct *vma = find_vma(current->mm, addr);
   ```

2. **取消映射**
   ```c
   do_munmap(current->mm, addr, len, NULL);
   ```

3. **释放资源**
   - 释放物理页面
   - 更新页表
   - 删除 VMA

### brk 系统调用详解

#### 函数签名

```c
SYSCALL_DEFINE1(brk, unsigned long, brk)
```

**功能**：
- 调整堆的边界
- 扩展或收缩堆内存
- 用于 malloc/free 实现

#### 实现流程

**位置**：`../common/mm/mmap.c`

**关键步骤**：

1. **获取当前堆边界**
   ```c
   unsigned long newbrk, oldbrk = current->mm->brk;
   ```

2. **调整堆边界**
   ```c
   if (brk > oldbrk) {
       // 扩展堆
       do_brk_flags(oldbrk, brk - oldbrk, 0, NULL);
   } else if (brk < oldbrk) {
       // 收缩堆
       do_munmap(current->mm, brk, oldbrk - brk, NULL);
   }
   ```

3. **更新堆边界**
   ```c
   current->mm->brk = brk;
   ```

#### 堆内存管理

**机制**：
- 堆是进程的线性地址空间的一部分
- 通过 brk 系统调用调整堆大小
- malloc/free 在堆上分配和释放内存

### mprotect 系统调用详解

#### 函数签名

```c
SYSCALL_DEFINE3(mprotect, unsigned long, start, size_t, len, unsigned long, prot)
```

**功能**：
- 修改内存区域的保护属性
- 可以改变读、写、执行权限

**参数**：
- `start`：内存区域起始地址
- `len`：内存区域长度
- `prot`：新的保护标志

#### 实现流程

**位置**：`../common/mm/mprotect.c`

**关键步骤**：

1. **查找 VMA**
   ```c
   struct vm_area_struct *vma = find_vma(current->mm, start);
   ```

2. **修改保护属性**
   ```c
   do_mprotect_pkey(start, len, prot, -1);
   ```

3. **更新页表**
   - 修改页表项的权限位
   - 刷新 TLB

### madvise 系统调用详解

#### 函数签名

```c
SYSCALL_DEFINE3(madvise, unsigned long, start, size_t, len, int, behavior)
```

**功能**：
- 向内核提供内存使用建议
- 帮助内核优化内存管理

**常用建议**：
- `MADV_NORMAL`：正常访问模式
- `MADV_RANDOM`：随机访问模式
- `MADV_SEQUENTIAL`：顺序访问模式
- `MADV_WILLNEED`：预取页面
- `MADV_DONTNEED`：释放页面

### mincore 系统调用详解

#### 函数签名

```c
SYSCALL_DEFINE3(mincore, unsigned long, start, size_t, len, unsigned char __user *, vec)
```

**功能**：
- 检查内存页是否驻留在物理内存中
- 返回每个页面的驻留状态

## 实际应用

### 追踪内存映射

```bash
# 追踪 mmap 系统调用
strace -e trace=mmap,munmap,brk <application>

# 查看进程的内存映射
cat /proc/<pid>/maps

# 查看内存使用
cat /proc/<pid>/smaps
```

### 分析内存使用

```bash
# 统计内存相关系统调用
strace -c -e trace=mmap,munmap,brk,mprotect <application>

# 查看堆使用情况
cat /proc/<pid>/status | grep -i heap
```

## 与你的性能问题的关联

内存管理类系统调用与你的性能问题密切相关：

### 1. mmap 性能影响

**可能的问题**：
- 频繁的 mmap/munmap 调用
- 大内存映射的开销
- 内存映射与文件系统的交互

**分析方法**：
```bash
# 统计 mmap 调用
strace -c -e trace=mmap,munmap <go_application>
```

### 2. brk 性能影响

**可能的问题**：
- 频繁的堆扩展/收缩
- 堆碎片化
- GO 语言的大对象分配

**分析方法**：
```bash
# 统计 brk 调用
strace -c -e trace=brk <go_application>

# 查看堆使用
cat /proc/<pid>/status | grep -E "VmData|VmStk|VmExe"
```

### 3. 内存回收交互

**可能的问题**：
- GO GC 与内核内存回收的冲突
- 大对象分配触发内存回收
- 内存压力导致性能下降

**分析思路**：
- 监控 mmap/brk 调用频率
- 监控内存回收活动
- 分析两者的时间关系

### 4. 与升级问题的关联

**你的升级问题**（2024-05_r2 → 2024-11_r3）可能涉及：

1. **内存管理策略变更**
   - mmap 实现的变化
   - 内存回收策略的变化
   - 可能影响 GO 应用的内存使用

2. **内存分配器变更**
   - brk 处理的变化
   - 可能影响堆内存分配

3. **内存映射优化**
   - mmap 优化的副作用
   - 可能影响某些使用模式

**分析方法**：
```powershell
# 对比两个版本的内存管理变更
cd ../common
git diff android13-5.15-2024-05_r2 android13-5.15-2024-11_r3 -- mm/mmap.c mm/mprotect.c
```

## 总结

### 核心要点

1. **mmap 是内存映射的主要接口**：支持文件映射和匿名映射
2. **brk 管理堆内存**：通过调整堆边界分配内存
3. **mprotect 修改内存保护**：可以动态改变内存权限
4. **与内存管理子系统紧密交互**：通过 VMA 和页表管理

### 关键系统调用

- **mmap/munmap**：创建/取消内存映射
- **brk**：调整堆边界
- **mprotect**：修改内存保护
- **madvise**：提供内存使用建议
- **mincore**：检查内存驻留

### 与性能问题的关联

- **频繁的内存操作**：mmap/brk 的频繁调用
- **内存回收交互**：与内核内存回收的冲突
- **堆碎片化**：影响内存分配效率
- **升级影响**：内存管理策略的变更
