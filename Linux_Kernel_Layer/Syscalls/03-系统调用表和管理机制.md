# 系统调用表和管理机制

## 学习目标

- 理解 sys_call_table 的定义和初始化
- 掌握系统调用号的分配和管理
- 深入理解 SYSCALL_DEFINE 宏
- 了解系统调用的注册机制
- 理解系统调用表的查找和调用过程
- 掌握 ARM64 特定的系统调用实现

## 背景介绍

系统调用表是内核管理所有系统调用的核心数据结构。理解系统调用表的管理机制，有助于理解系统调用的分发过程，以及如何添加新的系统调用。

## 核心概念

### sys_call_table 的定义

#### 系统调用表结构

**位置**：`../common/arch/arm64/kernel/sys.c`

**定义**：
```c
const syscall_fn_t sys_call_table[__NR_syscalls] = {
    [0 ... __NR_syscalls - 1] = __arm64_sys_ni_syscall,
#include <asm/unistd.h>
};
```

**特点**：
- 数组类型：`syscall_fn_t`（函数指针数组）
- 数组大小：`__NR_syscalls`（系统调用总数）
- 默认值：所有未定义的系统调用指向 `__arm64_sys_ni_syscall`（未实现系统调用）
- 通过 `#include <asm/unistd.h>` 填充具体系统调用

#### syscall_fn_t 类型

**定义**：
```c
typedef long (*syscall_fn_t)(const struct pt_regs *regs);
```

**特点**：
- 函数指针类型
- 参数：`pt_regs` 结构体指针（包含所有寄存器）
- 返回值：`long`（系统调用返回值）

### 系统调用号的分配和管理

#### 系统调用号定义

**位置**：`../common/arch/arm64/include/uapi/asm/unistd.h`

**格式**：
```c
#define __NR_open 56
#define __NR_close 57
#define __NR_read 63
#define __NR_write 64
// ...
```

**特点**：
- 每个系统调用有唯一的编号
- 编号从 0 开始
- 编号是稳定的（向后兼容）

#### 系统调用号的使用

**在系统调用表中**：
```c
#define __SYSCALL(nr, sym) [nr] = __arm64_##sym,
#include <asm/unistd.h>
```

**展开后**：
```c
[__NR_open] = __arm64_sys_open,
[__NR_close] = __arm64_sys_close,
[__NR_read] = __arm64_sys_read,
// ...
```

### SYSCALL_DEFINE 宏详解

#### 宏定义

**位置**：`../common/include/linux/syscalls.h`

**基本形式**：
```c
SYSCALL_DEFINE0(name)           // 0 个参数
SYSCALL_DEFINE1(name, type1, arg1)  // 1 个参数
SYSCALL_DEFINE2(name, type1, arg1, type2, arg2)  // 2 个参数
// ... 最多 6 个参数
```

#### 宏展开

**示例**：
```c
SYSCALL_DEFINE2(open, const char __user *, filename, int, flags)
```

**展开为**：
```c
long __arm64_sys_open(const struct pt_regs *regs)
{
    return __se_sys_open(regs->regs[0], regs->regs[1]);
}

static long __se_sys_open(const char __user *filename, int flags)
{
    long ret = __do_sys_open(filename, flags);
    return ret;
}

static long __do_sys_open(const char __user *filename, int flags)
{
    // 实际的系统调用实现
    // ...
}
```

#### 参数提取

**__se_sys_* 函数**：
- 从 `pt_regs` 中提取参数
- 进行类型转换
- 调用 `__do_sys_*` 函数

**参数映射**：
- `regs->regs[0]` → 第一个参数
- `regs->regs[1]` → 第二个参数
- `regs->regs[2]` → 第三个参数
- ...

#### 安全检查

**用户空间指针检查**：
- `__user` 标记表示用户空间指针
- 需要验证指针有效性
- 防止内核空间访问用户空间无效指针

**示例**：
```c
SYSCALL_DEFINE2(open, const char __user *, filename, int, flags)
{
    // filename 是用户空间指针
    // 需要验证和复制
    char *kernel_filename;
    kernel_filename = strndup_user(filename, PATH_MAX);
    // ...
}
```

### 系统调用的注册机制

#### 自动注册

**机制**：
- 通过 `#include <asm/unistd.h>` 自动注册
- 系统调用号在头文件中定义
- 系统调用表通过宏展开填充

**流程**：
1. 定义系统调用号（`unistd.h`）
2. 实现系统调用函数（使用 `SYSCALL_DEFINE` 宏）
3. 系统调用表自动包含该函数

#### 系统调用表初始化

**位置**：`../common/arch/arm64/kernel/sys.c`

**初始化过程**：
```c
const syscall_fn_t sys_call_table[__NR_syscalls] = {
    // 默认值：所有未定义的系统调用
    [0 ... __NR_syscalls - 1] = __arm64_sys_ni_syscall,
    
    // 通过包含 unistd.h 填充具体系统调用
    #include <asm/unistd.h>
};
```

**unistd.h 中的定义**：
```c
#define __SYSCALL(nr, sym) [nr] = __arm64_##sym,
__SYSCALL(__NR_open, sys_open)
__SYSCALL(__NR_close, sys_close)
// ...
```

### 系统调用表的查找和调用

#### 查找过程

**位置**：`../common/arch/arm64/kernel/syscall.c`

**关键代码**：
```c
static void invoke_syscall(struct pt_regs *regs, unsigned int scno,
                           unsigned int sc_nr,
                           const syscall_fn_t syscall_table[])
{
    long ret;
    
    if (scno < sc_nr) {
        syscall_fn_t syscall_fn;
        // 使用 array_index_nospec 防止 Spectre 攻击
        syscall_fn = syscall_table[array_index_nospec(scno, sc_nr)];
        ret = __invoke_syscall(regs, syscall_fn);
    } else {
        ret = do_ni_syscall(regs, scno);
    }
    
    syscall_set_return_value(current, regs, 0, ret);
}
```

**步骤**：
1. 检查系统调用号有效性（`scno < sc_nr`）
2. 使用 `array_index_nospec` 安全地访问数组（防止 Spectre 攻击）
3. 从系统调用表获取函数指针
4. 调用系统调用函数
5. 设置返回值

#### 调用过程

**__invoke_syscall 函数**：
```c
static long __invoke_syscall(struct pt_regs *regs, syscall_fn_t syscall_fn)
{
    return syscall_fn(regs);
}
```

**过程**：
- 直接调用函数指针
- 传递 `pt_regs` 结构体
- 返回系统调用结果

### ARM64 特定的系统调用实现

#### __arm64_sys_* 函数

**命名约定**：
- 所有 ARM64 系统调用函数以 `__arm64_sys_` 开头
- 例如：`__arm64_sys_open`, `__arm64_sys_read`

**实现**：
- 从 `pt_regs` 提取参数
- 调用通用实现函数
- 返回结果

#### 兼容性支持

**32 位兼容**：
```c
#ifdef CONFIG_COMPAT
const syscall_fn_t compat_sys_call_table[__NR_compat_syscalls] = {
    // 32 位系统调用表
};
#endif
```

**特点**：
- 支持 32 位应用程序
- 使用独立的系统调用表
- 参数转换和兼容性处理

### 未实现的系统调用

#### __arm64_sys_ni_syscall

**定义**：
```c
long __arm64_sys_ni_syscall(const struct pt_regs *regs)
{
    return -ENOSYS;
}
```

**作用**：
- 处理未实现的系统调用
- 返回 `-ENOSYS` 错误
- 作为系统调用表的默认值

### 系统调用表的保护

#### 只读保护

**机制**：
- 系统调用表标记为 `const`
- 编译时确定，运行时不可修改
- 防止恶意修改系统调用表

#### 安全访问

**array_index_nospec**：
- 防止 Spectre 变体 1 攻击
- 在数组访问前插入屏障
- 防止推测执行泄露数据

## 实际应用

### 添加新系统调用

**步骤**：

1. **定义系统调用号**
   ```c
   // arch/arm64/include/uapi/asm/unistd.h
   #define __NR_my_syscall 450
   ```

2. **实现系统调用**
   ```c
   // kernel/my_syscall.c
   SYSCALL_DEFINE1(my_syscall, int, arg)
   {
       // 实现
       return 0;
   }
   ```

3. **系统调用表自动包含**
   - 通过 `#include <asm/unistd.h>` 自动注册

### 查看系统调用表

```bash
# 在内核中查看系统调用表（需要 root 权限）
cat /proc/kallsyms | grep sys_call_table

# 使用 SystemTap 追踪系统调用
stap -e 'probe kernel.function("invoke_syscall") { printf("%d\n", $scno) }'
```

## 与你的性能问题的关联

系统调用表管理可能与你的性能问题相关：

1. **系统调用查找开销**：数组查找的时间复杂度是 O(1)，但仍有开销
2. **函数指针调用**：间接调用可能影响性能
3. **安全检查开销**：`array_index_nospec` 等安全检查的开销

**分析方法**：
```bash
# 统计系统调用频率
strace -c <application>

# 查看特定系统调用的性能
perf record -e syscalls:sys_enter_* <application>
```

## 总结

### 核心要点

1. **系统调用表是函数指针数组**：存储所有系统调用的处理函数
2. **SYSCALL_DEFINE 宏简化定义**：自动生成参数提取和类型转换代码
3. **系统调用号唯一标识**：每个系统调用有唯一的编号
4. **自动注册机制**：通过包含头文件自动注册系统调用

### 关键机制

- **系统调用表**：`sys_call_table` 数组存储所有系统调用函数
- **SYSCALL_DEFINE 宏**：简化系统调用定义
- **参数提取**：从 `pt_regs` 自动提取参数
- **安全检查**：防止 Spectre 攻击和无效访问
