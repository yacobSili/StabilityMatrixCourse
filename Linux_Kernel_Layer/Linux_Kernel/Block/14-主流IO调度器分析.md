# 主流 IO 调度器分析

## 学习目标

- 了解主流 IO 调度器的特点和算法
- 理解各调度器的适用场景
- 掌握调度器的选择建议
- 了解调度器的配置和调优方法

## 概述

Linux 内核提供了多种 IO 调度器，每种调度器都有其特定的算法和适用场景。选择合适的调度器可以显著提高 IO 性能。

本文档分析主流 IO 调度器的特点、算法和使用建议。

---

## 一、IO 调度器概览

### 主流调度器列表

| 调度器 | 特点 | 适用场景 |
|--------|------|---------|
| **None** | 无调度器，FIFO 顺序 | 现代 SSD、高性能场景 |
| **MQ-Deadline** | 基于截止时间的调度 | 通用场景、平衡吞吐量和延迟 |
| **Kyber** | 基于延迟的调度 | 低延迟要求、交互式应用 |
| **BFQ** | 基于公平性的调度 | 多任务场景、公平性要求高 |

---

## 二、None（无调度器）

### 特点

**算法**：
- 不使用 IO 调度器
- 请求按 FIFO 顺序处理
- 直接发送给驱动

**优势**：
- **零开销**：无调度器开销
- **低延迟**：请求直接发送，延迟最低
- **简单**：实现简单，易于理解

**劣势**：
- **无优化**：不进行请求合并和重排序
- **不公平**：可能造成请求饥饿

### 适用场景

**推荐使用**：
- 现代 SSD 设备（内部有调度）
- 性能要求极高的场景
- 不需要请求重排序的场景

**不推荐使用**：
- 传统机械硬盘
- 需要请求合并的场景
- 多任务竞争的场景

### 配置方法

```bash
# 设置调度器为 none
echo none > /sys/block/sda/queue/scheduler
```

---

## 三、MQ-Deadline

### 特点

**算法**：
- **基于截止时间**：每个请求有截止时间
- **读优先**：读请求优先于写请求
- **批处理**：相同方向的请求批量处理

**关键参数**：
- `read_expire`：读请求截止时间（默认 500ms）
- `write_expire`：写请求截止时间（默认 5s）
- `fifo_batch`：批处理大小（默认 16）

**优势**：
- **平衡性能**：平衡吞吐量和延迟
- **防止饥饿**：写请求不会长时间等待
- **通用性强**：适用于大多数场景

**劣势**：
- **参数敏感**：需要根据场景调整参数
- **算法简单**：不如 BFQ 公平

### 适用场景

**推荐使用**：
- 通用场景
- 需要平衡吞吐量和延迟
- 桌面和服务器系统

**配置示例**：
```bash
# 设置调度器为 mq-deadline
echo mq-deadline > /sys/block/sda/queue/scheduler

# 调整参数
echo 1000 > /sys/block/sda/queue/iosched/read_expire
echo 5000 > /sys/block/sda/queue/iosched/write_expire
```

## 四、Kyber

### 特点

**算法**：
- **基于延迟**：控制请求的延迟
- **多域调度**：按请求类型（读/写/丢弃）分类
- **动态调整**：根据延迟动态调整队列深度

**关键参数**：
- `read_latency`：读请求延迟目标（默认 2ms）
- `write_latency`：写请求延迟目标（默认 10ms）
- `async_depth`：异步请求深度

**优势**：
- **低延迟**：专注于降低延迟
- **自适应**：根据延迟动态调整
- **适合交互**：适合交互式应用

**劣势**：
- **吞吐量可能较低**：优先考虑延迟而非吞吐量
- **参数调优复杂**：需要根据场景调整

### 适用场景

**推荐使用**：
- 低延迟要求
- 交互式应用
- 桌面系统

**配置示例**：
```bash
# 设置调度器为 kyber
echo kyber > /sys/block/sda/queue/scheduler

# 调整延迟目标
echo 1000000 > /sys/block/sda/queue/iosched/read_latency_ns
echo 5000000 > /sys/block/sda/queue/iosched/write_latency_ns
```

## 五、BFQ

### 特点

**算法**：
- **基于公平性**：保证每个进程公平获得 IO 带宽
- **权重分配**：根据进程权重分配 IO 资源
- **低延迟**：优化交互式应用的延迟

**关键参数**：
- `low_latency`：低延迟模式（默认开启）
- `slice_idle`：空闲时间片
- `timeout_sync`：同步请求超时

**优势**：
- **公平性强**：保证进程间公平
- **低延迟**：优化交互式应用
- **适合多任务**：适合多任务竞争场景

**劣势**：
- **开销较大**：算法复杂，开销较高
- **吞吐量可能较低**：优先考虑公平性

### 适用场景

**推荐使用**：
- 多任务场景
- 需要公平性的场景
- 桌面系统

**配置示例**：
```bash
# 设置调度器为 bfq
echo bfq > /sys/block/sda/queue/scheduler

# 调整参数
echo 1 > /sys/block/sda/queue/iosched/low_latency
```

## 六、调度器选择建议

### 选择指南

**根据设备类型选择**：

| 设备类型 | 推荐调度器 | 原因 |
|---------|-----------|------|
| **NVMe SSD** | None 或 MQ-Deadline | 硬件性能好，不需要复杂调度 |
| **SATA SSD** | MQ-Deadline 或 Kyber | 需要一定调度优化 |
| **机械硬盘** | MQ-Deadline 或 BFQ | 需要请求合并和重排序 |
| **eMMC/UFS** | MQ-Deadline | 平衡性能和延迟 |

**根据应用场景选择**：

| 应用场景 | 推荐调度器 | 原因 |
|---------|-----------|------|
| **高性能服务器** | None | 追求极致性能 |
| **通用桌面** | MQ-Deadline | 平衡性能和延迟 |
| **交互式应用** | Kyber 或 BFQ | 低延迟要求 |
| **多任务竞争** | BFQ | 公平性要求 |

### Android 系统中的选择

**Android 系统**：
- **默认**：MQ-Deadline（大多数设备）
- **高性能设备**：None（某些高端设备）
- **低延迟设备**：Kyber（某些设备）

## 七、调度器性能对比

### 性能指标

| 调度器 | 吞吐量 | 延迟 | 公平性 | CPU 开销 |
|--------|--------|------|--------|---------|
| **None** | 最高 | 最低 | 低 | 最低 |
| **MQ-Deadline** | 高 | 低 | 中 | 低 |
| **Kyber** | 中 | 最低 | 中 | 中 |
| **BFQ** | 中 | 低 | 最高 | 高 |

### 选择建议

**性能优先**：None
**平衡性能**：MQ-Deadline
**低延迟优先**：Kyber
**公平性优先**：BFQ

## 总结

### 核心要点

1. **主流调度器**：
   - **None**：无调度器，性能最高
   - **MQ-Deadline**：基于截止时间，通用场景
   - **Kyber**：基于延迟，低延迟场景
   - **BFQ**：基于公平性，多任务场景

2. **选择原则**：
   - 根据设备类型选择
   - 根据应用场景选择
   - 根据性能要求选择

3. **配置和调优**：
   - 通过 sysfs 配置调度器
   - 根据场景调整参数
   - 监控性能指标

### 关键概念

- **调度器算法**：决定请求的调度顺序
- **调度器参数**：影响调度器的行为
- **调度器切换**：可以动态切换调度器
