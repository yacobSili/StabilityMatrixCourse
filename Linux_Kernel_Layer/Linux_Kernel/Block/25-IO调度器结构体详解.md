## 三、struct deadline_data - mq-deadline 调度器全局数据

**源码位置**: `block/mq-deadline.c:84-108`

**在此阶段的作用**: 调度器的核心数据结构，保存三个优先级队列和配置参数。

### 3.1 完整结构体定义

```c
struct deadline_data {
    /*
     * run time data
     */
    struct dd_per_prio per_prio[DD_PRIO_COUNT];  // 3个优先级队列

    /* Data direction of latest dispatched request. */
    enum dd_data_dir last_dir;      // 上次派发方向（READ/WRITE）
    unsigned int batching;          // 当前批次计数
    unsigned int starved;           // 写饥饿计数

    /*
     * settings that change how the i/o scheduler behaves
     */
    int fifo_expire[DD_DIR_COUNT];  // FIFO超时时间（读/写）
    int fifo_batch;                 // 批处理大小
    int writes_starved;             // 写饥饿阈值
    int front_merges;               // 是否允许前向合并
    u32 async_depth;                // 异步请求深度限制
    int prio_aging_expire;          // 优先级老化时间

    spinlock_t lock;                // 保护调度器数据
    spinlock_t zone_lock;           // 分区写锁（zoned设备）
};

// 优先级枚举
enum dd_prio {
    DD_RT_PRIO   = 0,    // 实时优先级
    DD_BE_PRIO   = 1,    // Best Effort
    DD_IDLE_PRIO = 2,    // 空闲优先级
    DD_PRIO_MAX  = 2,
};

enum { DD_PRIO_COUNT = 3 };

// 数据方向
enum dd_data_dir {
    DD_READ  = READ,     // 0
    DD_WRITE = WRITE,    // 1
};

enum { DD_DIR_COUNT = 2 };
```

### 3.2 核心字段详解 ⭐⭐⭐

| 字段名 | 类型 | 默认值 | 作用 |
|--------|------|--------|------|
| **优先级队列** ||||
| `per_prio[3]` | `struct dd_per_prio` | - | **三个独立队列**<br/>RT/BE/IDLE各有独立的红黑树和FIFO |
| **调度状态** ||||
| `last_dir` | `enum dd_data_dir` | DD_WRITE | **上次派发方向**<br/>用于批处理优化（连续同方向） |
| `batching` | `unsigned int` | 0 | **当前批次计数**<br/>与fifo_batch对比 |
| `starved` | `unsigned int` | 0 | **写饥饿计数**<br/>读请求连续多少次忽略写 |
| **配置参数** ||||
| `fifo_expire[2]` | `int` | [500ms, 5000ms] | **FIFO超时**<br/>[0]=读超时<br/>[1]=写超时 |
| `fifo_batch` | `int` | 16 | **批处理大小**<br/>同方向连续处理的最大请求数 |
| `writes_starved` | `int` | 2 | **写饥饿阈值**<br/>读连续忽略写超过此值后优先写 |
| `prio_aging_expire` | `int` | 10000ms | **优先级老化时间**<br/>低优先级等待超时后优先处理 |
| **其他** ||||
| `async_depth` | `u32` | 3/4 * tag_shift | 异步请求Tag深度限制 |
| `front_merges` | `int` | 1 | 是否允许前向合并 |
| `lock` | `spinlock_t` | - | 保护per_prio数据 |

**详细说明**：

**三个优先级队列的布局**：
```
deadline_data
├─ per_prio[DD_RT_PRIO]     优先级 0 (最高)
│  ├─ dispatch (链表)
│  ├─ sort_list[READ] (红黑树)
│  ├─ sort_list[WRITE] (红黑树)
│  ├─ fifo_list[READ] (FIFO)
│  └─ fifo_list[WRITE] (FIFO)
│
├─ per_prio[DD_BE_PRIO]     优先级 1 (默认)
│  ├─ dispatch
│  ├─ sort_list[READ]
│  ├─ sort_list[WRITE]
│  ├─ fifo_list[READ]
│  └─ fifo_list[WRITE]
│
└─ per_prio[DD_IDLE_PRIO]   优先级 2 (最低)
   ├─ dispatch
   ├─ sort_list[READ]
   ├─ sort_list[WRITE]
   ├─ fifo_list[READ]
   └─ fifo_list[WRITE]
```

**读写平衡机制**：
```c
// __dd_dispatch_request() 简化逻辑
if (!list_empty(&per_prio->fifo_list[DD_READ])) {
    // 有读请求
    if (deadline_fifo_request(dd, per_prio, DD_WRITE) &&
        (dd->starved++ >= dd->writes_starved))  // 写已饥饿？
        goto dispatch_writes;  // 是：优先处理写，starved=0
    
    // 否：继续处理读，starved++
    data_dir = DD_READ;
}

// 效果: 最多连续2次读后必须处理1次写（writes_starved=2）
```

### 3.3 配置参数的实际影响

| 参数 | 默认值 | 修改方式 | 影响 |
|------|-------|---------|------|
| `read_expire` | 500ms | `/sys/block/sda/queue/iosched/read_expire` | 读请求在FIFO中的最长等待时间 |
| `write_expire` | 5000ms | `/sys/block/sda/queue/iosched/write_expire` | 写请求在FIFO中的最长等待时间 |
| `writes_starved` | 2 | `/sys/block/sda/queue/iosched/writes_starved` | 连续几次读后必须处理写 |
| `fifo_batch` | 16 | `/sys/block/sda/queue/iosched/fifo_batch` | 同方向最多连续处理几个 |

## 四、struct dd_per_prio - 每个优先级的数据

**源码位置**: `block/mq-deadline.c:75-82`

**在此阶段的作用**: 每个优先级独立的队列和统计信息。

### 4.1 完整结构体定义

```c
/*
 * Deadline scheduler data per I/O priority (enum dd_prio). Requests are
 * present on both sort_list[] and fifo_list[].
 */
struct dd_per_prio {
    struct list_head dispatch;              // 待派发队列
    struct rb_root sort_list[DD_DIR_COUNT]; // 红黑树（按扇区排序）
    struct list_head fifo_list[DD_DIR_COUNT]; // FIFO队列（按时间排序）
    struct request *next_rq[DD_DIR_COUNT];  // 下一个顺序请求指针
    struct io_stats_per_prio stats;         // 统计信息
};

// 统计信息
struct io_stats_per_prio {
    uint32_t inserted;      // 已插入请求数
    uint32_t merged;        // 已合并请求数
    uint32_t dispatched;    // 已派发请求数
    atomic_t completed;     // 已完成请求数
};
```

### 4.2 核心字段详解 ⭐⭐⭐

| 字段名 | 类型 | 作用 | 在调度中的角色 |
|--------|------|------|--------------|
| **队列结构** ||||
| `dispatch` | `struct list_head` | **待派发队列**<br/>at_head插入时使用 | 最高优先级<br/>先于sort_list和fifo_list |
| `sort_list[2]` | `struct rb_root` | **红黑树（按扇区排序）**<br/>[0]=读树<br/>[1]=写树 | 顺序IO优化<br/>减少磁头移动 |
| `fifo_list[2]` | `struct list_head` | **FIFO队列（按时间排序）**<br/>[0]=读队列<br/>[1]=写队列 | 防止请求饿死<br/>超时保证 |
| `next_rq[2]` | `struct request *` | **下一个顺序请求**<br/>指向红黑树中下一个节点 | 批处理优化<br/>快速查找 |
| **统计信息** ||||
| `stats` | `struct io_stats_per_prio` | 统计计数器 | debugfs输出<br/>性能分析 |

**详细说明**：

**同一个 request 的三种组织方式**：
```
request X (sector 1008-1015, READ, RT优先级)
  │
  ├─ queuelist ────▶ 挂在 fifo_list[READ] 链表
  │                  (按时间排序)
  │
  ├─ rb_node ──────▶ 挂在 sort_list[READ] 红黑树
  │                  (按扇区排序)
  │
  └─ 被 next_rq[READ] 指向（如果是下一个顺序请求）

选择时：
1. 首先看 dispatch 队列
2. 其次考虑批处理（next_rq）
3. 然后检查 FIFO 是否有超时
4. 最后从红黑树选择顺序请求
```

**红黑树按扇区排序示例**：
```
sort_list[READ] (红黑树)

              [1008-1015]
              /           \
        [1000-1007]    [1020-1023]
                       /
                  [1016-1019]

中序遍历: 1000 → 1008 → 1016 → 1020  (扇区递增)
目的: 减少磁头移动（HDD）或优化访问模式（SSD）
```

**FIFO队列按时间排序示例**：
```
fifo_list[READ] (FIFO链表)

HEAD ──▶ [rq1, time=T0] ──▶ [rq2, time=T1] ──▶ [rq3, time=T2] ──▶ TAIL
         最老的请求                                              最新的请求

fifo_time = 插入时间 + fifo_expire[READ]  (jiffies)

检查超时:
if (time_after_eq(jiffies, rq->fifo_time))
    return 1;  // 已超时，必须立即处理
```

### 4.3 Request 在调度器中的状态

```
阶段A: 插入时
┌─────────────────────────────────────────────────────────────────┐
│ dd_insert_request()                                             │
│   ├─ 获取 ioprio_class ──▶ IOPRIO_CLASS_RT                     │
│   ├─ 映射到 dd_prio ──▶ DD_RT_PRIO (0)                         │
│   ├─ per_prio = &dd->per_prio[DD_RT_PRIO]                      │
│   │                                                             │
│   └─ if (at_head) {                                            │
│         list_add(&rq->queuelist, &per_prio->dispatch);         │
│     } else {                                                    │
│         deadline_add_rq_rb(per_prio, rq);  // 红黑树            │
│         rq->fifo_time = jiffies + dd->fifo_expire[data_dir];   │
│         list_add_tail(&rq->queuelist, &per_prio->fifo_list);   │
│     }                                                           │
└─────────────────────────────────────────────────────────────────┘

阶段B: 派发选择时
┌─────────────────────────────────────────────────────────────────┐
│ dd_dispatch_request()                                           │
│   ├─ 1. 检查低优先级老化 (dd_dispatch_prio_aged_requests)       │
│   │     for (prio = BE; prio <= IDLE; prio++)                  │
│   │         if (等待时间 > prio_aging_expire)                   │
│   │             return 低优先级请求;  // 防饥饿                  │
│   │                                                             │
│   └─ 2. 按优先级顺序选择 (for prio = RT → BE → IDLE)           │
│         rq = __dd_dispatch_request(dd, &per_prio[prio]);       │
│         if (rq || dd_queued(dd, prio))                         │
│             break;  // 当前优先级有请求就不看更低优先级          │
└─────────────────────────────────────────────────────────────────┘
```

## 五、调度器选择逻辑详解

### 5.1 __dd_dispatch_request() 流程

**源码位置**: `block/mq-deadline.c:423-534`

```c
static struct request *__dd_dispatch_request(struct deadline_data *dd,
                                             struct dd_per_prio *per_prio,
                                             unsigned long latest_start)
{
    struct request *rq, *next_rq;
    enum dd_data_dir data_dir;

    // ===== 第1优先级: dispatch 队列 =====
    if (!list_empty(&per_prio->dispatch)) {
        rq = list_first_entry(&per_prio->dispatch, struct request, queuelist);
        if (started_after(dd, rq, latest_start))
            return NULL;
        list_del_init(&rq->queuelist);
        goto done;
    }

    // ===== 第2优先级: 继续批处理 =====
    rq = deadline_next_request(dd, per_prio, dd->last_dir);
    if (rq && dd->batching < dd->fifo_batch)
        goto dispatch_request;  // 继续当前方向

    // ===== 第3优先级: 选择数据方向 =====
    if (!list_empty(&per_prio->fifo_list[DD_READ])) {
        // 有读请求
        if (deadline_fifo_request(dd, per_prio, DD_WRITE) &&
            (dd->starved++ >= dd->writes_starved))
            goto dispatch_writes;  // 写已饥饿

        data_dir = DD_READ;
        goto dispatch_find_request;
    }

    if (!list_empty(&per_prio->fifo_list[DD_WRITE])) {
dispatch_writes:
        dd->starved = 0;  // 重置饥饿计数
        data_dir = DD_WRITE;
        goto dispatch_find_request;
    }

    return NULL;  // 没有请求

dispatch_find_request:
    // ===== 第4优先级: FIFO vs 扇区排序 =====
    next_rq = deadline_next_request(dd, per_prio, data_dir);
    if (deadline_check_fifo(per_prio, data_dir) || !next_rq) {
        // FIFO有超时 或 没有顺序请求
        rq = deadline_fifo_request(dd, per_prio, data_dir);
    } else {
        // 继续顺序IO
        rq = next_rq;
    }

    if (!rq)
        return NULL;

    dd->last_dir = data_dir;
    dd->batching = 0;

dispatch_request:
    if (started_after(dd, rq, latest_start))
        return NULL;

    dd->batching++;
    deadline_move_request(dd, per_prio, rq);  // 从红黑树和FIFO移除

done:
    per_prio->stats.dispatched++;
    rq->rq_flags |= RQF_STARTED;
    return rq;
}
```

### 5.2 选择决策树

```
                         开始
                          │
                          ▼
                ┌──────────────────┐
                │ dispatch队列     │──▶ 有 ──▶ 返回队首请求 (最高优先级)
                │ 是否为空？        │
                └────────┬─────────┘
                         │ 空

                ┌──────────────────┐
                │ batching <       │──▶ 是 ──▶ 继续顺序IO (next_rq)

                └────────┬─────────┘

                └────────┬─────────┘
```

## 六、请求插入流程详解

### 6.1 dd_insert_request() 代码分析

**源码位置**: `block/mq-deadline.c:768-820`

```c
static void dd_insert_request(struct blk_mq_hw_ctx *hctx, 
                               struct request *rq, bool at_head)
{
    struct request_queue *q = hctx->queue;
    struct deadline_data *dd = q->elevator->elevator_data;
    const enum dd_data_dir data_dir = rq_data_dir(rq);  // 读(0)或写(1)
    
    // 1. 获取请求的IO优先级
    u16 ioprio = req_get_ioprio(rq);
    u8 ioprio_class = IOPRIO_PRIO_CLASS(ioprio);
    
    // 2. 映射到调度器优先级
    enum dd_prio prio = ioprio_class_to_prio[ioprio_class];
    struct dd_per_prio *per_prio = &dd->per_prio[prio];

    // 3. 更新统计
    if (!rq->elv.priv[0]) {
        per_prio->stats.inserted++;
        rq->elv.priv[0] = (void *)(uintptr_t)1;  // 标记已插入
    }

    // 4. 尝试合并
    LIST_HEAD(free);
    if (blk_mq_sched_try_insert_merge(q, rq, &free)) {
        blk_mq_free_requests(&free);
        return;  // 合并成功
    }

    trace_block_rq_insert(rq);

    // 5. 插入队列
    if (at_head) {
        // at_head: 直接插入dispatch队列头部
        list_add(&rq->queuelist, &per_prio->dispatch);
        rq->fifo_time = jiffies;
    } else {
        // 正常插入: 同时插入红黑树和FIFO
        
        // 插入红黑树（按扇区排序）
        deadline_add_rq_rb(per_prio, rq);
        
        // 添加到合并哈希表
        if (rq_mergeable(rq)) {
            elv_rqhash_add(q, rq);
            if (!q->last_merge)
                q->last_merge = rq;
        }

        // 插入FIFO（按时间排序）
        rq->fifo_time = jiffies + dd->fifo_expire[data_dir];
        list_add_tail(&rq->queuelist, &per_prio->fifo_list[data_dir]);
    }
}
```

## 七、优先级调度实例

### 7.1 三个优先级的请求分布

```
假设当前队列状态:

per_prio[DD_RT_PRIO] (优先级0)
  dispatch:   空
  sort_list:  [1000-1007(R)]
  fifo_list:  [1000-1007(R)]  fifo_time=T0+500ms

per_prio[DD_BE_PRIO] (优先级1)
  dispatch:   空
  sort_list:  [2000-2007(R)], [2010-2015(W)], [2020-2023(R)]
  fifo_list:  READ:  [2000, 2020]  
              WRITE: [2010]

per_prio[DD_IDLE_PRIO] (优先级2)
  dispatch:   空
  sort_list:  [3000-3007(W)]
  fifo_list:  [3000]  fifo_time=T0+5000ms

当前时间: T0 + 600ms

dd_dispatch_request() 调用流程:
1. dd_dispatch_prio_aged_requests() ──▶ 检查BE和IDLE是否老化
   - BE最老请求: T0+0, 已等待600ms < 10000ms, 未老化
   - IDLE最老请求: T0+0, 已等待600ms < 10000ms, 未老化
   - 返回 NULL

2. 按优先级循环 (prio = 0 → 1 → 2):
   - prio=0 (RT): 
     __dd_dispatch_request(&per_prio[0])
     ├─ dispatch为空
     ├─ 检查FIFO: [1000] fifo_time=T0+500ms < T0+600ms ──▶ 已超时!
     └─ 返回请求 [1000-1007(R)]  ✓ 选中

结果: 返回 RT优先级的读请求
```

### 7.2 写饥饿保护实例

```
场景: 连续读请求到来，写请求在等待

初始状态:
  last_dir = DD_READ
  batching = 0
  starved = 0

第1次派发:
  ├─ fifo_list[READ] 不为空
  ├─ fifo_list[WRITE] 不为空
  ├─ starved++ (starved=1) < writes_starved(2)
  └─ 选择读请求, last_dir=READ, batching=1

第2次派发:
  ├─ batching(1) < fifo_batch(16), 继续批处理
  └─ 选择下一个读请求, batching=2

第3次派发:
  ├─ batching(2) >= fifo_batch, 停止批处理
  ├─ fifo_list[READ] 不为空
  ├─ fifo_list[WRITE] 不为空
  ├─ starved++ (starved=2) >= writes_starved(2)  ──▶ 触发!
  └─ 选择写请求, last_dir=WRITE, starved=0

结果: 最多连续2次读后必须处理1次写
```

### 8.3 调优建议

| 场景 | 建议配置 | 原因 |
|------|---------|------|
| **数据库** | `read_expire=100`<br/>`writes_starved=4` | 读优先，但允许更多读<br/>批量刷脏页时不阻塞读 |
| **视频流** | `fifo_batch=32`<br/>`writes_starved=1` | 大块连续读写<br/>写重要性提升 |
| **编译服务器** | 默认配置 | 读写混合平衡 |
| **SSD设备** | `fifo_batch=8`<br/>考虑使用`none` | SSD无寻道时间<br/>批处理意义减小 |

## 九、总结与下篇预告

### 9.1 本文要点

1. **elevator_queue** - 调度器接口
   - `type`: 调度器类型和操作函数表
   - `elevator_data`: 指向 `deadline_data`

2. **deadline_data** - 调度器全局配置
   - `per_prio[3]`: 三个优先级队列
   - 配置参数：`fifo_expire`, `writes_starved`, `prio_aging_expire`

3. **dd_per_prio** - 每个优先级的队列
   - `dispatch`: 最高优先级
   - `sort_list`: 按扇区排序（红黑树）
   - `fifo_list`: 按时间排序（FIFO）

4. **调度策略**
   - 优先级：RT > BE > IDLE（除非老化）
   - 方向平衡：读优先，但写不能饿死
   - 请求选择：dispatch → 批处理 → FIFO超时 → 扇区顺序

### 9.2 Request 在调度器中的组织

```
一个request同时存在于:
1. rb_node ──▶ sort_list[dir] 红黑树
2. queuelist ──▶ fifo_list[dir] FIFO链表
3. hash ──▶ 合并哈希表 (可选)
4. elv.priv[0] ──▶ 标记已插入

派发时从红黑树和FIFO同时移除
```
