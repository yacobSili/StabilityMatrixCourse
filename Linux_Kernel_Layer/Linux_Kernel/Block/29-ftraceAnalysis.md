## 设备号解析

```bash
# 设备号格式：major:minor
```

## 关键细节分析

### 1. 请求类型

```bash
RA = Read Ahead (预读)
R  = Read (同步读)

# om.mediatek.ims 的请求是 RA
# → 这是文件系统的预读优化
# → 预测应用接下来会读取这些数据
# → 提前加载到页缓存

# grpc-timer-0 的请求是 R
# → 同步读取，应用正在等待数据
```

### 2. 扇区映射

```bash
逻辑设备 (dm-4) → 物理设备 (8,32)
扇区转换:
  3767264 → 14334944

计算偏移:
  14334944 - 3767264 = 10567680 sectors
  10567680 * 512 bytes = 5,410,652,160 bytes ≈ 5.4 GB

这个偏移可能是：
  - dm-4 在物理磁盘上的起始位置
  - 加密层的偏移
  - 分区表的偏移
```

### 3. I/O 优先级

```bash
be,0,4
↑  ↑ ↑
│  │ └─ 标志位
│  └─── 优先级级别 (0 = 最高)
└────── I/O 类别 (be = Best Effort)

# 这个请求的优先级很高
# 可能是 IMS (IP Multimedia Subsystem) 的关键服务
```

### 4. 请求合并

```bash
kworker 的两个请求：
  请求 1: 5295040 + 8 (4KB)
  请求 2: 5295048 + 8 (4KB)
          ↓
  合并后: 15862720 + 16 (8KB)

优势：
  - 减少硬件命令数量
  - 提高总线利用率
  - 降低延迟
```

### 5. 延迟分析

```bash
om.mediatek.ims 的 I/O 延迟：

queue:    110984 (提交到硬件)
complete: 114200 (硬件完成)
延迟:     3216 微秒 = 3.2 毫秒
```

## 为什么有多层 Device Mapper？

### 常见的 dm 层次结构

```bash

  ↓
dm-4 (下层)

3. 设备快照

4. 虚拟化层
```
