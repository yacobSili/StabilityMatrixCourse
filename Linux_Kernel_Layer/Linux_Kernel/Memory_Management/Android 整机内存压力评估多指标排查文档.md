# Android 整机内存压力评估多指标排查文档

# 一、核心前提：Memory PSI 无法完全反映整机内存压力

Memory PSI（Pressure Stall Information）的核心设计目标是衡量「用户进程因内存分配阻塞的时间占比」，其统计锚点为「进程主动申请内存时的阻塞行为」。该指标存在天然统计盲区，仅能反映“已影响用户进程运行的显性压力”，无法覆盖内核后台操作、资源隐性占用等场景的内存压力，因此不能单独作为 Android 整机内存压力的评估依据，需结合多指标交叉验证。

## 1.1 Memory PSI 的统计范围（仅显性阻塞场景）

Memory PSI 的统计数据仅来源于以下两类与用户进程阻塞强相关的场景，范围具有明显局限性：

- **Direct Reclaim（直接回收）**：进程分配内存失败时触发，同步阻塞当前进程，阻塞时间会被完整记录，是 Memory PSI 数据的主要来源；

- **kswapd 唤醒瞬间**：kswapd 是内核后台内存回收线程，其持续工作过程不阻塞用户进程，仅在被唤醒的瞬间可能被记录少量压力数据，几乎可忽略。

## 1.2 Memory PSI 未覆盖的隐性内存压力场景

以下场景中系统存在显著内存压力，但因不直接阻塞用户进程的内存申请，无法被 Memory PSI 统计，易导致整机内存压力被低估：

1. **kswapd 后台持续回收压力**：当系统内存水位低于阈值时，kswapd 会异步持续回收内存（如回收页缓存、匿名页），此过程不阻塞任何用户进程。即使 kswapd 长期高负载运行（如内存碎片严重、页缓存堆积导致频繁回收），Memory PSI 也无明显反馈，但会引发 I/O 升高、CPU 被过度占用等问题，整机内存压力已处于较高水平；

2. **内存碎片导致的隐性压力**：系统长期运行后易产生内存碎片，此时物理内存总剩余量充足，但缺乏连续大页内存，导致需要大页内存的进程（如 GPU 驱动、SurfaceFlinger）分配失败，引发卡顿、功能异常等问题。由于未触发 Direct Reclaim，Memory PSI 数值无变化，无法反映此类压力；

3. **页缓存过度占用的潜在压力**：内核页缓存（Page Cache）会优先占用空闲内存缓存磁盘文件数据，当页缓存占比过高时，系统可用内存水位持续走低，后续进程申请内存时会触发更剧烈的 Direct Reclaim。但这种“潜在压力”在转化为进程阻塞前，无法被 Memory PSI 提前感知；

4. **内核态内存过度占用压力**：Memory PSI 主要关注用户态进程的内存分配阻塞，而内核态内存（如 slab 缓存、内核数据结构、驱动内存）的过度占用（如 slab 泄漏）会持续挤压用户态可用内存。此过程中无用户进程阻塞，Memory PSI 数值正常，但整机内存压力会持续累积，最终可能触发 OOM；

5. **交换分区（ZRAM/ZSWAP）的隐性开销压力**：开启 ZRAM 后，内存页的压缩/解压缩过程、ZRAM 自身的内存占用均属于内存压力的一部分，但这些开销不会被 Memory PSI 统计（除非压缩/解压缩直接导致进程阻塞），导致实际内存压力被低估。

# 三、指标联动分析方法

单一指标无法全面评估内存压力，需通过“指标联动”判断整机状态，核心分析逻辑如下：

1. **Memory PSI 高时**：需结合 Direct Reclaim 频率、LMK/OOM 日志验证是否为真实用户进程阻塞；同时查看 kswapd CPU 占比、si/so 数据，判断压力是否已扩散至后台回收和交换分区；

2. **Memory PSI 正常但 kswapd 高负载**：大概率是页缓存过度占用或内存碎片导致的后台回收压力，需进一步查看 PageCache 占比、buddyinfo 碎片指数；

3. **可用内存低但无明显阻塞**：重点排查 Slab 缓存、页缓存占用，以及是否存在进程内存泄漏（通过 procrank、memleak 验证）；

4. **频繁大页分配失败**：即使 Memory PSI 正常，也属于严重内存碎片压力，需结合 buddyinfo 优化内存分配策略（如开启大页合并、调整进程内存申请方式）。

# 四、总结

Memory PSI 是“用户进程内存阻塞”的被动指标，而非“整机内存资源”的主动评估指标，其核心价值在于反映“压力对用户体验的直接影响”，但无法覆盖内核后台、内存碎片、内核态占用等隐性压力场景。在 Android 整机内存压力评估中，需以 Memory PSI 为基础，结合内存占用、回收行为、内核状态、交换开销等多维度指标交叉验证，才能全面、准确判断系统内存状态，并定位压力根源。
