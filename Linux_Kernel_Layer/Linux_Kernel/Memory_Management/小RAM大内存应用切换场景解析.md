这是一个非常经典且具有挑战性的内存压力场景（Memory Pressure Scenario）。3GB 物理内存在面对两个“大型游戏”时，绝对属于资源紧缺状态。

既然你之前问到了 **Page Cache**，我们在分析这个过程时，会重点关注 **Page Cache（文件页）** 和 **Anonymous Pages（匿名页，即堆栈内存）** 是如何互斥和竞争的。

以下是基于 Linux 内核（Android/桌面 Linux 通用逻辑）的内存调度全过程推演：

### 第一阶段：启动游戏 A（内存的第一次占领）

1.  **加载代码与资源（Page Cache 填充）：**
    *   当你点击游戏 A 图标，内核通过 `mmap` 或 `read` 读取游戏的可执行文件（.exe/ELF）和大量的资源文件（纹理、模型）。
    *   **关键点：** 这些数据首先被读入 **Page Cache**。此时，系统的 `Buff/Cache` 占用率飙升。
    *   如果此时剩余空闲内存（Free）很多，内核会很痛快地分配。

2.  **动态内存分配（匿名页生成）：**
    *   游戏开始运行，初始化引擎，分配堆内存（Heap）来存放变量、对象、游戏状态。
    *   这些内存被称为 **匿名页（Anonymous Pages）**，因为它们背后没有文件支撑，纯粹是内存里的数据。

3.  **稳定状态（3分钟后）：**
    *   假设游戏 A 占用了 2GB 内存（其中 1GB 是 Page Cache 的资源，1GB 是堆栈数据）。
    *   系统剩余：1GB（留给 OS 内核 + 桌面环境）。此时内存处于“健康但略显拥挤”的状态。

---

### 第二阶段：回到桌面，启动游戏 B（内存危机爆发）

当你按下 Home 键回到桌面，并点击游戏 B 的瞬间，**内存调度器（Memory Allocator）** 开始面临巨大的压力。假设游戏 B 启动也需要 2GB 内存，而系统只剩 500MB 空闲，**缺口高达 1.5GB**。

内核会按照以下优先级进行“抢地盘”：

#### 1. 软柿子阶段：驱逐未修改的 Page Cache（Clean Pages）
内核首先会看向 **Page Cache**。
*   **动作：** 内核检查 LRU（最近最少使用）链表。它发现游戏 A 的很多纹理、地图数据虽然在内存里，但因为你切到了后台，暂时没用了。
*   **处理：** 对于那些**只读的**（代码段）或者**未修改的**（读取的纹理）文件页，内核直接**丢弃（Evict/Drop）**。
*   **后果：** 释放出了几百 MB 空间给游戏 B。
*   *注意：这步不需要写磁盘，速度很快。*

#### 2. 硬骨头阶段：处理脏页和匿名页（Dirty & Anonymous）
光丢弃 Page Cache 不够（因为游戏 A 的 1GB 堆栈数据是不能直接丢弃的，否则游戏 A 就崩溃了）。此时内核必须做艰难的决定：

*   **脏页回写（Page Writeback）：** 如果 Page Cache 里有修改过的数据（比如游戏 A 的日志或存档），内核必须先把它写回磁盘，才能释放这块内存。这会产生 IO，导致系统变卡。
*   **匿名页交换（Swapping）：** 这是最关键的一步。游戏 A 的 1GB 动态数据（人物位置、血量等）必须找地方存。
    *   **如果有 Swap 分区/文件（PC）或 zRAM（手机）：** 内核会将游戏 A 的这些内存数据压缩（zRAM）或写入硬盘（Swap）。
    *   **现象：** 你的硬盘灯狂闪，或者手机开始发热。游戏 B 的启动速度明显变慢，因为 CPU 在忙着搬运游戏 A 的数据。

#### 3. 阻塞与直接回收（Direct Reclaim）
当空闲内存低于**低水位线（Low Watermark）**时：
*   **kswapd 线程**（后台回收线程）会疯狂工作。
*   但如果游戏 B 请求内存的速度太快，kswapd 来不及回收，内核就会触发 **Direct Reclaim（直接回收）**。
*   **后果：** 游戏 B 的分配内存请求会被**阻塞**，直到内核强行回收一部分内存。用户端体验就是：**点击图标后，界面卡死几秒钟没有任何反应。**

---

### 第三阶段：终局（两种可能的结局）

由于物理内存只有 3GB，且两个游戏都很大，Swap/zRAM 也是有限的，通常会走向以下两个结局之一：

#### 结局 A：Thrashing（系统颠簸/抖动）—— 如果你没开启激进的杀进程策略
*   游戏 B 勉强启动了，占用了大部分内存。
*   游戏 A 的数据大部分被挤到了 Swap/磁盘里。
*   **如果你此时切回游戏 A：** 系统需要把游戏 B 的数据 Swap 出去，再把游戏 A 的数据从磁盘 Swap 进来。
*   **结果：** 极其卡顿，甚至鼠标/触摸屏都动不了。因为 CPU 绝大部分时间都在等待磁盘 IO，而不是运行游戏逻辑。

#### 结局 B：OOM Killer / LMK（更现实的结局）—— 尤其是 Android 或现代 Linux
内核发现无论怎么回收 Page Cache，无论怎么 Swap，内存都捉襟见肘，甚至影响到了系统的稳定性（System Thrashing）。

*   **触发机制：** 触发 **OOM (Out of Memory) Killer** 或者 Android 的 **LMK (Low Memory Killer)**。
*   **计算分数：** 内核会给所有进程打分（`oom_score`）。游戏 B 是前台进程（正在启动），分数低（受保护）。游戏 A 是后台进程，占用内存大，分数高（容易被杀）。
*   **处决：** 内核发送 `SIGKILL` 信号给游戏 A。
*   **现象：** 游戏 B 顺利启动。当你下次想切回游戏 A 时，发现它**重新启动**了（因为之前的进程已经被杀死了）。

### 总结图解

在这个过程中，Page Cache 和 匿名页的命运如下：

| 内存类型 | 归属 | 调度动作 | 耗时 |
| :--- | :--- | :--- | :--- |
| **Page Cache (Clean)** | 游戏 A 的代码/纹理 | **直接丢弃**。下次切回 A 时需重新从磁盘读取。 | 极快 |
| **Page Cache (Dirty)** | 游戏 A 的存档/日志 | **刷盘后释放**。 | 慢 (IO) |
| **Anonymous Pages** | 游戏 A 的堆/栈数据 | **Swap/压缩**。如果没空间或策略不允许，游戏 A 被 **Kill**。 | 很慢 / 致命 |

所以在 3GB 内存下，**Page Cache 会被瞬间牺牲殆尽**，紧接着就是**Swap 机制与 OOM Killer 的博弈**。通常情况下，为了保住前台的游戏 B，后台的游戏 A 会被无情杀掉。
