# PSI与ANR深度关联解析：CPU、MEM、IO压力下的根因判定

在Android系统级ANR（应用无响应）排查中，传统性能指标（如CPU使用率、剩余内存、iowait）常因“视角局限”导致根因误判。而PSI（Pressure Stall Information，压力停滞信息）作为Linux内核引入的新型指标，能从CPU、内存（MEM）、IO三大核心资源维度，精准捕捉进程因资源竞争陷入阻塞的状态，成为定位ANR资源类根因的“核心工具”。本文将聚焦PSI与ANR的内在关联，拆解三大资源压力下的ANR触发逻辑、PSI维度解读及关键阈值，为ANR排查提供标准化判定依据。

## 一、PSI核心概念：窗口、维度与解读基础

PSI通过“时间窗口”和“阻塞范围”两个维度，量化资源压力对进程的影响，其统计逻辑与进程状态强绑定，无关资源本身的使用率，这也是其能精准关联ANR的核心原因。

### 1. 时间窗口：10s/60s/300s的不同价值

PSI默认提供三个时间窗口的统计结果，分别对应不同压力场景，ANR排查需优先关注短期窗口（10s），中期/长期窗口辅助判断趋势：

* **10秒窗口（avg10）**：反映短期突发压力，是ANR根因判定的核心窗口。ANR触发多为短期资源枯竭导致核心进程阻塞，此窗口能精准匹配ANR发生时段的压力状态；

* **60秒窗口（avg60）**：反映中期压力趋势，用于判断ANR是“突发单次问题”还是“持续压力累积导致”；

* **300秒窗口（avg300）**：反映长期压力基线，若此窗口数值偏高，说明系统存在持续性资源瓶颈，ANR可能反复发生。


### 2. 阻塞范围：some与full的核心差异

每种资源的PSI均包含两个维度，分别对应不同范围的进程阻塞，直接决定ANR的严重程度和影响范围：

* **some**：特定时间窗口内，**至少有一个进程**因该资源竞争陷入阻塞的时间占比。反映资源压力的“普遍程度”，是ANR排查的首要关注维度；

* **full**：特定时间窗口内，**所有进程**均因该资源竞争陷入阻塞的时间占比。反映资源压力的“极端程度”，一旦升高，系统近乎卡死，ANR成为必然。


核心原则：ANR排查中，优先通过“资源类型+10秒窗口+some维度”锁定主因，full维度用于判定极端场景的严重程度。

结合三大资源PSI特性，ANR排查可按以下步骤锁定根因，避免单一指标误判：

5. 核心进程验证：通过ANR日志堆栈，确认system_server等核心进程是否因该资源阻塞（最终验证依据）。
