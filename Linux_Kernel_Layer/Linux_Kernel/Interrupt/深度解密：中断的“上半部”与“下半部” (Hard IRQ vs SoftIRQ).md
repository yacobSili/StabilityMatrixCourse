# 深度解密：中断的“上半部”与“下半部” (Hard IRQ vs SoftIRQ)

在 Linux 内核中，中断并不是“一气呵成”的。为了防止 CPU 被某个硬件长时间霸占，Linux 将中断处理强行劈成了两半：**上半部 (Top Half)** 和 **下半部 (Bottom Half)**。

这就像医院的急诊流程：
1.  **急诊分诊台（上半部）**：只负责看你一眼，确认还没死，挂个号，立刻叫下一个。**（求快）**
2.  **专科医生治疗（下半部）**：负责缝针、开药、写病历、办理住院。**（求细）**

---

## 一、 为什么要分两半？

想象一下，网卡收到了一个 1KB 的数据包，触发了中断。
如果 CPU 在中断里直接开始解析 TCP/IP 协议、校验数据、拷贝内存、通知应用程序……这可能需要耗时 100 微秒。
在这 100 微秒里，CPU 是**屏蔽中断**的（听不到外界声音）。如果此时鼠标动了，或者磁盘写完了，CPU 统统不知道。结果就是：鼠标卡顿，数据丢失。

**解决方案：**
*   **上半部**：只做最最紧急的事（比如告诉网卡“我听到了”），耗时 0.1 微秒，立刻恢复监听其他中断。
*   **下半部**：把剩下的 99.9 微秒的活儿（解析数据），留到稍后 CPU 有空时再慢慢做。

---

## 二、 详细对比：Hard IRQ vs SoftIRQ

### 1. 上半部：Hard IRQ (硬中断)
这是物理信号直接触发的代码。

*   **别名**：Interrupt Service Routine (ISR), Top Half.
*   **任务**：
    1.  应答硬件（Ack）：告诉磁盘/网卡“信号收到了，别再叫了”。
    2.  读取最关键的基础信息（如中断状态寄存器）。
    3.  **调度下半部**：把繁重的工作挂到 SoftIRQ 队列里，说“待会处理”。
*   **特点**：
    *   **极快**：通常纳秒级。
    *   **关中断**：运行时，CPU 往往会屏蔽当前 CPU 上的其他中断（或所有中断）。
    *   **不可被抢占**：天王老子来了也得等它跑完。
*   **日志特征**：
    *   标志位通常显示为 **`d.h..`** (h = HardIRQ)。
    *   常见事件：`irq_handler_entry`, `irq_handler_exit`。

### 2. 下半部：SoftIRQ (软中断)
这是被上半部唤醒的“打工仔”。

*   **别名**：Bottom Half, Software Interrupt.
*   **任务**：
    1.  处理数据（你的日志就在做这个：`block_bio_complete`）。
    2.  解析网络协议栈 (TCP/IP)。
    3.  唤醒等待数据的进程。
*   **特点**：
    *   **稍慢**：微秒到毫秒级。
    *   **开中断**：运行时，如果有新的硬件中断（Hard IRQ）来了，**SoftIRQ 会被立刻打断**。
    *   **借用上下文**：通常“借用”中断发生时正在运行的那个进程（如 `ims`）的身体来运行。
*   **日志特征**：
    *   标志位显示为 **`..s..`** (s = SoftIRQ)。
    *   常见事件：`block_rq_complete`, `netif_receive_skb`, `tasklet_hi_action`.

## 三、 结合你的日志看全过程

让我们把镜头放慢，看看你的 IMS I/O 是如何完成的：

#### 第一步：物理时刻 (Hard IRQ) - *日志里没开启这部分抓取*
1.  UFS 磁盘控制器完成读取，拉高电平。
2.  CPU 暂停 `ims`，跳转到 UFS 驱动的 ISR。
3.  **状态**：CPU 标志位是 `d.h..`。
4.  **动作**：CPU 读一下硬件寄存器，确认“哦，是读取完成了”。然后设置一个标志位：`raise_softirq(BLOCK_SOFTIRQ)`。
5.  **结束**：ISR 退出，CPU 恢复响应其他中断。

#### 第二步：稍后片刻 (SoftIRQ) - *你看到的日志*
1.  内核检查到刚才立了个 flag (BLOCK_SOFTIRQ)，于是开始处理下半部。
2.  **状态**：CPU 标志位变成 `..s..`。
3.  **动作**：
    *   运行 `block_rq_complete`：从硬件队列里移除请求。
    *   运行 `block_bio_complete`：通知文件系统数据好了。
    *   如果需要，唤醒正在睡眠等待数据的进程。
4.  这就是为什么你的日志里全是 `s`，因为繁重的收尾工作都在这里。

## 四、 总结对比表

| 特征 | 上半部 (Hard IRQ) | 下半部 (SoftIRQ) |
| :--- | :--- | :--- |
| **触发源** | 硬件电信号 | 软件代码 (由上半部触发) |
| **响应速度** | 极速 (立刻) | 稍后 (延迟执行) |
| **是否屏蔽中断** | 是 (通常屏蔽) | **否** (这是关键！它能被 HardIRQ 打断) |
| **ftrace 标志** | `h` (如 `d.h..`) | **`s`** (如 `..s..`) |
| **主要工作** | 签收快递 | 拆快递、分类、通知收件人 |
| **你的日志位置** | (未记录) | **这里** (`block_bio_complete`) |

### 为什么这对分析很重要？
1.  **Hard IRQ 耗时过长**：会导致系统**丢中断**（鼠标跳帧、网络丢包）。
2.  **SoftIRQ 耗时过长**：会导致**系统卡顿**（进程得不到 CPU 调度），或者出现 `ksoftirqd` 占用率 100% 的情况。

所以，你在日志里看到的 `om.mediatek.ims ... ..s..` 实际上是 IMS 进程正在“替”磁盘驱动跑完那漫长的“下半部”流程。
