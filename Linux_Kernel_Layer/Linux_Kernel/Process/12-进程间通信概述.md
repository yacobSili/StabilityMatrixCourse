# 进程间通信概述

## 学习目标

- 理解进程间通信（IPC）的必要性和分类
- 掌握各种 IPC 机制的特点和适用场景
- 了解 IPC 机制与进程管理的关系
- 理解 Android IPC 的特殊性

## 概述

进程间通信（Inter-Process Communication，IPC）是操作系统提供的让进程之间交换数据和信息的机制。由于进程有独立的地址空间，进程之间不能直接访问对方的内存，需要通过内核提供的 IPC 机制进行通信。

---

## 一、IPC 机制分类

### 按通信方式分类

| 类型 | 机制 | 特点 |
|-----|------|-----|
| 信号 | Signal | 异步通知，携带信息有限 |
| 管道 | Pipe, FIFO | 单向、字节流 |
| 消息传递 | Message Queue | 消息边界、优先级 |
| 共享内存 | Shared Memory | 最高效、需要同步 |
| 套接字 | Socket | 支持网络、全双工 |
| 远程过程调用 | RPC, Binder | 跨进程方法调用 |

### 按通信模型分类

```
1. 数据传输模型
   ┌──────────┐     数据复制     ┌──────────┐
   │  进程 A  │ ───────────────► │  进程 B  │
   └──────────┘     (管道/消息)   └──────────┘

2. 共享内存模型
   ┌──────────┐                  ┌──────────┐
   │  进程 A  │                  │  进程 B  │
   └────┬─────┘                  └────┬─────┘
        │                             │
        └───────────┬─────────────────┘
                    │
              ┌─────▼─────┐
              │ 共享内存   │
              └───────────┘

3. 同步通知模型
   ┌──────────┐     信号       ┌──────────┐
   │  进程 A  │ ─────────────► │  进程 B  │
   └──────────┘               └──────────┘
```

## 二、各种 IPC 机制概览

### 1. 信号（Signal）

**特点**：
- 异步通知机制
- 携带信息有限（仅信号编号和简单数据）
- 用于通知事件，不适合传输数据

**适用场景**：
- 进程终止通知（SIGCHLD）
- 中断处理（SIGINT, SIGTERM）
- 定时器（SIGALRM）

```c
// 发送信号
kill(pid, SIGTERM);

// 处理信号
signal(SIGTERM, handler);
sigaction(SIGTERM, &act, NULL);
```

### 2. 管道（Pipe）

**特点**：
- 单向通信
- 字节流，无消息边界
- 匿名管道：仅限父子进程
- 命名管道（FIFO）：任意进程

**适用场景**：
- Shell 管道：`ls | grep foo`
- 父子进程简单通信

```c
// 匿名管道
int fd[2];
pipe(fd);
// fd[0]: 读端
// fd[1]: 写端

// 命名管道
mkfifo("/tmp/myfifo", 0666);
```

### 3. 消息队列（Message Queue）

**特点**：
- 消息有边界
- 支持优先级
- 独立于发送/接收进程的生命周期

**适用场景**：
- 多生产者多消费者模型
- 需要消息优先级的场景

```c
// System V
int msgid = msgget(key, IPC_CREAT | 0666);
msgsnd(msgid, &msg, size, 0);
msgrcv(msgid, &msg, size, type, 0);

// POSIX
mqd_t mq = mq_open("/queue", O_CREAT | O_RDWR, 0666, &attr);
mq_send(mq, msg, size, priority);
mq_receive(mq, msg, size, &priority);
```

### 4. 共享内存（Shared Memory）

**特点**：
- 最高效（无数据复制）
- 需要同步机制（信号量、互斥锁）
- 独立于进程生命周期

**适用场景**：
- 大量数据交换
- 高性能要求

```c
// System V
int shmid = shmget(key, size, IPC_CREAT | 0666);
void *ptr = shmat(shmid, NULL, 0);

// POSIX
int fd = shm_open("/shm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, size);
void *ptr = mmap(NULL, size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
```

### 5. 信号量（Semaphore）

**特点**：
- 同步机制，不传输数据
- 计数器，支持 P/V 操作
- 用于保护共享资源

**适用场景**：
- 共享内存同步
- 资源计数

```c
// System V
int semid = semget(key, 1, IPC_CREAT | 0666);
semop(semid, &op, 1);

// POSIX
sem_t *sem = sem_open("/sem", O_CREAT, 0666, 1);
sem_wait(sem);
sem_post(sem);
```

### 6. 套接字（Socket）

**特点**：
- 支持网络通信
- 支持本地通信（Unix Domain Socket）
- 全双工
- 多种协议（TCP, UDP）

**适用场景**：
- 网络通信
- 跨机器 IPC
- 需要全双工的场景

```c
// Unix Domain Socket
int fd = socket(AF_UNIX, SOCK_STREAM, 0);
struct sockaddr_un addr = {.sun_family = AF_UNIX, .sun_path = "/tmp/socket"};
connect(fd, (struct sockaddr *)&addr, sizeof(addr));
```

## 三、IPC 机制对比

### 性能对比

| 机制 | 数据复制次数 | 延迟 | 吞吐量 |
|-----|------------|------|-------|
| 共享内存 | 0 | 最低 | 最高 |
| Binder | 1 | 低 | 高 |
| 管道 | 2 | 中 | 中 |
| Socket | 2 | 中-高 | 中 |
| 消息队列 | 2 | 中 | 中 |

### 功能对比

| 机制 | 方向 | 边界 | 同步/异步 | 网络支持 |
|-----|------|-----|----------|---------|
| 信号 | 单向 | 无 | 异步 | 否 |
| 管道 | 单向 | 无 | 同步 | 否 |
| 消息队列 | 双向 | 有 | 同步 | 否 |
| 共享内存 | 双向 | 无 | 需自行同步 | 否 |
| Socket | 双向 | 可选 | 同步 | 是 |
| Binder | 双向 | 有 | 同步 | 否 |

### 选择建议

```
需要网络通信？
  ├─ 是 → Socket
  └─ 否
      ├─ 需要最高性能？
      │   ├─ 是 → 共享内存 + 同步机制
      │   └─ 否
      │       ├─ 简单通知？
      │       │   ├─ 是 → 信号
      │       │   └─ 否
      │       │       ├─ 父子进程？
      │       │       │   ├─ 是 → 匿名管道
      │       │       │   └─ 否
      │       │       │       ├─ Android 系统？
      │       │       │       │   ├─ 是 → Binder
      │       │       │       │   └─ 否
      │       │       │       │       ├─ 需要消息边界？
      │       │       │       │       │   ├─ 是 → 消息队列
      │       │       │       │       │   └─ 否 → FIFO 或 Socket
```

## 四、IPC 与进程管理的关系

### 进程创建时的 IPC 继承

```c
// fork 时的 IPC 继承
pid_t pid = fork();
if (pid == 0) {
    // 子进程继承：
    // - 打开的文件描述符（包括管道、socket）
    // - 共享内存映射
    // - 信号处理设置
}
```

### 进程退出时的 IPC 清理

```c
// 进程退出时：
// - 关闭所有文件描述符（包括管道、socket）
// - 共享内存引用计数减少
// - 信号量引用计数减少
// 注意：System V IPC 资源不会自动删除
```

### 进程状态与 IPC

```
进程 A                    进程 B
   │                         │
   │ send() 阻塞             │
   │ ─────────────────────►  │
   │ TASK_INTERRUPTIBLE      │
   │                         │ recv() 阻塞
   │  ◄──────────────────────│
   │                         │ TASK_INTERRUPTIBLE
   │                         │
```

## 五、Android IPC 概述

### Android IPC 特点

1. **Binder 为主**：跨进程调用的标准方式
2. **Socket 为辅**：系统服务启动阶段
3. **共享内存**：大数据传输（如图形缓冲区）
4. **Intent**：应用层消息传递

### Android IPC 架构

```
┌─────────────────────────────────────────────────────────────┐
│                     应用层                                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │   App A     │  │   App B     │  │   App C     │         │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘         │
│         │                │                │                 │
│         └────────────────┼────────────────┘                 │
│                          │ Intent/AIDL                      │
├──────────────────────────┼──────────────────────────────────┤
│                          │                                  │
│  ┌───────────────────────▼───────────────────────────────┐ │
│  │              Framework 层                              │ │
│  │   ActivityManagerService, ServiceManager              │ │
│  └───────────────────────┬───────────────────────────────┘ │
│                          │ Binder                          │
├──────────────────────────┼──────────────────────────────────┤
│                          │                                  │
│  ┌───────────────────────▼───────────────────────────────┐ │
│  │              Native 层                                 │ │
│  │   libbinder, servicemanager                           │ │
│  └───────────────────────┬───────────────────────────────┘ │
│                          │ ioctl                           │
├──────────────────────────┼──────────────────────────────────┤
│                          │                                  │
│  ┌───────────────────────▼───────────────────────────────┐ │
│  │              Kernel 层                                 │ │
│  │   /dev/binder, binder driver                          │ │
│  └───────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### Binder 简介

**特点**：
- 一次数据拷贝（比管道/socket 更高效）
- 支持对象传递
- 支持跨进程调用
- 安全性（进程 UID/PID 验证）

**核心组件**：
- Binder 驱动（/dev/binder）
- ServiceManager（服务注册中心）
- Binder 对象（服务端）
- Binder 代理（客户端）

## 六、System V IPC vs POSIX IPC

### 对比

| 特性 | System V | POSIX |
|-----|----------|-------|
| 历史 | 较老 | 较新 |
| 接口 | msgget/shmget/semget | mq_open/shm_open/sem_open |
| 命名 | 整数 key | 字符串名称 |
| 权限 | 类 Unix | 类文件 |
| 可移植性 | 广泛支持 | 需要 librt |

### System V IPC

```c
// 消息队列
int msgid = msgget(ftok("/tmp", 'A'), IPC_CREAT | 0666);

// 共享内存
int shmid = shmget(ftok("/tmp", 'B'), 4096, IPC_CREAT | 0666);

// 信号量
int semid = semget(ftok("/tmp", 'C'), 1, IPC_CREAT | 0666);

// 查看 IPC 资源
// ipcs -a
```

### POSIX IPC

```c
// 消息队列
mqd_t mq = mq_open("/myqueue", O_CREAT | O_RDWR, 0666, NULL);

// 共享内存
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);

// 信号量
sem_t *sem = sem_open("/mysem", O_CREAT, 0666, 1);

// 资源在 /dev/shm 和 /dev/mqueue
```

## 七、IPC 安全性

### 权限控制

```c
// System V IPC 使用类似文件的权限
int shmid = shmget(key, size, IPC_CREAT | 0600);
// 0600: 只有所有者可读写

// 获取/设置 IPC 权限
struct shmid_ds buf;
shmctl(shmid, IPC_STAT, &buf);
buf.shm_perm.mode = 0644;
shmctl(shmid, IPC_SET, &buf);
```

### Android Binder 安全

```java
// Binder 调用时可以获取调用者的 UID/PID
int uid = Binder.getCallingUid();
int pid = Binder.getCallingPid();

// 检查权限
if (checkCallingPermission("android.permission.SOME_PERMISSION")
    != PackageManager.PERMISSION_GRANTED) {
    throw new SecurityException("Permission denied");
}
```

3. **选择依据**：性能、功能、复杂度、适用场景
4. **Android IPC**：以 Binder 为核心
