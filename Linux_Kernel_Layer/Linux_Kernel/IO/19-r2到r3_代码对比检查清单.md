## 具体检查点

### 1. dispatch budget机制

#### 检查点1: budget获取逻辑
```c
// 在 blk-mq-sched.c 中
budget_token = blk_mq_get_dispatch_budget(q);
if (budget_token < 0)
    break;
```
**检查**：
- `blk_mq_get_dispatch_budget()`的实现是否变化
- budget获取失败的条件是否更严格
- budget获取失败后的处理逻辑是否变化

#### 检查点2: budget释放时机
```c
// 在 blk-mq.c 中
static void blk_mq_release_budgets(struct request_queue *q,
                                    struct list_head *list)
{
    list_for_each_entry(rq, list, queuelist) {
        int budget_token = blk_mq_get_rq_budget_token(rq);
        if (budget_token >= 0)
            blk_mq_put_dispatch_budget(q, budget_token);
    }
}
```
**检查**：
- budget释放的时机是否变化
- 是否有budget泄漏的情况
- budget释放是否及时

**检查**：

## 预期发现

基于问题现象，预期会在以下方面发现变化：

1. **budget机制更严格**：budget获取失败的条件更严格，导致swap IO请求被阻塞
2. **tag分配更保守**：tag分配策略更保守，导致swap IO请求无法及时获得tag
3. **dispatch限制更严格**：max_dispatch或break条件更严格，导致swap IO请求无法及时dispatch
4. **swap IO优先级降低**：REQ_SWAP的处理变化，导致swap IO优先级降低

这些变化都会导致swap IO变慢，进而引发你观察到的恶性循环。
