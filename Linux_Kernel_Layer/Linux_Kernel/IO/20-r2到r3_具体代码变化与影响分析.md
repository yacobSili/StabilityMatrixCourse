# r2到r3具体代码变化与影响分析

## 版本信息
- **升级路径**: android13-5.15-2024-05_r2 → android13-5.15-2024-11_r3
- **变化文件**: `block/blk-mq.c` (增加24行代码)

## 具体代码变化

### 变化1: blk_mq_mark_tag_wait()函数

**文件**: `block/blk-mq.c`  
**函数**: `blk_mq_mark_tag_wait()`  
**Commit**: 89e0e66682e1 - "blk-mq: fix IO hang from sbitmap wakeup race"  
**位置**: 第1182-1196行

#### 代码变化
```c
// r2版本（变化前）
__add_wait_queue(wq, wait);

/*
 * It's possible that a tag was freed in the window between the
 * allocation failure and adding the hardware queue to the wait
 * queue.
 */
ret = blk_mq_get_driver_tag(rq);

// r3版本（变化后）
__add_wait_queue(wq, wait);

/*
 * Add one explicit barrier since blk_mq_get_driver_tag() may
 * not imply barrier in case of failure.
 *
 * Order adding us to wait queue and allocating driver tag.
 *
 * The pair is the one implied in sbitmap_queue_wake_up() which
 * orders clearing sbitmap tag bits and waitqueue_active() in
 * __sbitmap_queue_wake_up(), since waitqueue_active() is lockless
 *
 * Otherwise, re-order of adding wait queue and getting driver tag
 * may cause __sbitmap_queue_wake_up() to wake up nothing because
 * the waitqueue_active() may not observe us in wait queue.
 */
smp_mb();  // ← 新增：内存屏障

/*
 * It's possible that a tag was freed in the window between the
 * allocation failure and adding the hardware queue to the wait
 * queue.
 */
ret = blk_mq_get_driver_tag(rq);
```

#### 变化说明
- **新增内容**: 在`__add_wait_queue()`和`blk_mq_get_driver_tag()`之间添加了`smp_mb()`内存屏障
- **目的**: 修复sbitmap wakeup race导致的IO hang问题
- **影响**: 确保wait queue添加和driver tag获取之间的顺序

#### 造成的结果

**正面影响**：
- 修复了IO hang问题（sbitmap wakeup race）

**负面影响（导致IO劣化）**：
1. **内存屏障增加延迟**
   - `smp_mb()`是完整的内存屏障，会强制CPU等待所有内存操作完成
   - 在高IO压力下，这个屏障会增加每个IO请求的处理延迟
   - 对于swap IO这种频繁的操作，累积延迟显著

2. **swap IO请求被阻塞**
   - 当swap IO请求无法获取driver tag时，会调用`blk_mq_mark_tag_wait()`
   - 内存屏障增加了等待时间，导致swap IO请求堆积
   - 大量swap IO请求等待tag，IO压力升高

3. **kswapd0被阻塞**
   - kswapd0执行swap操作时，如果无法获取tag，会被阻塞在`blk_mq_mark_tag_wait()`
   - 内存屏障增加了阻塞时间
   - kswapd0的CPU使用率下降（因为等待IO），但实际工作时间更长

4. **恶性循环形成**
   ```
   swap IO请求 → 无法获取tag → blk_mq_mark_tag_wait() → smp_mb()延迟
   → swap IO变慢 → kswapd0被阻塞 → 内存回收变慢
   → 需要swap更多页面 → 更多IO请求 → IO压力进一步增大
   → 更多请求等待tag → 更长的等待时间 → 循环加剧
   ```

### 变化2: blk_mq_delay_run_hw_queues()函数

**文件**: `block/blk-mq.c`  
**函数**: `blk_mq_delay_run_hw_queues()`  
**Commit**: b69cdb7373f4 - "blk-mq: avoid extending delays of active hctx from blk_mq_delay_run_hw_queues"  
**位置**: 第1727-1734行

#### 代码变化
```c
// r2版本（变化前）
queue_for_each_hw_ctx(q, hctx, i) {
    if (blk_mq_hctx_stopped(hctx))
        continue;
    /*
     * Dispatch from this hctx either if there's no hctx preferred
     * by IO scheduler or if it has requests that bypass the
     * scheduler.
     */
    if (!sq_hctx || sq_hctx == hctx ||
        !list_empty_careful(&hctx->dispatch))
        blk_mq_delay_run_hw_queue(hctx, msecs);
}

// r3版本（变化后）
queue_for_each_hw_ctx(q, hctx, i) {
    if (blk_mq_hctx_stopped(hctx))
        continue;
    /*
     * If there is already a run_work pending, leave the
     * pending delay untouched. Otherwise, a hctx can stall
     * if another hctx is re-delaying the other's work
     * before the work executes.
     */
    if (delayed_work_pending(&hctx->run_work))  // ← 新增：检查pending work
        continue;
    /*
     * Dispatch from this hctx either if there's no hctx preferred
     * by IO scheduler or if it has requests that bypass the
     * scheduler.
     */
    if (!sq_hctx || sq_hctx == hctx ||
        !list_empty_careful(&hctx->dispatch))
        blk_mq_delay_run_hw_queue(hctx, msecs);
}
```

#### 变化说明
- **新增内容**: 在设置延迟运行之前，检查`hctx->run_work`是否已经pending
- **目的**: 避免一个hctx重新延迟另一个hctx的work，导致I/O stall
- **影响**: 如果run_work已经pending，跳过设置新的延迟

#### 造成的结果

**正面影响**：
- 避免了hctx之间的相互延迟导致的I/O stall

**负面影响（可能导致IO劣化）**：
1. **swap IO的hctx可能被跳过**
   - 如果处理swap IO的hctx已经有pending的run_work
   - 新的延迟设置会被跳过，可能导致swap IO请求延迟处理
   - 在高IO压力下，这个检查可能导致swap IO的hctx无法及时调度

2. **延迟累积**
   - 如果swap IO的hctx一直有pending work
   - 新的延迟请求会被跳过，可能导致swap IO请求等待时间过长
   - 在高IO压力下，延迟累积会导致swap IO变慢

3. **与变化1的协同效应**
   - 变化1导致swap IO请求在`blk_mq_mark_tag_wait()`中等待更长时间
   - 变化2可能导致swap IO的hctx无法及时调度
   - 两个变化叠加，导致swap IO性能显著下降

### 变化3: __blk_mq_get_tag()中的hctx_may_queue检查

**文件**: `block/blk-mq-tag.c`  
**函数**: `__blk_mq_get_tag()`  
**位置**: 第77-88行

#### 代码逻辑（r3版本）
```c
static int __blk_mq_get_tag(struct blk_mq_alloc_data *data,
                            struct sbitmap_queue *bt)
{
    if (!data->q->elevator && !(data->flags & BLK_MQ_REQ_RESERVED) &&
            !hctx_may_queue(data->hctx, bt))
        return BLK_MQ_NO_TAG;

    if (data->shallow_depth)
        return __sbitmap_queue_get_shallow(bt, data->shallow_depth);
    else
        return __sbitmap_queue_get(bt);
}
```

#### 关键检查逻辑
`hctx_may_queue()`函数检查硬件队列是否可以分配更多tag：

```c
static inline bool hctx_may_queue(struct blk_mq_hw_ctx *hctx,
                                  struct sbitmap_queue *bt)
{
    unsigned int depth, users;

    if (!hctx || !(hctx->flags & BLK_MQ_F_TAG_QUEUE_SHARED))
        return true;

    if (bt->sb.depth == 1)
        return true;

    // 计算每个活跃队列允许的tag数量
    users = atomic_read(&hctx->tags->active_queues);
    if (!users)
        return true;

    depth = max((bt->sb.depth + users - 1) / users, 4U);
    return __blk_mq_active_requests(hctx) < depth;
}
```

#### 可能的变化和影响

**如果r2到r3之间`hctx_may_queue()`的逻辑变得更严格**：

1. **tag分配更保守**
   - `depth`的计算可能变得更保守（例如最小值从4U增加到更大值）
   - 导致每个硬件队列允许的tag数量减少
   - swap IO请求更容易被拒绝分配tag

2. **active_queues统计的影响**
   - 如果`active_queues`的统计逻辑变化
   - 可能导致`users`值不准确
   - 影响`depth`的计算，进而影响tag分配

3. **与变化1的协同效应**
   - 如果tag分配更保守，更多swap IO请求会进入`blk_mq_mark_tag_wait()`
   - 结合`smp_mb()`的延迟，导致swap IO等待时间显著增加

#### 造成的结果

**负面影响**：
1. **swap IO请求更容易被拒绝tag**
   - `hctx_may_queue()`检查更严格
   - swap IO请求在tag分配阶段就被拒绝
   - 需要进入等待队列，增加延迟

2. **tag资源竞争加剧**
   - 在高IO压力下，多个硬件队列竞争共享tag
   - `hctx_may_queue()`限制每个队列的tag数量
   - swap IO的硬件队列可能无法获得足够的tag

3. **与内存屏障叠加**
   - tag分配失败 → 进入`blk_mq_mark_tag_wait()` → `smp_mb()`延迟
   - 双重延迟导致swap IO性能显著下降

### 变化4: __blk_mq_get_driver_tag()中的hctx_may_queue检查

**文件**: `block/blk-mq.c`  
**函数**: `__blk_mq_get_driver_tag()`  
**位置**: 第1075-1097行

#### 代码逻辑（r3版本）
```c
static bool __blk_mq_get_driver_tag(struct request *rq)
{
    struct sbitmap_queue *bt = rq->mq_hctx->tags->bitmap_tags;
    unsigned int tag_offset = rq->mq_hctx->tags->nr_reserved_tags;
    int tag;

    blk_mq_tag_busy(rq->mq_hctx);

    if (blk_mq_tag_is_reserved(rq->mq_hctx->sched_tags, rq->internal_tag)) {
        bt = rq->mq_hctx->tags->breserved_tags;
        tag_offset = 0;
    } else {
        if (!hctx_may_queue(rq->mq_hctx, bt))  // ← 关键检查
            return false;
    }

    tag = __sbitmap_queue_get(bt);
    if (tag == BLK_MQ_NO_TAG)
        return false;

    rq->tag = tag + tag_offset;
    return true;
}
```

#### 关键检查点
在获取driver tag之前，会调用`hctx_may_queue()`检查硬件队列是否可以分配更多tag。

#### 可能的变化和影响

**如果r2到r3之间这个检查被添加或变得更严格**：

1. **driver tag获取失败增加**
   - 在dispatch阶段，driver tag获取失败
   - 需要进入`blk_mq_mark_tag_wait()`等待
   - 结合`smp_mb()`延迟，导致swap IO dispatch延迟增加

2. **dispatch效率下降**
   - 每次dispatch都需要检查`hctx_may_queue()`
   - 如果检查失败，需要等待tag释放
   - 导致swap IO请求无法及时dispatch

3. **与变化1的协同效应**
   - `hctx_may_queue()`检查失败 → 进入`blk_mq_mark_tag_wait()` → `smp_mb()`延迟
   - 两个检查点都增加延迟，导致swap IO性能显著下降

#### 造成的结果

**负面影响**：
1. **swap IO dispatch延迟**
   - driver tag获取失败
   - 需要等待tag释放
   - 在高IO压力下，等待时间显著增加

2. **IO请求堆积**
   - dispatch延迟导致IO请求在调度器队列中堆积
   - swap IO请求无法及时发送到驱动
   - IO压力进一步升高

### 变化5: blk_mq_prep_dispatch_rq()中的budget检查

**文件**: `block/blk-mq.c`  
**函数**: `blk_mq_prep_dispatch_rq()`  
**位置**: 第1286-1321行

#### 代码逻辑（r3版本）
```c
static enum prep_dispatch blk_mq_prep_dispatch_rq(struct request *rq,
                                                  bool need_budget)
{
    struct blk_mq_hw_ctx *hctx = rq->mq_hctx;
    int budget_token = -1;

    if (need_budget) {
        budget_token = blk_mq_get_dispatch_budget(rq->q);
        if (budget_token < 0) {
            blk_mq_put_driver_tag(rq);
            return PREP_DISPATCH_NO_BUDGET;  // ← 返回失败
        }
        blk_mq_set_rq_budget_token(rq, budget_token);
    }

    if (!blk_mq_get_driver_tag(rq)) {
        if (!blk_mq_mark_tag_wait(hctx, rq)) {
            if (need_budget)
                blk_mq_put_dispatch_budget(rq->q, budget_token);
            return PREP_DISPATCH_NO_TAG;
        }
    }

    return PREP_DISPATCH_OK;
}
```

#### 关键机制
- **budget机制**：只有SCSI驱动实现了`get_budget/put_budget`
- **budget获取失败**：如果无法获取budget，直接返回`PREP_DISPATCH_NO_BUDGET`
- **tag获取失败**：如果无法获取driver tag，进入`blk_mq_mark_tag_wait()`

#### 可能的变化和影响

**如果r2到r3之间budget检查变得更严格**：

1. **budget获取失败增加**
   - SCSI驱动的budget限制更严格
   - swap IO请求无法获取budget
   - 直接返回失败，无法dispatch

2. **与tag等待的交互**
   - budget获取成功，但tag获取失败
   - 进入`blk_mq_mark_tag_wait()`，结合`smp_mb()`延迟
   - 导致swap IO dispatch延迟增加

3. **资源竞争加剧**
   - budget和tag都是有限资源
   - 在高IO压力下，swap IO请求更难获取这些资源
   - 导致swap IO性能下降

#### 造成的结果

**负面影响**：
1. **swap IO dispatch被阻塞**
   - budget获取失败导致swap IO请求无法dispatch
   - 需要等待budget释放
   - 在高IO压力下，等待时间显著增加

2. **资源竞争**
   - budget和tag都是有限资源
   - 多个IO请求竞争这些资源
   - swap IO请求可能无法及时获得资源

## 综合影响分析

### 根本原因

**主要根因**：`blk_mq_mark_tag_wait()`函数中添加的`smp_mb()`内存屏障是导致IO劣化的主要原因。

**次要根因**：
1. `hctx_may_queue()`检查可能变得更严格，导致tag分配更保守
2. `blk_mq_delay_run_hw_queues()`中的pending work检查可能延迟swap IO的hctx调度
3. budget检查机制（如果使用SCSI驱动）可能限制swap IO的dispatch

### 影响链条
```
1. swap IO请求提交到blk-mq
   ↓
2. 尝试获取tag（变化3/4：hctx_may_queue检查可能更严格）
   ↓
3. tag获取失败 → 调用blk_mq_mark_tag_wait()等待tag
   ↓
4. smp_mb()内存屏障增加延迟（变化1：主要延迟来源）
   ↓
5. swap IO请求堆积，IO压力升高
   ↓
6. kswapd0执行swap操作时被阻塞
   ↓
7. 内存回收变慢，需要swap更多页面
   ↓
8. 更多swap IO请求 → 更多请求等待tag
   ↓
9. blk_mq_delay_run_hw_queues()可能跳过swap IO的hctx（变化2）
   ↓
10. budget检查可能失败（变化5：SCSI驱动budget限制）
   ↓
11. swap IO进一步延迟
   ↓
12. 恶性循环：IO压力持续升高 → PSI IO压力达到94.55%
```

### 各变化的贡献度分析

**主要贡献（直接影响swap IO延迟）**：
1. **变化1（smp_mb()内存屏障）**：★★★★★
   - 直接影响：每次tag等待都增加内存屏障延迟
   - 影响范围：所有无法获取tag的swap IO请求
   - 延迟增加：每个请求增加数微秒到数十微秒

2. **变化3/4（hctx_may_queue检查）**：★★★★☆
   - 直接影响：tag分配更保守，更多请求进入等待
   - 影响范围：共享tag队列的所有IO请求
   - 延迟增加：增加tag分配失败的概率

**次要贡献（间接影响swap IO延迟）**：
3. **变化2（delayed_work_pending检查）**：★★★☆☆
   - 间接影响：可能延迟swap IO的hctx调度
   - 影响范围：特定场景下的swap IO hctx
   - 延迟增加：增加hctx调度的延迟

4. **变化5（budget检查）**：★★☆☆☆
   - 间接影响：SCSI驱动的budget限制
   - 影响范围：使用SCSI驱动的设备
   - 延迟增加：budget获取失败时的延迟

5. **变化6（nr_budgets处理）**：★☆☆☆☆
   - 间接影响：可能影响批量dispatch效率
   - 影响范围：批量IO场景
   - 延迟增加：相对较小

### 关键证据支持
1. **IO压力先升高**：内存屏障导致swap IO变慢，IO压力先升高
2. **kswapd0被阻塞**：内存屏障增加了等待时间，kswapd0的CPU使用率下降
3. **IO压力远高于内存压力**：IO劣化是根本原因，内存压力是结果

### 问题严重性
- **ANR激增20倍**：swap IO变慢导致大量进程等待IO，触发ANR
- **PSI IO压力94.55%**：IO请求大量堆积，IO压力异常高
- **swap风暴**：内存回收变慢，需要swap更多页面，形成恶性循环

## 修复建议

### 短期方案

1. **优化变化1（最优先）**：
   - 移除`blk_mq_mark_tag_wait()`中的`smp_mb()`，使用更轻量级的同步机制
   - 或使用更精确的内存屏障（如`smp_wmb()`或`smp_rmb()`）替代`smp_mb()`
   - 预期效果：直接减少swap IO的等待延迟

2. **优化变化3/4**：
   - 检查`hctx_may_queue()`的逻辑是否在r2到r3之间变得更严格
   - 如果确实变严格，考虑放宽对swap IO的检查
   - 预期效果：减少tag分配失败的概率

3. **优化变化2**：
   - 调整`blk_mq_delay_run_hw_queues()`的逻辑，确保swap IO的hctx能够及时调度
   - 考虑为swap IO的hctx提供优先级
   - 预期效果：减少swap IO的hctx调度延迟

### 长期方案

1. **重新设计tag等待机制**：
   - 避免使用完整内存屏障，使用更高效的同步机制
   - 考虑使用RCU或其他无锁机制
   - 预期效果：从根本上减少tag等待的延迟

2. **优化swap IO路径**：
   - 为swap IO提供专门的快速路径，减少等待时间
   - 考虑为swap IO预留tag或budget
   - 预期效果：确保swap IO的优先级和及时处理

3. **改进资源分配策略**：
   - 优化tag和budget的分配策略
   - 考虑为swap IO提供资源保障
   - 预期效果：减少swap IO的资源竞争

4. **性能监控和调优**：

## 总结

### 核心问题

**主要根因**：`blk_mq_mark_tag_wait()`函数中添加的`smp_mb()`内存屏障导致swap IO请求等待时间增加，引发IO劣化和swap风暴。

**次要根因**：
1. `hctx_may_queue()`检查可能变得更严格，导致tag分配更保守
2. `blk_mq_delay_run_hw_queues()`函数中添加的pending work检查可能进一步延迟swap IO的处理
3. budget检查机制（如果使用SCSI驱动）可能限制swap IO的dispatch

### 变化优先级

1. **变化1（smp_mb()内存屏障）**：★★★★★ - 主要延迟来源
2. **变化3/4（hctx_may_queue检查）**：★★★★☆ - 增加tag分配失败概率
3. **变化2（delayed_work_pending检查）**：★★★☆☆ - 可能延迟hctx调度
4. **变化5（budget检查）**：★★☆☆☆ - SCSI驱动相关
5. **变化6（nr_budgets处理）**：★☆☆☆☆ - 影响相对较小

### 最终结果

- **IO压力升高**：PSI IO压力达到94.55%
- **kswapd0被阻塞**：内存屏障和tag等待导致kswapd0被阻塞
- **内存回收变慢**：swap IO变慢导致内存回收不及时
- **ANR激增20倍**：大量进程等待IO，触发ANR
- **swap风暴**：内存回收变慢 → 需要swap更多页面 → 更多IO请求 → 恶性循环

### 修复建议优先级

1. **立即修复**：优化或移除`smp_mb()`内存屏障（变化1）
2. **短期修复**：检查并优化`hctx_may_queue()`检查逻辑（变化3/4）
3. **中期优化**：改进`blk_mq_delay_run_hw_queues()`逻辑（变化2）
4. **长期优化**：重新设计tag等待机制，优化swap IO路径
