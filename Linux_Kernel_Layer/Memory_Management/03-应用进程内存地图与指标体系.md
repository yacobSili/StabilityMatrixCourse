# 应用进程内存地图与指标体系

## 学习目标

- 掌握应用进程内存的“地图”视角：maps、smaps、区域类型
- 理解 RSS、PSS、USS 等指标的含义与使用场景
- 建立匿名页、文件页、共享页的解释框架
- 能结合 `/proc/<pid>/` 与 dumpsys meminfo 做基本定位

## 概述

本文从 **应用进程** 视角介绍“内存长什么样、怎么量”：通过 `/proc/<pid>/maps` 与 `smaps` 看地址空间布局，通过 RSS/PSS/USS 等指标理解“谁在用多少内存”，并区分匿名页、文件页、共享页，为后续 ART 堆、Native 分配、图形内存等文章建立统一的解释框架。

## 一、进程内存“地图”：maps 与 smaps

### 1.1 `/proc/<pid>/maps`

每个进程的虚拟地址空间由内核维护，用户态可通过 `cat /proc/<pid>/maps` 查看“地图”：一段段连续虚拟地址区间及其属性。

**典型输出示例（节选）**：

```
address           perms offset  dev   inode   pathname
...
00400000-0040a000 r-xp 00000000 fd:00 12345   /system/bin/app_process64
0040a000-0040b000 r--p 0000a000 fd:00 12345   /system/bin/app_process64
0040b000-0040c000 rw-p 0000b000 fd:00 12345   /system/bin/app_process64
...
7f1234000000-7f1234800000 rw-p 00000000 00:00 0   [heap]
7f1234800000-7f1234c00000 rw-p 00000000 00:00 0   [anon:dalvik-*]
...
7ffd00000000-7ffd00200000 rw-p 00000000 00:00 0   [stack]
```

**列含义简要**：

| 列 | 含义 |
|----|------|
| address | 虚拟地址区间 [start-end) |
| perms | r=读 w=写 x=执行 p=私有 s=共享 |
| offset | 在文件中的偏移（匿名区多为 0） |
| dev / inode | 设备与 inode（文件映射时有值） |
| pathname | 区间名称：可执行/so、[heap]、[stack]、[anon:...] 等 |

**常见区间类型**：

- **可执行与 so**：程序与动态库的代码、数据段（文件映射）
- **[heap]**：进程堆，通常由 brk 扩展（匿名）
- **[anon:dalvik-*]**：ART 虚拟机堆等（匿名）
- **[stack]**：主线程栈（匿名）
- **匿名 /dev/ashmem**：ashmem 共享内存等

### 1.2 `/proc/<pid>/smaps` 与 smaps_rollup

`smaps` 在 maps 的基础上，为 **每一段** 增加更细的统计，例如：

- **Size / Rss / Pss / Shared_Clean / Shared_Dirty / Private_Clean / Private_Dirty** 等

`smaps_rollup`（若内核支持）则给出 **进程级汇总** 的一行，便于快速看总量。

**典型 smaps 段示例**：

```
7f1234000000-7f1234800000 rw-p 00000000 00:00 0   [heap]
Size:                8192 kB
Rss:                 2048 kB
Pss:                 2048 kB
Shared_Clean:          0 kB
Shared_Dirty:          0 kB
Private_Clean:         0 kB
Private_Dirty:      2048 kB
Referenced:         2048 kB
Anonymous:          2048 kB
...
```

## 二、核心指标：RSS、PSS、USS

### 2.1 RSS（Resident Set Size）

- **定义**：该段（或进程）当前在 **物理内存中** 的页数 × 页大小，即“实际占用的物理内存”。
- **特点**：包含与其他进程 **共享** 的页（如 so、共享库），所以各进程 RSS 相加会 **重复计算** 共享部分。
- **用途**：看“这台机器上该进程占了多少物理页”，但不能直接相加得到系统总占用。

### 2.2 PSS（Proportional Set Size）

- **定义**：对共享页按 **共享该页的进程数** 平分；本进程 PSS = 独占页 + 共享页/共享进程数。
- **特点**：各进程 PSS 相加 ≈ 系统总物理占用（不含内核等），适合 **跨进程对比、加总**。
- **用途**：Android 常用 PSS 做“进程内存排名”、oom_adj 与 LMK 策略输入。

### 2.3 USS（Unique Set Size）

- **定义**：仅统计 **本进程独占** 的物理页（Private）。
- **特点**：不含任何共享页，USS ≤ PSS ≤ RSS。
- **用途**：看“杀掉该进程能释放多少物理内存”；分析泄漏时关注 USS 增长。

### 2.4 关系小结

| 指标 | 含义 | 是否含共享 | 典型用途 |
|------|------|------------|----------|
| RSS | 实际驻留物理内存 | 含 | 单进程占机量 |
| PSS | 按比例分摊的驻留 | 含（按比例） | 排名、加总、策略 |
| USS | 独占物理内存 | 不含 | 释放量、泄漏分析 |

## 三、匿名页、文件页、共享页

### 3.1 匿名页（Anonymous）

- **来源**：堆（brk/mmap 匿名）、栈、显式 mmap 匿名区、部分 shmem。
- **特点**：不对应磁盘文件；回收时若未 swap，只能通过丢弃或 swap 到 zram/swap 设备。
- **在 smaps 中**：Anonymous、Private_Dirty 等往往对应匿名使用。

### 3.2 文件页（File-backed）

- **来源**：可执行文件、so、mmap 的文件（如 APK 资源、数据库文件）。
- **特点**：有对应文件，可与 PageCache 协同；回收时可写回后丢弃，再按需从文件读回。
- **在 smaps 中**：pathname 为文件路径，Shared_Clean/Private_Clean 等可能来自文件缓存。

### 3.3 共享页（Shared）

- **来源**：多进程映射同一文件（so、jar）、ashmem、DMA-BUF 等。
- **特点**：物理页被多个进程算进各自 RSS，但 PSS 只算一份（按进程数平分）。
- **在 smaps 中**：Shared_Clean / Shared_Dirty 表示共享部分；perms 中 `s` 表示共享映射。

### 3.4 解释框架（如何“看”一段内存）

1. **看 maps**：这段是堆、栈、so、文件映射还是匿名 mmap？
2. **看 smaps**：Rss/Pss/Private/Shared 各多少；Anonymous 与 file-backed 比例。
3. **结合路径与名字**：如 [heap] 与 [anon:dalvik-*] 对应应用堆与 ART；so 与 jar 对应文件页与共享。

## 四、Android 侧的呈现：dumpsys meminfo

`dumpsys meminfo <pid>` 在 Android 上对 PSS 等做了 **按类型汇总**（Java Heap、Native Heap、Stack、Graphics、Code 等），便于应用开发与排查。

- **PSS Total**：进程总 PSS，与 `/proc/<pid>/smaps` 汇总一致。
- **Java Heap / Native Heap**：与 ART、Native 分配器对应（详见 [04-ART堆与GC的内存观](04-ART堆与GC的内存观.md)、[09-malloc到内核：brk与mmap及分配策略](09-malloc到内核-brk与mmap及分配策略.md)）。
- **Graphics**：与 Gralloc/GPU 相关（详见 [20-Gralloc与BufferQueue的内存路径](20-Gralloc与BufferQueue的内存路径.md)、[21-图形内存问题定位框架](21-图形内存问题定位框架.md)）。

本文建立的是“地图 + 指标”的统一框架，各类型的深入机制在对应专题文章中展开。

## 五、实践建议

1. **看单进程**：`cat /proc/<pid>/maps` 看布局，`cat /proc/<pid>/smaps` 或 `smaps_rollup` 看 Rss/Pss/Private。
2. **看系统**：`/proc/meminfo` 看全局；Android 上 `dumpsys meminfo` 看各进程 PSS 排名与分类。
3. **定位增长**：对比前后 smaps/dumpsys，看哪类（Java/Native/Graphics/Code 等）在涨，再结合 maps 定位到具体区间。
4. **共享与泄漏**：PSS 不涨但 RSS 涨，可能是共享页增多；USS 持续涨往往提示独占页泄漏（需结合 ART/Native 分析）。

---

## 总结

- **maps**：进程虚拟地址空间“地图”；**smaps**：每段的 Rss/Pss/Private/Shared 等细项。
- **RSS**：实际驻留；**PSS**：按比例分摊（可加总）；**USS**：独占（看释放与泄漏）。
- **匿名页 / 文件页 / 共享页**：决定“这段内存从哪来、怎么回收、算在谁头上”，是后续 ART、Native、图形内存分析的统一解释框架。
