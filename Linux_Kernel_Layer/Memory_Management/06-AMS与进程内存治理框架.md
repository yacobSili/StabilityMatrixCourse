# AMS 与进程内存治理框架

## 学习目标

- 理解 ActivityManagerService 在内存治理中的角色
- 掌握进程重要性（adj、oom_adj）与 trim 回调的协同
- 明确 AMS 与 lmkd 的边界与协同方式
- 能结合系统行为理解“何时 trim、何时杀进程”

## 概述

本文介绍 Android Framework 层的 **进程内存治理框架**：AMS 如何维护进程重要性、如何下发 trim 回调、如何与内核侧的 lmkd 协同，以及“治理”与“杀进程”的边界。为 [07-lmkd与低内存治理链路](07-lmkd与低内存治理链路.md) 和 [18-memcg与PSI：Android低内存信号层](18-memcg与PSI：Android低内存信号层.md) 建立 Framework 侧上下文。

---

## 一、AMS 在内存治理中的角色

### 1.1 职责概览

- **进程重要性**：根据前后台、组件状态、可见性等计算 **oom_adj / adj**，供 lmkd 等做“杀谁”的决策输入。
- **内存回调**：在内存压力时向应用下发 **onTrimMemory**、**onLowMemory** 等，引导应用释放非关键资源。
- **统计与上报**：汇总各进程 PSS 等，供 dumpsys meminfo、statsd 等使用；不直接杀进程，杀进程由 lmkd 根据内核压力与 adj 执行。

### 1.2 与 lmkd 的边界

- **AMS**：维护 adj、下发 trim、提供进程列表与重要性；**不直接调内核 OOM Killer**。
- **lmkd**：用户态守护进程，监听 **PSI/memcg** 等压力信号，结合 **adj** 决定是否杀进程、杀哪个；杀进程时通过 **proc 接口** 或 **binder** 与 AMS 协同（如通知进程已杀、更新列表）。

因此：**“压力信号”来自内核（PSI/memcg），“策略输入”来自 AMS（adj），“执行杀进程”在 lmkd。**

---

## 二、进程重要性：adj / oom_adj

### 2.1 含义

- **oom_adj**（旧）/ **adj**（新）：数值越大，进程越“可杀”；数值越小，越重要（如前台可见应用）。
- 由 AMS 根据 **进程类型**（前台/可见/服务/缓存/空）与 **组件状态** 计算并写入 **proc**（如 `/proc/<pid>/oom_score_adj`），供 lmkd 读取。

### 2.2 典型分类（概念性）

| 类型 | 含义 | adj 范围（示例） |
|------|------|------------------|
| 前台 | 正在与用户交互 | 最低（如 0） |
| 可见 | 可见但非前台 | 较低 |
| 服务 | 后台服务 | 中等 |
| 缓存 | 缓存进程，无服务 | 较高 |
| 空 | 无任何组件 | 最高（优先杀） |

具体数值与策略因版本而异；本文侧重“AMS 负责算、lmkd 负责用”。

### 2.3 与 PSS 的关系

- lmkd 杀进程时往往结合 **adj + 内存占用（如 PSS）**：在同类 adj 下优先杀占用大的，或在高压力下放宽“不杀”的 adj 阈值。AMS 提供 PSS 统计，lmkd 从 proc 或通过 binder 获取。

---

## 三、Trim 回调：onTrimMemory / onLowMemory

### 3.1 目的

- 在 **尚未到杀进程** 的阶段，先让应用 **主动释放** 非关键内存（缓存、临时 Bitmap、未用对象），缓解压力，减少被杀的几率与系统卡顿。

### 3.2 级别（onTrimMemory）

- 不同 **level** 表示不同压力程度（如 TRIM_MEMORY_RUNNING_MODERATE/LOW、TRIM_MEMORY_BACKGROUND 等）。
- 应用可根据 level 决定释放力度：如后台时释放大图缓存，压力大时进一步收缩。

### 3.3 与“回收链路”的衔接

- 压力信号来自内核（PSI/memcg）→ lmkd；lmkd 可 **先** 通过 AMS 向应用发 trim，**再** 视情况杀进程。
- 因此：**trim 是“治理”的软手段，杀进程是硬手段**；AMS 负责把“系统希望应用省内存”的意图通过回调传达给应用。

---

## 四、AMS 与 lmkd 的协同流程（概念）

1. **内核**：内存紧张时产生 **PSI** 或 **memcg** 事件。
2. **lmkd**：收到事件后，读取各进程 **adj**（及可选 PSS），按策略排序。
3. **lmkd**：可先请求 AMS 向部分进程发 **trim**；若压力仍高，则选择进程 **杀**（如向 `/proc/<pid>/oom_score_adj` 或通过接口触发）。
4. **AMS**：更新进程列表、通知应用进程已死等；继续维护 adj 与统计。

具体协议与接口见 [07-lmkd与低内存治理链路](07-lmkd与低内存治理链路.md)。

---

## 五、实践建议

1. **应用侧**：实现 **onTrimMemory** / **onLowMemory**，按 level 释放缓存与大对象，有助于在压力下少占 PSS、降低被杀概率。
2. **排查**：若“应用被后台杀”，看 **adj** 与 **PSS** 是否偏高；若“系统卡顿”，看是否压力大导致频繁 trim 与回收（见 [16-回收与压力：LRU与vmscan与compaction](16-回收与压力-LRU与vmscan与compaction.md)、[17-ZRAM与Swap：Android设备的压缩交换链路](17-ZRAM与Swap：Android设备的压缩交换链路.md)）。
3. **不要依赖“不杀”**：adj 再低，在极端压力下仍可能被杀；关键数据应持久化或可恢复。

---

## 总结

- **AMS** 负责进程重要性（adj）、trim 回调、统计与上报；**不直接杀进程**。
- **lmkd** 根据内核压力（PSI/memcg）与 **adj**（及 PSS）决定是否先 trim、再杀进程；杀进程由 lmkd 执行。
- **Trim** 是软治理手段，**杀进程** 是硬手段；两者协同构成 Android 的“低内存治理链路”。
