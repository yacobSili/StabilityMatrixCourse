# 应用内存优化与常见坑

## 学习目标

- 掌握应用侧常见内存问题：Bitmap、线程栈、mmap 文件、Native 泄漏
- 理解启动与常驻阶段的内存形态差异
- 能结合 maps/smaps 与 dumpsys 做基本定位与优化方向判断
- 避免常见误区（如“GC 后 PSS 必须降”、“free 后 RSS 立刻降”等）

## 概述

本文在 [03-应用进程内存地图与指标体系](03-应用进程内存地图与指标体系.md) 和 [04-ART堆与GC的内存观](04-ART堆与GC的内存观.md) 的基础上，从 **工程实践** 角度梳理应用层内存优化与常见坑：Bitmap、线程栈、mmap、Native 泄漏、启动与常驻内存，并给出与“地图/指标”的对应关系。

## 一、Bitmap 与图片内存

### 1.1 为何容易占大量 PSS

- Bitmap 像素数据在 **Native 堆** 或 **共享/匿名** 区间（视实现与 API 等级而定），解码后常为 **宽×高×每像素字节** 的连续块。
- 大图或未采样加载会直接推高 **Native Heap** 或 **Graphics** 相关 PSS；多张未回收则累积明显。

### 1.2 常见坑与优化方向

- **不采样直接加载大图**：应按显示尺寸采样（Options.inSampleSize、inPreferredConfig 等），减少解码后尺寸。
- **多副本**：Java 侧 Bitmap 对象与 Native 像素缓冲可能在不同区间；注意 API 与版本差异，避免重复持有。
- **生命周期**：在列表/大图场景及时 recycle（若仍适用）或置空引用，便于 GC 回收；结合 dumpsys 的 Native/Graphics 看是否持续增长。

### 1.3 在“地图”中的位置

- 在 smaps 中可能表现为 **匿名大块** 或 **ashmem** 段；dumpsys 中常归入 **Native Heap** 或 **Graphics**。结合 [21-图形内存问题定位框架](21-图形内存问题定位框架.md) 区分 GPU/GraphicBuffer 与解码缓冲。

## 二、线程栈

### 2.1 栈大小与数量

- 每线程栈默认大小（如 1MB）在创建时预留虚拟空间；**首次触及** 才会占用物理页，故线程多不一定立刻 RSS 暴涨，但 **虚拟空间** 会累加。
- 栈过深（递归、大局部变量）会导致该线程栈的 Rss 上升，甚至栈溢出。

### 2.2 常见坑

- **无节制开线程**：虚拟地址与可能的 Rss 都会增加；应使用线程池、协程等控制并发与栈总量。
- **大局部变量**：大数组、大对象放栈上会推高该栈段的 Rss。

### 2.3 在 maps 中的位置

- 每线程栈对应一段 **[stack]** 或带 tid 的栈区间；smaps 中看 **Stack** 总 Rss/Pss 与线程数是否匹配预期。

## 三、mmap 文件

### 3.1 为何影响内存

- mmap 文件会把 **文件区间** 映射进地址空间；访问时通过 **PageCache** 占用物理内存（文件页），计入本进程 **Rss**（共享部分按 PSS 分摊）。
- 大文件、多进程 mmap 同一文件（如 so、数据库）会带来共享页与 PSS 的平衡；只读共享通常 PSS 较低。

### 3.2 常见坑与优化

- **整库/大文件 mmap 且随机访问**：容易把大量文件页拉进内存，推高 PSS；可考虑按需读、限制 mmap 范围或使用常规 read。
- **写回**：脏页需写回磁盘，与 [15-PageCache与文件映射：与FS的业务往来](15-PageCache与文件映射：与FS的业务往来.md)、[16-回收与压力：LRU与vmscan与compaction](16-回收与压力-LRU与vmscan与compaction.md) 相关；压力大时可能触发回收与 IO。

### 3.3 在 maps 中的位置

- pathname 为 **文件路径**；smaps 中为 **file-backed**，Shared_* / Private_* 反映共享与独占。

## 四、Native 泄漏

### 4.1 表现

- **Native Heap**（或对应匿名段的 Rss）**持续上升** 且不随 GC 回落；进程长期存在时更明显。
- 可能原因：JNI 或 so 中 malloc/new 未 free、句柄/资源未关闭、缓存无限增长等。

### 4.2 定位思路

- **dumpsys meminfo** 看 Native Heap 趋势；**smaps** 看哪几段匿名/堆在涨。
- 使用 **LeakSanitizer（LSan）**、**AddressSanitizer（ASan）** 或 **HWASan**（见 [10-Android分配器概览：scudo与jemalloc](10-Android分配器概览：scudo与jemalloc.md) 与 Native 调试文档）辅助定位分配栈。
- 结合 [09-malloc到内核：brk与mmap及分配策略](09-malloc到内核-brk与mmap及分配策略.md)：了解“free 后 RSS 不一定立刻降”（归还给内核的时机与碎片），避免误判。

### 4.3 常见坑

- **误以为 free 后 RSS 立刻降**：用户态分配器可能保留 arena 或块，只有归还内核（munmap/brk 收缩）时 Rss 才明显降；PSS 同理。
- **只关注 Java Heap**：Native 泄漏只看 Java 会漏掉，必须看 Native Heap 与 smaps 匿名段。

## 五、启动与常驻内存形态

### 5.1 启动阶段

- **冷启动**：加载 so、类、资源、首屏 UI，Java Heap、Native、Code、Graphics 等会快速上升，随后可能因 GC/trim 略有回落。
- **热启动**：部分已缓存，增长量通常小于冷启动；若热启动仍很高，需看是否有重复加载、未释放的缓存。

### 5.2 常驻阶段

- 理想情况：业务稳定后 **PSS 在某一区间波动**（GC、后台回收、压力回调会带来小幅波动）。
- **持续爬升**：可能是泄漏（Java 或 Native）、缓存无界、列表/大图未回收等；需结合时间线看 Java/Native/Graphics 哪类在涨。

### 5.3 与“治理”的衔接

- 系统在内存压力下会发 **onTrimMemory**、**onLowMemory** 等；应用可在此释放非关键缓存、缩小图片缓存等，缓解被杀与卡顿。详见 [06-AMS与进程内存治理框架](06-AMS与进程内存治理框架.md)、[07-lmkd与低内存治理链路](07-lmkd与低内存治理链路.md)。

## 六、常见误区小结

| 误区 | 事实 |
|------|------|
| GC 后 PSS 一定会明显降 | GC 主要回收对象，物理页常保留给堆复用，PSS 不一定降 |
| free 后 RSS 立刻降 | 用户态分配器可能保留内存，只有归还内核时 Rss 才明显降 |
| 只优化 Java Heap 即可 | Native/Graphics/栈/mmap 同样会推高 PSS，需一起看 |
| 线程数多一定 RSS 暴涨 | 栈按需占物理页，但虚拟与潜在 Rss 仍会随线程数增加 |

---

## 总结

- **Bitmap**：注意采样与生命周期，关注 Native/Graphics PSS。
- **线程栈**：控制线程数与大局部变量，看 smaps 中 Stack。
- **mmap 文件**：注意大文件与共享，对应 file-backed 与 PageCache。
- **Native 泄漏**：看 Native Heap 与匿名段趋势，配合 LSan/ASan；理解 free 与 Rss 的关系。
- **启动与常驻**：区分冷/热启动与常驻形态，结合 trim 与系统压力做优化。
